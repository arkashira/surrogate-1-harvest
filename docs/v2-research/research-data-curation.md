## 1. Data deduplication at scale

### 1.1 Why dedup matters for code more than for text

Roughly **40% of permissively licensed files** in The Stack are near-duplicates (BigCode 2022 finding, confirmed in The Stack v2 / OpenCoder 2024). Code corpora carry vendored deps, generated boilerplate, and forks — all duplicate-prone. Memorization of duplicates is the #1 mechanism behind "model leaked an internal path / commit" — exactly Surrogate-1's bug. Dedup recall is roughly proportional to leakage reduction.

Token economics: dedup typically removes **~75% of raw tokens** at exact stage and **~6% additional** at fuzzy stage (OpenCoder pipeline numbers, RefineCode 755B → 960B+derived).

### 1.2 Algorithms — exact, MinHash LSH, SimHash, suffix-array, Bloom

| Algorithm | When to use | Latency | Recall (code) | Memory |
|---|---|---|---|---|
| **MD5 / SHA-256 exact** | First pass, character-perfect dupes | O(n) | low (only exact) | O(n) hashes |
| **MinHash + LSH** | Near-dupe (Jaccard ≥ 0.7) | O(n × num_perm) | high (~0.96 precision @ 256 perm) | O(n × num_perm) |
| **SimHash** (64-128 bit) | Hamming-distance near-dupe | faster than MinHash | medium | low |
| **Suffix-array (ExactSubstr)** | Long verbatim copies (RefinedWeb uses this in addition to MinHash) | high RAM | very high for substring | O(n) |
| **Bloom filter** | Streaming exact dedup at scale (Dolma uses for paragraph-level) | sub-linear | exact | tiny |
| **LSHBloom** (NeurIPS 2024 paper) | Internet-scale hybrid | very fast | matches MinHash | very low |

**Recommended for Surrogate-1 v2**: MD5 first pass (cheap, removes ~75% trash) → MinHash LSH near-dedup (catches the long tail).

### 1.3 BigCode / The Stack canonical parameters

From BigCode dedup blog and `bigcode-project/bigcode-dataset/near_deduplication/minhash_deduplication.py`:

```
num_perm  = 256
ngram_size = 5      # 5-gram word shingles, NOT 1-gram. 5-grams kill false positives on code.
threshold  = 0.7    # Jaccard similarity. 0.7 is empirically aggressive; 0.85 too lax.
shingles   = ngrams(NON_ALPHA.split(content), 5)   # split on non-alphanumeric, then 5-gram
hash_func  = sha1
bands      = computed from threshold (B,R such that B*R = 256, optimized for FPR)
```

OpenCoder used a more aggressive variant: `num_perm=2048, bands=16, rows=128, ngram=5` (2048 perms = better recall, expensive but justified for code).

CodeParrot (predecessor) used `(256, 0.8, 1)` — unigrams. **Don't use unigrams for code** — 5-grams cut FP rate by ~5×.

For Surrogate-1 v2 specifically (115M raw → target ~30-50M after dedup), **256 perm + 5-gram + 0.7 thresh** is the safe pick: matches BigCode results, 8× cheaper than OpenCoder's 2048-perm config, sufficient quality for an 11B-class model.

```python
from datatrove.pipeline.dedup import MinhashDedupCluster, MinhashDedupFilter, MinhashDedupSignature
from datatrove.pipeline.dedup.minhash import MinhashConfig, MinhashDedupBuckets

from datatrove.pipeline.writers import JsonlWriter

cfg = MinhashConfig(
    num_buckets=14, hashes_per_bucket=8,    # bands=14, rows=8 → ~112 total perms; for code prefer 32×8=256
    n_grams=5,                              # CRITICAL: 5 for code
    seed=1,
)

# Stage 1: signatures
LocalPipelineExecutor(
    pipeline=[
        JsonlReader("data/raw"),
        MinhashDedupSignature(output_folder="data/sig", config=cfg),
    ],
    workers=64,
).run()

# Stage 2: bucketing → 3: clustering → 4: filtering (4 stages, see datatrove example)
```

### 1.5 Code-specific dedup — AST-aware vs. syntactic vs. semantic

Three levels of code dedup, escalating cost vs. recall:

1. **Syntactic (MinHash on text)** — what BigCode / OpenCoder / Stack v2 do. Catches near-text-dupes including reformatted code. **Default for Surrogate-1.**
2. **AST-aware** — parse to AST, hash the structure. Catches functional dupes with renamed identifiers / reformatted whitespace. Tools: `tree-sitter` + custom hasher, or `astor`. Used in research papers, not yet in production pipelines for full corpora — too expensive at >1B docs. **Skip for v2** — diminishing returns above MinHash.
3. **Semantic (embedding-based)** — embed each function with code-embedding model (e.g. `nomic-embed-code`, `Salesforce/codet5p-110m-embedding`), then nearest-neighbor + cluster. NeMo Curator supports this. Catches translations between languages, structurally different but functionally equivalent code. **Worth adding as stage 3** if v2 budget permits — improves SDLC quality measurably.

### 1.6 Recommended Surrogate-1 v2 dedup pipeline

```
Raw 115M pairs
  ↓ Stage 1: SHA-256 exact dedup on assistant-response field   (datatrove)
Expected: ~75M after — drop ~35% exact dupes
  ↓ Stage 2: MinHash LSH dedup (256 perm, 5-gram, thresh 0.7)  (datatrove or NeMo Curator)
Expected: ~50-60M after — drop ~20% near-dupes
  ↓ Stage 3 (optional): Semantic dedup via BGE-en or code-embed   (NeMo Curator)
Expected: ~40-50M, ~10-20% addl. — clusters truly equivalent reformulations
```

Decontamination cross-check: also MinHash against HumanEval, MBPP, LiveCodeBench, SWE-Bench eval sets. **Critical** — current v1 likely contaminated.

## 2. Quality filtering for code datasets

Three filter families, escalating in cost and quality:

### 2.1 Heuristic / rule-based (cheap, broad)

OpenCoder's `opc_data_filtering` repo has **~100 rules** organized into:

- **Natural-language filters**: file size (>8MB drop), line count, avg line length, max line length, alphabet ratio (printable / total)
- **General code filters**: variable count, avg function length, comment density, fraction of repeated lines, ratio of code-to-comment
- **Language-specific**: Python `pass`-statement frequency, C `goto` count, HTML visible-text ratio, JSON/YAML char-count bounds (50-5000)

Reproducing the OpenCoder canonical thresholds:

```python
# from opc_data_filtering rules (selected; full list ~100)
RULES = {
    "max_file_size_bytes":      8 * 1024 * 1024,    # drop if larger
    "min_lines":                3,
    "max_avg_line_len":         100,
    "max_max_line_len":         1000,
    "min_alphabet_ratio":       0.25,
    "max_repeated_line_frac":   0.30,    # drops auto-generated, fixture-heavy files
    "min_alphanumeric_frac":    0.40,
    "html_visible_text_ratio":  0.20,    # HTML files only
    "json_yaml_char_min":       50,
    "json_yaml_char_max":       5000,
}
```

Install + run:

```bash
git clone https://github.com/OpenCoder-llm/opc_data_filtering
cd opc_data_filtering && pip install -r requirements.txt
jupyter lab data_cleaning_example.ipynb
```

**Surrogate-1 applicability**: critical. Apply BEFORE classifier filter — heuristics remove obvious junk cheaply.

DeepSeek-Coder-V2 added similar rules: HTML visible-text-ratio ≥ 20% AND ≥ 100 chars; JSON/YAML char count [50, 5000]. After heuristics + dedup their corpus = **821B code tokens**.

### 2.2 Perplexity-based (KenLM, n-gram LM)

Train a small KenLM 5-gram model on a known-good code subset (e.g. top-starred Python) → score each candidate doc → drop high-perplexity outliers.

```bash
pip install kenlm
# Train 5-gram on the curated seed corpus
kenlm/build/bin/lmplz -o 5 < seed_corpus.txt > model.arpa
kenlm/build/bin/build_binary model.arpa model.bin

# Score each doc
import kenlm
lm = kenlm.Model("model.bin")
score = lm.score(doc, bos=False, eos=False) / len(doc.split())  # per-token logprob
```

Useful but **rarely the deciding signal for code** — code-vs-natural-language perplexity is noisy. Dolma uses KenLM for natural language only.

### 2.3 Classifier-based (highest quality, recommended for v2)

#### `HuggingFaceFW/fineweb-edu-classifier` (text)

Trained on 450K Llama3-70B annotations. F1 = 0.82 binary at score≥3 threshold. Use for the natural-language portion of your pretraining data (e.g., StackExchange, code tutorials).

```python
from transformers import AutoTokenizer, AutoModelForSequenceClassification
clf = AutoModelForSequenceClassification.from_pretrained("HuggingFaceFW/fineweb-edu-classifier")
# Score 0-5; drop anything with score < 3
```

#### `HuggingFaceTB/stack-edu-classifier-{python,javascript,...}` (code, 15 languages)

This is **the** code-specific quality classifier of 2025. Each language has its own 124M-param StarEncoder fine-tuned on 500K Llama-3.1-70B annotations. Threshold = 3 across languages (Java is exception: threshold 2 works better).

Languages covered: python, javascript, typescript, java, cpp, c, csharp, php, swift, rust, go, markdown, html, css, sql.

```python
# Per-language: load classifier, score, filter
from transformers import AutoTokenizer, AutoModelForSequenceClassification
clf = AutoModelForSequenceClassification.from_pretrained(
    "HuggingFaceTB/stack-edu-classifier-python"
)
# Used to build Stack-Edu, the SmolLM2 training mix.
```

**This was the missing piece in BigCode's first iteration.** Stack-Edu (filtered Stack v2) noticeably outperforms raw Stack v2 on MultiPL-E.

#### `nvidia/nemocurator-fineweb-mixtral-edu-classifier`

Same idea, Mixtral annotated. Smaller (~150M), CPU-friendly. Use if H100 allocation is tight.

#### `kenhktsui/fineweb-edu-fasttext-classifier` (fastText, ultra-fast)

CPU classifier, **>2000 docs/sec**. Use for first-pass at trillion-scale; reserve BERT-class classifier for final cut.

```bash
pip install fasttext
import fasttext
clf = fasttext.load_model("./fasttext_model.bin")
labels, probs = clf.predict(text)
```

#### Ultra-FineWeb (2025 paper, classifier improvement)

Lightweight fastText classifier (vector dim 256, n-gram=3, 3 epochs) trained on a tighter seed of "verified high-quality" docs. Improves +3.61pt over FineWeb, +1.3pt over FineWeb-Edu. Recipe is in arXiv 2505.05427.

### 2.4 Code-specific signals (compilable, AST-valid, test-coverage)

Beyond classifier scores, code uniquely allows execution-based signals. **Use these in a final pass to elevate top quartile of training data**:

- **Parse-ability** via `tree-sitter` per language — drop unparseable files. Cheap.
- **Type-check pass** (Python: pyright; TypeScript: tsc) — used by StarCoder2-Instruct seed selection. Drops broken snippets.
- **Lint cleanliness** (ruff, eslint) — soft signal, optional.
- **Test execution status** — for SFT data, pre-execute the example. NVIDIA's `OpenCodeInstruct` ships a per-row `average_test_score ∈ [0,1]` and `test_execution_status`. **Always filter SFT samples to test_score ≥ 0.7.**

### 2.5 Surrogate-1 v2 quality-filter recipe

```
After dedup → quality filter pipeline:

Stage A: Heuristic rules (OpenCoder ~100 rules)
  Drop: file size > 8MB, < 3 lines, repeated-line frac > 30%, alphanumeric frac < 40%
  Keep: passes all gates

Stage B: AST validity (tree-sitter per language)
  Drop: unparseable files

Stage C: Stack-Edu classifier per language (threshold 3, except Java threshold 2)
  Drop: classifier score < 3

Stage D (SFT only): Execution + test-score filter
  Drop: average_test_score < 0.7

Estimated retention: 25-40% of original (consistent with OpenCoder's RefineCode reduction ratio)
```

Benchmark expectation: applying this pipeline to a clean base recovers OpenCoder-level results — HumanEval 68.9% (base), 83.5% (instruct), MBPP 79.9% / 79.1%. For Surrogate-1 v2 (smaller model), expect proportionally smaller but consistent gains.

## 3. Instruction quality for SFT

The big shift 2024 → 2026: **stop scraping ShareGPT-style logs, start synthesizing**. SOTA SFT data is now nearly all model-generated, with execution-verified rejection.

### 3.1 Magpie — extract instructions from aligned LLMs

ICLR 2025 paper. Insight: aligned chat models like Llama-3-Instruct, when fed only the left-side template up to the user-message slot, will *autoregressively complete the user message itself*. So you get instructions for free.

```bash
git clone https://github.com/magpie-align/magpie
cd magpie
conda create -n magpie python=3.10 -y
conda activate magpie
pip install -r requirements.txt
cd scripts && bash magpie.sh    # config inside; pick model + n_samples
```

Compatible source models: Llama 3.3 / 3.1 / 3, Qwen 2.5 / 2, Phi 3, Gemma-2 (production-ready). Filterable but viable: Llama 2, Vicuna, Mistral, DeepSeek Coder.

**Empirical result**: Magpie 300K outperforms the Llama-3-8B-Instruct training mix (10M samples) on AlpacaEval, ArenaHard, WildBench. Performance scales with diversity, not raw count.

Top Magpie-team releases (as of 2026):
- Llama-3.3 mix (1M, Jan 2025)
- Reasoning V2 (250K CoT samples mixed from QwQ, Skywork-o1, Llama, Jan 2025)

**Surrogate-1 applicability**: high. Generate ~100-200K Magpie samples from a strong code model (Qwen2.5-Coder-32B or DeepSeek-Coder-V2). Add to SFT mix.

### 3.2 Evol-Instruct — WizardLM lineage

Evol-Instruct rewrites instructions step-by-step into more complex variants. Five operators: add-constraints, deepen, concretize, increase-reasoning, complicate-input. Original paper: arXiv 2304.12244 (Apr 2023, still the canonical reference).

WizardCoder (the code variant) used Evol-Instruct on Code-Alpaca seed → big HumanEval jumps. Datasets:

- `WizardLMTeam/WizardLM_evol_instruct_70k` (general)
- `WizardLMTeam/WizardLM_evol_instruct_V2_196k` (general v2)
- `theblackcat102/evol-codealpaca-v1` → re-distilled as `ise-uiuc/Magicoder-Evol-Instruct-110K` (decontaminated for HumanEval / MBPP)

Run your own evol on a code seed:

```python
# pseudo — use https://github.com/nlpxucan/WizardLM as starting point
SEED = load_dataset("sahil2801/CodeAlpaca-20k")
EVOLVED = []
for instruction in SEED:
    for op in ["add_constraints", "deepen", "concretize", "increase_reasoning", "complicate_input"]:
        new_instr = call_strong_llm(EVOL_PROMPT[op].format(instr=instruction))
        EVOLVED.append(new_instr)
```

### 3.3 Self-Instruct (the original, still relevant)

Generate seed instructions via few-shot from the model itself, filter by ROUGE-L diversity. Underpins most modern pipelines as the "first stage" before Magpie / Evol-Instruct.

### 3.4 Self-OSS-Instruct (StarCoder2-Instruct method) — the "always works" recipe

The **single most reproducible** synthesis pipeline of 2024-2025. Yields `bigcode/self-oss-instruct-sc2-exec-filter-50k` and StarCoder2-15B-Instruct (HumanEval 72.6).

Pipeline:
1. Filter The Stack v1 Python by Pyright (type-check) + dedup MinHash 0.5 + docstring quality (LLM judge with 7-shot).
2. From 250K filtered seeds, prompt StarCoder2-15B in-context to extract code concepts, then generate instructions. → 238K instructions.
3. For each, sample 10 (response, test) pairs at T=0.7. → 2.4M responses.
4. Sandbox-execute tests. Drop failing. → 500K passing.
5. One per instruction + dedup → 50K final.

**Key innovation**: entirely self-aligned, no GPT-4 distillation, fully permissive license.

Adapt for Surrogate-1: use Qwen2.5-Coder-32B or DeepSeek-Coder-V2 in place of StarCoder2-15B. Keep the seed-→-extract-→-generate-→-execute-filter pipeline intact.

### 3.5 Genie / OrcaAgentInstruct (Microsoft, agentic flows)

`microsoft/orca-agentinstruct-1M-v1` (1M pairs, subset of larger ~25M dataset used to post-train Orca-3-Mistral). Generated via multi-agent flows with GPT-4-Azure. Fine-tuning yields:
- +40% AGIEval
- +19% MMLU
- +54% GSM8K
- +38% BBH
- +45% AlpacaEval

Cleaned variant: `mlabonne/orca-agentinstruct-1M-v1-cleaned`.

**Surrogate-1 applicability**: include the cleaned variant for non-coding general capability — coding+general blend at ~85/15 to 75/25 is the sweet spot per OpenHermes-2.5 ablations (7-14% code in non-code datasets boosted non-code benchmarks).

### 3.6 Diversity vs. quantity — the 2025 consensus

Research consensus (DataRater, DCLM, Magpie, Tulu 3) is consistent: **diversity beats quantity** above ~100K samples.

Operationalize:
- **K-means cluster** the SFT pool on instruction embeddings (e.g., bge-large-en).
- Sample uniformly across clusters.
- For each cluster, prefer high quality-classifier scores.
- Cap per-cluster count → forces diversity.

Tulu 3's recipe (allenai/tulu-3-sft-mixture, 939K samples): 18 sources, ~147K explicitly code (Evol CodeAlpaca 107K + Tulu-3-Persona-Python 35K + TableGPT 5K). Coding responses generated by Claude 3.5 Sonnet (highest quality at the time); other responses by GPT-4o.

## 4. Hard negatives / preference data construction (DPO)

DPO + verifier-mined preferences is the dominant alignment recipe of 2025-2026 (replacing pure RLHF/PPO at small-mid scale).

### 4.1 RS-DPO — rejection-sampling + DPO baseline

NAACL 2024 (Amazon). Pipeline:
1. SFT a model.
2. For each prompt, sample k responses from SFT.
3. Score with a reward model.
4. Take highest-RM as `chosen`, lowest as `rejected`.
5. Train DPO.

Outperforms vanilla DPO, PPO, plain rejection sampling. **The default starting point** for new preference data.

### 4.2 Code-specific: Focused-DPO (ACL 2025 Findings)

Direct answer to the v1 problem. Key idea: don't apply DPO uniformly across the response; localize the DPO loss to the **error-prone tokens** where chosen and rejected diverge.

Pipeline:
1. Concurrently generate code AND tests for each problem.
2. PageRank-style scoring across (code, test) pairs to rank.
3. Match common prefix and suffix → identify the error-prone span.
4. Train DPO with token-level masking on that span.

Paper: arXiv 2502.11475. Reproducible, reduces noise from irrelevant tokens. **Strongly recommended for Surrogate-1 v2** — current v1 leakage tokens (paths, tags) are precisely the kind of localized error this method targets.

### 4.3 Test-execution mining — the "always trustworthy signal"

Construct preference pairs by:
1. Have the model generate N candidates per problem.
2. Run each against unit tests (or compile/parse + linter for non-test domains).
3. `chosen` = passes all tests AND has highest readability/quality score.
4. `rejected` = fails ≥1 test (preferred) or compiles but quality-low.

If no tests exist:
- Synthesize tests with another model (NVIDIA's OpenCodeInstruct method — ships with tests).
- Cross-validate via mutual verification (rStar-Coder method): generate solutions and tests, accept only mutually consistent pairs.

### 4.4 Voting-based preference (Best-of-N self-consensus)

For non-test-able domains (architecture explanations, code review):
1. Sample N=5-8 responses from a strong model.
2. Have a stronger judge (or majority self-consistency) vote.
3. Highest-voted = chosen, lowest = rejected.

Less reliable than test-execution but applicable broadly.

### 4.5 Preference datasets ready to use

| Dataset | Size | Construction | Domain |
|---|---|---|---|
| `Vezora/Code-Preference-Pairs` | 55K | Buggy variants surgically transplanted into known-good code | Code DPO, 127 langs |
| `martin-wey/CodeUltraFeedback` (TOSEM 2025) | ~10K | Coding preferences via LLM evaluators | Code DPO with multi-criteria judging |
| Focused-DPO repo paired data | (custom) | Error-point identified from generated code+tests | High-quality code DPO |
| `argilla/distilabel-math-preference-dpo` | ~7K | Distilabel pipeline | Math DPO (useful supplement) |
| `argilla/distilabel-intel-orca-dpo-pairs` | ~13K | Distilabel + Orca | General DPO |

### 4.6 Recommended Surrogate-1 v2 DPO data plan

Mix of three sources, ~20-50K pairs total (DPO works at modest scale):

```

+ Focused-DPO localized loss training    — instead of standard DPO loss
```

## 5. Sanitization patterns — eliminating Surrogate-1's specific leak class

Surrogate-1 v1 leaked:
- `/home/hermes/.surrogate/state/orchestrate/...` (filesystem path)
- `# generated via cerebras:llama3.1-8b` (provider tag)

Both are **assistant-side leaks** — they appear in target responses, NOT user inputs. So filter at the response field, not the prompt field.

### 5.1 Three-layer sanitization stack

Layer 1: regex (catches structured patterns, fast)
Layer 2: NER model (StarPII for code, gliner-pii for text) — catches names, custom keys
Layer 3: detect-secrets + truffleHog — catches AWS / GitHub / Slack / Stripe API keys with high precision

Use all three. Single layer fails 5-15% of the time per published numbers; stacked layers approach 99%.

### 5.2 Production-ready Python regex set for Surrogate-1

```python
"""
surrogate1_sanitize.py — apply to BOTH user and assistant fields, but
particularly aggressive on assistant-side. Apply BEFORE adding to SFT mix.
"""
import re
from typing import Iterable

# ---------------------------- 1. INTERNAL PATHS -----------------------------
# Catches both your specific leak ("/home/hermes/.surrogate/...") and general
# absolute Unix/Windows paths that shouldn't be in a generic answer.
INTERNAL_PATH_PATTERNS = [
    # User home + dotted hidden state (covers /home/hermes/.surrogate, /Users/X/.claude, etc.)
    re.compile(r"/(?:home|Users)/[A-Za-z0-9._-]+/\.[A-Za-z0-9._-]+(?:/[^\s)\]\"'`]+)*"),
    # Any path under specific internal namespaces (extend as you discover more)
    re.compile(r"/(?:home|Users)/hermes(?:/[^\s)\]\"'`]+)*"),
    re.compile(r"/(?:home|Users)/[^/\s]+/(?:\.surrogate|\.orchestrate|\.synth)(?:/[^\s)\]\"'`]+)*"),
    # Windows-style internal
    re.compile(r"[Cc]:\\\\Users\\\\[A-Za-z0-9._-]+\\\\\.[A-Za-z0-9._-]+(?:\\\\[^\s\"'`]+)*"),
]

# ----------------------- 2. LLM-PROVIDER LEAK TAGS --------------------------
# The "# generated via cerebras:llama3.1-8b" class — covers comments, docstrings,
# headers that name the generation source.
LLM_PROVIDER_PATTERNS = [
    re.compile(r"#\s*generated\s+(?:by|via|with|using)\s+[A-Za-z0-9_:./-]+", re.IGNORECASE),
    re.compile(r"//\s*generated\s+(?:by|via|with|using)\s+[A-Za-z0-9_:./-]+", re.IGNORECASE),
    re.compile(r"<!--\s*generated\s+(?:by|via|with|using)\s+[A-Za-z0-9_:./-]+\s*-->", re.IGNORECASE),
    re.compile(r"\"\"\"[\s\S]*?(?:generated|produced|written)\s+(?:by|via|with|using)\s+[A-Za-z0-9_:./-]+[\s\S]*?\"\"\""),
    # Provider-named tags
    re.compile(r"\b(?:GPT|Claude|Gemini|Llama|Cerebras|Together|Groq|Anthropic|OpenAI|Mistral|Cohere|DeepSeek)[-_:][A-Za-z0-9._-]+\s+(?:response|output|completion|generation)\b", re.IGNORECASE),
    re.compile(r"\b(?:as\s+an\s+AI|as\s+a\s+language\s+model|I\s+am\s+(?:an\s+AI|Claude|GPT|Llama|Gemini))\b", re.IGNORECASE),
]

# --------------------------- 3. CHAT-TEMPLATE LEAKS -------------------------
# Stops "<|im_start|>user", "<assistant>", "[INST]", "Human:", "Assistant:" leaks
CHAT_TEMPLATE_PATTERNS = [
    re.compile(r"<\|(?:im_start|im_end|begin_of_text|end_of_text|system|user|assistant|eot_id|start_header_id|end_header_id|fim_prefix|fim_middle|fim_suffix)\|>"),
    re.compile(r"\[/?INST\]|\[/?SYS\]"),
    re.compile(r"^\s*(?:Human|User|Assistant|System|AI):\s*", re.MULTILINE),
    re.compile(r"<(?:s|/s|bos|eos|pad|unk)>"),
]

# -------------------- 4. EMAILS / IPs / PHONE / URLs WITH CREDS -------------
EMAIL_PATTERN = re.compile(
    r"\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}\b"
)
IPV4_PATTERN = re.compile(
    r"\b(?:25[0-5]|2[0-4]\d|[01]?\d?\d)(?:\.(?:25[0-5]|2[0-4]\d|[01]?\d?\d)){3}\b"
)
IPV6_PATTERN = re.compile(
    r"\b(?:[A-F0-9]{1,4}:){7}[A-F0-9]{1,4}\b", re.IGNORECASE
)
PHONE_PATTERN = re.compile(
    r"\b(?:\+?\d{1,3}[-.\s]?)?\(?\d{3}\)?[-.\s]?\d{3}[-.\s]?\d{4}\b"
)
URL_WITH_CREDENTIALS = re.compile(
    r"https?://[^:@\s]+:[^@\s]+@[^\s/$.?#].[^\s]*"
)

# -------------------- 5. API KEYS / SECRETS / TOKENS ------------------------
# Most are detected best by detect-secrets, but baseline regex catches the obvious ones.
API_KEY_PATTERNS = [
    re.compile(r"\b(?:sk|pk)-[A-Za-z0-9]{20,}\b"),                     # OpenAI / Stripe
    re.compile(r"\bAKIA[0-9A-Z]{16}\b"),                                # AWS access key
    re.compile(r"\bASIA[0-9A-Z]{16}\b"),                                # AWS temp credential
    re.compile(r"\baws_secret_access_key\s*=\s*[A-Za-z0-9/+=]{40}\b"),  # AWS secret
    re.compile(r"\bgh[ps]_[A-Za-z0-9]{36,}\b"),                         # GitHub PAT/server
    re.compile(r"\bxox[baprs]-[A-Za-z0-9-]{10,}\b"),                    # Slack
    re.compile(r"\b(?:AIza)[A-Za-z0-9_\-]{35}\b"),                      # Google API key
    re.compile(r"-----BEGIN (?:RSA|DSA|EC|OPENSSH|PRIVATE) PRIVATE KEY-----[\s\S]+?-----END [A-Z]+ PRIVATE KEY-----"),
    re.compile(r"\b[A-Za-z0-9]{16,}-[A-Za-z0-9]{16,}\b"),               # Generic compound tokens
    re.compile(r"(?:password|passwd|pwd|secret|token|api[_-]?key)\s*[:=]\s*['\"]?[^\s'\"]{8,}", re.IGNORECASE),
]

# ---------- 6. NAMES / USERNAMES / HOSTNAMES (heuristic, supplement NER) ----
USERNAME_LIKE = re.compile(
    r"\b(?:user|username|login|account)\s*[:=]\s*['\"]?[A-Za-z][A-Za-z0-9._-]{2,}\b", re.IGNORECASE
)
INTERNAL_HOSTNAME = re.compile(
    # extend with your internal namespaces:
    r"\b[a-z0-9-]+\.(?:internal|local|corp|prod|stg|staging)(?:\.[a-z]+)*\b"
)

# ------------------ 7. SCRUBBING DRIVER ------------------------------------
def sanitize(text: str, replacement_strategy: str = "placeholder") -> tuple[str, dict]:
    """
    Apply the full sanitization pipeline. Returns (cleaned_text, hits_summary).

    replacement_strategy:
        "placeholder"  -> replace with <REDACTED:KIND>
        "drop"         -> drop the whole document if any pattern matches
        "blank"        -> replace match with empty string
    """
    hits = {}
    out = text

    def sub(name: str, pattern: re.Pattern, kind: str):
        nonlocal out
        c = len(pattern.findall(out))
        if c:
            hits[name] = hits.get(name, 0) + c
            if replacement_strategy == "blank":
                out = pattern.sub("", out)
            elif replacement_strategy == "placeholder":
                out = pattern.sub(f"<REDACTED:{kind}>", out)

    # Apply in order — paths and provider tags first, since they're our specific bug.
    for i, p in enumerate(INTERNAL_PATH_PATTERNS):
        sub(f"path_{i}", p, "PATH")
    for i, p in enumerate(LLM_PROVIDER_PATTERNS):
        sub(f"provider_{i}", p, "PROVIDER")
    for i, p in enumerate(CHAT_TEMPLATE_PATTERNS):
        sub(f"chat_{i}", p, "CHAT_TPL")
    sub("email", EMAIL_PATTERN, "EMAIL")
    sub("ipv4", IPV4_PATTERN, "IP")
    sub("ipv6", IPV6_PATTERN, "IP")
    sub("phone", PHONE_PATTERN, "PHONE")
    sub("url_with_creds", URL_WITH_CREDENTIALS, "URL_CREDS")
    for i, p in enumerate(API_KEY_PATTERNS):
        sub(f"key_{i}", p, "KEY")
    sub("username", USERNAME_LIKE, "USERNAME")
    sub("hostname", INTERNAL_HOSTNAME, "HOST")

    if replacement_strategy == "drop" and hits:
        return "", hits

    return out, hits


# ----------------- 8. FILE-LEVEL DEDUP OF IDENTICAL ASSISTANT RESPONSES ----
def dedup_responses_by_assistant(records: Iterable[dict], assistant_field: str = "response") -> Iterable[dict]:
    """
    Drops records where the assistant_field is byte-identical to one already seen.
    Combine with MinHash for near-dedup.
    """
    seen: set[str] = set()
    for rec in records:
        v = rec.get(assistant_field, "").strip()
        if not v or v in seen:
            continue
        seen.add(v)
        yield rec
```

#### Run it on the Surrogate-1 dataset

```bash
pip install datasets detect-secrets transformers torch
python - <<'PY'
from datasets import load_from_disk, Dataset
from surrogate1_sanitize import sanitize, dedup_responses_by_assistant

ds = load_from_disk("./surrogate1_v1_raw")  # 115M pairs

cleaned = []
hit_total = {}
for row in ds:
    inst, hits1 = sanitize(row["instruction"], replacement_strategy="placeholder")
    resp, hits2 = sanitize(row["response"],   replacement_strategy="drop")  # AGGRESSIVE on assistant
    if not resp:
        continue   # drop entire record on assistant-side leak hit
    row["instruction"] = inst
    row["response"]    = resp
    cleaned.append(row)
    for k, v in {**hits1, **hits2}.items():
        hit_total[k] = hit_total.get(k, 0) + v

cleaned = list(dedup_responses_by_assistant(cleaned))
Dataset.from_list(cleaned).save_to_disk("./surrogate1_v1_sanitized")
print("Hit summary:", hit_total)
PY
```

**Recommendation**: aggressive `drop` strategy on assistant-side fields for path/provider patterns (these should NEVER be in answers). `placeholder` on user-side (preserves training signal where appropriate).

## 9. Open questions / Surrogate-1-specific things to investigate further
