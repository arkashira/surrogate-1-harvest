## retro
- **What went well**
  - Consistent ADR template adoption across 30+ files (uniform structure, IDs, target).
  - Clear product scope lock (NEW-PRODUCT) and stack baseline (Python + FaaS) maintained.
  - Rapid generation cadence (sub-minute intervals) shows automated pipeline throughput.
  - Duplicate detection intent in ADR title signals proactive governance.

- **What didn’t**
  - High duplication: 30+ near-identical ADRs (same ID, title, truncated stack) indicate noisy automation or missing deduplication.
  - Truncated JSON stacks (“Python + F”) block reproducibility and dependency tracking.
  - No versioning or change-log links between ADR revisions; drift invisible.
  - Missing decision rationale/outcomes; titles repeat but no context or acceptance criteria.
  - No owner or review metadata; accountability gap.

- **Process improvements**
  - Gate automation: require diff vs prior ADR; suppress duplicates, enforce changelog entry on updates.
  - Enforce schema: mandatory fields (owner, status, date, rationale, stack full JSON) via lint step.
  - Truncation fix: validate stack JSON completeness and pin dependency versions before merge.
  - ADR lifecycle: adopt amend/revoke actions and semantic ADR IDs (e.g., sequential per component).
  - CI quality gate: fail build on truncated content or missing fields; publish ADR index on merge.

- **Quality score**
  - 4/10: Strong template uptake and cadence, but severe duplication, truncation, and missing rationale undermine trust and usability.
