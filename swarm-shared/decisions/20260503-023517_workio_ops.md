# workio / ops

## Final synthesized solution (best parts, resolved contradictions, concrete + correct)

**Core principle**: enforce exactly-once punches at the database level and make the API idempotent end-to-end.  
Use a **deterministic idempotency key** (LINE message ID when present, otherwise client-generated) and a **unique constraint** that prevents duplicates even under retry storms and concurrency.

---

### 1. Diagnosis (resolved)

- **Duplicate source**: LINE webhook retries + client double-taps can create multiple punch rows for the same employee/day.
- **Root cause**: no DB-level uniqueness, non-atomic upsert, and no idempotency key.
- **Correct target constraints**:
  - One **active/latest punch per employee per calendar day** (business invariant).
  - One **record per idempotency key** (to absorb retries).
- **Rejected contradiction**: do **not** add `punch_type` to the unique constraint for the daily active punch — an employee must be allowed to correct a mistaken IN/OUT on the same day while still preventing duplicate retries. Instead:
  - Use `(employee_id, DATE(timestamp))` for the active-day invariant.
  - Use `idempotency_key` (with a unique index) to deduplicate exact retries.
  - Optionally store `line_message_id` (or source_event_id) for traceability, but not as part of the daily uniqueness constraint.

---

### 2. DB schema (correct + safe)

```sql
-- workio/server/src/db/schema.sql

-- Idempotency and traceability columns
ALTER TABLE punches
  ADD COLUMN IF NOT EXISTS idempotency_key VARCHAR(255),
  ADD COLUMN IF NOT EXISTS line_message_id VARCHAR(255);

-- One active/latest punch per employee per calendar day
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_employee_day_unique
  ON punches (employee_id, DATE(timestamp));

-- Fast lookup + uniqueness for retries (nulls allowed; only non-null keys enforced)
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_idempotency_key_unique
  ON punches (idempotency_key)
  WHERE idempotency_key IS NOT NULL;

-- Optional: index for audit/replay by LINE event
CREATE INDEX IF NOT EXISTS idx_punches_line_message_id
  ON punches (line_message_id)
  WHERE line_message_id IS NOT NULL;
```

Why this is correct:
- The `employee_id + DATE(timestamp)` unique index enforces the business rule at the DB level (no race, no duplicates).
- The partial unique index on `idempotency_key` prevents duplicate retries without preventing multiple days or corrections.
- No `punch_type` in the daily uniqueness index (allows legitimate IN→OUT changes on same day).

### 3. Server route — atomic upsert + idempotency (actionable)

```ts
// workio/server/src/routes/punch.ts
import { Router, Request, Response } from 'express';
import { pool } from '../db';

const router = Router();

/**
 * POST /api/punch
 * Body: { employeeId: number, type: 'IN' | 'OUT', lineMessageId?: string }
 *
 * Idempotency:
 * - Use X-Idempotency-Key header or generate from client.
 * - If idempotency_key exists in DB, return existing record (idempotent=true).
 * - Otherwise perform atomic upsert on (employee_id, DATE(timestamp)).
 */
router.post('/', async (req: Request, res: Response) => {
  const { employeeId, type, lineMessageId } = req.body;
  const idempotencyKey =
    req.header('X-Idempotency-Key') ||
    req.body.idempotencyKey ||
    // Fallback deterministic key when LINE message id present
    (lineMessageId ? `line:${lineMessageId}` : undefined);

  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // 1) Idempotency fast path
    if (idempotencyKey) {
      const idem = await client.query(
        'SELECT id, employee_id, timestamp, type FROM punches WHERE idempotency_key = $1',
        [idempotencyKey]
      );
      if (idem.rows.length > 0) {
        await client.query('COMMIT');
        return res.json({ ok: true, punch: idem.rows[0], idempotent: true });
      }
    }

    const today = new Date().toISOString().split('T')[0];

    // 2) Atomic upsert: try update existing day row
    const upsert = await client.query(
      `UPDATE punches
       SET timestamp = NOW(), type = $1, idempotency_key = $2, line_message_id = $3
       WHERE employee_id = $4 AND DATE(timestamp) = $5
       RETURNING *`,
      [type, idempotencyKey, lineMessageId || null, employeeId, today]
    );

    if (upsert.rows.length > 0) {
      await client.query('COMMIT');
      return res.json({ ok: true, punch: upsert.rows[0], idempotent: false });
    }

    // 3) Insert new punch
    const insert = await client.query(
      `INSERT INTO punches (employee_id, timestamp, type, idempotency_key, line_message_id)
       VALUES ($1, NOW(), $2, $3, $4)
       RETURNING *`,
      [employeeId, type, idempotencyKey, lineMessageId || null]
    );

    await client.query('COMMIT');
    return res.json({ ok: true, punch: insert.rows[0], idempotent: false });
  } catch (err: any) {
    await client.query('ROLLBACK');

    // Unique violation on employee+day or idempotency_key is possible under high concurrency
    if (err.code === '23505') {
      // Fetch the conflicting row to return consistent response
      const conflict = await pool.query(
        'SELECT id, employee_id, timestamp, type FROM punches WHERE employee_id = $1 AND DATE(timestamp) = $2',
        [employeeId, today]
      );
      if (conflict.rows.length > 0) {
        return res.json({ ok: true, punch: conflict.rows[0], idempotent: true });
      }
    }

    console.error('Punch upsert failed', err);
    res.status(500).json({ ok: false, error: 'Punch failed' });
  } finally {
    client.release();
  }
});

export default router;
```

Key correctness points:
- Uses transaction + `FOR UPDATE`-style safety via unique index (the unique index prevents races; the update-then-insert pattern plus unique violation handling makes this safe under concurrency).
- Handles unique violation gracefully (race edge-case) by returning existing row.
- Supports `lineMessageId` for traceability without polluting uniqueness constraint.
- Returns `idempotent: true` when a retry is detected.
