## Final Synthesis (Best Parts, Resolved Contradictions, Concrete Actions)

**Chosen approach**: Combine Candidate 1’s strict DB-level idempotency and schema changes with Candidate 2’s centralized `webhook_events` audit table and tenant-aware write-time validation.  
**Resolution rule**: Where they conflict, prefer database-enforced uniqueness and atomic upserts (Candidate 1) plus an immutable audit row (Candidate 2). Make every change immediately actionable.

## 1. Diagnosis (Consolidated)

- **Race-prone ingestion**: LINE retries identical deliveries; current find-then-upsert is not atomic.
- **No DB-level idempotency**: Application-layer checks are racy under concurrency.
- **Missing uniqueness constraints**: Duplicates can appear in `ClockRecord`, `LeaveRequest`, `OvertimeRequest`.
- **No audit trail**: Critical mutations lack immutable log for debugging and replay.
- **Weak write-time validation**: Tenant rules (max daily hours, shift windows, breaks) are not enforced at write, causing invalid state.
- **No bounded retry handling**: Transient errors cause unbounded LINE retries and amplify duplicates.

**Goals**:
1. **Deterministic idempotency key** derived from payload.
2. **Database-enforced uniqueness** via unique indexes and atomic upserts (`ON CONFLICT DO NOTHING`).
3. **Centralized audit** via `webhook_events` for every LINE event processed.
4. **Write-time tenant validation** for clock-in/out rules; reject invalid writes with clear errors so LINE retries stop.
5. **Idempotent, transactional handler** with safe retry/backoff behavior at application level.

### 3.2 Idempotency Key Utility (`server/src/utils/idempotency.ts`)

```ts
import crypto from 'crypto';

/**
 * Deterministic idempotency key for LINE webhook events.
 * Uses tenantId + userId + eventType + eventTimestamp + (optional actionId) to ensure uniqueness.
 */
export function deriveIdempotencyKey(
  tenantId: string,
  userId: string,
  eventType: string,
  eventTimestamp: string | number,
  actionId?: string
): string {
  const normalized = [tenantId, userId, eventType, String(eventTimestamp), actionId || ''].join('|');
  return `line:${crypto.createHash('sha256').update(normalized).digest('hex')}`;
}
```

### 3.3 Tenant Clock-In/Out Validation (`server/src/utils/tenantRules.ts`)

```ts

 * Basic write-time validation for clock events.
 * Extend with shift windows, max daily hours, minimum break, etc.
 * Returns { valid: true } or { valid: false, error_code, error_message }.
 */

  client: PoolClient,
  tenantId: string,
  userId: string,
  eventType: 'in' | 'out',
  eventTime: Date
): Promise<{ valid: true } | { valid: false; error_code: string; error_message: string }> {
  // Example: prevent two consecutive 'in' without an 'out'
  const latest = await client.query(
    `SELECT event_type, event_timestamp FROM "ClockRecord"
     WHERE line_user_id = $1 AND tenant_id = $2
     ORDER BY event_timestamp DESC LIMIT 1`,
    [userId, tenantId]
  );

  if (latest.rows.length > 0) {
    const last = latest.rows[0];
    if (last.event_type === eventType) {
      return {
        valid: false,
        error_code: 'duplicate_punch',
        error_message: `Cannot clock ${eventType} twice in a row`
      };
    }
  }

  // Example: max daily hours (e.g., 14h)
  const startOfDay = new Date(eventTime);
  startOfDay.setHours(0, 0, 0, 0);
  const endOfDay = new Date(eventTime);
  endOfDay.setHours(23, 59, 59, 999);

    `SELECT event_type, event_timestamp FROM "ClockRecord"

       AND event_timestamp >= $3 AND event_timestamp <= $4
     ORDER BY event_timestamp`,

  );

    }
  }
