# vanguard / ops

## 1. Diagnosis

- No persistent ops state for dataset ingestion runs (date window, repo, status) — forces manual re-entry and makes retries error-prone.
- No CDN-bypass ingestion path — training scripts likely call HF API during data load and will hit 429 rate limits.
- No Lightning Studio reuse guard — risk of recreating studios and burning quota (80hr/mo waste).
- No pre-flight file-list snapshot — training jobs can’t run CDN-only without a deterministic file manifest.
- No idempotent ingestion artifact layout — mixed-schema parquet files likely written to `enriched/` with extra metadata columns, breaking Surrogate-1 schema expectations.

## 2. Proposed change

Add a lightweight ops controller (`/opt/axentx/vanguard/ops/ingest_ctl.py`) plus a run manifest template (`/opt/axentx/vanguard/ops/manifest.json`) that:

- Accepts CLI args: `--repo`, `--date`, `--page-size`, `--max-folders`, `--out-dir`.
- Calls `list_repo_tree(path, recursive=False)` once per date folder, saves JSON manifest.
- Downloads via HF CDN (`resolve/main/...`) without Authorization.
- Projects each file to `{prompt,response}` on parse and writes to `batches/mirror-merged/{date}/{slug}.parquet` (no `source`, no `ts`).
- Reuses a running Lightning Studio named `vanguard-ingest` if available; otherwise starts one (L40S) and tags it for reuse.
- Exits 0 only when manifest + parquet outputs exist and schema passes quick validation.

Scope: single new file + optional env config; no changes to existing app code.
