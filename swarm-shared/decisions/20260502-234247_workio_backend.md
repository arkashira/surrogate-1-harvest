# workio / backend

## Final Synthesis — Idempotent, Atomic Punch Clock for LINE Webhooks

### Diagnosis (merged, highest-impact)
- **Missing idempotency key** on punch rows causes duplicates from LINE redeliveries and concurrent deliveries.
- **Non-atomic read-then-insert** allows race conditions (double clock-in, orphaned sessions).
- **No server-enforced “one active session per user”** constraint makes correctness dependent on fragile app-level locking.
- **No stable deduplication key tied to LINE event + user + day** means retries within LINE’s retry window (minutes–hours) are treated as new events.
- **No audit trail for replays** and no safe handling of constraint violations under concurrency.

### Single Goal
Make punch creation **idempotent and state transitions atomic at the database level**, using a deterministic idempotency key and partial unique constraints, so retries and concurrent deliveries cannot corrupt state.

## 1. Schema (migration)

```sql
-- server/src/db/migrations/001_punches_idempotency_and_active_session.sql

-- Add idempotency key (nullable for existing rows)
ALTER TABLE punches
  ADD COLUMN idempotency_key VARCHAR(128) NULL;

-- Deduplicate by idempotency key (only enforce on non-null keys)
CREATE UNIQUE INDEX idx_punches_idempotency_key
  ON punches (idempotency_key)
  WHERE idempotency_key IS NOT NULL;

-- Enforce at most one active (clocked-in) session per user
CREATE UNIQUE INDEX idx_punches_one_active
  ON punches (user_id)
  WHERE clocked_out_at IS NULL;

-- Optional audit helper: record LINE message ID and event timestamp for traceability
ALTER TABLE punches
  ADD COLUMN line_message_id VARCHAR(64) NULL,
  ADD COLUMN line_event_timestamp TIMESTAMPTZ NULL;
```

**Notes**
- Use a migration tool (e.g., `pg-migrate`, `knex`, or plain SQL applied via CI/CD).
- Backfill existing rows with deterministic keys if you need to reconcile history; otherwise leave them NULL and rely on new events.

## 2. Deterministic Idempotency Key

```ts
// server/src/services/punch.ts
function makeIdempotencyKey({
  lineMessageId,
  userId,
  eventType,
  timestamp,
}: {
  lineMessageId: string;
  userId: string;
  eventType: 'clock_in' | 'clock_out';
  timestamp: Date;
}): string {
  // Stable across retries: message + user + event + calendar day (YYYY-MM-DD)
  const day = timestamp.toISOString().slice(0, 10); // YYYY-MM-DD
  return `${lineMessageId}:${userId}:${eventType}:${day}`;
}
```

Why this shape:
- `lineMessageId` is unique per LINE message.
- Adding `userId` and `eventType` prevents cross-user or cross-type collisions.
- Adding the calendar day prevents stale retries from affecting future days.

## 3. Atomic Service Implementation

```ts
// server/src/services/punch.ts
import { Pool, PoolClient } from 'pg';

const pool = new Pool({ connectionString: process.env.DATABASE_URL });

type PunchEvent = {
  userId: string;
  lineMessageId: string;
  eventType: 'clock_in' | 'clock_out';
  latitude?: number;
  longitude?: number;
  timestamp: Date;
};

export async function createPunch(event: PunchEvent) {
  const idempotencyKey = makeIdempotencyKey(event);
  const client: PoolClient = await pool.connect();
  try {
    await client.query('BEGIN');

    // 1) Idempotency check (fast, indexed)
    const idem = await client.query(
      `SELECT id FROM punches WHERE idempotency_key = $1`,
      [idempotencyKey]
    );
    if (idem.rows.length > 0) {
      await client.query('ROLLBACK');
      return { created: false, existing: idem.rows[0], reason: 'duplicate' };
    }

    if (event.eventType === 'clock_in') {
      // 2a) Ensure no active session (enforced by idx_punches_one_active)
      const active = await client.query(
        `SELECT id FROM punches
         WHERE user_id = $1 AND clocked_out_at IS NULL`,
        [event.userId]
      );
      if (active.rows.length > 0) {
        await client.query('ROLLBACK');
        return { created: false, existing: active.rows[0], reason: 'already_clocked_in' };
      }

      // 3a) Insert new session
      await client.query(
        `INSERT INTO punches
           (user_id, clocked_in_at, latitude_in, longitude_in, idempotency_key, line_message_id, line_event_timestamp, tenant_id)
         VALUES
           ($1, $2, $3, $4, $5, $6, $7, (SELECT tenant_id FROM users WHERE id = $1))`,
        [
          event.userId,
          event.timestamp,
          event.latitude,
          event.longitude,
          idempotencyKey,
          event.lineMessageId,
          event.timestamp,
        ]
      );
    } else {
      // 2b) Clock-out: find latest active session
      const latest = await client.query(
        `SELECT id FROM punches
         WHERE user_id = $1 AND clocked_out_at IS NULL
         ORDER BY clocked_in_at DESC LIMIT 1`,
        [event.userId]
      );
      if (latest.rows.length === 0) {
        await client.query('ROLLBACK');
        return { created: false, reason: 'no_active_punch_to_clock_out' };
      }

      // 3b) Update session with clock-out and idempotency key
      await client.query(
        `UPDATE punches
         SET clocked_out_at = $1,
             latitude_out = $2,
             longitude_out = $3,
             idempotency_key = $4,
             line_message_id = $5,
             line_event_timestamp = $6
         WHERE id = $7`,
        [
          event.timestamp,
          event.latitude,
          event.longitude,
          idempotencyKey,
          event.lineMessageId,
          event.timestamp,
          latest.rows[0].id,
        ]
      );
    }

    await client.query('COMMIT');
    return { created: true };
  } catch (err: any) {
    await client.query('ROLLBACK');

    // Unique violation on idempotency_key or one_active index => duplicate or invalid state
    if (err.code === '23505') {
      return { created: false, reason: 'constraint_violation', detail: err.message };
    }

    // Re-throw unexpected errors for monitoring/alerting
    throw err;
  } finally {
    client.release();
  }
}
```

Key correctness choices:
- All checks and writes occur in a single transaction with `BEGIN`/`COMMIT`.
- The partial unique index `idx_punches_one_active` is the source of truth for “one active session”.
- Idempotency key check is performed first to short-circuit duplicates quickly.
- Constraint violations are mapped to safe, non-throwing responses so retries don’t cascade.
