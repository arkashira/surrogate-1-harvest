## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: Android app (no root, stock VPNService) that establishes a local SOCKS5 listener hidden behind domain-fronted CF Workers; Rust control plane on CF Workers + Supabase for ephemeral session/port state; one obfuscated transport (shadowsocks over WS + domain-front) with automatic endpoint failover; no TUN interface; basic WAUS/U counters stored in Postgres; free-tier only (CF Workers + Supabase); manual bridge list for testing; end-to-end encrypted control channel; minimal UI for connect/disconnect and last-mile latency display.

## Open questions
- Which obfuscation primitives are safest against DPI in target jurisdictions (domain-front vs. meek vs. scramblesuit vs. vless+XTLS)?
- Can we avoid TUN entirely on stock Android while still routing per-app traffic reliably (VPNService API constraints)?
- How will we distribute and rotate bridge/endpoint lists without centralized discovery that can be blocked?
- What telemetry (if any) is acceptable for WAUS/U without risking user deanonymization?
- Do CF Workers limits (CPU-ms, request size) allow us to proxy long-lived streams or only short-lived handshakes?
- How will we handle captive portals and DNS hijacking without leaking real DNS?
