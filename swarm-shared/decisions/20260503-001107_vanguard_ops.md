### 1. Diagnosis
* No persisted manifest per `(repo, dateFolder)` exists, leading to repeated enumeration via authenticated HF API for every training launch, which burns quota and risks 429 rate limit errors.
* The current implementation likely uses `load_dataset(streaming=True)` on heterogeneous repos, triggering PyArrow `CastError` on mixed schema files.
* Training scripts do not utilize the HF CDN bypass strategy, resulting in unnecessary API calls and potential rate limit issues.
* The project lacks a mechanism to pre-flight or cache available files, causing users to pick invalid or missing paths and triggering late failures.
* There is no implementation of the knowledge-rag pipeline to provide contextual insights for business research.

### 2. Proposed change
The proposed change involves creating a persisted manifest per `(repo, dateFolder)` and implementing the HF CDN bypass strategy. This will be achieved by modifying the `granite-business-research.sh` script and the training data ingestion pipeline.

### 3. Implementation
