## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: single strategy pipeline (research snapshot → signal generator → rule-based risk guardrails → simulated execution) with deterministic replay via Redis Streams + Supabase Postgres audit; <500ms median signal-to-execution on synthetic data; API to submit research snapshots and query pipeline_run + audit_event; no live broker integration (simulated fills); OSS stack deployable to CF Workers + Supabase free tier.

## Open questions
- Which brokers/data vendors must be supported in v0.1 and what auth flows do they require?
- Deterministic replay: will Redis Streams + persisted snapshots suffice or do we need event-sourcing (e.g., append-only Postgres log)?
- Latency budget breakdown: target per-stage budgets to meet <500ms median (network, risk, queue, execution)?
- Risk engine integration: in-process guardrails vs external service; what SLAs are required?
- Idempotency and exactly-once execution guarantees: dedup keys and broker idempotency tokens strategy?
- Compliance retention: how long must audit_event and raw snapshots be retained and who retains them?
- Worker hosting: CF Workers for API + queue consumers, or HF space for dev/test; production worker placement?
