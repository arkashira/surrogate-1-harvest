## v0.1 scope
v0.1 ships in 2 weeks: CLI + FastAPI service that connects to one AWS account and one GCP project, lists Lambda + Cloud Functions schedules, persists them in Supabase Postgres, computes duplicates/drifted (cron vs deployed code hash) and orphaned (no target function) findings, exposes a GET /audits endpoint with JSON results, and exports a CSV report. No UI, no auto-remediation, no queue workers.

## Open questions
- Which identity model will customers prefer: bring-your-own long-lived keys, OIDC federation, or per-scan service-account impersonation?
- What is the acceptable scan cadence (hourly vs daily) and retention window for schedule history?
- How will we define 'duplicate' (identical cron+target vs overlapping time windows) and 'drifted' (code vs schedule mismatch) across accounts?
- Do we need write-back (auto-disable/delete) or read-only recommendations for the v0.1 scope?
