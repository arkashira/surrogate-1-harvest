## Deployment
hybrid (Android app on device; CF Workers for config/egress helpers; Supabase for auth/metadata; optional HF Space for public build artifacts)

## v0.1 scope
v0.1 in 2 weeks: Android app (minSdk 24) that establishes an obfuscated SOCKS5 proxy via a Rust core (compiled to JNI), uses randomized TUN-like routing through VpnService to redirect traffic into the proxy, supports one DoH-based obfs mode, stores minimal config in local SQLite, and pairs with a CF Worker that serves signed obfs profiles and acts as a lightweight egress helper; Supabase Postgres holds user/device metadata and session logs (WAUS/U aggregation). No kill-switch, no multi-hop, no UI beyond connect/disconnect and minimal status.

## Open questions
- Which obfuscation primitives minimize DPI fingerprintability while preserving throughput on high-loss paths (domain-fronting vs. DoH tunneling vs. traffic morphing)?
- How will we distribute and rotate endpoint/obfs profiles without a central signal that can be blocked (CF Workers + decentralized fallback)?
- What is the strategy to hide the TUN interface name and SOCKS5 bind port from local apps and system scans (randomized names, ephemeral ports, namespace tricks on Android)?
- How do we measure WAUS/U reliably without creating a privacy-leaking telemetry pipeline?
- Can we rely on Android VpnService in a way that avoids triggering Knox/MDM flags on hardened devices, or do we need a tun-socket-only fallback?
