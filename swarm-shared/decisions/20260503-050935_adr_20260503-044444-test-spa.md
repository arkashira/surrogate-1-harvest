## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: CLI + FastAPI service that scans configured AWS accounts and GCP projects, persists schedules to Supabase Postgres, detects duplicates (identical cron+target), drifted (cron/timezone changes), and orphaned (missing function) schedules, and exposes a JSON report and simple counts endpoint; no UI, no automated remediation, free-tier only (CF Workers optional proxy, Supabase free tier, OSS deps).

## Open questions
- Which AWS accounts and GCP projects will be in scope for v0.1 and how are credentials provided (OIDC, API keys, service account JSON)?
- What defines 'drifted' for each platform (time skew, expression change, timezone change) and what tolerance thresholds should apply?
- How will orphaned schedules be detected (missing target function vs. disabled trigger) and what permissions are required?
- Will customers prefer read-only reporting or automated safe-retire actions, and what approval workflow is acceptable?
- What retention and privacy constraints apply to storing cloud metadata in Supabase Postgres?
