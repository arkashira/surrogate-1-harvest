# workio / design

### Final Synthesis (Best Parts + Correctness + Actionability)

**Core diagnosis (merged, de-duplicated)**
- LINE webhook ingestion is **not idempotent**: retries with the same LINE message can create duplicate punches.
- Punch upsert is **non-atomic** across `(tenant, employee, date, event)` and vulnerable to race conditions.
- There is **no storage-layer enforcement** of tenant-scoped uniqueness or message-id deduplication.
- Tenant isolation is **application-only**; no compound unique index prevents cross-tenant or duplicate punches.

**Chosen approach (pragmatic + robust)**
- Use **storage-layer constraints** as the source of truth:
  - Compound unique index on `punches(tenant_id, employee_id, punch_date, event_type)` to enforce one punch per window and provide tenant isolation.
  - Dedicated idempotency table `punch_line_message_dedup(line_message_id PK, ...)` to make LINE retries safe.
- Perform **atomic upsert + dedup in a single transaction** (`INSERT ... ON CONFLICT DO UPDATE`) so races and retries cannot corrupt state.
- Add **early middleware fast-path** for known duplicate LINE messages to short-circuit retries cheaply.
- Keep **idempotent response semantics**: return success (200/204) for duplicates so LINE stops retrying.

### Implementation (concrete, production-ready)

#### 1) Schema changes (`workio/server/src/db/schema.sql`)
```sql
-- Idempotency table for LINE message deduplication
CREATE TABLE IF NOT EXISTS punch_line_message_dedup (
  line_message_id  VARCHAR(255) PRIMARY KEY,
  tenant_id        INTEGER NOT NULL,
  employee_id      INTEGER NOT NULL,
  punch_id         INTEGER NOT NULL REFERENCES punches(id) ON DELETE CASCADE,
  created_at       TIMESTAMPTZ DEFAULT NOW()
);

-- Enforce one punch per tenant/employee/date/event and provide tenant isolation
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_tenant_employee_date_event
  ON punches (tenant_id, employee_id, punch_date, event_type);
```

#### 2) Middleware: fast duplicate check (`workio/server/src/middleware/lineWebhookMiddleware.ts`)
```ts
import { pool } from '../db';
import { Request, Response, NextFunction } from 'express';

export async function lineWebhookDedup(
  req: Request,
  res: Response,
  next: NextFunction
) {
  const lineMessageId =
    req.headers['x-line-message-id'] || (req.body as any)?.lineMessageId;

  if (!lineMessageId) return next();

  const { rows } = await pool.query(
    `SELECT 1 FROM punch_line_message_dedup WHERE line_message_id = $1 LIMIT 1`,
    [lineMessageId]
  );

  if (rows.length > 0) {
    // Idempotent success: stop LINE retries
    return res.status(200).json({ ok: true, reason: 'duplicate' });
  }

  // Attach for downstream use
  (req as any).lineMessageId = lineMessageId;
  next();
}
```

Wire in your router:
```ts
import { lineWebhookDedup } from './middleware/lineWebhookMiddleware';
import { handlePunch } from './controllers/punchController';

router.post('/webhook/line', lineWebhookDedup, handlePunch);
```

#### 3) Controller: atomic upsert + dedup (`workio/server/src/controllers/punchController.ts`)
```ts
import { pool } from '../db';
import { Request, Response } from 'express';

export async function handlePunch(req: Request, res: Response) {
  const { tenantId, employeeId, eventType } = req.body;
  const lineMessageId = (req as any).lineMessageId;

  if (!tenantId || !employeeId || !eventType) {
    return res.status(400).json({ error: 'Missing required fields' });
  }

  const punchDate = new Date().toISOString().split('T')[0]; // YYYY-MM-DD
  const now = new Date();

  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // Atomic upsert for punch
    const upsertPunch = `
      INSERT INTO punches (tenant_id, employee_id, punch_date, event_type, punched_at, created_at)
      VALUES ($1, $2, $3, $4, $5, $5)
      ON CONFLICT (tenant_id, employee_id, punch_date, event_type)
      DO UPDATE SET punched_at = EXCLUDED.punched_at
      RETURNING id
    `;
    const punchRes = await client.query(upsertPunch, [
      tenantId,
      employeeId,
      punchDate,
      eventType,
      now,
    ]);
    const punchId = punchRes.rows[0].id;

    // Record LINE idempotency; ignore duplicate-key races (still idempotent)
    if (lineMessageId) {
      try {
        await client.query(
          `INSERT INTO punch_line_message_dedup (line_message_id, tenant_id, employee_id, punch_id)
           VALUES ($1, $2, $3, $4)`,
          [lineMessageId, tenantId, employeeId, punchId]
        );
      } catch (e: any) {
        // Duplicate dedup entry is fine; transaction remains valid
      }
    }

    await client.query('COMMIT');
    return res.status(200).json({ ok: true, punchId });
  } catch (err) {
    await client.query('ROLLBACK');
    console.error('Punch handling failed', err);
    return res.status(500).json({ error: 'Punch handling failed' });
  } finally {
    client.release();
  }
}
```

### Verification checklist (actionable)

1. **Schema validation**
   ```bash
   psql workio -c "\d punches"
   psql workio -c "\d punch_line_message_dedup"
   psql workio -c "\di idx_punches_tenant_employee_date_event"
   ```
   Confirm unique index and dedup table exist.

2. **Idempotency (LINE retry)**
   - POST same payload (same `lineMessageId`) twice → second returns `200` with `{ ok: true, reason: 'duplicate' }` (middleware) or `{ ok: true, punchId }` (controller dedup path).
   - Verify only one punch row exists for `(tenant, employee, date, event)`.

3. **Atomicity under concurrency**
   - Fire N concurrent requests with different `lineMessageId`s but same `(tenant, employee, date, event)` → exactly one punch row wins insert; others update `punched_at` (per `ON CONFLICT`). No duplicates or constraint violations.

4. **Tenant isolation**
   - Attempt to create punches for two tenants with same `(employee, date, event)` → both allowed (index is tenant-scoped).
   - Attempt to create two punches for same tenant with same `(employee, date, event)` → second upserts (does not create duplicate).

5. **Edge cases**
   - Missing `lineMessageId` → still processes punch (idempotency via punch uniqueness).
   - Dedup table insert fails (duplicate) → transaction still commits (idempotent behavior preserved).

---

### Key decisions (why this synthesis)

- **Storage-layer constraints** (unique index + dedup table) are authoritative; app logic can’t bypass them.
- **Single transaction** (`BEGIN` → upsert + dedup → `COMMIT`) guarantees atomicity across punch and idempotency record.
- **Middleware fast-path** reduces load for known duplicates while still allowing controller to handle races safely.
- **Ignore duplicate-key on dedup insert** keeps behavior correct under races without complex locking.
- **Explicit tenant columns in all constraints** enforce tenant isolation at the database level.
