# workio / frontend

## 1. Diagnosis

- Frontend clock-in/out can fire multiple rapid requests while previous request is in-flight, creating duplicate punches despite backend fixes.
- No optimistic UI + rollback on network error; users see inconsistent state after failed requests.
- Clock-in/out button lacks loading/disabled state, inviting double-taps and race submissions.
- Location permission prompt can be triggered late (after user taps), causing UX friction and failed GPS verification.
- No client-side debounce/throttle or request deduplication for the same user+date+open-punch action.

## 2. Proposed change

File: `workio/src/components/TimeClock.tsx` (or equivalent main clock UI component)  
Scope: add request deduplication, optimistic state, loading/disabled UI, and early location permission pre-check.

## 3. Implementation

```tsx
// workio/src/components/TimeClock.tsx
import { useState, useRef, useCallback } from 'react';
import { toast } from 'sonner';

type PunchStatus = 'idle' | 'pending' | 'committing' | 'error';

export function TimeClock() {
  const [status, setStatus] = useState<PunchStatus>('idle');
  const [lastAction, setLastAction] = useState<'clock_in' | 'clock_out' | null>(null);
  const inFlightRef = useRef<AbortController | null>(null);
  const locationGrantedRef = useRef<boolean | null>(null);

  // Pre-check location permission early (can be called on mount or before UI reveal)
  const checkLocationPermission = useCallback(async () => {
    if (!navigator.geolocation) {
      toast.error('Geolocation not supported');
      return false;
    }
    if (locationGrantedRef.current === true) return true;
    return new Promise<boolean>((resolve) => {
      navigator.permissions?.query({ name: 'geolocation' as PermissionName }).then((result) => {
        if (result.state === 'granted') {
          locationGrantedRef.current = true;
          resolve(true);
        } else if (result.state === 'prompt') {
          // will prompt on actual call; treat as potentially ok
          resolve(true);
        } else {
          toast.error('Location permission denied');
          resolve(false);
        }
      }).catch(() => resolve(true)); // fallback to attempt
    });
  }, []);

  // Deduplicated punch action
  const punch = useCallback(async (type: 'clock_in' | 'clock_out') => {
    // Prevent duplicate in-flight requests for same action
    if (status === 'pending' || status === 'committing') {
      toast.info('Request already in progress');
      return;
    }

    // Abort previous controller if exists (cleanup)
    if (inFlightRef.current) {
      inFlightRef.current.abort();
    }
    const ac = new AbortController();
    inFlightRef.current = ac;

    const ok = await checkLocationPermission();
    if (!ok) {
      setStatus('error');
      toast.error('Location permission required');
      return;
    }

    // Optimistic update
    setStatus('pending');
    setLastAction(type);
    const now = new Date().toISOString();

    try {
      const res = await fetch(`/api/punches/${type}`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ timestamp: now }),
        signal: ac.signal,
      });

      setStatus('committing');

      if (!res.ok) {
        const err = await res.json().catch(() => ({}));
        throw new Error(err.message || `Server error ${res.status}`);
      }

      const data = await res.json();
      toast.success(`Clocked ${type === 'clock_in' ? 'in' : 'out'}`);
      // Optionally update global state/store here
      setStatus('idle');
      setLastAction(null);
    } catch (err: any) {
      if (err.name === 'AbortError') {
        toast.info('Request cancelled');
      } else {
        toast.error(err.message || 'Failed to record punch');
        setStatus('error');
      }
    } finally {
      if (inFlightRef.current === ac) {
        inFlightRef.current = null;
      }
      if (status !== 'idle') setStatus('idle');
    }
  }, [status, checkLocationPermission]);

  const handleClockIn = useCallback(() => punch('clock_in'), [punch]);
  const handleClockOut = useCallback(() => punch('clock_out'), [punch]);

  const isWorking = lastAction === 'clock_in' && status !== 'committing';

  return (
    <div className="flex flex-col items-center gap-4 p-6">
      <div className="text-lg font-medium">
        {isWorking ? 'คุณกำลังทำงาน' : 'นอกเวลางาน'}
      </div>

      <div className="flex gap-3">
        <button
          onClick={handleClockIn}
          disabled={status === 'pending' || status === 'committing' || isWorking}
          className={`px-6 py-3 rounded-lg font-medium transition ${
            isWorking || status === 'pending' || status === 'committing'
              ? 'bg-gray-200 text-gray-500 cursor-not-allowed'
              : 'bg-green-600 text-white hover:bg-green-700'
          }`}
        >
          {status === 'pending' && lastAction === 'clock_in' ? 'กำลังบันทึก...' : 'Clock In'}
        </button>

        <button
          onClick={handleClockOut}
          disabled={status === 'pending' || status === 'committing' || !isWorking}
          className={`px-6 py-3 rounded-lg font-medium transition ${
            !isWorking || status === 'pending' || status === 'committing'
              ? 'bg-gray-200 text-gray-500 cursor-not-allowed'
              : 'bg-red-600 text-white hover:bg-red-700'
          }`}
        >
          {status === 'pending' && lastAction === 'clock_out' ? 'กำลังบันทึก...' : 'Clock Out'}
        </button>
      </div>

      {status === 'error' && (
        <div className="text-sm text-red-600">
          เกิดข้อผิดพลาด กรุณาลองใหม่
        </div>
      )}
    </div>
  );
}
```

If no existing `TimeClock` component, create it and wire into the main dashboard. Also ensure backend endpoints accept idempotency (e.g., client-generated punch idempotency key header) for stronger deduplication.

## 4. Verification

- Manual: rapidly click Clock In multiple times — only one request should reach the server (check Network tab). Button should show loading and disable while pending.
- Network failure: simulate offline or 500 response — toast should show error, button returns to enabled, and retry works.
- Location: deny location permission — flow should block with clear error; grant permission — flow proceeds.
- Double-tab across separate actions (Clock In then quickly Clock Out) — second action should be blocked while first is pending, then allowed after success.
