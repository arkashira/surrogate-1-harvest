## Deployment

## v0.1 scope
v0.1 in 2 weeks: CLI-first MVP that connects to one AWS account and one GCP project via configured credentials, snapshots schedules into Supabase Postgres, runs local duplicate/drifted/orphan detection using croniter, outputs a JSON report and retires candidates via dry-run flag; no queue/cache, deployed as a CF Worker route for report retrieval and manual trigger.

## Open questions
- How will customers authenticate/authorize cross-account reads for Lambda and Cloud Functions (least-privilege IAM roles vs service account keys)?
- What staleness threshold (e.g., no invocation in 90 days) should default for flagging stale schedules?
- Will we require read-only replica access to production platforms or rely on exported inventory APIs to avoid runtime impact?
- How should drift be defined: cron syntax differences, effective next-fire mismatches, or timezone discrepancies?
