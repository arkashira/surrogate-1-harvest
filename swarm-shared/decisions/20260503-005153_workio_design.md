# workio / design

### Final synthesized solution (correct + actionable)

**Core diagnosis**  
- LINE webhook retries (at-least-once) + no DB uniqueness on “open punch” + no transactional upsert ⇒ duplicate open punches per user.  
- Race/retry window exists between app-layer checks and DB write.  
- Observability is weak (ops blind to duplicates).

**Chosen strategy (resolve contradictions)**  
- **DB-level partial unique constraint** is the source-of-truth guard (Candidate 2).  
- **Idempotency via LINE event_id** (provided by LINE) + lightweight idempotency table is safer and simpler than inventing `user_id-timestamp` keys (Candidate 2).  
- **Transactional upsert** with proper rollback/retry handling prevents races (Candidate 2).  
- **Short in-memory dedupe** (LRU) absorbs burst retries cheaply before DB write (Candidate 2).  
- **App-layer idempotency key** (Candidate 1) is replaced by LINE’s own `webhookEventId`; avoids inventing a new key format and clock-sync pitfalls.

---

### 1) DB schema changes (single source of truth)

```sql
-- workio/server/src/db/schema.sql

-- 1) Enforce exactly one open punch per user
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_one_open_per_user
ON punches (user_id)
WHERE clock_out_at IS NULL;

-- 2) Lightweight idempotency ledger for LINE webhook events
CREATE TABLE IF NOT EXISTS line_webhook_idempotency (
  event_id   TEXT PRIMARY KEY,
  user_id    INTEGER NOT NULL,
  punch_id   INTEGER NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Optional: index for cleanup jobs
CREATE INDEX IF NOT EXISTS idx_idempotency_expiry
ON line_webhook_idempotency (created_at);
```

### 2) Middleware: fast in-memory dedupe + event_id extraction

```typescript
// workio/server/src/middleware/lineWebhook.ts
import type { Request, Response, NextFunction } from 'express';
import LRU from 'lru-cache';

const recentEvents = new LRU<string, boolean>({
  max: 10_000,
  ttl: 60_000,
});

export function lineWebhookGuard(req: Request, res: Response, next: NextFunction) {
  const event = req.body?.events?.[0] ?? req.body;
  const eventId = event?.webhookEventId ?? event?.event_id;

  if (!eventId) {
    return res.status(400).json({ error: 'missing_event_id' });
  }

  // Fast in-memory dedupe for retries within same process
  if (recentEvents.has(eventId)) {
    return res.json({ ok: true, duplicate: true });
  }

  // Attach for downstream handlers
  req.body.event_id = eventId;
  req.body.signature = req.headers['x-line-signature'] as string;

  recentEvents.set(eventId, true);
  next();
}
```

Wire it:

```typescript
// workio/server/src/routes/line.ts
import express from 'express';
import { lineWebhookGuard } from '../middleware/lineWebhook';
import { handleLineWebhook } from '../controllers/lineController';

const router = express.Router();
router.post('/webhook', lineWebhookGuard, handleLineWebhook);
export default router;
```

### 3) Transactional, idempotent punch handler

```typescript
// workio/server/src/routes/attendance.ts
import { pool } from '../db';
import type { Request, Response } from 'express';

export async function recordPunch(req: Request, res: Response) {
  const { user_id, type, latitude, longitude } = req.body;
  const eventId = req.body.event_id || `manual-${Date.now()}`;
  const client = await pool.connect();

  try {
    await client.query('BEGIN');

    // Clock-out: close any open punch
    if (type === 'out') {
      await client.query(
        `UPDATE punches
         SET clock_out_at = NOW(),
             updated_at   = NOW(),
             clock_out_lat = $1,
             clock_out_lng = $2
         WHERE user_id = $3
           AND clock_out_at IS NULL`,
        [latitude, longitude, user_id]
      );
    }

    // Clock-in: insert only if constraint allows
    if (type === 'in') {
      const insertRes = await client.query(
        `INSERT INTO punches (user_id, clock_in_at, clock_in_lat, clock_in_lng)
         VALUES ($1, NOW(), $2, $3)
         RETURNING id`,
        [user_id, latitude, longitude]
      );

      await client.query(
        `INSERT INTO line_webhook_idempotency (event_id, user_id, punch_id)
         VALUES ($1, $2, $3)`,
        [eventId, user_id, insertRes.rows[0].id]
      );
    }

    await client.query('COMMIT');
    return res.json({ ok: true });
  } catch (err: any) {
    await client.query('ROLLBACK');

    // Duplicate open punch (race/retry)
    if (err.code === '23505' && err.constraint === 'idx_punches_one_open_per_user') {
      return res.status(409).json({ error: 'already_clocked_in' });
    }

    // Duplicate event id (idempotent replay)
    if (err.code === '23505' && err.constraint === 'line_webhook_idempotency_pkey') {
      const dup = await client.query(
        `SELECT punch_id FROM line_webhook_idempotency WHERE event_id = $1`,
        [eventId]
      );
      return res.json({ ok: true, duplicate: true, punch_id: dup.rows[0]?.punch_id });
    }

    console.error('recordPunch failure', { err, eventId, user_id, type });
    return res.status(500).json({ error: 'internal' });
  } finally {
    client.release();
  }
}
```

### 4) Logging/observability (minimal but critical)

Add structured logs/metrics at these points:
- When `lineWebhookGuard` drops a duplicate `event_id` (in-memory).  
- When `recordPunch` returns `409` (duplicate open punch).  
- When `recordPunch` returns duplicate via idempotency table.  

Example log line:
```json
{
  "level": "info",
  "msg": "duplicate_punch_prevented",
  "event_id": "ev_123",
  "user_id": 42,
  "reason": "already_clocked_in"
}
```

### 5) Verification checklist

1. **DB constraint works**  
   - Try inserting two `clock_out_at IS NULL` rows for same user in one transaction ⇒ second insert fails with `23505`.

2. **Idempotency works**  
   - Replay same LINE `webhookEventId` ⇒ returns `{ ok: true, duplicate: true }` and no new punch row.

3. **Clock-out safety**  
   - Multiple `type=out` replays for same open punch ⇒ only one row updated; no extra rows.

4. **Race handling**  
   - Fire two concurrent `type=in` requests for same user (same eventId or different) ⇒ only one open punch persists; others get `409` or idempotent duplicate response.

5. **Cleanup**  
   - Schedule periodic purge of `line_webhook_idempotency` older than N days (e.g., 7) via cron/pg_cron.

---

**Why this wins**  
- Correctness: DB constraint is the single source of truth; app logic cannot bypass it.  
- Actionability: Small, focused changes (schema + middleware + transactional handler) that can ship in <2 hours.  
- Observability: Logs/metrics make duplicates visible instead of silent.
