## Deployment
hf space

## v0.1 scope
v0.1 in 2 weeks: deterministic workflow engine (research→signal→risk→execution) with Postgres-backed immutable audit, Redis-cached risk limits, pre-built mock connectors to one broker and one data vendor, compliance guardrails (block/allow rules), and replay endpoint to rerun workflows from hashed inputs. Deployed as a containerized HF Space with Supabase free tier and Redis OSS sidecar; CLI to submit workflows and export audit trails.

## Open questions
- What deterministic replay SLA do we need (exact input hashing, event sourcing, or full event log)?
- Which brokers/OMS/EMS and data vendors must be supported in v0.1 and what auth patterns do they require (OAuth, API keys, mTLS)?
- What compliance guardrail rules are non-negotiable for launch (max notional, restricted tickers, time-of-day, concentration limits)?
- Target sub-second latency budget per step and end-to-end — which components must be edge-deployed vs. colocated?
- Retention and immutability requirements for audit logs (WORM, export to object storage, GDPR/right-to-erasure conflicts)?
