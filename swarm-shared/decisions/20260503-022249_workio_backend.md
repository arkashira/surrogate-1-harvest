# workio / backend

## Final Synthesis — One Correct, Actionable Answer

**Root cause**: duplicate LINE webhook deliveries + non-atomic upsert allow multiple punches for the same employee/date.  
**Required fix**: enforce idempotency at the DB level (stable dedupe key + unique constraint) and make punch upsert atomic within a transaction that includes audit/idempotency records.

---

### 1. DB schema changes (run once)

```sql
-- Idempotency table for LINE deliveries (fast, lightweight dedupe)
CREATE TABLE IF NOT EXISTS line_deliveries (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id        UUID NOT NULL,
  line_delivery_id TEXT NOT NULL,   -- stable across retries (use LINE deliveryId or deterministic composite)
  employee_id      UUID NOT NULL,
  event_type       TEXT NOT NULL,
  payload          JSONB NOT NULL,
  created_at       TIMESTAMPTZ DEFAULT now(),
  UNIQUE (tenant_id, line_delivery_id)
);

-- Strong uniqueness for active punches per tenant/employee/date/source.
-- If you also need to prevent overlapping punch times, add an EXCLUDE constraint
-- (see note below). This index is the last line of defense.
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_tenant_employee_date_source
  ON punches (tenant_id, employee_id, punch_date, source)
  WHERE deleted_at IS NULL;
```

**Notes**:
- Use the official `deliveryId` from LINE webhook metadata when available.  
  If unavailable, use a deterministic composite:  
  `source.userId : timestamp : type` (all primitives from the event).  
- Keep `tenant_id` on events or derive it from `line_user_id → employee → tenant`.  
- If overlapping punches (clock-in/clock-out times) must be prevented, add:  
  ```sql
  ALTER TABLE punches
    ADD CONSTRAINT prevent_overlapping_punches
    EXCLUDE USING gist (
      tenant_id WITH =,
      employee_id WITH =,
      tsrange(punch_at, COALESCE(punch_out_at, punch_at + interval '1 second'), '[]') WITH &&
    ) WHERE (deleted_at IS NULL);
  ```
  (Requires `btree_gist` extension.)

### 2. Webhook handler (single source of truth)

File: `workio/server/src/routes/line/webhook.ts`

```ts
import { Router, Request, Response } from 'express';
import { pool } from '../../db';
import { verifySignature } from '../../utils/line-signature';

const router = Router();

/**
 * Idempotent LINE webhook handler.
 * Guarantees:
 * - Exactly-once punch per (tenant, employee, date, source) for retries.
 * - Atomic punch + idempotency record (all-or-nothing).
 */
router.post('/webhook/line', verifySignature, async (req: Request, res: Response) => {
  const body = req.body;
  const events = body?.events;
  if (!Array.isArray(events) || events.length === 0) {
    return res.status(400).json({ error: 'invalid_payload' });
  }

  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    for (const ev of events) {
      // 1) Build stable dedupe key
      const deliveryId = ev.deliveryId ||
        (ev.source?.userId && ev.timestamp && ev.type
          ? `${ev.source.userId}:${ev.timestamp}:${ev.type}`
          : null);
      if (!deliveryId) continue; // malformed event; skip

      // 2) Idempotency short-circuit
      const dup = await client.query(
        `SELECT 1 FROM line_deliveries WHERE tenant_id = $1 AND line_delivery_id = $2`,
        [ev.source?.tenantId || null, deliveryId]
      );
      if (dup.rows.length > 0) continue;

      // 3) Resolve employee
      const emp = await client.query(
        `SELECT id, tenant_id FROM employees WHERE line_user_id = $1 AND tenant_id = $2`,
        [ev.source?.userId, ev.source?.tenantId || null]
      );
      if (emp.rows.length === 0) continue; // unknown user; skip
      const employee = emp.rows[0];

      // 4) Punch datetime
      const punchAt = ev.timestamp ? new Date(Number(ev.timestamp)) : new Date();
      const punchDate = punchAt.toISOString().split('T')[0]; // YYYY-MM-DD

      // 5) Atomic punch upsert
      await client.query(
        `INSERT INTO punches (tenant_id, employee_id, punch_date, punch_at, source, created_at, updated_at)
         VALUES ($1, $2, $3, $4, $5, now(), now())
         ON CONFLICT (tenant_id, employee_id, punch_date, source)
         DO UPDATE SET punch_at = EXCLUDED.punch_at, updated_at = now()
         WHERE punches.deleted_at IS NULL`,
        [employee.tenant_id, employee.id, punchDate, punchAt, 'line']
      );

      // 6) Record delivery (idempotency)
      await client.query(
        `INSERT INTO line_deliveries (tenant_id, line_delivery_id, employee_id, event_type, payload)
         VALUES ($1, $2, $3, $4, $5)`,
        [employee.tenant_id, deliveryId, employee.id, ev.type, ev]
      );

      // 7) Optional: emit audit row here in same transaction if required
      // await client.query(`INSERT INTO punch_audits (...) VALUES (...)`);
    }

    await client.query('COMMIT');
    res.status(200).json({ ok: true });
  } catch (err) {
    await client.query('ROLLBACK');
    console.error('LINE webhook transaction failed', err);
    res.status(500).json({ error: 'processing_failed' });
  } finally {
    client.release();
  }
});

export default router;
```

### 3. Verification plan (minimal, high-value)

1. **Idempotency test**  
   - POST identical event payload twice.  
   - Assert: exactly 1 punch row and 1 `line_deliveries` row.

2. **Concurrency test**  
   - Fire 10–20 parallel requests with the same event.  
   - Assert: no unique-constraint violations; exactly 1 punch row.

3. **Transaction rollback test**  
   - Force error after punch insert but before commit.  
   - Assert: no punch row and no delivery row persisted.

4. **Manual smoke**  
   - Expose endpoint via ngrok, configure LINE webhook.  
   - Clock in twice in quick succession.  
   - Query `punches` and `line_deliveries` to confirm single punch.

### 4. Operational notes

- **Tenant derivation**: if `tenantId` is not present on LINE events, resolve it via `line_user_id → employee → tenant` before step 2.  
- **Soft deletes**: the unique index and upsert include `WHERE deleted_at IS NULL` so soft-deleted rows don’t block new punches.  
- **Performance**: `line_deliveries` is append-only and indexed on `(tenant_id, line_delivery_id)` for fast dedupe checks.  
- **Monitoring**: alert on repeated `processing_failed` responses or rising 4xx/5xx rates from LINE.
