# workio / quality

## Final synthesized implementation (correct + actionable)

**Core problem**: duplicate punches caused by missing idempotency, missing DB uniqueness, UI spam, and timezone/day-boundary ambiguity.  
**Resolution priorities**:  
1. Prevent duplicates at the DB layer (hard constraint).  
2. Make API idempotent with an idempotency key.  
3. Normalize punch date to tenant timezone so “day” is unambiguous.  
4. Add audit trail for reconciliations.  
5. Make frontend resilient (optimistic state + debounce + unique client keys).

### 1) DB schema (single source of truth)

```sql
-- server/src/db/schema.sql
-- Punches: one IN/OUT per tenant/user/calendar-day (normalized to tenant TZ)
CREATE TABLE IF NOT EXISTS punches (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id       UUID NOT NULL REFERENCES tenants(id),
  user_id         UUID NOT NULL REFERENCES users(id),
  punch_date      DATE NOT NULL,        -- normalized to tenant TZ (YYYY-MM-DD)
  punch_ts        TIMESTAMPTZ NOT NULL, -- exact moment with TZ
  type            VARCHAR(10) NOT NULL CHECK (type IN ('IN', 'OUT')),
  latitude        DOUBLE PRECISION,
  longitude       DOUBLE PRECISION,
  line_user_id    TEXT,
  idempotency_key TEXT NOT NULL,
  created_at      TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE (tenant_id, user_id, punch_date, type)
);

-- Audit trail for edits/reconciliations
CREATE TABLE IF NOT EXISTS punch_audit (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  punch_id   UUID REFERENCES punches(id) ON DELETE SET NULL,
  tenant_id  UUID NOT NULL,
  user_id    UUID NOT NULL,
  action     VARCHAR(16) NOT NULL, -- CREATE, REVERT, CORRECT
  old_value  JSONB,
  new_value  JSONB,
  reason     TEXT,
  by_user_id UUID REFERENCES users(id),
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Fast idempotency and uniqueness lookups
CREATE INDEX IF NOT EXISTS idx_punches_idempotency ON punches (tenant_id, idempotency_key);
CREATE INDEX IF NOT EXISTS idx_punches_user_date   ON punches (tenant_id, user_id, punch_date);
```

Notes:  
- `UNIQUE (tenant_id, user_id, punch_date, type)` prevents duplicate IN or OUT on the same normalized day.  
- Keep both `punch_date` (normalized day) and `punch_ts` (exact moment) for reporting and audits.  
- Index on idempotency key makes the idempotency check O(log N).

### 2) Backend: idempotent punch handler

```ts
// server/src/routes/punch.ts
import { Router } from 'express';
import { db } from '../db';
import { normalizeToTenantDate, getTenantTimeZone } from '../utils/time';

const router = Router();

router.post('/punch', async (req, res) => {
  const { tenantId, userId, type, latitude, longitude, lineUserId } = req.body;
  const idempotencyKey = req.header('Idempotency-Key') || req.body.idempotencyKey;

  if (!idempotencyKey) {
    return res.status(400).json({ error: 'Idempotency-Key required' });
  }
  if (!['IN', 'OUT'].includes(type)) {
    return res.status(400).json({ error: 'Invalid type' });
  }

  try {
    await db.query('BEGIN');

    // Idempotency check (exact key match)
    const existing = await db.queryOne(
      `SELECT id, punch_date, punch_ts, type FROM punches WHERE tenant_id = $1 AND idempotency_key = $2`,
      [tenantId, idempotencyKey]
    );

    if (existing) {
      await db.query('COMMIT');
      return res.status(200).json({ ok: true, duplicate: true, punch: existing });
    }

    const now = new Date();
    const { punchDate, punchTs } = await normalizeToTenantDate(now, tenantId);

    const { rows } = await db.query(
      `INSERT INTO punches (tenant_id, user_id, punch_date, punch_ts, type, latitude, longitude, line_user_id, idempotency_key)
       VALUES ($1,$2,$3,$4,$5,$6,$7,$8,$9)
       RETURNING id, punch_date, punch_ts, type`,
      [tenantId, userId, punchDate, punchTs, type, latitude, longitude, lineUserId, idempotencyKey]
    );

    // Audit
    await db.query(
      `INSERT INTO punch_audit (punch_id, tenant_id, user_id, action, new_value, by_user_id)
       VALUES ($1,$2,$3,$4,$5,$6)`,
      [rows[0].id, tenantId, userId, 'CREATE', rows[0], userId]
    );

    await db.query('COMMIT');
    return res.status(201).json({ ok: true, punch: rows[0] });
  } catch (err: any) {
    await db.query('ROLLBACK');
    // Unique violation (race or constraint) -> treat as idempotent success
    if (err.code === '23505') {
      // Best-effort: return the existing punch for this tenant/user/date/type
      const fallback = await db.queryOne(
        `SELECT id, punch_date, punch_ts, type FROM punches WHERE tenant_id = $1 AND user_id = $2 AND punch_date = (
           SELECT punch_date FROM punches WHERE idempotency_key = $3
         ) AND type = $4`,
        [tenantId, userId, idempotencyKey, type]
      ).catch(() => null);
      return res.status(200).json({ ok: true, duplicate: true, punch: fallback });
    }
    console.error('Punch error', err);
    return res.status(500).json({ error: 'Server error' });
  }
});
```

Key points:  
- Idempotency key is required and checked first.  
- Unique constraint is the final safety net.  
- On `23505` (unique violation), treat as success to avoid client retries creating more issues.  
- All writes are transactional.

### 3) Time normalization (tenant-aware day boundary)

```ts

import { utcToZonedTime, format } from 'date-fns-tz';

}

export async function normalizeToTenantDate(ts: Date, tenantId: string) {
  const tz = await getTenantTimeZone(tenantId);
  const zoned = utcToZonedTime(ts, tz);
  const punchDateStr = format(zoned, 'yyyy-MM-dd');
  return {
    punchDate: punchDateStr,
    punchTs: ts.toISOString()
  };
}
```

Why this matters:  
- Prevents punches from landing on the wrong calendar day when users or servers are in different zones or during DST transitions.
