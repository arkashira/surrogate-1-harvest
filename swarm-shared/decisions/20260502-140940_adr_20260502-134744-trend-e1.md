## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: ingest API for two identity sources (e.g., auth + payments), deterministic identity merge via trait hash + embedding similarity (threshold configurable), async attribution job via CF Workers, Postgres+pgvector storage, Redis cache for locks, basic GET endpoints for identity clusters and incidents, and a minimal CLI to trigger/inspect attribution jobs — no customer-facing UI, no billing, free-tier only.

## Open questions
- Which upstream services will emit identity signals and what is the canonical schema?
- What is the deterministic deduplication threshold (embedding similarity + trait hash) to avoid over/under-merging?
- How will we handle PII retention and deletion requests across merged identity clusters?
- What SLA/latency targets for attribution jobs (async vs synchronous) are acceptable for v0.1?
- Who will consume the API (internal CLI, internal dashboard, or partner integrations) and what auth model should we use?
