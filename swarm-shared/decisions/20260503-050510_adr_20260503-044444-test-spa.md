## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: CLI-first (FastAPI server optional) that connects to one AWS account and one GCP project, lists all Lambda + Cloud Functions schedules, computes duplicates (by cron+target+timezone hash), drift (next_run_at deviates >5 min from expected), and orphans (target missing or permission denied), persists results to Supabase Postgres, exports CSV/JSON report and summary counts; no UI, no remediation actions, no auth beyond local API key.

## Open questions
- Which accounts/projects and regions will be in scope for v0.1 and how will credentials be provided (assumed roles vs API keys)?
- What defines 'stale' for v0.1 (e.g., no runs in 90 days, no code changes in 180 days, or disabled targets)?
- How will we detect orphaned schedules (schedule exists but target function/permission missing) without destructive reads?
- Do we need write-back to disable/delete schedules automatically or only report and recommend?
- What RBAC and tenancy model is required (single tenant vs multi-tenant) for the free-tier launch?
