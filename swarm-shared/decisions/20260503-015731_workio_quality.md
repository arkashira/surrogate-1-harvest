## Final consolidated solution  
*(Strongest, most correct + actionable parts from both candidates, contradictions resolved in favor of correctness and deployability)*

---

### 1. Diagnosis (merged)

- **No idempotency key on LINE webhook handler** — identical `X-Line-Signature` + payload replays (network retries, user double-taps) create duplicate clock-in/out rows.
- **Non-atomic read-then-write upsert** — `findOne` → conditional `insert` races under concurrency and can corrupt state.
- **No transactional boundary for punch + audit/notification** — partial failures leave DB consistent but downstream state (notifications, summaries) divergent.
- **Missing DB-level uniqueness** — uniqueness enforced only in app code; concurrent requests bypass it.
- **No replay protection / dedupe cache** — repeated deliveries within minutes (common with LINE) are treated as new events.
- **Schema does not enforce single active punch per user per date** — business rule lives only in code.

---

### 2. Design decisions (resolved)

- **Idempotency mechanism**: use a deterministic `delivery_id` derived from `X-Line-Signature` + body + timestamp (or use LINE’s delivery ID if available). Store it in `line_webhook_deliveries` and reference it in `punches.line_delivery_id`.
- **Uniqueness scope**:
  - **Idempotency**: unique `delivery_id` prevents exact replays.
  - **Business rule**: one punch per `(user_id, punch_date, punch_type)` for normal operations (prevents duplicate “in” or “out” on same day). Implemented as a partial unique index on non-deleted rows.
- **Atomicity**: all writes (idempotency record + punch + audit/notification events) inside a single DB transaction with `ON CONFLICT DO NOTHING` for safe upserts.
- **Concurrency safety**: rely on DB constraints + transaction isolation rather than advisory locks for simplicity and correctness.
- **Schema changes**: additive only (nullable columns, IF NOT EXISTS) to minimize rollout risk.

### 3. Implementation

#### 3.1 DB schema / migration (`server/src/db/schema.sql` or migration file)

```sql
-- Idempotency table for LINE webhook deliveries
CREATE TABLE IF NOT EXISTS line_webhook_deliveries (
  delivery_id TEXT PRIMARY KEY,
  received_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Add delivery reference to punches (nullable for legacy/backfill)
ALTER TABLE punches
  ADD COLUMN IF NOT EXISTS line_delivery_id TEXT;

-- Idempotent punch insert per delivery
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_user_date_type_delivery
  ON punches (user_id, punch_date, punch_type, line_delivery_id)
  WHERE line_delivery_id IS NOT NULL;

-- Business rule: at most one non-deleted punch per user/date/type
CREATE UNIQUE INDEX IF NOT EXISTS idx_user_date_punch_type_active
  ON punches (user_id, punch_date, punch_type)
  WHERE deleted_at IS NULL;
```

#### 3.2 Webhook handler (`server/src/services/lineWebhookService.ts`)

```ts
import { Pool } from 'pg';
import crypto from 'crypto';

const pool = new Pool({ connectionString: process.env.DATABASE_URL });

export async function handleLineWebhook(req: any, res: any) {
  const signature = req.headers['x-line-signature'] as string;
  const body = JSON.stringify(req.body);
  const timestamp = req.headers['x-line-request-timestamp'] || Date.now();

  // Deterministic delivery id from signature + body + timestamp
  const deliveryId = crypto
    .createHash('sha256')
    .update(signature + body + timestamp)
    .digest('hex');

  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // Idempotency check
    const { rows } = await client.query(
      'SELECT 1 FROM line_webhook_deliveries WHERE delivery_id = $1',
      [deliveryId]
    );
    if (rows.length > 0) {
      await client.query('COMMIT');
      return res.status(200).json({ ok: true, reason: 'duplicate' });
    }

    // Record delivery (atomic)
    await client.query(
      'INSERT INTO line_webhook_deliveries (delivery_id) VALUES ($1)',
      [deliveryId]
    );

    const events = req.body.events || [];
    for (const ev of events) {
      if (ev.type !== 'message' || ev.message.type !== 'text') continue;
      const text = ev.message.text.trim().toLowerCase();
      const userId = ev.source.userId;
      const now = new Date();
      const punchDate = now.toISOString().split('T')[0];
      const punchType = text.includes('in') || text.includes('เข้า') ? 'in' : 'out';

      // Atomic upsert: if conflict on unique index, do nothing (idempotent)
      await client.query(
        `INSERT INTO punches (user_id, punch_date, punch_type, line_delivery_id, created_at, updated_at)
         VALUES ($1, $2, $3, $4, $5, $5)
         ON CONFLICT (user_id, punch_date, punch_type, line_delivery_id) DO NOTHING`,
        [userId, punchDate, punchType, deliveryId, now]
      );

      // Optional: emit internal event for notifications/summary (within same tx if needed)
      // await client.query('INSERT INTO punch_events(...) ...')
    }

    await client.query('COMMIT');
    return res.status(200).json({ ok: true });
  } catch (err) {
    await client.query('ROLLBACK');
    console.error('LINE webhook tx failed', err);
    return res.status(500).json({ error: 'internal' });
  } finally {
    client.release();
  }
}
```

### 4. Verification

1. **Idempotency test**  
   Send identical payload twice within seconds:
   ```bash
   curl -X POST https://your-domain.com/webhook/line \
     -H "Content-Type: application/json" \
     -H "X-Line-Signature: <sig>" \
     -d '{"events":[{"type":"message","message":{"type":"text","text":"in"},"source":{"userId":"U123"}}]}'
   ```
   - First call → `200 {ok:true}` and one row in `punches`.  
   - Second call → `200 {ok:true, reason:"duplicate"}` and **no new punch row**.

2. **DB constraint test**  
   Attempt direct insert violating unique index:
   ```sql
   INSERT INTO punches(user_id,punch_date,punch_type,line_delivery_id)
   VALUES ('U123','2026-05-03','in','abc123');
   -- second identical -> ERROR: duplicate key violates unique constraint
   ```

3. **Concurrency test**  
   Fire 10 parallel requests with same payload; verify exactly one punch row created and one delivery row.

4. **LINE retry simulation**  
   Use ngrok to expose local dev, configure LINE console webhook, send message, observe no duplicates in UI/DB after LINE retries.
