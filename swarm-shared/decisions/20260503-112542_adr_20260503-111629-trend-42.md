## Deployment
cf workers (free-tier Workers + D1 (Postgres-compatible) + KV/Queues; fallback to Supabase for Postgres + Redis via free-tier add-on)

## v0.1 scope
v0.1 in 2 weeks: a single POST /v1/chat/completions endpoint that accepts OpenAI-compatible payloads, maps model_alias to a failover route across two providers (OpenAI + Anthropic), enforces per-provider RPM/TPM via Redis counters, logs requests to Supabase Postgres, exposes basic Prometheus metrics, and supports API-key auth for gateway clients (no frontend).

## Open questions
- Which provider auth patterns must be supported at v0.1 (API key header vs custom header vs OAuth)?
- What is the canonical request/response schema across providers (OpenAI-compatible or custom)?
- How will global rate limits be enforced (token bucket vs sliding window) and where (edge vs origin)?
- What PII/redaction approach will be used (regex, presets, or pluggable filters) and where does it run?
- How will failover/backoff and circuit-breaking thresholds be configured per provider/route?
- Will usage/cost attribution be per API key, org, or project — and how will limits be applied?
