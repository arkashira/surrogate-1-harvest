## Deployment
cf workers

## v0.1 scope
v0.1 ships a single static binary (Linux x86_64) with: 1) TUN/WireGuard-like tunnel via boringtun, 2) TLS-fragment + meek fronting modes to hide SOCKS5/TUN artifacts, 3) local SQLite-backed profiles and kill-switch (fail-closed), 4) minimal client-side telemetry hashed for privacy, 5) CF Workers as optional lightweight bridge for domain-fronting endpoints; goal: 2-week MVP enabling WAU-ZB measurement in lab tests simulating DPI probes.

## Open questions
- Which obfuscation primitives minimize DPI detectability while remaining performant on low-end devices (e.g., TLS-fragment vs. meek vs. HTTP/2 mux)?
- How will we distribute/rotate bridge endpoints and PSKs without centralized infrastructure (CF Workers + IPFS/HF)?
- What client-side heuristics define 'blocked' reliably across regions without leaking telemetry that deanonymizes users?
- How do we implement a robust fail-closed kill-switch across TUN/SOCKS5 on each target OS (Windows/Linux/macOS) without admin/root at runtime?
- Can we reuse existing pluggable transports (obfs4, snowflake) as optional backends while keeping the client zero-config?
