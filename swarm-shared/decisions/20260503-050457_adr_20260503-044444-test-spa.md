## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: CLI + FastAPI service that scans Lambda and Cloud Functions in one customer account, persists schedules to Supabase Postgres, detects duplicates (identical cron+target), drifted (cron changed since last scan), and orphans (no deployed function), exposes /audit/run (POST) and /schedules (GET) endpoints, and outputs a simple JSON report. No frontend, no scheduler, no multi-tenant RBAC.

## Open questions
- Which IAM/service-account read-only scopes are acceptable to customers for scanning Lambda and Cloud Functions?
- How often should scans run (hourly/daily) to balance freshness and rate limits?
- Will we store raw function configs or only hashes to minimize PII/surface area?
- How do we map 'owner' for orphan detection (tags, naming convention, or external CMDB)?
