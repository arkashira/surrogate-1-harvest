# workio / backend

## 1. Diagnosis
- Missing tenant cache on `/webhook/line` path: every LINE event triggers DB tenant lookup → DB thrash and higher 429/timeout risk under traffic spikes.
- No request-scoped LRU for `channelId → tenantId`; repeated signature-verified requests re-query the same tenant row.
- Tenant resolution runs **after** signature verification (correct), but still does `SELECT … WHERE channel_id = $1` on every event (inefficient).
- No bounded cache eviction: unbounded map or repeated queries can grow memory under many tenants/channel rotations.
- Cold-start penalty for new/rotated channels: first event pays full DB roundtrip; subsequent events should be near-zero cost.

## 2. Proposed change
File: `/opt/axentx/workio/server/src/middleware/tenant.ts`  
Scope: add a lightweight, request-scoped LRU cache (TTL 10 min, max 500 entries) for `channelId → tenant` and wire it into the existing tenant resolver used by `/webhook/line`.

```typescript
// /opt/axentx/workio/server/src/middleware/tenant.ts
import { Request, Response, NextFunction } from 'express';
import { LRUCache } from 'lru-cache';
import pool from '../db/index.js';

// Lightweight, in-process cache: channelId -> tenant row (TTL 10m, max 500)
const tenantCache = new LRUCache<string, {
  id: string;
  name: string;
  channel_id: string;
  line_channel_secret: string;
  line_access_token: string;
  settings: Record<string, any>;
  created_at: Date;
  updated_at: Date;
}>({
  max: 500,
  ttl: 10 * 60 * 1000, // 10 minutes
  allowStale: false,
  updateAgeOnGet: true,
});

/**
 * Resolve tenant by channelId with LRU cache.
 * Attaches `req.tenant` for downstream handlers (webhook, auth, etc).
 */
export async function resolveTenant(
  req: Request,
  res: Response,
  next: NextFunction
) {
  // Prefer channelId from body (LINE webhook) or header; fallback to param/query
  const channelId =
    req.body?.channelId ||
    req.body?.destination ||
    req.headers['x-line-channel-id'] ||
    req.query.channelId ||
    req.params.channelId;

  if (!channelId || typeof channelId !== 'string') {
    return res.status(400).json({ error: 'channelId missing' });
  }

  // Fast path: cache hit
  const cached = tenantCache.get(channelId);
  if (cached) {
    (req as any).tenant = cached;
    return next();
  }

  // Slow path: DB lookup (fail-fast)
  try {
    const result = await pool.query(
      `SELECT id, name, channel_id, line_channel_secret, line_access_token, settings, created_at, updated_at
       FROM tenants
       WHERE channel_id = $1
       LIMIT 1`,
      [channelId]
    );

    if (result.rows.length === 0) {
      return res.status(404).json({ error: 'Tenant not found' });
    }

    const tenant = result.rows[0];
    tenantCache.set(channelId, tenant);
    (req as any).tenant = tenant;
    next();
  } catch (err) {
    next(err);
  }
}

/**
 * Optional: invalidate tenant cache (useful on tenant update/rotation).
 * Call from tenant update endpoints or admin ops.
 */
export function invalidateTenantCache(channelId: string) {
  tenantCache.delete(channelId);
}
```

Wire into webhook route (example):
```typescript
// /opt/axentx/workio/server/src/routes/webhook/line.ts
import { Router } from 'express';
import { resolveTenant } from '../../middleware/tenant.js';
import { verifyLineSignature } from '../../utils/line.js';

const router = Router();

// Order: signature -> tenant resolution -> handler
router.post('/webhook/line', verifyLineSignature, resolveTenant, (req, res) => {
  const tenant = (req as any).tenant;
  // ... process LINE events using tenant.line_access_token etc.
  res.json({ ok: true });
});

export default router;
```

## 4. Verification
1. Start backend: `cd /opt/axentx/workio/server && npm run dev`
2. Send a LINE webhook payload (or simulate with curl) to `/webhook/line` with valid `destination` (channelId). Confirm 200 and correct tenant resolution.
3. Check logs/metrics: second request with same `channelId` should skip the DB query (cache hit). You can verify by:
   - Temporarily logging “cache hit”/“cache miss” in `resolveTenant`, or
   - Enabling DB query logging and confirming only one `SELECT … WHERE channel_id = $1` per channelId within 10 minutes.
4. Validate invalidation: call `invalidateTenantCache(channelId)` (or update tenant row) and confirm next request fetches fresh row and repopulates cache.
5. Load test (light): fire ~100 rapid requests for same channelId; DB should see one query, response times should drop after first request.
