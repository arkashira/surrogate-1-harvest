## Deployment
cf workers

## v0.1 scope
v0.1 ships in 2 weeks: a single Rust CLI client (Linux x86_64) that (1) creates a local SOCKS5 listener on a randomized port, (2) tunnels traffic through a Cloudflare Worker fronted HTTPS endpoint using TLS1.3 with traffic shaping mimicking common browser flows, (3) applies basic DPI countermeasures (packet size padding, timing jitter, plausible SNI), (4) enforces a fail-closed kill-switch that blocks non-stealth traffic if the stealth path fails, (5) stores session state locally in SQLite, and (6) emits minimal, pseudonymous telemetry to Supabase via CF Workers Queue to compute WAU-ZB.

## Open questions
- Which DPI signatures are highest risk in target regions (SNI, packet lengths, TTL, burst timing)?
- Can we rely on Cloudflare Workers as fronting endpoints without account attribution?
- What local port/TUN naming conventions minimize fingerprintability across desktop/mobile?
- How will clients obtain fresh obfuscation profiles and revocation lists without centralized tracking?
- What kill-switch granularity is required (per-app, system-wide, TUN-only) and how to enforce it client-side?
- How to validate WAU-ZB without deanonymizing users or trusting client-reported telemetry blindly?
