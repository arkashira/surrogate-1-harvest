## Deployment
cf workers (free tier) + Supabase (free tier) — deploy API as Worker, run scheduled scans via Workers Cron Triggers; fallback HF space for batch jobs if compute limits hit

## v0.1 scope
v0.1 in 2 weeks: CLI-first MVP (FastAPI backend deployed to CF Workers) that connects to one AWS account and one GCP project, lists all schedules, detects exact-duplicate cron hashes and orphaned schedules (no invocation logs in 30 days), writes results into Supabase Postgres, and exposes a minimal JSON API and markdown report output; no UI, no auto-remediation, no queue/cache.

## Open questions
- Which IAM scopes can we safely request from customers to list/describe Lambda and Cloud Functions without requiring elevated write permissions?
- How will we authenticate cross-account AWS and cross-project GCP targets (assume-role, service account keys, or OAuth tokens)?
- What staleness threshold (e.g., no invocation in 30/60/90 days) should define an 'orphan' schedule for different customer risk profiles?
- Should duplicate detection be exact cron match or semantic equivalence (rate vs cron, timezone normalization)?
- Where should long-running scans run if CF Workers CPU/time limits are too tight (HF space vs customer self-hosted runner)?
