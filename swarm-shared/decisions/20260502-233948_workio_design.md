# workio / design

**Final Synthesized Recommendation: Idempotent, Race-Free Punch Processing**

**Core Problem**  
Punch records are vulnerable to duplicates and state corruption from LINE webhook redeliveries and concurrent requests. The current read-then-insert pattern is non-atomic, validation is weak, error handling is unclear, and observability is poor.

**Chosen Solution (Resolved for Correctness + Actionability)**  
Implement **idempotency keys with atomic upsert (not optimistic concurrency)** and strict request validation, clear errors, and structured logging. Prefer atomic upsert over optimistic concurrency because:
- It is safer under high concurrency (no “read stale data then overwrite” window).
- It is simpler to reason about for deduplication: same key → same row.
- It still prevents duplicates and state corruption when paired with a unique constraint.

Reserve optimistic locking (version/timestamp) only if you must support mid-air collision detection on business fields; for deduplication, atomic upsert + unique idempotency key is sufficient and more robust.

---

**Concrete Implementation Plan**

1) Schema change (PostgreSQL example; adapt to Sequelize/TypeScript)  
```sql
ALTER TABLE punches
  ADD COLUMN idempotency_key TEXT NOT NULL,
  ADD CONSTRAINT uniq_punch_idempotency_key UNIQUE (idempotency_key);

-- Optional: partial index to help active punches lookups
CREATE INDEX idx_punches_user_active ON punches(user_id)
  WHERE clock_out_time IS NULL;
```

If using Sequelize (JavaScript), match Candidate 1’s field but use `allowNull: false` and unique constraint. If using TypeScript (Candidate 2), add `@Unique` decorator or equivalent.

2) Atomic upsert endpoint (preferred over read-then-write)  
```typescript
// src/controllers/punch.ts
import { Punch } from '../models/punch';
import { validatePunchRequest } from '../validators/punch';
import { logger } from '../utils/logger';

export async function createOrUpdatePunch(req, res) {
  const { idempotency_key, user_id, clock_in, clock_out } = req.body;

  // 1) Validate
  const validation = validatePunchRequest(req.body);
  if (!validation.valid) {
    logger.warn('Invalid punch request', { body: req.body, errors: validation.errors });
    return res.status(400).json({ error: 'Invalid request', details: validation.errors });
  }

  // 2) Atomic upsert (Postgres ON CONFLICT; adapt for your dialect)
  try {
    const [punch, created] = await Punch.upsert(
      {
        idempotency_key,
        user_id,
        clock_in_time: clock_in ? new Date() : null,
        clock_out_time: clock_out ? new Date() : null,
      },
      {
        returning: true,
        conflictFields: ['idempotency_key'],
      }
    );

    logger.info('Punch processed', { idempotency_key, created, punch_id: punch.id });
    return res.json({ punch, created });
  } catch (err) {
    // Unique violation or other DB error
    if (err.name === 'SequelizeUniqueConstraintError') {
      // This should rarely happen due to upsert, but handle gracefully
      const existing = await Punch.findOne({ where: { idempotency_key } });
      logger.warn('Idempotency conflict resolved', { idempotency_key, punch_id: existing.id });
      return res.json({ punch: existing, created: false });
    }

    logger.error('Punch upsert failed', { idempotency_key, error: err.message, stack: err.stack });
    return res.status(500).json({ error: 'Failed to process punch' });
  }
}
```

Notes:
- Use `ON CONFLICT (idempotency_key) DO UPDATE` semantics (or Sequelize `upsert`) to guarantee atomicity.
- If your ORM/dialect lacks true upsert, use a transaction with `INSERT ... ON CONFLICT` raw SQL for safety.
- Do not allow idempotency_key mismatch updates (Candidate 2’s 400 on mismatch) unless you explicitly want to reject replays with different payloads; for webhook replays, returning the existing record is safer and more idempotent.

3) Validation rules (apply before upsert)  
- `idempotency_key`: required, string, max length (e.g., 128).  
- `user_id`: required, valid user.  
- At least one of `clock_in` or `clock_out` truthy (or enforce one active punch per user via application logic).  
- Reject future timestamps beyond a small grace window.

4) Error handling contract  
- 400 for invalid requests (with machine-readable details).  
- 409 for idempotency conflicts only if you choose to surface them (otherwise 200 with existing record).  
- 500 for system errors.  
- Always include a stable error code and correlation ID in responses.

5) Logging and observability  
- Structured logs: idempotency_key, user_id, created/updated, punch_id, latency, outcome.  
- Metrics: punch_requests_total, punch_duplicates_total, punch_errors_total.  
- Alert on spikes in duplicates or errors.

6) Tests (minimal, high-value)  
```typescript
describe('Punch idempotency', () => {
  it('should not create duplicate punches for same idempotency_key', async () => {
    const key = 'key-123';
    await request(app).post('/punch').send({ idempotency_key: key, user_id: 1, clock_in: true });
    const res = await request(app).post('/punch').send({ idempotency_key: key, user_id: 1, clock_in: true });
    expect(res.body.created).toBe(false);
    const punches = await Punch.findAll({ where: { idempotency_key: key } });
    expect(punches).toHaveLength(1);
  });

  it('should reject invalid punch requests with 400', async () => {
    const res = await request(app).post('/punch').send({});
    expect(res.status).toBe(400);
  });
});
```
