### Diagnosis

Both candidates identify similar issues with the current implementation of the `/punch` endpoint:

1. **Lack of Server-Side Idempotency**: This leads to duplicate punch records due to LINE webhook replays or client double-taps.
2. **Insufficient Client-Side Handling**: The absence of optimistic state and debounce on punch buttons allows for rapid repeated submissions, queuing concurrent requests.
3. **Invalid State Transitions**: Without monotonic state transition checks, invalid sequences (e.g., clocking in when already clocked in) can be persisted.
4. **Race Conditions**: Loose `INSERT` statements without conflict handling can produce duplicates under load instead of safe no-ops.
5. **Missing Correlation IDs**: The absence of correlation ID propagation from client to server hampers duplicate detection and tracing across logs.

### Proposed Change

To address these issues, the proposed changes include:

#### Backend

1. **Implement Idempotency**: Add an idempotency key to the `/punch` endpoint. This can be supplied by the client as an `X-Idempotency-Key` header or derived from the LINE event ID.
2. **Enforce State Transitions**: Implement a check to ensure monotonic state transitions (e.g., cannot clock in if the last punch is already “in”).
3. **Conflict Handling**: Modify the `INSERT` statement to use `ON CONFLICT DO NOTHING` or an upsert mechanism on a unique constraint `(user_id, idempotency_key)` to prevent duplicate records.
4. **Correlation ID Propagation**: Ensure that correlation IDs are properly propagated from the client to the server and logged for tracing and duplicate detection.

#### Frontend

1. **Optimistic State**: Implement an optimistic state for the punch button to prevent rapid repeated submissions.
2. **Debounce**: Add a debounce to the punch button to prevent queuing concurrent requests.
3. **Idempotency Key Generation**: Generate an idempotency key for each punch action and attach it to the request as an `X-Idempotency-Key` header.

### Implementation

The implementation details provided in Candidate 1 offer a concrete approach to addressing the identified issues:

- **Backend (Node/Express)**: Implement idempotency middleware and modify the `/punch` route to include idempotency checks, state transition validation, and conflict handling.
- **Frontend (React)**: Modify the `PunchButton` component to include optimistic state, debounce, and generate an idempotency key for each punch action.

### SQL Schema Adjustment

Add an `idempotency_keys` table to store idempotency keys with their corresponding responses for a short TTL (e.g., 5 minutes) to handle replayed requests.

```sql
CREATE TABLE IF NOT EXISTS idempotency_keys (
    key TEXT PRIMARY KEY,
    response_status INTEGER NOT NULL,
    response_body JSONB NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    expires_at TIMESTAMPTZ NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_idempotency_expires ON idempotency_keys (expires_at);
```
