# workio / frontend

## 1. Diagnosis

- **No optimistic UI for clock in/out**: users see no immediate feedback on punch actions; perceived latency increases double-tap risk and poor UX.
- **Missing client-side idempotency key plumbing**: frontend does not generate or attach `X-Idempotency-Key` (or equivalent) for punch requests, leaving duplicate prevention entirely to backend and vulnerable to race/retry duplicates.
- **Punch actions lack local state + rollback**: on slow networks or errors, UI is stuck with no clear retry path and no local cache of last punch state.
- **No debounce/guard against double-tap**: rapid successive taps can queue multiple identical requests before first resolves.
- **Hard to test/verify behavior**: punch UI is purely imperative (no extracted hook/component) and tightly coupled to message/toast side-effects.

## 2. Proposed change

- **Files**: `workio/src/features/clock/components/ClockButton.tsx` (create) and `workio/src/features/clock/hooks/useClockPunch.ts` (create).  
- **Scope**: extract punch logic into a small hook that generates idempotency key, performs optimistic update, retries on failure, and exposes state; wire into existing ClockButton (or create one if missing).  
- **Lines**: new files; minimal edits to parent page to swap in new button/hook.

## 3. Implementation

Create the hook (`workio/src/features/clock/hooks/useClockPunch.ts`):

```ts
import { useState, useCallback } from 'react';
import { v4 as uuidv4 } from 'uuid';

type PunchType = 'IN' | 'OUT';
type Status = 'idle' | 'pending' | 'success' | 'error';

interface LastPunch {
  type: PunchType;
  timestamp: string;
  location?: { lat: number; lng: number };
}

export function useClockPunch() {
  const [status, setStatus] = useState<Status>('idle');
  const [lastPunch, setLastPunch] = useState<LastPunch | null>(null);
  const [error, setError] = useState<string | null>(null);

  const punch = useCallback(async (type: PunchType, location?: { lat: number; lng: number }) => {
    // Guard rapid double-tap
    if (status === 'pending') return;

    const idempotencyKey = uuidv4();
    const now = new Date().toISOString();
    const optimistic: LastPunch = { type, timestamp: now, location };

    // Optimistic update
    setStatus('pending');
    setError(null);
    setLastPunch(optimistic);

    try {
      const res = await fetch('/api/punch', {
        method: 'POST',
        headers: {
          'Content-Type': 'application/json',
          'X-Idempotency-Key': idempotencyKey,
        },
        body: JSON.stringify({
          type,
          timestamp: now,
          location,
        }),
      });

      if (!res.ok) {
        const msg = await res.text().catch(() => `Request failed ${res.status}`);
        throw new Error(msg);
      }

      const data = await res.json();
      // backend may return canonical punch; prefer it if present
      setLastPunch(data || optimistic);
      setStatus('success');
    } catch (err: any) {
      // Rollback optimistic state on failure so UI can reflect reality
      setLastPunch(null);
      setStatus('error');
      setError(err.message || 'Failed to punch');
      // Optionally: retry with exponential backoff here or expose retry()
    }
  }, [status]);

  const reset = useCallback(() => {
    setStatus('idle');
    setError(null);
  }, []);

  return { status, lastPunch, error, punch, reset };
}
```

Create the button component (`workio/src/features/clock/components/ClockButton.tsx`):

```tsx
import React from 'react';
import { useClockPunch } from '../hooks/useClockPunch';

interface ClockButtonProps {
  type: 'IN' | 'OUT';
  onPunched?: () => void;
}

export function ClockButton({ type, onPunched }: ClockButtonProps) {
  const { status, error, punch } = useClockPunch();

  const handleClick = async () => {
    // Simple geolocation; fallback to null if denied/unavailable
    let location: { lat: number; lng: number } | undefined;
    if (navigator.geolocation) {
      try {
        const pos = await new Promise<GeolocationPosition>((resolve, reject) =>
          navigator.geolocation.getCurrentPosition(resolve, reject, {
            enableHighAccuracy: true,
            timeout: 5000,
          })
        );
        location = {
          lat: pos.coords.latitude,
          lng: pos.coords.longitude,
        };
      } catch {
        // ignore; proceed without location
      }
    }

    await punch(type, location);
    if (onPunched) onPunched();
  };

  const isPending = status === 'pending';

  return (
    <div className="flex flex-col items-center gap-2">
      <button
        onClick={handleClick}
        disabled={isPending}
        className={`
          px-6 py-3 rounded-lg font-semibold text-white transition
          ${type === 'IN' ? 'bg-green-600 hover:bg-green-700' : 'bg-red-600 hover:bg-red-700'}
          disabled:opacity-60 disabled:cursor-not-allowed
        `}
      >
        {isPending ? 'กำลังบันทึก...' : type === 'IN' ? 'Clock In' : 'Clock Out'}
      </button>
      {error && <p className="text-sm text-red-500">{error}</p>}
    </div>
  );
}
```

## 4. Verification

1. **Manual happy path**:  
   - Open app, click "Clock In".  
   - Button shows "กำลังบันทึก..." then returns to idle.  
   - Network tab shows single POST with `X-Idempotency-Key` header (UUID).  
   - Backend responds 200; optimistic state matches server response.

2. **Double-tap guard**:  
   - Rapidly click "Clock In" multiple times.  
   - Only one request is sent (subsequent clicks blocked while `status === 'pending'`).

3. **Offline / error rollback**:  
   - Disable network, click "Clock In".  
   - Button shows error text; optimistic state is cleared (no false "last punch" remains).  
   - Re-enable network and retry — succeeds.

4. **Idempotency across retries**:  
   - Use browser devtools to simulate a timeout (delay response >10s) and retry.  
   - Same `X-Idempotency-Key` is not reused across separate clicks (new key per click), but within a single click retries (if you implement internal retry) would reuse same key — backend dedupe works.

5. **Geolocation**:  
   - Allow location → request payload includes `location`.  
   - Deny location → request payload omits location (no failure).
