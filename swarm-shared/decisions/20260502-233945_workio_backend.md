# workio / backend

**Synthesized Final Recommendation: Idempotent, Atomic Punch Processing with Concurrency Control & Observability**

---

### Diagnosis (Consolidated)

All candidates correctly identify the same root causes:

- **Missing idempotency** on punch records causes duplicates from LINE webhook redeliveries.
- **Non-atomic read-then-write** allows race conditions between concurrent deliveries or updates.
- **Weak error handling and logging** make failures hard to detect and debug.
- **Missing concurrency control** allows inconsistent punch state across parallel operations.

---

### Chosen Solution (Best of All Candidates + Corrections)

Implement **idempotency at the API/webhook layer**, **atomic upsert at the database layer**, **optimistic concurrency for state transitions**, and **structured logging + error handling**.

**Why this combination:**
- Candidate 1’s optimistic concurrency pattern is correct in intent but flawed in implementation (manual `version` checks outside Sequelize). We fix this using Sequelize’s built-in `version: true`.
- Candidate 2 separates concerns cleanly (controller vs service) and correctly returns `200 OK` for duplicates.
- Candidate 3 emphasizes logging/error handling, which is essential for production webhooks.

#### 2. Controller: Idempotent Create with Atomic Upsert

`server/src/controllers/punchController.js`
```js
const { Punch } = require('../models');
const logger = require('../utils/logger');

exports.createPunch = async (req, res) => {
  const { idempotency_key } = req.body;
  if (!idempotency_key) {
    logger.warn('Missing idempotency_key in punch creation', { body: req.body });
    return res.status(400).json({ error: 'idempotency_key is required' });
  }

  try {
    // Use transaction + findOrCreate for atomicity
    const result = await Punch.findOrCreate({
      where: { idempotency_key },
      defaults: req.body,
    });

    const [punch, created] = result;
    if (created) {
      logger.info('Punch created', { id: punch.id, idempotency_key });
      return res.status(201).json(punch);
    } else {
      logger.info('Duplicate punch delivery ignored', { idempotency_key });
      return res.status(200).json(punch);
    }
  } catch (error) {
    if (error.name === 'SequelizeUniqueConstraintError') {
      // Race: another request created same key concurrently
      const existing = await Punch.findOne({ where: { idempotency_key } });
      logger.info('Concurrent duplicate resolved', { idempotency_key });
      return res.status(200).json(existing);
    }
    logger.error('Failed to create punch', { error: error.message, idempotency_key });
    return res.status(500).json({ error: 'Internal server error' });
  }
};
```

#### 3. Service: Optimistic Concurrency for State Updates

`server/src/services/punchService.js`
```js
const { Punch } = require('../models');
const logger = require('../utils/logger');

exports.updatePunchState = async (id, newState, expectedVersion) => {
  const punch = await Punch.findByPk(id);
  if (!punch) {
    logger.warn('Punch not found for update', { id });
    throw new Error('Punch not found');
  }

  try {
    const updated = await punch.update({ state: newState }, {
      where: { id, version: expectedVersion || punch.version },
    });
    logger.info('Punch state updated', { id, newState, version: updated.version });
    return updated;
  } catch (error) {
    if (error.name === 'SequelizeOptimisticLockError') {
      logger.warn('Optimistic lock failure on punch update', { id, expectedVersion });
      throw new Error('Concurrent modification detected. Please retry.');
    }
    logger.error('Failed to update punch state', { id, error: error.message });
    throw error;
  }
};
```

#### 5. Verification Steps

1. **Idempotency Test**: Send same webhook payload twice with same `idempotency_key`. Verify:
   - Only one row created.
   - Second request returns `200` with existing record.

2. **Concurrency Test**: Simulate two simultaneous requests with same key. Verify:
   - No duplicates.
   - No crashes.

3. **Optimistic Lock Test**:
   - Fetch punch, note version.
   - Simulate concurrent update by manually incrementing version in DB.
   - Attempt update with stale version → should throw concurrency error.

4. **Logging Test**:
   - Check logs for duplicate deliveries and concurrency errors.

### Why This Is Better

- **Correctness**: Uses Sequelize-native optimistic locking (`version: true`) instead of manual checks.
- **Atomicity**: `findOrCreate` inside transaction prevents race conditions.
- **Actionability**: Clear code, logging, and verification steps.
- **Resilience**: Handles webhook redelivery, concurrency, and conflicts gracefully.
