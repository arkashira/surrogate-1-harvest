# workio / quality

## Final Synthesized Answer (Best Parts Combined)

### 1. Diagnosis (merged, highest-risk items first)
- **No idempotency at DB level**: LINE retries with same `X-Line-Signature` (or same event) produce duplicate punch rows.
- **Non-atomic find-then-upsert**: `findOne → if exists update else insert` races under retries/concurrency → duplicates or lost updates.
- **Missing constraints**: DB allows multiple open punches (`clock_out IS NULL`) per user/date and allows exact duplicate rows.
- **No replay/time-window protection**: old signatures or events could be replayed beyond a reasonable window.
- **No application-level short-circuit**: signature is verified but not persisted; no fast-path for already-processed deliveries.
- **No safe retry/backoff semantics**: transient failures leave client ambiguous.

### 2. Proposed Change (concrete, minimal surface)
- **Files**:
  - `workio/server/src/db/schema.sql` — add idempotency table + punch constraints/indexes.
  - `workio/server/src/routes/line.ts` (or equivalent handler) — atomic upsert with idempotency key and single transaction per webhook.
  - Optional: `workio/server/src/services/lineWebhookService.ts` — extract core logic if you prefer service-layer separation.
- **Scope**: per-webhook single transaction: verify signature → insert idempotency key (unique) → upsert punch with deterministic fields using `INSERT ... ON CONFLICT`/`UPDATE` → commit.

#### 3.2 Webhook handler (`workio/server/src/routes/line.ts`)
```ts
import { Request, Response } from 'express';
import crypto from 'crypto';
import { pool } from '../db';

const LINE_CHANNEL_SECRET = process.env.LINE_CHANNEL_SECRET || '';
const REPLAY_WINDOW_MS = 5 * 60 * 1000; // 5 minutes

function verifySignature(body: string, signature: string): boolean {
  const hash = crypto
    .createHmac('sha256', LINE_CHANNEL_SECRET)
    .update(body, 'utf8')
    .digest('base64');
  return crypto.timingSafeEqual(Buffer.from(hash), Buffer.from(signature));
}

export async function handleLineWebhook(req: Request, res: Response) {
  const signature = req.get('X-Line-Signature') || '';
  const bodyRaw = JSON.stringify(req.body);

  // 1) Basic signature presence and replay window (fast reject)
  if (!signature) return res.status(401).send('Missing signature');
  const now = Date.now();
  const ts = req.body.events?.[0]?.timestamp
    ? new Date(req.body.events[0].timestamp * 1000).getTime()
    : now;
  if (Math.abs(now - ts) > REPLAY_WINDOW_MS) {
    return res.status(400).send('Stale webhook');
  }

  // 2) Verify signature
  if (!verifySignature(bodyRaw, signature)) {
    return res.status(401).send('Invalid signature');
  }

  const events = req.body.events || [];
  if (!events.length) return res.status(200).send('OK');

  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // 3) Idempotency key: prefer signature+body-hash to cover exact delivery
    const idemKey = crypto
      .createHash('sha256')
      .update(signature + ':' + Buffer.from(bodyRaw).toString('base64'))
      .digest('hex');

    const idemInsert = await client.query(
      `INSERT INTO line_webhook_idempotency (key) VALUES ($1) ON CONFLICT (key) DO NOTHING RETURNING key`,
      [idemKey]
    );

    if (idemInsert.rowCount === 0) {
      await client.query('COMMIT');
      return res.status(200).send('OK (duplicate)');
    }

    // 4) Process events (expect one actionable event per webhook in most cases)
    for (const ev of events) {
      if (ev.type !== 'message') continue;

      const userId = ev.source.userId; // map to your internal user_id as needed
      const ts = new Date(ev.timestamp * 1000);

      const action = (ev.message.text || '').trim().toLowerCase();
      const isClockIn = action === 'clock in' || action === 'เข้างาน';
      const isClockOut = action === 'clock out' || action === 'เลิกงาน';
      if (!isClockIn && !isClockOut) continue;

      const today = new Date(ts.getFullYear(), ts.getMonth(), ts.getDate());

      if (isClockIn) {
        // Atomic insert; conflict on open punch prevents duplicate clock-ins
        await client.query(
          `INSERT INTO punches (user_id, clock_in, date, location, created_at)
           VALUES ($1, $2, $3, $4, NOW())
           ON CONFLICT (user_id, date) WHERE clock_out IS NULL DO NOTHING`,
          [userId, ts, today, JSON.stringify({ lat: null, lng: null })]
        );
      } else {
        // Atomic clock-out of the open punch
        await client.query(
          `UPDATE punches
           SET clock_out = $1, updated_at = NOW()
           WHERE user_id = $2 AND date = $3 AND clock_out IS NULL`,
          [ts, userId, today]
        );
      }
    }

    await client.query('COMMIT');
    return res.status(200).send('OK');
  } catch (err) {
    await client.query('ROLLBACK');
    console.error('Webhook processing failed', err);
    return res.status(500).send('Server error');
  } finally {
    client.release();
  }
}
```

2. **Duplicate prevention**:
   - Replay same signed payload twice → second returns `200 OK (duplicate)` and no extra punch rows.
   - Concurrent requests for same user/date → at most one open punch (`clock_out IS NULL`).
3. **Clock-out idempotency**:
   - Repeated clock-out events for same user/date do not create rows or corrupt timestamps.
4. **Signature & replay**:
   - Invalid signature → `401`.
   - Stale timestamp (>5 min) → `400`.
5. **Metrics**:
   - `line_webhook_idempotency` row count grows only for unique deliveries.

### 5. Notes & Decisions (resolved contradictions)
- **Idempotency key choice**: use hash of `signature:bodyBase64` (not signature alone) to cover body-level changes while remaining collision-safe for this domain.
- **Replay window**: enforce 5-minute window at handler entry for fast rejection; keeps idempotency table small and reduces replay risk.
- **Atomicity**: single transaction per webhook with `INSERT ... ON CONFLICT` and unique idempotency key prevents races and duplicates.
- **Constraints**: partial unique index on `(user_id, date)
