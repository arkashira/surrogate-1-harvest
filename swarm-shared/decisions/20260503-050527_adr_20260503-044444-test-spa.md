## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: CLI-only (Python) that authenticates via customer-supplied AWS/GCP credentials, lists Lambda schedules (EventBridge rules) and Cloud Functions triggers, persists metadata to a local DuckDB, runs duplicate/drifted/orphaned detection using croniter, and outputs a JSON report + summary counts. No hosted frontend, no queue, no cache; deployable as a CF Worker script stub (but runs locally for v0.1).

## Open questions
- Which identity/assume-role pattern will the auditor use to read Lambda schedules and Cloud Functions triggers per customer (cross-account vs single-project)?
- How will we detect orphaned schedules (schedules with no target or deleted targets) given platform-specific APIs and eventual consistency?
- What retention and PII policy will we apply to discovered metadata and findings?
- Will customers run this as a CLI/CI job or as a hosted worker (CF) on a schedule — and how do we scope multi-tenant isolation?
- How do we define and compute 'drifted' (time skew vs expression change vs timezone drift) and acceptable thresholds?
