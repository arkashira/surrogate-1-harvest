## Deployment
cf workers (free-tier Durable Objects + Queues; deploy via Wrangler; Postgres via Supabase free tier)

## v0.1 scope
v0.1 (2 weeks): read-only multi-account scanner + Postgres-backed inventory and drift/duplicate/orphan detection. CLI + FastAPI endpoints to trigger an audit, list schedules with state, and export CSV/JSON. Supports AWS Lambda (EventBridge rules + Lambda function ARNs) and GCP Cloud Functions (Cloud Scheduler jobs). Deploy via CF Workers (Durable Object job coordinator) with Supabase free-tier Postgres. No write-back, no UI, no auth beyond bearer token.

## Open questions
- Which AWS accounts and GCP projects will be in scope for v0.1 and how will credentials be provided (OIDC, API keys, or manual tokens)?
- What is the acceptable latency for schedule detection (real-time webhook vs periodic scan) and how often should audits run?
- How should duplicates be grouped (exact cron match, same target, same tags) and what is the canonical retention rule?
- Do we need write-back (auto-disable/delete) in v0.1 or read-only reports + suggested actions only?
- What auth model for consumers (CF Workers Auth token, Supabase Row Level Security, or API keys)?
