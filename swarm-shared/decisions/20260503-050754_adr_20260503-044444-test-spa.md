## Deployment
cf workers

## v0.1 scope
CLI-only v0.1 in 2 weeks: authenticate to AWS and GCP, list EventBridge/Lambda schedules and PubSub/Cloud Function schedules, store canonical Schedule records in Supabase Postgres, run local duplicate/drift/orphan detection, emit JSON/HTML report with findings and remediation hints; no queue/cache/workers, no auto-remediation, no frontend UI.

## Open questions
- Which identity model will the CLI use for cross-cloud reads (assume AWS/GCP creds via env/profiles or workload identity)?
- How will we define 'last_run_at' reliably across Lambda (CloudWatch/Logs) and Cloud Functions (Stackdriver) without extra ingestion costs?
- What retention window for schedule history is sufficient to detect drift/orphans while minimizing storage?
- Will orphan detection require reconciling against IaC state (Terraform/Serverless.yml) or only platform runtime state?
- Do we need writeback (auto-disable/delete) or read-only reports for v0.1?
