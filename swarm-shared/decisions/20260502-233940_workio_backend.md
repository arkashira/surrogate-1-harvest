Candidate 4:
## 1. Diagnosis

- **No idempotency guard on LINE webhook punches** — redeliveries and concurrent deliveries create duplicate clock-in/out rows.
- **Non-atomic read-then-insert for punch state** — races between concurrent webhooks produce multiple open punches or lost state.
- **Missing unique constraint on `(employee_id, status, date)` for open punches** — allows multiple concurrent “in” records per employee.
- **No optimistic concurrency or versioning on punch rows** — updates can silently overwrite concurrent state changes.
- **Webhook handler does not deduplicate by LINE `deliveryId`/`timestamp`** — same event processed multiple times under load.

## 2. Proposed change

- **Scope**: `server/src/models/punch.js` (or `.ts`), `server/src/controllers/punchController.js`, and `server/src/db/schema.sql`.
- **Goal**: Add idempotency via `idempotency_key` (unique, indexed) + unique partial index for open punches + atomic upsert in a single transaction.
- **Deliverable**: Backend-only change that prevents duplicates and races for clock-in/out via LINE webhook.

## 3. Implementation

### 3.1 Schema change (`server/src/db/schema.sql`)

Add `idempotency_key` and constraints:

```sql
-- Add idempotency_key to punches if not present
ALTER TABLE punches
  ADD COLUMN IF NOT EXISTS idempotency_key VARCHAR(255);

-- Unique idempotency key (one key = one processed event)
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_idempotency_key
  ON punches (idempotency_key)
  WHERE idempotency_key IS NOT NULL;

-- Prevent multiple open "in" punches per employee (partial unique index)
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_open_in_per_employee
  ON punches (employee_id)
  WHERE status = 'in' AND deleted_at IS NULL;

-- Optional: created_at index for fast time-range queries
CREATE INDEX IF NOT EXISTS idx_punches_employee_created
  ON punches (employee_id, created_at DESC);
```
