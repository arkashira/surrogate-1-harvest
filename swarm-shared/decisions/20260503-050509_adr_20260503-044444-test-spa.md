## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: CLI + FastAPI service that discovers Lambda and Cloud Functions cron/rate schedules (read-only), persists them to Supabase Postgres, computes duplicates (by hash), drift (expression vs observed runs), and orphans (schedule with no target), exposes findings via REST endpoints and a simple JSON report — deployable as a CF Worker with Supabase free tier; no UI, no scheduler execution, no remediation actions.

## Open questions
- Which IAM/service-account read-only scopes are acceptable to customers for schedule enumeration?
- How frequently should discovery jobs run (hourly vs daily) to balance freshness and quota/cost?
- Will we store raw event logs or only schedule metadata to detect orphaned schedules reliably?
- How do we deduplicate across regions/accounts without false positives for intentionally duplicated schedules?
- What retention period for findings balances auditability and storage on free tier?
