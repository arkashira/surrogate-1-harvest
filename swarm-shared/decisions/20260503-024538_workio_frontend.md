# workio / frontend

### Final consolidated solution

**Problem summary**  
- No optimistic UI → perceived latency, double-tap risk, and poor UX.  
- No idempotency key → duplicate punches on retries or double-clicks.  
- UI waits for server round-trip → janky UX and stale reads across tabs/devices.  
- No offline/retry queue → actions lost on spotty connectivity.  
- Missing clear visual states → confusion, errors, and support load.

**Chosen approach (correctness + actionability)**  
Adopt Candidate 2’s robust primitives (optimistic UI, idempotency, in-memory queue with retries, react-query invalidation/rollback) and Candidate 1’s pragmatic guidance (concrete component locations, simple verification steps). Resolve contradictions in favor of correctness:

- **Idempotency**: always send `X-Idempotency-Key` (or `Idempotency-Key`) using `crypto.randomUUID()`/`uuid`. Server must honor it to prevent duplicate punches.
- **Optimistic UI**: update local state immediately on click; rollback on error. Keep server as source of truth via react-query invalidation.
- **Queue + retry**: in-memory queue with exponential backoff and max retries; persist to IndexedDB only if offline-first is required (start simple).
- **Visual feedback**: idle → pending → success/error with clear affordances and disabled button during submit.
- **Race handling**: cancel outgoing queries on mutate, invalidate on settle, and avoid manual state thrashing.

### Implementation (single final artifact)

File: `workio/src/features/clock/ClockButton.tsx` (or `ClockIn/ClockOut` if you prefer split components).

```tsx
// workio/src/features/clock/ClockButton.tsx
import { useState, useCallback, useRef } from 'react';
import { useMutation, useQueryClient } from '@tanstack/react-query';
import { v4 as uuidv4 } from 'uuid';

type PunchType = 'IN' | 'OUT';

interface PunchRequest {
  type: PunchType;
  location?: GeolocationPosition;
  idempotencyKey: string;
  timestamp: string; // ISO
}

// In-memory queue; promote to IndexedDB if offline-first is required later.
const pendingQueue: Array<{
  request: PunchRequest;
  resolve: (value: unknown) => void;
  reject: (reason?: any) => void;
  retries: number;
}> = [];

const MAX_RETRIES = 3;
const RETRY_DELAY_MS = 2000;

export function ClockButton() {
  const queryClient = useQueryClient();
  const [optimisticState, setOptimisticState] = useState<{
    status: 'idle' | 'pending' | 'success' | 'error';
    lastPunch?: PunchType;
    ts?: string;
  }>({ status: 'idle' });

  const processingRef = useRef(false);

  const punchMutation = useMutation({
    mutationFn: async (req: PunchRequest) => {
      const res = await fetch('/api/punch', {
        method: 'POST',
        headers: {
          'Content-Type': 'application/json',
          'X-Idempotency-Key': req.idempotencyKey,
        },
        body: JSON.stringify(req),
      });

      if (!res.ok) {
        const err: any = new Error(`Punch failed: ${res.status}`);
        err.status = res.status;
        throw err;
      }

      return res.json();
    },
    onMutate: async (req) => {
      setOptimisticState({
        status: 'pending',
        lastPunch: req.type,
        ts: req.timestamp,
      });

      await queryClient.cancelQueries({ queryKey: ['punches'] });
      await queryClient.cancelQueries({ queryKey: ['today-summary'] });

      const previous = queryClient.getQueryData(['today-summary']);
      return { previous };
    },
    onError: (_err, _req, context) => {
      setOptimisticState((s) => ({ ...s, status: 'error' }));
      if (context?.previous) {
        queryClient.setQueryData(['today-summary'], context.previous);
      }
    },
    onSettled: () => {
      queryClient.invalidateQueries({ queryKey: ['today-summary'] });
      queryClient.invalidateQueries({ queryKey: ['punches'] });
    },
  });

  const processQueue = useCallback(async () => {
    if (processingRef.current || pendingQueue.length === 0) return;
    processingRef.current = true;

    while (pendingQueue.length > 0) {
      const item = pendingQueue[0];
      try {
        await punchMutation.mutateAsync(item.request);
        item.resolve({ ok: true });
        pendingQueue.shift();
      } catch (err) {
        item.retries += 1;
        if (item.retries >= MAX_RETRIES) {
          item.reject(err);
          pendingQueue.shift();
          setOptimisticState((s) => ({ ...s, status: 'error' }));
        } else {
          // Exponential backoff before retry
          await new Promise((r) => setTimeout(r, RETRY_DELAY_MS * Math.pow(2, item.retries - 1)));
          continue;
        }
      }
    }

    processingRef.current = false;
  }, [punchMutation]);

  const enqueuePunch = useCallback(
    (type: PunchType) => {
      return new Promise((resolve, reject) => {
        const req: PunchRequest = {
          type,
          idempotencyKey: uuidv4(),
          timestamp: new Date().toISOString(),
        };

        pendingQueue.push({ request: req, resolve, reject, retries: 0 });
        processQueue();
      });
    },
    [processQueue]
  );

  const handleClockIn = () => enqueuePunch('IN');
  const handleClockOut = () => enqueuePunch('OUT');

  const isSubmitting = optimisticState.status === 'pending' || punchMutation.isLoading;

  return (
    <div style={{ display: 'flex', gap: 12, alignItems: 'center' }}>
      <button onClick={handleClockIn} disabled={isSubmitting} aria-busy={isSubmitting}>
        {isSubmitting && optimisticState.lastPunch === 'IN' ? 'Clocking in...' : 'Clock In'}
      </button>

      <button onClick={handleClockOut} disabled={isSubmitting} aria-busy={isSubmitting}>
        {isSubmitting && optimisticState.lastPunch === 'OUT' ? 'Clocking out...' : 'Clock Out'}
      </button>

      {optimisticState.status === 'success' && (
        <div role="status" style={{ color: 'green' }}>
          {optimisticState.lastPunch === 'IN' ? 'Clocked in' : 'Clocked out'}
        </div>
      )}

      {optimisticState.status === 'error' && (
        <div role="alert" style={{ color: 'red' }}>
          Failed to punch. Please try again.
        </div>
      )}
    </div>
  );
}
```

### Key correctness decisions (why this wins)

1. **Idempotency is non-negotiable**  
   - Always include `X-Idempotency-Key`. Server must dedupe by this key. This prevents double-punches from retries, double-clicks, or network replays.

2. **Optimistic UI with safe rollback**  
   - Update local optimistic state immediately for responsiveness.  
   - Use react-query `onMutate` to snapshot previous data and rollback on error.  
   - Invalidate queries on `onSettled` to reconcile with server truth.

3. **Queue + retry for resilience**  
   - In-memory queue with exponential backoff and max retries handles spotty networks without losing punches.  
   - Avoids naive “fire-and-forget” that loses actions. Promote to IndexedDB only if offline-first is a hard requirement (adds complexity).

4. **Clear visual states**  
   - `idle | pending | success | error` with disabled buttons and aria-busy reduces double-taps and support load.

5. **Race and concurrency safety**  
   - `processingRef` prevents concurrent queue processing.  
   - Cancel outgoing queries during mutate to avoid clobbering optimistic state.
