# workio / ops

### Final Synthesis (Correct + Actionable)

**Core diagnosis (merged, prioritized)**
- LINE retries identical webhook deliveries with the same `X-Line-Signature` on network errors/timeouts → duplicate punches.
- Read-then-write upsert without an atomic idempotency key is vulnerable to races and double-punches under concurrent replays.
- Punch + state/notification updates must be in a single transactional boundary; partial updates corrupt employee/time-state consistency.
- Idempotency must be keyed by a stable, verifiable value (message ID or signature-derived hash) scoped to tenant, enforced at DB level.
- Handler must tolerate retries safely (skip duplicates, avoid thundering herd) and surface clear outcomes to LINE (2xx).

**Chosen approach**
- Add a lightweight idempotency table keyed by `(tenant_id, line_message_id)` with a unique constraint (simpler and safer than signature hash per event; avoids false misses when payload normalization varies).
- Use a DB transaction per event (or per batch if events are independent) that:
  1) inserts idempotency record (or detects duplicate),
  2) performs punch upsert,
  3) marks delivery processed,
  4) commits atomically.
- Return success for duplicates (200) so LINE stops retrying; return 5xx only on transient failures to trigger LINE retry.
- Keep handler simple, defensive, and observable.

### 2) Handler implementation (Node + PostgreSQL)

```ts
// server/src/routes/line/webhook.ts
import { db } from '../../db';
import crypto from 'crypto';

function payloadHash(obj: unknown): string {
  return crypto.createHash('sha256').update(JSON.stringify(obj)).digest('hex');
}

export async function handleLineWebhook(req, res) {
  const signature = req.get('X-Line-Signature') || '';
  const tenantId = req.tenant?.id; // set by auth middleware
  if (!tenantId) return res.status(401).send('Unauthorized');

  const payload = req.body;
  const events = payload.events || [];
  if (!events.length) return res.status(200).send('OK');

  const client = await db.connect();
  try {
    await client.query('BEGIN');

    for (const ev of events) {
      // Stable idempotency key: prefer message.id; fallback to deterministic composite
      const msgId = ev.message?.id || `${ev.source?.userId || 'unknown'}:${ev.timestamp}:${ev.type}`;
      const phash = payloadHash(ev);

      // Idempotency check (within transaction)
      const dup = await client.query(
        `SELECT 1 FROM line_webhook_deliveries WHERE tenant_id = $1 AND line_message_id = $2`,
        [tenantId, msgId]
      );

      if (dup.rows.length) {
        // Already processed — skip but continue other events
        continue;
      }

      // Record delivery attempt
      await client.query(
        `INSERT INTO line_webhook_deliveries (tenant_id, line_message_id, event_type, payload_hash, created_at)
         VALUES ($1, $2, $3, $4, NOW())`,
        [tenantId, msgId, ev.type, phash]
      );

      // Process punch atomically
      if (ev.type === 'message' && ev.message?.type === 'text') {
        const text = (ev.message.text || '').trim().toLowerCase();
        const userId = ev.source?.userId;
        if (!userId) continue;

        const isClockIn = text === 'clock in';
        const isClockOut = text === 'clock out';
        if (!isClockIn && !isClockOut) continue;

        const now = new Date();
        // Adjust table/column names to your schema; this uses an upsert per (tenant, user, date)
        await client.query(
          `INSERT INTO punches (tenant_id, user_id, clock_in, clock_out, created_at)
           VALUES ($1, $2, $3, $4, $5)
           ON CONFLICT (tenant_id, user_id, date(created_at))
           DO UPDATE SET
             clock_in = CASE WHEN $3 IS NOT NULL THEN $3 ELSE punches.clock_in END,
             clock_out = CASE WHEN $4 IS NOT NULL THEN $4 ELSE punches.clock_out END`,
          [tenantId, userId, isClockIn ? now : null, isClockOut ? now : null, now]
        );
      }

      // Mark processed
      await client.query(
        `UPDATE line_webhook_deliveries SET processed_at = NOW() WHERE tenant_id = $1 AND line_message_id = $2`,
        [tenantId, msgId]
      );
    }

    await client.query('COMMIT');
    return res.status(200).send('OK');
  } catch (err) {
    await client.query('ROLLBACK').catch(() => {});
    console.error('LINE webhook handler error:', err);
    // Return 5xx so LINE will retry (transient failure). Do not retry on 4xx.
    return res.status(500).send('Internal Server Error');
  } finally {
    client.release();
  }
}
```

### 3) Verification plan

1. **Duplicate delivery test**  
   - Replay the same webhook payload with identical `X-Line-Signature` and events.  
   - Expect: exactly one `line_webhook_deliveries` row per `(tenant_id, line_message_id)` and one punch row; second request returns 200 and does not create duplicates.

2. **Concurrent replay test**  
   - Fire N parallel requests with the same payload.  
   - Expect: no race-induced double-punches; unique constraint prevents duplicate idempotency rows; punch upsert remains consistent.

3. **Transaction rollback test**  
   - Inject an error after idempotency insert but before commit (e.g., throw in punch logic).  
   - Expect: transaction rolls back; no partial idempotency row or punch row remains.

4. **Schema and index checks**  
   - Confirm `uq_tenant_line_message` unique constraint exists.  
   - Confirm `processed_at` index for cleanup jobs.

5. **Operational checks**  
   - Monitor logs for duplicate-skip messages and error rates.  
   - Periodically clean old unprocessed rows (if any) via `DELETE FROM line_webhook_deliveries WHERE processed_at IS NULL AND created_at < NOW() - INTERVAL '1 day'`.

### 4) Notes & corrections vs candidates

- **Idempotency key choice**: Use `line_message_id` + `tenant_id` (unique constraint) instead of signature-derived key to avoid false misses when payload normalization or whitespace varies. Signature can still be logged/hashed for audit.
- **UUID in punches table**: Not required; the idempotency table provides the deduplication guarantee.
- **Atomicity**: Transaction per event (or batch) ensures punch + idempotency record are committed together; no read-then-write races.
- **Error handling**: Return 200 for duplicates (stop LINE retries), 5xx for transient failures (allow LINE retry), 4xx for bad requests (do not retry).
- **Schema alignment**: Use `ON CONFLICT (tenant_id, user_id, date(created_at))` only if that matches your business rule; adjust to your actual punch uniqueness constraint.
