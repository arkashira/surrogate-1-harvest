## Deployment
cf workers (free-tier) + Supabase (free-tier Postgres + Auth/Edge Functions)

## v0.1 scope
v0.1 (2 weeks): CLI-first, read-only auditor that lists Lambda + Cloud Functions schedules, detects duplicates (same cron+target), drifted (cron vs observed next_run drift > threshold), and orphaned (target missing/disabled); outputs JSON report and populates Supabase audit tables; deployable via CF Workers + Supabase free tier; no UI, no remediation actions.

## Open questions
- Which AWS/GCP accounts/projects will be in scope for v0.1 and how will credentials be provided (keys vs OIDC)?
- Is read-only access sufficient or do we need write/remediation hooks to retire schedules automatically?
- What constitutes 'drifted' in practice — clock skew, timezone mismatches, or semantic cron changes vs actual runtime logs?
- How will orphaned schedules be detected (missing target function vs disabled target)?
- Will customers prefer CLI-first or API-first usage for CI/CD integration?
