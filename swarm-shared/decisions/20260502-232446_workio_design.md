# workio / design

## 1. Diagnosis

- No durable idempotency key on punch records → LINE webhook redeliveries and concurrent deliveries create duplicate clock-in/out rows.
- Punch insert is non-atomic (check-then-insert) → race between duplicate deliveries produces two rows for the same `user_id`/`date`/`event_type`.
- Missing unique constraint at DB level → duplicates can only be prevented by application logic; no last-line-of-defense.
- No `line_delivery_id` (or similar) column → retries and replays cannot be de-duplicated after the fact; reporting counts are inflated.
- Clock-state transition logic is split (app checks last punch then inserts) → under load or redelivery, two “clock-in” rows can be created for the same shift.

## 2. Proposed change

- **File**: `/opt/axentx/workio/server/src/db/schema.sql`
- **Scope**: 
  - Add `line_delivery_id VARCHAR(255)` to `punches` table.
  - Add unique constraint on `(user_id, date, event_type)` to enforce one clock-in/out per user per day per event (or use a state-machine constraint if multiple events allowed).
  - Add unique index on `line_delivery_id` (non-null) to guarantee idempotency for LINE redeliveries.
- **Secondary file**: `/opt/axentx/workio/server/src/routes/line/webhook.js` (or equivalent)
  - Accept and persist `line_delivery_id` from LINE event (use `deliveryId` or generate stable hash from event + timestamp + userId) and reject duplicates at insert via unique constraint.

## 3. Implementation

```sql
-- /opt/axentx/workio/server/src/db/schema.sql
-- Add columns + constraints to punches
ALTER TABLE punches
  ADD COLUMN IF NOT EXISTS line_delivery_id VARCHAR(255);

-- Ensure one punch per user/date/event_type (clock-in/out) per day
-- Adjust if business allows multiple events (e.g., in+out pairs); this enforces simple toggle.
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_user_date_event
  ON punches (user_id, date, event_type);

-- Idempotency for LINE redelivery
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_line_delivery_id
  ON punches (line_delivery_id)
  WHERE line_delivery_id IS NOT NULL;
```

```js
// /opt/axentx/workio/server/src/routes/line/webhook.js
// Inside LINE webhook handler (where event is processed)
async function handleLineEvent(req, res) {
  const { events } = req.body;
  const client = await pool.connect();
  try {
    await client.query('BEGIN');
    for (const event of events) {
      if (event.type !== 'message' && event.type !== 'postback') continue;

      const userId = event.source.userId;
      const date = new Date().toISOString().split('T')[0];
      // Stable idempotency key from LINE delivery (or build one)
      const lineDeliveryId = event.deliveryId || 
        `${userId}:${event.timestamp}:${event.type}:${event.message?.id || event.postback?.data || ''}`;

      // Determine event_type by user's last punch or payload
      const lastPunch = await client.query(
        `SELECT event_type FROM punches
         WHERE user_id = $1 AND date = $2
         ORDER BY created_at DESC LIMIT 1`,
        [userId, date]
      );

      const nextEventType = (!lastPunch.rows.length || lastPunch.rows[0].event_type === 'out') ? 'in' : 'out';

      try {
        await client.query(
          `INSERT INTO punches (user_id, date, event_type, line_delivery_id, created_at)
           VALUES ($1, $2, $3, $4, NOW())`,
          [userId, date, nextEventType, lineDeliveryId]
        );
      } catch (err) {
        // Unique violation on line_delivery_id or user/date/event -> duplicate, skip safely
        if (err.code === '23505') {
          // duplicate; continue processing other events
          continue;
        }
        throw err;
      }
    }
    await client.query('COMMIT');
    res.status(200).send('OK');
  } catch (err) {
    await client.query('ROLLBACK');
    console.error('LINE webhook error', err);
    res.status(500).send('ERROR');
  } finally {
    client.release();
  }
}
```

## 4. Verification

- **Unit**: Insert same `line_delivery_id` twice via test script; second insert must be ignored (no new row).
- **Constraint check**: Attempt to insert two rows for same `user_id`/`date`/`event_type`; second must fail with unique violation.
- **Integration**: Use ngrok to expose local webhook and replay a captured LINE event payload twice; verify only one punch row is created.
- **Dashboard**: Run `SELECT user_id, date, event_type, COUNT(*) FROM punches GROUP BY user_id, date, event_type HAVING COUNT(*) > 1;` — should return zero rows after fix is deployed and duplicates are cleaned.
