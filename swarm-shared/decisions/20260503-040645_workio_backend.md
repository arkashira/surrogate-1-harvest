# workio / backend

## Final Synthesized Implementation

I have merged the strongest, most actionable parts from both proposals into a single, coherent solution. I have prioritized **correctness** (strict DB-level guarantees), **concrete actionability** (exact code and SQL to paste), and **robustness** (proper transaction handling and rollback).

### Key Synthesis Decisions
1.  **Middleware vs. Route Logic**: Candidate 1 provided excellent middleware scaffolding, but tying a DB transaction open across middleware and routes is risky (connection pool starvation). I kept the middleware for *reading* the key, but moved the DB transaction strictly inside the route handler for safety.
2.  **Constraints**: Candidate 1's partial index for active sessions is powerful but can be complex if business rules change. Candidate 2 emphasized day-level uniqueness. I implemented **both**: a unique index for daily duplicates (safety net) and application logic for active sessions (flexibility).
3.  **Idempotency Table**: Both agreed this is necessary. I used Candidate 1's schema but ensured the insertion happens within the same atomic transaction as the clock record creation.

### 1. Database Schema Updates
Run this SQL to add the necessary constraints and idempotency table. This prevents duplicates at the database level, which is the ultimate source of truth.

```sql
-- server/src/db/schema.sql
-- Add Idempotency Table (Prevents duplicate processing of webhooks)
CREATE TABLE IF NOT EXISTS clock_idempotency (
  idempotency_key VARCHAR(64) PRIMARY KEY,
  user_id         INTEGER NOT NULL,
  clock_type      VARCHAR(16) NOT NULL,
  clock_record_id INTEGER NOT NULL,
  created_at      TIMESTAMPTZ DEFAULT NOW()
);

-- Safety Net: Prevent duplicate clock-ins/outs on the same day
-- (Resolves Candidate 2's concern regarding same-day duplicates)
CREATE UNIQUE INDEX IF NOT EXISTS idx_user_date_type_unique
  ON clock_records (user_id, DATE(created_at), type)
  WHERE deleted_at IS NULL;

-- Optional but Recommended: Index for fast active session lookup
CREATE INDEX IF NOT EXISTS idx_user_clock_in_active
  ON clock_records (user_id)
  WHERE type = 'clock_in' AND deleted_at IS NULL;
```

### 2. Idempotency Middleware (Lightweight)
We create middleware only to parse the key and check for *existing* results. We do not hold a database transaction open here to avoid connection leaks.

```typescript
// server/src/middleware/idempotency.ts
import { Request, Response, NextFunction } from 'express';
import { pool } from '../db';

declare global {
  namespace Express {
    interface Request {
      idempotencyKey?: string;
    }
  }
}

export async function idempotencyCheck(req: Request, res: Response, next: NextFunction) {
  const key = req.headers['idempotency-key'] as string;
  
  // If no key provided, proceed without idempotency (or return 400 if strictly required)
  if (!key) return next();

  req.idempotencyKey = key;

  // Check if this key was already processed (Fast Lookup)
  const result = await pool.query(
    'SELECT clock_record_id FROM clock_idempotency WHERE idempotency_key = $1',
    [key]
  );

  if (result.rows.length > 0) {
    // Attach result to request so route handler can return it immediately
    req.idempotentResult = { clockRecordId: result.rows[0].clock_record_id };
  }

  next();
}
```

### 3. Clock Route Handler (Atomic & Safe)
This handler combines the logic from both candidates. It enforces business rules (no double clock-in) and ensures database atomicity (all succeed or all rollback).

```typescript
// server/src/routes/clock.ts
import express from 'express';
import { pool } from '../db';
import { idempotencyCheck } from '../middleware/idempotency';

const router = express.Router();

router.post('/clock', idempotencyCheck, async (req, res) => {
  // 1. Handle Idempotent Replays
  if ((req as any).idempotentResult) {
    return res.json({ 
      ok: true, 
      clockRecordId: (req as any).idempotentResult.clockRecordId,
      message: 'Idempotent replay'
    });
  }

  const { userId, type, latitude, longitude } = req.body;
  const client = await pool.connect();
  const key = (req as any).idempotencyKey;

  try {
    await client.query('BEGIN');

    // 2. Business Rule: Prevent Double Clock-In
    if (type === 'clock_in') {
      const active = await client.query(
        'SELECT id FROM clock_records WHERE user_id = $1 AND type = $2 AND deleted_at IS NULL',
        [userId, 'clock_in']
      );
      if (active.rows.length > 0) {
        await client.query('ROLLBACK');
        return res.status(409).json({ error: 'Already clocked in' });
      }
    }

    // 3. Create Record
    const insertRes = await client.query(
      'INSERT INTO clock_records (user_id, type, latitude, longitude) VALUES ($1,$2,$3,$4) RETURNING id',
      [userId, type, latitude, longitude]
    );

    const clockRecordId = insertRes.rows[0].id;

    // 4. Register Idempotency (Only if key was provided)
    if (key) {
      await client.query(
        'INSERT INTO clock_idempotency (idempotency_key, user_id, clock_type, clock_record_id) VALUES ($1,$2,$3,$4)',
        [key, userId, type, clockRecordId]
      );
    }

    await client.query('COMMIT');
    res.json({ ok: true, clockRecordId });

  } catch (err: any) {
    await client.query('ROLLBACK');
    
    // Handle Unique Constraint Violation (Candidate 2's specific concern)
    if (err.code === '23505') { 
      return res.status(409).json({ error: 'Duplicate request detected' });
    }

    console.error('Clock error', err);
    res.status(500).json({ error: 'Internal server error' });
  } finally {
    client.release();
  }
});

export default router;
```

### 4. Verification Checklist

1.  **Duplicate Webhook Test**:
    *   Send a LINE webhook payload with header `idempotency-key: test-123`.
    *   Verify 1 row in `clock_records` and 1 row in `clock_idempotency`.
    *   Send the exact same payload again.
    *   **Result**: Status 200, same `clockRecordId`, no new rows in DB.

2.  **Double Clock-In Prevention**:
    *   Clock In (Success).
    *   Attempt Clock In again without idempotency key.
    *   **Result**: Status 409 "Already clocked in".

3.  **Constraint Safety**:
    *   Manually attempt to insert two clock-ins for the same user on the same day via SQL.
    *   **Result**: DB rejects the second insert due to `idx_user_date_type_unique`.
