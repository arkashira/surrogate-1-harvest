## Deployment
cf workers (free-tier) + Supabase (free-tier Postgres + edge functions) + Redis (free-tier via Upstash or Redis Cloud)

## v0.1 scope
v0.1 in 2 weeks: single instrument workflow (e.g., equity ETF) with research→signal→risk→execution lane; Supabase-backed immutable audit; Redis Streams for ordered replay; one broker connector (sandbox) and one OMS/data vendor connector; 3 compliance guardrails (max notional, venue allowlist, time-of-day); deterministic context snapshots and idempotent execution IDs; basic dashboardless API and logs for audit verification; NSM tracking via workflow_run.fee_generating flag.

## Open questions
- Which brokers/OMS and data vendors must be supported in v0.1 to prove NSM?
- What deterministic replay SLA is required (exact bitwise vs logical) and how will snapshots be versioned?
- What compliance guardrail set is non-negotiable for go-live (e.g., sanctions, concentration, venue/time limits)?
- How will idempotency and exactly-once execution be guaranteed across external broker APIs?
- What latency budget (p50/p99) is acceptable from signal to execution for target instruments?
- How will secrets/credentials be stored and rotated on free-tier infra (Supabase Vault vs CF Workers Secrets)?
