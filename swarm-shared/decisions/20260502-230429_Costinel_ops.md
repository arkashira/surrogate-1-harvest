# Costinel / ops

## Final Implementation Plan — Costinel ops quality increment (<2h)

**Unified objective:**  
Add a single read-only endpoint `GET /api/v1/cost-anomaly/signal/top-hub` (optional `?for_date=YYYY-MM-DD`) that returns the strongest hub insight from the knowledge graph as a cost-anomaly signal (Sense + Signal; no execution). Uses existing knowledge-rag/graph tooling; zero infra changes; fits <2h.

---

### Why this now (merged rationale)
- Pure read-only API change (no infra, no migrations).  
- Reuses proven patterns: top-hub review before planning (#knowledge-rag #graph #hub) and business-research/knowledge-rag integration (#business-research #knowledge-rag #graph).  
- Immediate ops value: surfaces MOC (or top hub) context for cost-anomaly review and dashboards.  
- Follows Costinel philosophy: detect and signal, don’t execute.

### Concrete implementation steps (merged + corrected)

1. **Locate or create the API router**  
   Identify where cost-anomaly routes live (likely `app/api/v1/cost_anomaly.py` or `routes/cost_anomaly.py`). Create or update the router file.

2. **Add lightweight service for top-hub insight**  
   Create `app/services/knowledge_rag.py` (or extend existing) that:
   - Calls knowledge-rag / graph to get top hub and related docs.
   - Returns a structured, typed insight: hub, score, summary, related_docs[], for_date.
   - Uses a deterministic fallback when graph is unavailable so ops can iterate immediately.

3. **Add endpoint**  
   `GET /api/v1/cost-anomaly/signal/top-hub`  
   Query params:
   - `for_date` (optional, default today) — format `YYYY-MM-DD`.

   Response shape:
   ```json
   {
     "signal_type": "top_hub_insight",
     "for_date": "2026-05-03",
     "hub": "MOC",
     "score": 0.92,
     "summary": "MOC shows strongest cross-account cost anomaly linkage this period.",
     "related_docs": [
       { "doc_id": "ri-coverage-2026-05-03", "title": "RI Coverage 2026-05-03", "score": 0.88 },
       { "doc_id": "anomalies-east-1", "title": "East-1 Anomalies", "score": 0.81 }
     ],
     "tags": ["#knowledge-rag", "#graph", "#hub", "#cost-anomaly", "#sense"]
   }
   ```

4. **Wire into main app**  
   Mount router under `/api/v1/cost-anomaly`.

5. **Add minimal tests**  
   - One unit test for endpoint shape and success path.  
   - One test for fallback behavior when graph is empty/unavailable.  
   - One test for invalid `for_date` format returning 400.

6. **Verify locally**  
   Start dev server, hit endpoint with and without `for_date`, confirm JSON shape, 200/400/500 behavior, and that fallback returns valid structure.

---

### Resolved contradictions (correctness + actionability)

- **Graph client availability:** Candidate 1 raised `NotImplementedError` in `_query_graph`, which would cause 500s in production. We resolve by preferring the fallback path as the safe default (deterministic, valid shape) and only calling a real graph client when one is configured. This keeps the endpoint useful during iteration.
- **Error handling:** Candidate 1’s snippet truncated on failure; we explicitly catch `ValueError` for bad dates (400), `NotImplementedError`/unavailable backend (501), and generic exceptions (500) with logging placeholders.
- **Response model consistency:** Both candidates used similar shapes; we unify under a single `TopHubSignalResponse` model and reuse it in both service and endpoint to avoid drift.
- **Dependency injection:** Keep `get_knowledge_rag_service` simple and explicit so wiring a real graph client later is trivial.
