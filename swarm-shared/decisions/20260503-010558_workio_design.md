# workio / design

## 1. Diagnosis

- Missing **database-level partial unique index** to enforce at most one open punch per user per calendar day (`clock_out_at IS NULL`), allowing duplicates under LINE webhook retries or client double-taps.
- App-layer `find-then-insert` is race-prone; concurrent LINE webhook deliveries can create multiple open punches for the same user/day.
- No **idempotency key** on punch writes, so retries (network/retry logic) are not safe.
- Clock-in handler does not atomically close previous open punch; instead relies on app logic that can lose races.
- No **constraint or trigger** preventing overlapping punches (clock-in before previous clock-out), which corrupts duration/report accuracy.

## 2. Proposed change

- **File**: `workio/server/src/db/schema.sql`
  - Add partial unique index: `CREATE UNIQUE INDEX CONCURRENTLY idx_punches_open_per_user_per_day ON punches(user_id, DATE(clock_in_at)) WHERE clock_out_at IS NULL;`
  - Add exclusion constraint (or trigger) to prevent overlapping punches (optional but high value).
- **File**: `workio/server/src/services/punchService.ts` (or equivalent)
  - Replace `find-then-insert` with `INSERT ... ON CONFLICT DO NOTHING` + return existing row; make handler idempotent.
  - Accept optional `idempotency_key` (UUID from client/LINE event id) stored on punch to dedupe retries.

### 3.2 Service change (`punchService.ts`)

```ts
// Minimal, safe clock-in that is idempotent and respects DB uniqueness
export async function clockIn(userId: number, lineEventId?: string): Promise<Punch> {
  const idempotencyKey = lineEventId || crypto.randomUUID();

  // Try insert; if duplicate open punch exists, return the existing one
  const result = await db.query<Punch>(
    `INSERT INTO punches (user_id, clock_in_at, idempotency_key)
     VALUES ($1, NOW(), $2)
     ON CONFLICT (user_id, DATE(clock_in_at))
     WHERE clock_out_at IS NULL
     DO UPDATE SET idempotency_key = EXCLUDED.idempotency_key
     RETURNING *`,
    [userId, idempotencyKey]
  );

  return result.rows[0];
}

// Clock-out remains unchanged but should reference the open punch deterministically
export async function clockOut(userId: number): Promise<Punch> {
  const result = await db.query<Punch>(
    `UPDATE punches
     SET clock_out_at = NOW()
     WHERE id = (
       SELECT id FROM punches
       WHERE user_id = $1 AND clock_out_at IS NULL
       ORDER BY clock_in_at DESC
       LIMIT 1
     )
     RETURNING *`,
    [userId]
  );

  if (!result.rows[0]) {
    throw new Error('No open punch found for user');
  }
  return result.rows[0];
}
```

### 3.3 Migration script (one-time)

Create `workio/server/src/db/migrations/20260504_add_open_punch_unique.sql`:

```sql
-- Up
CREATE UNIQUE INDEX CONCURRENTLY IF NOT EXISTS idx_punches_open_per_user_per_day
  ON punches (user_id, DATE(clock_in_at))
  WHERE clock_out_at IS NULL;

-- Down
DROP INDEX IF EXISTS idx_punches_open_per_user_per_day;
```

Run with `psql workio < server/src/db/migrations/20260504_add_open_punch_unique.sql` (or via your migration runner).

## 4. Verification

1. **Schema check**
   ```bash
   psql workio -c "\d punches"
   psql workio -c "SELECT indexname, indexdef FROM pg_indexes WHERE tablename = 'punches';"
   ```
   Confirm `idx_punches_open_per_user_per_day` exists and is partial (`WHERE clock_out_at IS NULL`).

2. **Race test (simulate concurrent clock-ins)**
   ```bash
   # In psql or via a small Node script, run two concurrent inserts for same user
   # Expect one success, one silently returns existing open punch (no duplicate)
   ```

3. **LINE webhook replay test**
   - Send same LINE event payload twice to `/webhook/line`.
   - Verify only one open punch row exists and second request returns the same punch (idempotency).

4. **Clock-out correctness**
   - Clock in, then clock out; verify `clock_out_at` populated and duration is sensible.
   - Attempt clock-out with no open punch; verify clear error.

5. **Reports/dashboard sanity**
   - Generate daily report for affected user; confirm no duplicate open punches inflate hours.
