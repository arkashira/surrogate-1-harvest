# workio / design

## 1. Diagnosis

- No idempotency key on punch records allows LINE webhook retries to create duplicate clock-in/out rows for same employee/date.
- Application-level upsert is non-atomic under concurrency; race between duplicate webhooks can bypass checks.
- Missing unique constraint means DB cannot enforce “one punch per employee per date” at storage level.
- Punch API accepts repeated payloads without stable dedupe token; retries accumulate noise in reports.
- No lightweight idempotency table for external webhook replays (LINE, future integrations) to cheaply reject duplicates before touching punch logic.

## 2. Proposed change

Add an idempotency layer at the persistence boundary:

- File: `workio/server/src/db/schema.sql` — add `punches.idempotency_key` (text, not null) and unique constraint `(tenant_id, idempotency_key)`.
- File: `workio/server/src/db/schema.sql` — add `punches.created_at` default now() if missing (for ordering).
- File: `workio/server/src/services/punchService.ts` — compute deterministic idempotency key from LINE webhook `deliveryId`/`timestamp`+`userId`+`eventType`, or accept client-supplied `idempotencyKey` for manual punches.
- File: `workio/server/src/routes/punch.ts` — insert via “insert … on conflict (tenant_id, idempotency_key) do nothing” and return existing row when conflicted.

Scope: single migration + two small code edits; <2h.

## 3. Implementation

### 3.1 DB schema (schema.sql)

```sql
-- Add idempotency key and unique constraint
ALTER TABLE punches
  ADD COLUMN IF NOT EXISTS idempotency_key TEXT NOT NULL DEFAULT '';

-- Backfill existing rows with stable keys (employee_id + date + type + created_at hash)
UPDATE punches
SET idempotency_key = CONCAT(
  'legacy-',
  employee_id, '-',
  to_char(date, 'YYYY-MM-DD'), '-',
  type, '-',
  to_char(created_at, 'YYYY-MM-DD"T"HH24:MI:SS')
)
WHERE idempotency_key = '';

-- Enforce uniqueness per tenant (allow same key across tenants)
ALTER TABLE punches
  ADD CONSTRAINT uniq_punch_tenant_idempotency
  UNIQUE (tenant_id, idempotency_key);

-- Ensure created_at exists and is defaulted
ALTER TABLE punches
  ALTER COLUMN created_at SET DEFAULT NOW();
```

### 3.2 Punch service (punchService.ts)

```ts
// Minimal change: compute or accept idempotencyKey and insert atomically
export async function createPunch(
  db: PoolClient,
  data: {
    tenant_id: string;
    employee_id: string;
    type: 'clock_in' | 'clock_out' | 'break_start' | 'break_end';
    timestamp: Date;
    location?: { lat: number; lng: number };
    idempotencyKey: string; // required
  }
) {
  const { tenant_id, employee_id, type, timestamp, location, idempotencyKey } = data;

  const result = await db.query(
    `INSERT INTO punches (tenant_id, employee_id, type, date, location, idempotency_key)
     VALUES ($1, $2, $3, $4, $5, $6)
     ON CONFLICT (tenant_id, idempotency_key) DO UPDATE
     SET employee_id = EXCLUDED.employee_id -- no-op update to return row
     RETURNING *`,
    [tenant_id, employee_id, type, timestamp, location, idempotencyKey]
  );

  return result.rows[0];
}
```

### 3.3 Punch route (punch.ts) — LINE webhook handler

```ts
import { v4 as uuidv4 } from 'uuid';

app.post('/webhook/line', async (req, res) => {
  const events = req.body.events || [];
  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    for (const ev of events) {
      if (ev.type !== 'message' && ev.type !== 'postback') continue;

      // Deterministic idempotency key from LINE delivery context
      const idempotencyKey = ev.source.userId + '-' + ev.timestamp + '-' + ev.type + '-' + (ev.message?.id || ev.postback?.data || '');

      // Map LINE event to punch type (simplified)
      const type = ev.type === 'message' ? 'clock_in' : 'clock_out';

      await punchService.createPunch(client, {
        tenant_id: 'tenant-from-middleware', // set by auth middleware
        employee_id: 'employee-id-from-db',  // resolved by middleware
        type,
        timestamp: new Date(ev.timestamp),
        location: ev.message?.location || undefined,
        idempotencyKey,
      });
    }

    await client.query('COMMIT');
    res.status(200).json({ ok: true });
  } catch (err) {
    await client.query('ROLLBACK');
    console.error(err);
    res.status(500).json({ error: 'internal' });
  } finally {
    client.release();
  }
});
```

### 3.4 Optional: client-supplied idempotency (for manual punches)

Accept header `Idempotency-Key: <uuid>` in punch creation endpoints and pass through to `createPunch`.

## 4. Verification

- Run migration locally: `psql workio < server/src/db/schema.sql` — should apply without errors and enforce unique constraint.
- Unit test: simulate two concurrent calls to `createPunch` with same `tenant_id` and `idempotencyKey`; expect exactly one row inserted and both calls return the same punch row.
- Integration test: replay same LINE webhook payload twice (same `ev.timestamp` and `ev.message.id`); verify second request returns 200 and does not create a second punch row.
- Query check: `SELECT tenant_id, idempotency_key, COUNT(*) FROM punches GROUP BY tenant_id, idempotency_key HAVING COUNT(*) > 1;` — should return zero rows.
