# workio / quality

## Final Synthesis — One Correct, Actionable Plan

**Core problem**: duplicate clock records caused by race conditions (client double-taps, retries, webhook replays) and weak location/validation handling.  
**Goal**: prevent duplicates end-to-end, reject bad location data early, and give users clear feedback.

---

### 1) Backend (source of truth)

- Add `idempotency_key` column + index to `clock_records`.
- Require an idempotency key on `/api/clock` (client-generated stable key: `userId:YYYY-MM-DD:type` or UUID for retries).
- Enforce uniqueness on `idempotency_key` (DB unique index) and use it for all upsert/idempotency checks.
- Validate strictly before insert:
  - `type` must be `in` or `out`.
  - Reject non-finite or out-of-range lat/lng.
  - Reject submissions with `accuracy` > threshold (e.g., 100 m) when provided.
- Enforce business rule: no double `in` without an intervening `out` per user per day (use a serializable or row-level lock/transaction to prevent races).
- Return clear status:
  - `200 OK` with `{ id, ts, duplicate: true }` for idempotent repeats.
  - `409 Conflict` for business rule violations (already clocked in).
  - `400` for bad location/payload.
  - `500` only for real server errors.
- Use a single transaction per request (BEGIN → checks → INSERT → COMMIT) with rollback on error.

**Schema change**
```sql
ALTER TABLE clock_records ADD COLUMN IF NOT EXISTS idempotency_key TEXT;
CREATE UNIQUE INDEX IF NOT EXISTS idx_clock_idempotency ON clock_records (idempotency_key);
CREATE INDEX IF NOT EXISTS idx_clock_user_ts ON clock_records (user_id, ts);
```

**Handler contract**
- Accept `{ userId, type, latitude, longitude, accuracy?, idempotencyKey }`.
- Derive key server-side if missing (`userId:YYYY-MM-DD:type`) but prefer client-supplied UUID for retries.
- Return existing record when key already seen (idempotent).

---

### 2) Frontend (UX + correctness)

- Clock button:
  - Immediately enter `pending` state on tap; disable both buttons while inflight.
  - Show clear labels: “Clocking in…”, “Clocking out…”.
  - On success, show brief confirmation; reset to idle after short delay.
  - On error, show inline message and allow retry.
- Location acquisition:
  - Request high accuracy with timeout.
  - If accuracy is poor or location fails, show actionable UI: “Location unavailable or inaccurate — retry or check permissions.”
  - Do not submit coordinates that fail accuracy/validity checks.
- Idempotency key:
  - Generate stable key for intentional taps (`userId:YYYY-MM-DD:type`).
  - For retries after failures, generate a UUID to avoid masking real duplicates while preserving idempotency for that retry.
- Network resilience:
  - Use exponential backoff for transient failures (max 2–3 retries).
  - Cancel pending requests on unmount.

---

### 3) Webhooks / External retries (LINE)

- Treat every webhook event as potentially replayed.
- Require LINE (or upstream) to include a delivery id or generate one server-side per event and use it as the idempotency key.
- Apply the same uniqueness check on that key so replays do not create duplicates.

---

### 4) Concrete Implementation Summary

**Backend (pseudocode flow)**
1. Parse and validate payload.
2. Validate coordinates and accuracy; reject if invalid/low-accuracy.
3. Begin transaction.
4. Check `idempotency_key` unique index; if exists, return existing record + `duplicate: true`.
5. Enforce business rule (no double-in) with a row-level check under transaction.
6. Insert record with `idempotency_key`.
7. Commit; return success.
8. Rollback on any error; return appropriate status.

**Frontend (pseudocode flow)**
1. On clock tap → set `pending`, acquire location (with accuracy check).
2. If location fails → show error, set `idle`, allow retry.
3. Build idempotency key (stable for user action; UUID for automated retries).
4. POST to `/api/clock`.
5. On 200 → show success, reset after delay.
6. On 409/400 → show specific message, allow correction/retry.
7. On network error → limited retry with backoff, then show error.

---

### 5) Why This Resolves Contradictions

- **Idempotency**: single source of truth (DB unique index on key) prevents duplicates from client, retries, or webhooks.
- **Location quality**: enforced server-side (range + accuracy) and client-side (pre-submit check) to avoid storing untrustworthy data.
- **UX**: optimistic/pending state prevents double-taps and informs users; errors are surfaced clearly.
- **Correctness + actionability**: schema change, index, transaction checks, and frontend state machine are concrete and implementable without ambiguity.
