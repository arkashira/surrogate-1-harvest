# workio / design

## 1. Diagnosis

- **Missing DB uniqueness guardrail**: no partial unique index to enforce at most one open punch per user per calendar day (`clock_out_at IS NULL`), allowing duplicates under LINE webhook retries or client double-taps.
- **Race-prone app upsert**: clock-in handler uses `find-then-insert` without locking or atomic upsert, creating race windows under concurrency.
- **No idempotency key**: LINE webhook retries (network hiccups, 5xx) can replay the same event; without idempotency, duplicate punches are created.
- **Clock-out ambiguity**: multiple open punches for same user/day make it undefined which row to close on clock-out, leading to incorrect hours.
- **No audit trail for edits**: updates to punches (manual corrections) lack `updated_at` and change tracking, reducing trust in reports.

## 2. Proposed change

- **Scope**: `workio/server/src/db/schema.sql` (add constraint + index), `workio/server/src/db/migrations/` (optional migration), and `workio/server/src/routes/webhook/line.ts` (idempotency + atomic upsert).
- **Goal**: enforce single open punch per user/day at DB level and make clock-in handler idempotent + race-safe.

## 3. Implementation

### 3.1 DB schema — add partial unique index and audit columns

```sql
-- workio/server/src/db/schema.sql
-- Add/modify within punches table definition

-- Ensure updated_at exists
ALTER TABLE punches ADD COLUMN IF NOT EXISTS updated_at TIMESTAMPTZ DEFAULT NOW();

-- Partial unique index: at most one open punch per user per calendar day
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_one_open_per_user_day
ON punches (user_id, DATE(clock_in_at))
WHERE clock_out_at IS NULL;

-- Optional: index for faster clock-out lookups
CREATE INDEX IF NOT EXISTS idx_punches_user_open
ON punches (user_id)
WHERE clock_out_at IS NULL;
```

### 3.2 Webhook handler — idempotent, atomic upsert

```ts
// workio/server/src/routes/webhook/line.ts
import { db } from '../../db';
import { sql } from 'kysely';

// Minimal in-memory dedupe for same-process retries (per worker)
// For multi-instance safety, use Redis or DB unique constraint as source of truth.
const recentIds = new Set<string>();

export async function handleLineClockIn(payload: {
  userId: string;
  lineEventId: string; // use as idempotency key
  clockInAt?: string;  // ISO timestamp from LINE or now()
  location?: { lat: number; lng: number };
}) {
  const { userId, lineEventId, clockInAt } = payload;

  // Fast in-process short dedupe (best-effort)
  if (recentIds.has(lineEventId)) return { ok: true, replayed: true };
  recentIds.add(lineEventId);
  setTimeout(() => recentIds.delete(lineEventId), 60_000);

  const now = clockInAt ? new Date(clockInAt) : new Date();

  // Atomic upsert:
  // - If an open punch exists (should be at most one due to partial index), do nothing.
  // - Otherwise insert new open punch.
  // Use ON CONFLICT on the partial index via unique constraint emulation:
  // We enforce uniqueness on (user_id, DATE(clock_in_at)) where clock_out_at IS NULL.
  // To upsert safely, attempt insert and ignore violation; or use advisory lock.
  // Simpler and safe: use advisory lock per user to serialize concurrent clock-ins.
  await db.transaction().execute(async (trx) => {
    // Advisory lock on user id (bigint) to serialize per-user clock-ins
    await sql`SELECT pg_advisory_xact_lock(${userId}::bigint)`.execute(trx);

    // Check for existing open punch
    const existing = await trx
      .selectFrom('punches')
      .select(['id', 'clock_in_at'])
      .where('user_id', '=', userId)
      .where('clock_out_at', 'is', null)
      .executeTakeFirst();

    if (existing) {
      // Open punch exists — do nothing (idempotent)
      return;
    }

    // Insert new open punch
    await trx
      .insertInto('punches')
      .values({
        user_id: userId,
        clock_in_at: now,
        clock_out_at: null,
        location: payload.location ? JSON.stringify(payload.location) : null,
        line_event_id: lineEventId,
        created_at: new Date(),
        updated_at: new Date(),
      })
      .execute();
  });

  return { ok: true };
}
```

### 3.3 Clock-out handler — close the open punch deterministically

```ts
// workio/server/src/routes/webhook/line.ts (add)
export async function handleLineClockOut(payload: {
  userId: string;
  lineEventId: string;
  clockOutAt?: string;
}) {
  const { userId, lineEventId, clockOutAt } = payload;
  const now = clockOutAt ? new Date(clockOutAt) : new Date();

  await db.transaction().execute(async (trx) => {
    // Advisory lock to serialize with clock-in for same user
    await sql`SELECT pg_advisory_xact_lock(${userId}::bigint)`.execute(trx);

    const open = await trx
      .selectFrom('punches')
      .select(['id', 'clock_in_at'])
      .where('user_id', '=', userId)
      .where('clock_out_at', 'is', null)
      .orderBy('clock_in_at', 'desc')
      .executeTakeFirst();

    if (!open) {
      // No open punch — optionally log or create correction record
      return;
    }

    await trx
      .updateTable('punches')
      .set({
        clock_out_at: now,
        line_event_id_out: lineEventId,
        updated_at: new Date(),
      })
      .where('id', '=', open.id)
      .execute();
  });

  return { ok: true };
}
```

## 4. Verification

1. **DB constraint test** — try inserting two open punches for same user same day:
   ```sql
   INSERT INTO punches (user_id, clock_in_at, clock_out_at) VALUES
     ('u1', '2026-05-04 09:00', NULL),
     ('u1', '2026-05-04 09:05', NULL);
   ```
   Second insert should fail with unique violation.

2. **Idempotency test** — call `handleLineClockIn` twice with same `lineEventId`; second call should be a no-op and not create a second punch.

3. **Concurrency test** — simulate parallel clock-ins (e.g., `Promise.all` with 10 concurrent calls) for same user; only one open punch should exist after all settle.

4. **Clock-out determinism** — create one open punch, call `handleLineClockOut`; verify `clock_out_at` is set on that row and no other open rows remain for the user.

5. **LINE retry simulation** — replay same webhook payload (same `lineEventId`) after a delay; confirm no duplicate punches and no errors.
