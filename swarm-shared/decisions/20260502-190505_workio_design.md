# workio / design

## 1. Diagnosis

- **No tenant cache on LINE webhook** — every webhook request performs a DB lookup (`channelId → tenant`); under burst traffic this amplifies DB load and increases latency/errors.
- **Missing health/readiness endpoints** — no `/health` or `/ready` for k8s/PM2/liveness and readiness probes; deployments risk routing traffic before DB/tenant checks pass.
- **No graceful degradation on tenant resolution failure** — if DB is slow or unavailable, LINE webhook returns 5xx and LINE may retry or drop events; no fallback or circuit-breaker behavior.
- **Repeated DB queries for static tenant config** — tenant-level settings (roles, features, limits) fetched per request instead of cached, causing redundant queries.
- **No observability hooks for webhook performance** — missing timing/metrics for tenant resolution and webhook handling; hard to detect regressions or noisy tenants.

## 2. Proposed change

Add an **LRU tenant cache (5–10m TTL)** keyed by `channelId → tenant` in the backend webhook layer, plus **`/health` and `/ready`** endpoints at the Express app root. Scope:

- `workio/server/src/app.ts` (or main Express file) — add routes and cache module import.
- `workio/server/src/services/tenantService.ts` (or new `tenantCache.ts`) — implement cache wrapper around existing tenant resolver.
- `workio/server/src/middleware/lineWebhook.ts` — use cached resolver instead of direct DB call.

## 3. Implementation

### 3.1 Add tenant cache service

Create `workio/server/src/services/tenantCache.ts`:

```ts
// workio/server/src/services/tenantCache.ts
import LRU from 'lru-cache';

export type Tenant = {
  id: string;
  channelId: string;
  name: string;
  // include minimal fields used by webhook/auth
  [k: string]: any;
};

const cache = new LRU<string, Tenant>({
  max: 500,
  ttl: 1000 * 60 * 10, // 10 minutes
  allowStale: false,
  updateAgeOnGet: true,
});

export function getTenantByChannelId(channelId: string): Tenant | undefined {
  return cache.get(channelId);
}

export function setTenantByChannelId(channelId: string, tenant: Tenant): void {
  cache.set(channelId, tenant);
}

export function deleteTenantByChannelId(channelId: string): void {
  cache.delete(channelId);
}

export function clearTenantCache(): void {
  cache.clear();
}
```

If your project uses an existing DB service (e.g., `tenantService.getByChannelId`), wrap it. Example adapter in `tenantService.ts`:

```ts
// workio/server/src/services/tenantService.ts
import { getTenantByChannelId, setTenantByChannelId } from './tenantCache';
import db from '../db'; // adjust import to your db client

export async function resolveTenantByChannelId(channelId: string) {
  const cached = getTenantByChannelId(channelId);
  if (cached) return cached;

  const tenant = await db.query('SELECT * FROM tenants WHERE channel_id = $1 LIMIT 1', [channelId])
    .then((res) => res.rows[0] || null);

  if (tenant) {
    setTenantByChannelId(channelId, tenant);
  }
  return tenant;
}

// Call this on tenant updates/inserts to keep cache fresh
export function invalidateTenantCache(channelId: string) {
  // simple delete; next resolve will repopulate
  require('./tenantCache').deleteTenantByChannelId(channelId);
}
```

### 3.2 Update LINE webhook to use cached resolver

In `workio/server/src/middleware/lineWebhook.ts` (or wherever webhook handler lives):

```ts
// workio/server/src/middleware/lineWebhook.ts
import { Router } from 'express';
import { resolveTenantByChannelId } from '../services/tenantService';

const router = Router();

router.post('/webhook/line', async (req, res) => {
  const channelId = req.headers['x-line-channel-id'] as string || req.body.destination;
  if (!channelId) {
    return res.status(400).json({ error: 'Missing channelId' });
  }

  try {
    const tenant = await resolveTenantByChannelId(channelId);
    if (!tenant) {
      // Do not leak info; LINE may retry. Return 200 to prevent retry storms if idempotent.
      console.warn(`No tenant found for channelId=${channelId}`);
      return res.status(200).json({ ok: true });
    }

    // Attach tenant to request for downstream handlers
    (req as any).tenant = tenant;

    // ... existing webhook processing logic (signature verify, event handling)
    // Example: await handleLineEvents(req.body.events, tenant);

    return res.status(200).json({ ok: true });
  } catch (err) {
    console.error('LINE webhook error', err);
    // Return 200 to avoid retry storms if error is transient; alternatively 503 for clear failure.
    return res.status(200).json({ ok: true });
  }
});

export default router;
```

### 3.3 Add health/readiness endpoints

In `workio/server/src/app.ts` (or main server file):

```ts
// workio/server/src/app.ts
import express from 'express';
import db from './db';
import lineWebhook from './middleware/lineWebhook';

const app = express();
app.use(express.json());

// Existing routes
app.use('/webhook/line', lineWebhook);

// Health (liveness) — lightweight
app.get('/health', (req, res) => {
  res.json({ status: 'ok', uptime: process.uptime() });
});

// Readiness — DB + at least one tenant (optional)
app.get('/ready', async (req, res) => {
  try {
    await db.query('SELECT 1');
    // optional: ensure at least one tenant exists
    const hasTenant = await db.query('SELECT 1 FROM tenants LIMIT 1').then((r) => r.rowCount > 0);
    if (!hasTenant) {
      return res.status(503).json({ status: 'not_ready', reason: 'no_tenants' });
    }
    return res.json({ status: 'ready' });
  } catch (err) {
    return res.status(503).json({ status: 'not_ready', error: String(err) });
  }
});

export default app;
```

If you use a separate server bootstrap (`server.ts`), mount `app` there.

### 3.4 Invalidate cache on tenant updates (best-effort)

Wherever tenants are created/updated (e.g., admin CRUD handlers), call:

```ts
import { invalidateTenantCache } from '../services/tenantService';
invalidateTenantCache(channelId);
```

This keeps cache fresh without adding complexity to every read.

## 4. Verification

1. **Cache behavior**
   - Start server and trigger a LINE webhook for a known `channelId`.
   - Check logs: first request should show DB query; second request within 10m should skip DB (verify via query logs or metrics).
   - Update tenant via admin UI/API and confirm cache is invalidated (next webhook fetches fresh row).

2. **Health endpoints**
   - `curl http://localhost:3000/health` → `{"status":"ok",...}`
   - `curl http://localhost:3000/ready` → `{"status":"ready"}` when DB is up and tenants exist; `503` when DB is down or no tenants.

3. **Load test (quick)**
   - Use `autocannon` or `ab` to send 50–100 POSTs to `/webhook/line` with same `channelId`.
   - Observe DB connections/queries: count should remain low (one initial query + cache hits).

4. **Failure mode**
   - Stop PostgreSQL and hit `/ready` → returns 503.
   - Send LINE webhook while DB is down (with cached tenant present) → should return 200 (cached path) and not crash.

5. **Metrics (optional)**
   - Add simple timers in middleware and log `tenantResolveMs`; confirm cached path
