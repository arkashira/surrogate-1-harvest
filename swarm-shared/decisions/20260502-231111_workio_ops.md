# workio / ops

## Final consolidated solution (correct + actionable)

**Core diagnosis (merged, de-duplicated)**
- No storage-level uniqueness on `(employee_id, punch_type, date)` → duplicates from retries/races.
- Write path is non-idempotent (read-then-insert) and racy under concurrent requests / LINE webhook redelivery.
- No index to enforce/fast-lookup the natural key; duplicate checks become slow scans.
- Missing transactional upsert / `ON CONFLICT` handling.
- (Optional but recommended) no idempotency key to make retries safe across distributed clients.

**Chosen approach (resolve contradictions in favor of correctness + simplicity)**
- Enforce uniqueness at the database level with a constraint (strongest guarantee).
- Make inserts idempotent with `ON CONFLICT DO NOTHING` + return existing row (no blind upserts that overwrite meaningful data).
- Add a supporting index for performance and to help the planner.
- Keep schema minimal: add `idempotency_key` only if you need cross-request retry safety; if added, enforce it with a unique index and prefer it for client retries while still keeping the `(employee_id, punch_type, date)` constraint for data integrity.
- Perform the constraint change safely via migration with rollback plan.

  UNIQUE (employee_id, punch_type, date);

-- 3) Fast lookup for conflict checks and day queries
CREATE INDEX IF NOT EXISTS idx_punches_employee_date_type
  ON punches (employee_id, date, punch_type);

-- 4) Optional: enforce idempotency_key uniqueness when present
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_idempotency_key
  ON punches (idempotency_key)
  WHERE idempotency_key IS NOT NULL;

### 2) Idempotent write path (preferred implementation)

Use a single `INSERT ... ON CONFLICT DO NOTHING` and, if no row was inserted, fetch the existing row. This avoids blind upserts that could overwrite valid data and keeps behavior predictable.

```ts
// workio/server/src/routes/punches.ts
import { Request, Response } from 'express';
import { db } from '../db';
import { punches } from '../db/schema';

export async function createPunchHandler(req: Request, res: Response) {
  const { employee_id, punch_type, date, timestamp, latitude, longitude, image_url, idempotency_key } = req.body;

  if (!employee_id || !punch_type || !date || !timestamp) {
    return res.status(400).json({ error: 'Missing required fields' });
  }

  try {
    // Try idempotency_key first when provided (fast path for retries)
    if (idempotency_key) {
      const [byIdempotency] = await db
        .select()
        .from(punches)
        .where(eq(punches.idempotency_key, idempotency_key))
        .limit(1);

      if (byIdempotency) {
        return res.status(200).json({ punch: byIdempotency, conflict: true });
      }
    }

    // Try insert; if conflict on (employee_id, punch_type, date) do nothing
    const [inserted] = await db
      .insert(punches)
      .values({
        employee_id,
        punch_type,
        date,
        timestamp: new Date(timestamp),
        latitude,
        longitude,
        image_url,
        idempotency_key,
      })
      .onConflictDoNothing()
      .returning();

    if (inserted) {
      return res.status(201).json({ punch: inserted, conflict: false });
    }

    // Race lost: fetch the existing row
    const [existing] = await db
      .select()
      .from(punches)
      .where(eq(punches.employee_id, employee_id))
      .where(eq(punches.punch_type, punch_type))
      .where(eq(punches.date, date))
      .limit(1);

    return res.status(200).json({ punch: existing, conflict: true });
  } catch (err) {
    console.error('Failed to create punch', err);
    return res.status(500).json({ error: 'Internal server error' });
  }
}
```

Notes:
- Do not perform blind upserts that overwrite `timestamp/latitude/longitude` unless you explicitly want last-write-wins semantics. The above preserves the first valid record and makes retries safe.
- If you do want to allow updates on retry (e.g., accept newer location/timestamp), use a targeted `ON CONFLICT ... DO UPDATE` only on nullable/optional fields and keep the natural-key constraint.

---

**Summary of choices**
- Constraint at DB level is the source of truth for duplicates.
- `ON CONFLICT DO NOTHING` + fetch-existing is safer and clearer than blind upserts.
- Added index for performance and correctness of conflict checks.
- Optional `idempotency_key` supported for distributed retries; if used, it’s indexed and checked first for fast retry handling.
