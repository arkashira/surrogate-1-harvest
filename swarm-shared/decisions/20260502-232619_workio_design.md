# workio / design

## Final consolidated solution (correct + actionable)

**Core diagnosis (merged, de-duplicated)**
- No idempotency key on punch records → LINE redeliveries and concurrent deliveries create duplicate rows.
- Non-atomic read-then-insert for clock-in/out → races produce two active punches or broken sequences.
- Webhook side effects are not transactional → partial success (punch saved but notification fails, or vice-versa) causes inconsistency.
- No persisted LINE delivery identifier → retries cannot be de-duplicated at the database level.
- Missing DB-level uniqueness for active punches and for idempotency → silent data corruption under concurrency.

**Single atomic, retry-safe design**
- Persist `line_delivery_id` and enforce a database unique constraint so duplicate deliveries are rejected.
- Use a single DB transaction that:
  1) de-duplicates by `line_delivery_id` (idempotent upsert),
  2) closes any previous open punch for the same user (if present),
  3) inserts the new punch,
  4) records the result, then
  5) sends LINE notification **after commit** (so notifications are not emitted for rolled-back work).
- Keep tenant/location/shift config concurrency fixes out of this change (separate ticket) to reduce scope and risk.

### 1) Schema change (`server/src/db/schema.sql`)

```sql
-- Add idempotency columns
ALTER TABLE punches
  ADD COLUMN line_delivery_id TEXT NULL,
  ADD COLUMN line_delivery_ts TIMESTAMPTZ NULL;

-- Idempotency guard: one row per LINE delivery
CREATE UNIQUE INDEX idx_punches_line_delivery_id
  ON punches (line_delivery_id)
  WHERE line_delivery_id IS NOT NULL;

-- Business rule guard: at most one active punch per user
CREATE UNIQUE INDEX idx_punches_one_active_per_user
  ON punches (user_id)
  WHERE clock_out_at IS NULL;
```

### 2) Service: idempotent punch upsert (`server/src/services/punchService.ts`)

```ts
import { PoolClient, QueryResult } from 'pg';

export interface PunchInput {
  userId: string;
  tenantId: string;
  lineDeliveryId: string;
  lineDeliveryTs: string; // ISO from LINE
  latitude?: number;
  longitude?: number;
  imageUrl?: string;
}

/**
 * Idempotently record a punch using line_delivery_id.
 * Returns the punch row (existing or new).
 */
export async function upsertPunchFromLine(
  client: PoolClient,
  input: PunchInput
): Promise<QueryResult['rows'][number]> {
  const {
    userId,
    tenantId,
    lineDeliveryId,
    lineDeliveryTs,
    latitude,
    longitude,
    imageUrl,
  } = input;

  const now = new Date().toISOString();

  // Deduplicate by line_delivery_id. If duplicate, return existing row.
  const upsertQ = `
    INSERT INTO punches (
      user_id,
      tenant_id,
      line_delivery_id,
      line_delivery_ts,
      clock_in_at,
      clock_out_at,
      latitude,
      longitude,
      image_url
    )
    VALUES ($1, $2, $3, $4, $5, NULL, $6, $7, $8)
    ON CONFLICT (line_delivery_id) DO UPDATE
      SET line_delivery_ts = EXCLUDED.line_delivery_ts
    RETURNING *;
  `;

  const res = await client.query(upsertQ, [
    userId,
    tenantId,
    lineDeliveryId,
    lineDeliveryTs,
    now,
    latitude ?? null,
    longitude ?? null,
    imageUrl ?? null,
  ]);

  return res.rows[0];
}

/**
 * Close any previous open punch for the same user.
 * Returns number of rows closed.
 */
export async function closePreviousOpenPunch(
  client: PoolClient,
  userId: string,
  tenantId: string,
  clockOutAt: string
): Promise<number> {
  const q = `
    UPDATE punches
    SET clock_out_at = $1
    WHERE user_id = $2
      AND tenant_id = $3
      AND clock_out_at IS NULL
    RETURNING id;
  `;
  const r = await client.query(q, [clockOutAt, userId, tenantId]);
  return r.rowCount;
}
```

### 3) Handler: transactional webhook (`server/src/handlers/lineWebhook.ts`)

```ts
import { pool } from '../db';
import { upsertPunchFromLine, closePreviousOpenPunch } from '../services/punchService';
import { sendLineReply } from '../integrations/line';

export async function handleLineWebhook(body: any) {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    const event = body.events?.[0];
    if (!event?.source?.userId || event.type !== 'message') {
      await client.query('ROLLBACK');
      return { status: 'ignored' };
    }

    const userId = event.source.userId;
    const deliveryId = event.delivery?.id || event.message?.id;
    if (!deliveryId) throw new Error('Missing delivery id');

    const tenantId = 'default'; // derive from user/tenant mapping in real code
    const lineDeliveryTs = new Date().toISOString();

    // Idempotent insert (safe under concurrent retries)
    const punch = await upsertPunchFromLine(client, {
      userId,
      tenantId,
      lineDeliveryId: deliveryId,
      lineDeliveryTs,
      latitude: event.message?.content?.latitude,
      longitude: event.message?.content?.longitude,
      imageUrl: event.message?.content?.originalContentUrl,
    });

    // If this is a newly inserted punch (not a duplicate), close any previous open punch
    if (punch.clock_out_at === null) {
      await closePreviousOpenPunch(client, userId, tenantId, lineDeliveryTs);
    }

    await client.query('COMMIT');

    // Notify after commit so we don't send messages for rolled-back work
    await sendLineReply(event.replyToken, `打卡成功 (${punch.clock_in_at})`);
    return { status: 'ok', punch };
  } catch (err) {
    await client.query('ROLLBACK').catch(() => {});
    console.error('Webhook transaction failed', err);
    throw err;
  } finally {
    client.release();
  }
}
```

**Deploy**

```bash
cd /opt/axentx/workio

```

**Verification**
1. **Idempotency**: POST same LINE payload twice (same `delivery.id`). Confirm only one punch row exists and second call returns the existing row.
   ```bash
   psql workio -c "SELECT line_delivery_id, COUNT(*) FROM punches GROUP BY line_delivery_id HAVING COUNT(*) > 1;"
   # Expect zero rows
   ```

2. **Active-punch invariant**: Fire two concurrent requests with different `delivery.id`s for the same user. Confirm at most one row with `clock_out_at IS NULL`.

3. **Transaction rollback**: Force error after punch insert but before commit. Confirm punch row is not persisted.

4. **LINE reply**: Confirm user receives exactly one “打卡成功” per unique delivery and no duplicate replies on redelivery.
