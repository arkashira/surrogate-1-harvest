### Final Proposal

**Diagnosis**: The system lacks idempotency handling for LINE webhooks, has race conditions on punch state transitions, lacks atomic upserts for punch close, and is missing unique constraints to enforce one open punch per user/tenant.

**Proposed Change**:
- Implement idempotency handling using a `webhook_event_idempotency` table with fields for `webhook_event_id`, `tenant_id`, `user_id`, `event_type`, and a timestamp for when the event was processed.
- Enforce one open punch per user/tenant using a partial unique index on the `punches` table.
- Convert punch create/update operations to use atomic upserts, ensuring that either a new punch is inserted or an existing one is updated without allowing concurrent requests to create multiple open punches for the same user/tenant.

**Implementation**:
```sql
-- Idempotency table for LINE webhook events
CREATE TABLE IF NOT EXISTS webhook_event_idempotency (
    webhook_event_id TEXT PRIMARY KEY,
    tenant_id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,
    event_type TEXT NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Partial index to enforce at most one open punch per user/tenant
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_open_unique ON punches (tenant_id, user_id) WHERE clock_out IS NULL;
```
