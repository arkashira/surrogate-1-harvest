## Deployment
cf workers (free-tier Workers + Durable Objects for queueing; Postgres via Supabase free tier)

## v0.1 scope
v0.1 in 2 weeks: CLI + FastAPI service that connects to one AWS account and one GCP project (service-account key), lists schedules into Supabase Postgres, detects duplicates (same cron+target+region), drifted (cron or target hash changed), and orphaned (target not found), exposes a GET /schedules?project_id=...&status=... endpoint and a POST /audit_runs to trigger an audit; no frontend, no automated remediation, no queueing yet — all synchronous with local rate-limiting and dry-run outputs.

## Open questions
- Which identity model will be used to access customer AWS/GCP resources (cross-account roles vs. customer-supplied API keys vs. OAuth)?
- How frequently should audits run by default (hourly/daily) and what are rate-limit/backoff constraints per platform?
- What constitutes an 'orphaned' schedule (target missing, permissions error, or both) and how will we verify safely?
- Will drift include IAM/permissions changes or only cron/target/parameter changes?
- How will customers retire schedules (automated dry-run + approve, or manual-only) and what safety guardrails are required?
