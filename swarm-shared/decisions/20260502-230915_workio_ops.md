## Final consolidated solution (correct + actionable)

**Core diagnosis (agreed by both candidates)**
- Duplicate punches happen because retries (LINE webhooks / clients) are accepted for the same `(employee_id, punch_type, date)`.
- No storage-layer enforcement and no transactional upsert allow races and double rows → payroll drift.
- Missing idempotency key prevents safe, deterministic retry handling.

**Chosen approach (resolve contradictions)**
- **Enforce uniqueness at the DB layer** with a **partial unique index** on `(employee_id, punch_type, DATE(punched_at))` where `deleted_at IS NULL`.  
  - This is stricter and safer than app-only checks and covers the normal daily punch case.
- **Add `idempotency_key` column + unique index** for webhook/API retries.  
  - Allows explicit deduplication when callers supply a key (LINE delivery ID, client request ID).
- **Use atomic `INSERT ... ON CONFLICT` (PostgreSQL)** in a single statement where possible.  
  - Avoids race conditions without long-held locks.
- **Keep behavior backward-compatible**: existing clients continue working; duplicates are suppressed and the existing punch is returned.
- **Do not require app-layer transaction for the normal path** (single upsert is atomic). Use a transaction only when business logic requires multiple dependent writes.

## 1) Schema changes (run once)

```sql
-- Add idempotency_key column if not present
ALTER TABLE punches
  ADD COLUMN IF NOT EXISTS idempotency_key VARCHAR(255);

-- Partial unique index: one non-deleted punch per employee+type+calendar day
-- This prevents accidental double punches on the same day.
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_unique_daily
  ON punches (employee_id, punch_type, DATE(punched_at))
  WHERE deleted_at IS NULL;

-- Unique index for idempotency keys (only on non-deleted rows)
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_idempotency
  ON punches (idempotency_key)
  WHERE idempotency_key IS NOT NULL AND deleted_at IS NULL;
```

**If the table already contains duplicates**, clean them once before creating the index:

```sql
-- Example cleanup (keep the earliest punch per day/type/employee)
DELETE FROM punches
WHERE ctid NOT IN (
  SELECT MIN(ctid)
  FROM punches
  WHERE deleted_at IS NULL
  GROUP BY employee_id, punch_type, DATE(punched_at)
);
```

## 2) API/webhook contract

- Accept optional `idempotency_key` in request body.
- For LINE webhooks, generate deterministic key from LINE fields, e.g.:
  - `line:{userId}:{timestamp}:{messageId}`  
    or  
  - `line:{deliveryId}` if available.
- Return **200 + existing punch** on duplicate (idempotent success), not 409/400.

## 3) Punch creation implementation (single upsert)

```ts
// server/src/routes/punch.ts
import { db } from '../db';
import { Request, Response } from 'express';

export async function createPunch(req: Request, res: Response) {
  const { employee_id, punch_type, punched_at, idempotency_key } = req.body;

  if (!employee_id || !punch_type || !punched_at) {
    return res.status(400).json({ error: 'Missing required fields' });
  }

  try {
    // Atomic upsert:
    // - If conflict on daily unique index, return existing punch.
    // - If idempotency_key provided and conflicts, return existing punch.
    const result = await db.query(
      `INSERT INTO punches (employee_id, punch_type, punched_at, idempotency_key)
       VALUES ($1, $2, $3, $4)
       ON CONFLICT (employee_id, punch_type, DATE(punched_at))
       DO UPDATE SET idempotency_key = COALESCE(EXCLUDED.idempotency_key, punches.idempotency_key)
       WHERE punches.deleted_at IS NULL
       RETURNING *`,
      [employee_id, punch_type, punched_at, idempotency_key || null]
    );

    // If conflict on idempotency_key or partial index, fetch existing row
    if (result.rowCount === 0) {
      const existing = await db.query(
        `SELECT * FROM punches
         WHERE employee_id = $1 AND punch_type = $2 AND DATE(punched_at) = DATE($3)
           AND deleted_at IS NULL
         ORDER BY punched_at DESC
         LIMIT 1`,
        [employee_id, punch_type, punched_at]
      );
      return res.status(200).json({ punch: existing.rows[0] });
    }

    return res.status(200).json({ punch: result.rows[0] });
  } catch (err: any) {
    // Unique violation (23505) — return existing punch
    if (err.code === '23505') {
      const existing = await db.query(
        `SELECT * FROM punches
         WHERE employee_id = $1 AND punch_type = $2 AND DATE(punched_at) = DATE($3)
           AND deleted_at IS NULL
         ORDER BY punched_at DESC
         LIMIT 1`,
        [employee_id, punch_type, punched_at]
      );
      return res.status(200).json({ punch: existing.rows[0] });
    }

    console.error('Punch create error', err);
    return res.status(500).json({ error: 'Internal server error' });
  }
}
```

## 4) LINE webhook idempotency helper

```ts
// server/src/middleware/lineWebhook.ts
export function handleLinePunchEvent(event: any) {
  // Deterministic idempotency key from LINE payload
  const idempotency_key = event.source?.userId && event.timestamp
    ? `line:${event.source.userId}:${event.timestamp}:${event.message?.id || event.mode || 'punch'}`
    : undefined;

  return {
    employee_id: mapLineUserIdToEmployee(event.source.userId),
    punch_type: inferPunchType(event), // 'clock_in' | 'clock_out'
    punched_at: new Date(event.timestamp).toISOString(),
    idempotency_key,
  };
}
```

## 6) Operational notes

- The partial unique index ensures normal daily punches cannot duplicate, even without an idempotency key.
- Idempotency key index gives explicit, cross-request safety for webhooks and API clients.
- Upsert is atomic at the DB level — no race windows.
- If future business rules require multiple punches per day per type (e.g., split shifts), relax the partial index and rely on `idempotency_key` for deduplication instead.
