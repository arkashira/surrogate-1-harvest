## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: single deterministic workflow (research->signal->risk check->execution) with FastAPI endpoints, Supabase Postgres audit trail, Redis-backed cache and idempotency, CF Workers deployment, one broker connector (REST), one risk limit type (notional), replayable audit log and basic NSM telemetry (successful workflow runs weighted by fee_generating).

## Open questions
- Deterministic replay: should workflow_run inputs/outputs be canonicalized (canonical JSON/CBOR) to guarantee bitwise audit equivalence across replays?
- Latency SLA: what is the p99 budget from trigger to execution ack (sub-500ms vs sub-1000ms) and which components are on the critical path?
- Compliance policy engine: rule format (YAML/JSON DSL vs Python functions) and hot-reload requirements for guardrail changes without redeploy?
- Idempotency and exactly-once: will CF Workers + Queues provide at-least-once and rely on idempotency keys, or do we need distributed transactions for risk_limit updates?
- Vendor connector SLAs and auth: which brokers/OMS/data vendors are in scope for v0.1 and what auth flows (OAuth2, API keys, mTLS) must be supported?
- Data residency and encryption: are audit entries required to be encrypted at rest with customer-managed keys for regulatory reasons?
- Replay sandbox: should rejected/failed runs be replayable in an isolated sandbox for compliance review before re-execution?
