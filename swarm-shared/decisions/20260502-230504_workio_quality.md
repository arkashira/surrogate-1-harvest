# workio / quality

## 1) Diagnosis

- **No idempotency key enforcement**: clock-in/out endpoints accept duplicate requests for the same `(employee_id, punch_type, date)` when clients retry on flaky mobile networks → double punches and payroll drift.
- **Non-transactional punch writes**: concurrent retries can insert multiple rows before uniqueness checks commit, creating phantom punches.
- **Missing unique constraint at DB layer**: schema allows duplicate `(employee_id, punch_type, punch_date)` rows; constraint is the single source-of-truth fix.
- **No idempotency table**: without storing request keys, retries cannot be safely de-duplicated across webhook redeliveries and client retries.
- **Race condition on concurrent punch attempts**: two near-simultaneous requests from same employee can both pass “no existing punch” check and insert two rows.

## 2) Proposed change

- **File**: `/opt/axentx/workio/server/src/db/schema.sql` — add unique constraint on punches.
- **File**: `/opt/axentx/workio/server/src/db/migrations/` (or inline schema) — add `punch_idempotency_keys` table for request-level deduplication.
- **File**: `/opt/axentx/workio/server/src/routes/punch.ts` (or equivalent) — wrap punch creation in transaction with `ON CONFLICT DO NOTHING` and idempotency upsert.

## 3) Implementation

### 3.1 DB schema — add constraint and idempotency table

```sql
-- /opt/axentx/workio/server/src/db/schema.sql
-- Add unique constraint to prevent duplicate punches
ALTER TABLE punches
  ADD CONSTRAINT uq_employee_punch_date_type
  UNIQUE (employee_id, punch_type, punch_date);

-- Idempotency keys for webhook/retry safety
CREATE TABLE IF NOT EXISTS punch_idempotency_keys (
  idempotency_key TEXT PRIMARY KEY,
  employee_id     INTEGER NOT NULL,
  punch_type      TEXT    NOT NULL,
  punch_date      DATE    NOT NULL,
  punch_id        INTEGER NOT NULL REFERENCES punches(punch_id),
  created_at      TIMESTAMPTZ DEFAULT NOW()
);

-- Index for fast lookup by key and cleanup
CREATE INDEX IF NOT EXISTS idx_idempotency_key ON punch_idempotency_keys(idempotency_key);
CREATE INDEX IF NOT EXISTS idx_idempotency_employee_date ON punch_idempotency_keys(employee_id, punch_date);
```

### 3.2 Punch route — transactional upsert with idempotency

```ts
// /opt/axentx/workio/server/src/routes/punch.ts
import { Router, Request, Response } from 'express';
import { pool } from '../db';

const router = Router();

/**
 * POST /punch
 * Body: { employee_id, punch_type, punch_date, idempotency_key, latitude?, longitude? }
 *
 * Guarantees:
 * - Exactly one punch per (employee_id, punch_type, punch_date)
 * - Idempotent for same idempotency_key
 */
router.post('/', async (req: Request, res: Response) => {
  const { employee_id, punch_type, punch_date, idempotency_key, latitude, longitude } = req.body;

  if (!employee_id || !punch_type || !punch_date || !idempotency_key) {
    return res.status(400).json({ error: 'Missing required fields' });
  }

  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // 1) Check idempotency key first (fast path)
    const idempotencyCheck = await client.query(
      `SELECT punch_id FROM punch_idempotency_keys WHERE idempotency_key = $1`,
      [idempotency_key]
    );

    if (idempotencyCheck.rows.length > 0) {
      await client.query('ROLLBACK');
      return res.status(200).json({
        message: 'Already processed',
        punch_id: idempotencyCheck.rows[0].punch_id,
        idempotent: true
      });
    }

    // 2) Try insert punch; unique constraint prevents duplicates
    const punchResult = await client.query(
      `INSERT INTO punches (employee_id, punch_type, punch_date, latitude, longitude, created_at)
       VALUES ($1, $2, $3, $4, $5, NOW())
       ON CONFLICT (employee_id, punch_type, punch_date) DO UPDATE
         SET updated_at = NOW()
       RETURNING punch_id, employee_id, punch_type, punch_date`,
      [employee_id, punch_type, punch_date, latitude, longitude]
    );

    const punch = punchResult.rows[0];

    // 3) Record idempotency key (unique by key)
    await client.query(
      `INSERT INTO punch_idempotency_keys (idempotency_key, employee_id, punch_type, punch_date, punch_id)
       VALUES ($1, $2, $3, $4, $5)`,
      [idempotency_key, employee_id, punch_type, punch_date, punch.punch_id]
    );

    await client.query('COMMIT');

    return res.status(201).json({
      message: 'Punch recorded',
      punch,
      idempotent: false
    });
  } catch (err: any) {
    await client.query('ROLLBACK');
    // Unique violation on idempotency key or punches should be handled above; fallback 409
    if (err.code === '23505') {
      return res.status(409).json({ error: 'Conflict: duplicate detected' });
    }
    console.error('Punch error:', err);
    return res.status(500).json({ error: 'Internal server error' });
  } finally {
    client.release();
  }
});

export default router;
```

### 3.3 Client/SDK guidance (optional but recommended)

- Require clients to send an `idempotency_key` (UUID v4) for every punch request.
- For LINE webhook handler, derive deterministic idempotency key from `deliveryId` + `eventType` + `userId` + timestamp bucket (e.g., `line:webhook:{deliveryId}:{event.message.id}`) to deduplicate redeliveries.

3. **Test idempotency flow**  
   Use `curl` or Postman:

   ```bash
   KEY=$(uuidgen)
   curl -X POST http://localhost:3000/punch \
     -H "Content-Type: application/json" \
     -d '{"employee_id":1,"punch_type":"clock_in","punch_date":"2026-05-02","idempotency_key":"'$KEY'"}'
   ```
   - First call → `201` with `idempotent: false`
   - Repeat same call → `200` with `idempotent: true` and same `punch_id`
   - Different `idempotency_key` but same `(employee_id, punch_type, punch_date)` → `201` but DB still has one row (unique constraint prevents second punch)

4. **Concurrency test**  
   Simulate race with parallel requests:
   ```bash
   for i in {1..10}; do
     curl -X POST http://localhost:3000/punch \
       -H "Content-Type: application/json" \
       -d '{"employee_id":99,"punch_type":"clock_out","punch_date":"2026-05-02-conc","idempotency_key":"race-'$RANDOM'"}' &
   done
   wait
   ```
   Verify only one punch row exists for `(99, clock_out, 2026-05-02-conc)`.

5. **LINE webhook redelivery simulation**  
   Send same LINE event payload twice with same derived idempotency key; confirm second is `200` with `idempotent: true` and no duplicate punch row.
