# workio / design

## Final synthesized solution (correct + actionable)

**Core diagnosis (merged, de-duplicated)**  
- LINE retries with the same `X-Line-Signature` and payload can replay and create duplicate punches.  
- Application-level “find → update/else insert” is racy under concurrent retries.  
- No DB-level uniqueness constraint prevents multiple active punches for the same `(user, tenant, date, shift_type)`.  
- No durable idempotency record makes retries invisible and debugging hard.  
- Clock-in/out state is derived in app code without a DB-enforced state machine, allowing invalid transitions under concurrency.

**Chosen approach (resolve contradictions in favor of correctness + concrete actionability)**  
- Keep the schema minimal and migration-friendly: add a lightweight idempotency table + a partial unique index on punches.  
- Perform all work in a single DB transaction per webhook batch with **atomic upsert** and **idempotency insert-first** semantics.  
- Handle `in`/`out` as a simple state transition on the active punch row, guarded by the unique index and transaction.  
- Avoid heavier queue/outbox machinery for now; this is implementable in ≤2 hours and directly fixes the race and replay problems.

---

### 1) Schema changes (`workio/server/src/db/schema.sql`)

```sql
-- Idempotency for LINE webhook replays (lightweight)
CREATE TABLE IF NOT EXISTS line_webhook_idempotency (
  idempotency_key VARCHAR(255) PRIMARY KEY,
  event_type      VARCHAR(50)  NOT NULL,
  user_id         VARCHAR(255) NOT NULL,
  tenant_id       VARCHAR(255) NOT NULL,
  payload_hash    VARCHAR(64)  NOT NULL,
  created_at      TIMESTAMPTZ  DEFAULT NOW()
);

-- Ensure at most one active punch per user/tenant/date/shift_type
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_unique_active
  ON punches (user_id, tenant_id, punch_date, shift_type)
  WHERE status = 'active';

-- Optional traceability; keep nullable for backward compatibility
ALTER TABLE punches
  ADD COLUMN IF NOT EXISTS idempotency_key VARCHAR(255);

CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_idempotency
  ON punches (idempotency_key)
  WHERE idempotency_key IS NOT NULL;
```

**Why this is correct + actionable**  
- `line_webhook_idempotency` primary key gives strict deduplication per signature.  
- Partial unique index enforces “one active punch” at storage level, eliminating the race.  
- Nullable `idempotency_key` avoids breaking existing rows; index only applies when present.

### 2) Webhook handler (`workio/server/src/routes/line/webhook.ts`)

```ts
import { Request, Response } from 'express';
import crypto from 'crypto';
import { pool } from '../../db';

export async function lineWebhookHandler(req: Request, res: Response) {
  const signature = req.get('X-Line-Signature');
  const body = req.body;

  if (!signature || !body?.events?.length) {
    return res.status(400).send('Bad request');
  }

  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    for (const event of body.events) {
      if (event?.type !== 'message' || event.message?.type !== 'text') continue;

      const userId = event.source.userId;
      const text = String(event.message.text).trim().toLowerCase();
      const tenantId = 'default'; // replace with real tenant resolution
      const now = new Date();
      const punchDate = now.toISOString().split('T')[0];
      const shiftType = text.includes('out') ? 'out' : 'in';
      const idempotencyKey = signature; // LINE retries reuse signature
      const payloadHash = crypto.createHash('sha256').update(JSON.stringify(event)).digest('hex');

      // 1) Idempotency guard (atomic)
      const idem = await client.query(
        `INSERT INTO line_webhook_idempotency (idempotency_key, event_type, user_id, tenant_id, payload_hash)
         VALUES ($1, $2, $3, $4, $5)
         ON CONFLICT (idempotency_key) DO NOTHING
         RETURNING idempotency_key`,
        [idempotencyKey, 'punch', userId, tenantId, payloadHash]
      );

      if (idem.rowCount === 0) continue; // already processed

      // 2) Atomic punch upsert for 'in' (create or keep active)
      if (shiftType === 'in') {
        await client.query(
          `INSERT INTO punches (user_id, tenant_id, punch_date, shift_type, status, clock_in_time, idempotency_key)
           VALUES ($1, $2, $3, $4, 'active', $5, $6)
           ON CONFLICT (user_id, tenant_id, punch_date, shift_type)
           WHERE status = 'active'
           DO UPDATE SET clock_in_time = EXCLUDED.clock_in_time, updated_at = NOW()`,
          [userId, tenantId, punchDate, shiftType, now, idempotencyKey]
        );
      }

      // 3) State transition for 'out' (close active 'in')
      if (shiftType === 'out') {
        await client.query(
          `UPDATE punches
           SET status = 'completed', clock_out_time = $1, updated_at = NOW()
           WHERE user_id = $2 AND tenant_id = $3 AND punch_date = $4
             AND shift_type = 'in' AND status = 'active'`,
          [now, userId, tenantId, punchDate]
        );

        // Optional: insert an 'out' punch row for record-keeping
        await client.query(
          `INSERT INTO punches (user_id, tenant_id, punch_date, shift_type, status, clock_out_time, idempotency_key)
           VALUES ($1, $2, $3, $4, 'completed', $5, $6)
           ON CONFLICT (user_id, tenant_id, punch_date, shift_type)
           WHERE status = 'active'
           DO NOTHING`,
          [userId, tenantId, punchDate, shiftType, now, idempotencyKey]
        );
      }
    }

    await client.query('COMMIT');
    res.status(200).json({ ok: true });
  } catch (err) {
    await client.query('ROLLBACK');
    console.error('Webhook processing failed', err);
    res.status(500).send('Server error');
  } finally {
    client.release();
  }
}
```

**Why this is correct + actionable**  
- Idempotency insert-first blocks replays before any state change.  
- `ON CONFLICT ... WHERE status='active'` enforces “one active punch” at the DB level, removing the race.  
- All operations are in a single transaction, so partial failures roll back cleanly.  
- `in` upsert is atomic; `out` transitions the active `in` row to `completed` and optionally records an `out` row without creating duplicate actives.

### 3) Verification checklist (concrete, reproducible)

1. **Schema migration**  
   - Apply `schema.sql`. Confirm:  
     - `line_webhook_idempotency` exists with PK.  
     - `idx_punches_unique_active` exists.  
     - `punches.idempotency_key` column exists.

2. **Idempotency test (replay)**  
   - Send identical webhook payload + `X-Line-Signature` twice.  
   - Assert:  
     - `line_webhook_idempotency` row count = 1.  
     - No duplicate punches created.

3. **Race-condition test (concurrency)**  
   - Fire two concurrent requests with same signature.  
   - Assert:  
     - Only one active punch exists (`SELECT COUNT(*) FROM punches WHERE status='active'` = 1).

4. **State transition test**  
   - Send `in` → punch row with `status='active'`, `shift_type='in'`.  
   - Send `out` → same row `status='completed'` with `clock_out_time`.  
   - Assert:  
     - No active `in` punch remains.  
     - No duplicate active punches.

5. **Log/traceability check**  
   - After retries, verify `idempotency_key` on punches
