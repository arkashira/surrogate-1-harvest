# workio / frontend

## Final synthesized plan (best parts + resolved contradictions + concrete actions)

**Core diagnosis (merged, non-redundant)**
- No real-time subscription layer: frontend relies on polling and hard refresh; LINE webhook–driven clock/leave/OT changes are invisible until reload.
- Multi-tab/device divergence: state changes in one tab (or device) are not reflected in others, causing stale UI and double-submit risk.
- Missing optimistic UI and rollback: network failures leave UI in indeterminate state with no clear recovery.
- No client idempotency handling: retries on flaky networks can create duplicate requests the server may process twice.
- Repeated polling wastes bandwidth and introduces jank.

**Chosen stack (resolved)**
- Real-time transport: **SSE** (simple, HTTP-based, auto-reconnect, tenant-scoped) — preferred over WebSockets for lower ops overhead and compatibility with existing HTTP infra.
- Cross-tab sync: **BroadcastChannel** (native, low latency) — preferred over localStorage events for performance and structured cloning. Keep BroadcastChannel usage minimal and synchronous to avoid ordering pitfalls.
- State/event bus: **Zustand** with devtools for typed domain events and global store.
- Idempotency: **client-generated idempotency keys** (UUID) attached to mutating requests; server must treat keys as dedupe tokens (at least 24–48h TTL).
- Optimistic UI: **optimistic update + rollback** helpers in hooks; queue failed events for retry with backoff and user-facing recovery actions.

## 1. Backend contract (minimal, production-ready)

**Route**
- `GET /api/stream` (tenant-scoped) → `text/event-stream`
- Query: `tenantId` (required)

**Event schema (canonical)**
```ts
// src/lib/events.ts
export type DomainEvent =
  | { type: 'clock'; payload: { userId: string; tenantId: string; clockIn: string; clockOut?: string | null; idempotencyKey?: string } }
  | { type: 'leave'; payload: { id: string; tenantId: string; status: 'approved' | 'rejected' | 'pending'; idempotencyKey?: string } }
  | { type: 'ot'; payload: { id: string; tenantId: string; status: 'approved' | 'rejected' | 'pending'; idempotencyKey?: string } }
  | { type: 'notification'; payload: { id: string; tenantId: string; title: string; body: string; actionUrl?: string } };
```

**Behavior (required for correctness)**
- Authenticate/authorize tenant via session or token (do not rely on query alone in prod).
- Set headers:
  - `Content-Type: text/event-stream`
  - `Cache-Control: no-cache`
  - `Connection: keep-alive`
  - `X-Accel-Buffering: no` (nginx)
- Send keepalive lines (`:\n\n`) every 15s.
- On domain events (e.g., after LINE webhook processing), write:
  - `data: <JSON>\n\n`
- On client disconnect, clean up server-side listeners immediately.
- Idempotency: mutating endpoints (clock/leave/ot) must accept `Idempotency-Key` header (client UUID) and return `409` if key already processed (with the original response), or process and record the key.

## 2. Frontend: real-time layer (SSE + BroadcastChannel + Zustand)

**SSE client (robust)**
- Exponential backoff with jitter, max retry cap, and per-tenant singleton pattern to avoid duplicate connections.
- Dedupe via last-event-id or client-side sequence IDs if server supports it.

`src/lib/sse.ts`
```ts
export type DomainEvent = import('./events').DomainEvent;

const DEFAULT_RETRY_MS = 1000;
const MAX_RETRY_MS = 30_000;

export class SseClient {
  private url: string;
  private listeners = new Set<(e: DomainEvent) => void>();
  private es: EventSource | null = null;
  private retryMs = DEFAULT_RETRY_MS;
  private tenantId: string;
  private reconnectTimer: ReturnType<typeof setTimeout> | null = null;

  constructor(tenantId: string, url = '/api/stream') {
    this.tenantId = tenantId;
    this.url = `${url}?tenantId=${encodeURIComponent(tenantId)}`;
  }

  start() {
    if (this.es) return;
    this.es = new EventSource(this.url);
    this.es.onmessage = (ev) => {
      try {
        const data: DomainEvent = JSON.parse(ev.data);
        this.listeners.forEach((l) => l(data));
        this.retryMs = DEFAULT_RETRY_MS;
      } catch {
        // ignore malformed frames
      }
    };
    this.es.onerror = () => {
      this.close();
      this.scheduleReconnect();
    };
  }

  private scheduleReconnect() {
    if (this.reconnectTimer) return;
    this.reconnectTimer = setTimeout(() => {
      this.reconnectTimer = null;
      this.start();
    }, this.retryMs);
    this.retryMs = Math.min(this.retryMs * 2, MAX_RETRY_MS);
  }

  close() {
    if (this.reconnectTimer) {
      clearTimeout(this.reconnectTimer);
      this.reconnectTimer = null;
    }
    if (this.es) {
      this.es.close();
      this.es = null;
    }
  }

  subscribe(fn: (e: DomainEvent) => void) {
    this.listeners.add(fn);
    return () => this.listeners.delete(fn);
  }
}
```

- Use BroadcastChannel only to forward events between tabs; keep Zustand as source of truth.
- Avoid using BroadcastChannel for large payloads; forward minimal event objects.
