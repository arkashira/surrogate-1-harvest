## Deployment
cf workers

## v0.1 scope
v0.1 ships in 2 weeks: a single Rust CLI binary (Linux x86_64) that (1) fetches encrypted config from Supabase via CF Workers, (2) establishes an obfuscated TLS tunnel masquerading as normal HTTPS using shadowsocks-rust + domain-fronting through CF Workers, (3) creates a userspace tunnel with smoltcp/boringtun to avoid kernel TUN artifacts, (4) implements fail-closed kill-switch that blocks all non-tunnel traffic on detection, and (5) emits local, anonymized session_telemetry to Supabase for WAU-ZB calculation.

## Open questions
- Which DPI signatures are most prevalent in target regions (SNI, JA3, packet timing, TUN interface names)?
- Can we rely solely on domain-fronting/HTTP(S) masquerade via CF Workers, or do we need fallback transports (QUIC, DoH)?
- How will clients discover updated obfuscation endpoints without a centralized beacon that can be blocked?
- What is the acceptable false-positive rate for kill-switch triggers to avoid unnecessary denial-of-service for users?
- How will WAU-ZB telemetry be collected and verified client-side without leaking user identifiers?
