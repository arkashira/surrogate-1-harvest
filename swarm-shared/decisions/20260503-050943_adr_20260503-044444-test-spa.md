## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: CLI + FastAPI service that discovers Lambda schedules (CloudWatch Events rules) and Cloud Function schedules (Cloud Scheduler jobs) for a single customer account, persists normalized Schedule records in Supabase Postgres, detects duplicates (same cron+target), drifted (expression changed), and orphaned (target missing) schedules via hash diffing, exposes a GET /schedules?diff=... endpoint and a simple JSON report; no frontend, no queue, no cache, deployable as a CF Worker or local CLI, OSS-only dependencies.

## Open questions
- Which cloud credentials model will customers use (assume-role, workload identity, or read-only keys) and how do we scope least privilege?
- How frequently should we run discovery scans (hourly vs daily) to balance freshness vs quota/cost on free tier?
- What is the canonical definition of 'orphaned' (no function exists, function deleted, permissions revoked, or no recent logs) and which signals are available within free-tier limits?
- How will customers onboard: provide cross-account roles, per-platform tokens, or connect via Supabase Auth + OAuth proxies?
- Do we store raw log samples or only aggregated last_known_run to stay within Supabase free storage limits?
