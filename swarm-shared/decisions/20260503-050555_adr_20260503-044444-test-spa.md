## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: CLI-first MVP (hosted on CF Workers) that connects to one AWS account and one GCP project via user-supplied credentials, discovers Lambda + Cloud Scheduler entries, persists schedules to Supabase Postgres, computes duplicates (identical cron+target), drifted (same target with different cron), and orphaned (target function missing), and exports a JSON/CSV report plus simple GET endpoints for listing issues — no UI, no queue, no scheduler, manual trigger only.

## Open questions
- Which AWS accounts and GCP projects/scopes will be supported in v0.1 and how are credentials provided (OIDC, service keys, or user-supplied tokens)?
- How will drift be defined: exact cron string match, or semantic equivalence (rate vs cron, timezone normalization, payload checksum)?
- What retention policy for discovered metadata and audit_run history given free-tier Postgres limits?
- Will orphaned detection require live existence checks on target functions or rely only on platform APIs and tags?
- How will customers retire schedules: advisory report only, or automated safe-retire with dry-run and approval flow?
