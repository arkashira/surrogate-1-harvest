# workio / design

## 1. Diagnosis

- **No idempotency key enforcement on LINE webhook ingestion** — duplicate events from LINE retries create duplicate punches/leave/OT records.
- **Race-prone non-atomic upsert in clock-in/out path** — `findOne` → conditional `update`/`insert` allows concurrent requests to create multiple active punches.
- **Missing tenant-scoped uniqueness constraints** — same employee can have multiple active punches across tenants or overlapping intervals.
- **No compensating transaction / rollback for partial failures** — e.g., punch succeeds but notification fails, leaving inconsistent state.
- **Frontend allows rapid double-submit while request is in-flight** — UX permits multiple clicks before confirmation, compounding duplicates.

## 2. Proposed change

**Scope**: `workio/server/src/routes/webhook/line.ts` (or equivalent) + `workio/server/src/db/schema.sql` + `workio/src/components/ClockButton.tsx`

- Add `line_event_id` column + unique constraint `(tenant_id, line_event_id)` to deduplicate at DB level.
- Convert clock-in/out upsert to atomic `INSERT ... ON CONFLICT (employee_id, tenant_id) WHERE clocked_out_at IS NULL DO UPDATE` (single active punch per employee).
- Add idempotency middleware that returns `200 OK` immediately for seen `line_event_id`.
- Frontend: disable button + show pending state until confirmation; add client-side short debounce.

## 3. Implementation

### 3.1 DB schema (add idempotency + active punch constraint)

```sql
-- workio/server/src/db/schema.sql
-- Add idempotency column to punches
ALTER TABLE punches ADD COLUMN IF NOT EXISTS line_event_id TEXT;
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_line_event_id
  ON punches (tenant_id, line_event_id)
  WHERE line_event_id IS NOT NULL;

-- Ensure one active punch per employee (nullable clocked_out_at = active)
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_one_active
  ON punches (tenant_id, employee_id)
  WHERE clocked_out_at IS NULL;
```

### 3.2 Idempotency middleware + atomic upsert

```ts
// workio/server/src/middleware/idempotency.ts
import type { Request, Response, NextFunction } from 'express';
import { pool } from '../db';

export async function lineIdempotency(
  req: Request & { body?: { events?: Array<{ source?: { userId?: string }; message?: any; timestamp?: number; id?: string; type?: string }> } },
  res: Response,
  next: NextFunction
) {
  const events = req.body?.events || [];
  for (const ev of events) {
    if (!ev?.id) continue;
    const exists = await pool.oneOrNone(
      `SELECT 1 FROM punches WHERE line_event_id = $1 AND tenant_id = $2`,
      [ev.id, req.headers['x-tenant-id']]
    ).catch(() => null);
    if (exists) {
      // Already processed — short-circuit safely
      return res.status(200).json({ ok: true, reason: 'duplicate_event' });
    }
  }
  next();
}
```

```ts
// workio/server/src/routes/webhook/line.ts
import { Router } from 'express';
import { lineIdempotency } from '../../middleware/idempotency';
import { pool } from '../../db';

const router = Router();

router.post('/line', lineIdempotency, async (req, res) => {
  const tenantId = req.headers['x-tenant-id'];
  const events = req.body?.events || [];

  for (const ev of events) {
    if (ev.type !== 'message' || ev.message?.type !== 'text') continue;
    const userId = ev.source?.userId;
    const text = ev.message.text?.trim();
    const lineEventId = ev.id;

    if (!userId || !text || !lineEventId) continue;

    // Resolve employee by LINE userId + tenant
    const employee = await pool.oneOrNone(
      `SELECT id FROM employees WHERE tenant_id = $1 AND line_user_id = $2`,
      [tenantId, userId]
    ).catch(() => null);

    if (!employee) continue;

    // Atomic upsert: one active punch per employee
    // If there's an active punch (clocked_out_at IS NULL), update it; otherwise insert.
    // We also store line_event_id for idempotency.
    const now = new Date();
    await pool.none(
      `
        INSERT INTO punches (tenant_id, employee_id, clocked_in_at, clocked_out_at, line_event_id, location, note)
        VALUES ($1, $2, $3, NULL, $4, NULL, $5)
        ON CONFLICT (tenant_id, employee_id)
        WHERE clocked_out_at IS NULL
        DO UPDATE SET clocked_out_at = $3, line_event_id = EXCLUDED.line_event_id
        RETURNING *
      `,
      [tenantId, employee.id, now, lineEventId, text]
    ).catch((err) => {
      // If conflict on unique line_event_id, ignore (idempotency already handled)
      if (err.code !== '23505') throw err;
    });
  }

  res.status(200).json({ ok: true });
});

export default router;
```

### 3.3 Frontend: disable double-submit + pending state

```tsx
// workio/src/components/ClockButton.tsx
import { useState } from 'react';
import { clockIn } from '../api/clock';

export function ClockButton() {
  const [pending, setPending] = useState(false);

  const handleClock = async () => {
    if (pending) return;
    setPending(true);
    try {
      await clockIn();
    } finally {
      setPending(false);
    }
  };

  return (
    <button
      onClick={handleClock}
      disabled={pending}
      className={`px-4 py-2 rounded text-white font-medium ${
        pending ? 'bg-gray-400 cursor-not-allowed' : 'bg-blue-600 hover:bg-blue-700'
      }`}
    >
      {pending ? 'กำลังบันทึก...' : 'ลงเวลาทำงาน'}
    </button>
  );
}
```

## 4. Verification

1. **DB constraints**: run migration and confirm indexes exist:
   ```bash
   psql workio -c "\d punches"
   psql workio -c "\d idx_punches_line_event_id"
   psql workio -c "\d idx_punches_one_active"
   ```

2. **Idempotency test**:
   - Send identical LINE event payload twice (same `id`) within seconds.
   - Expect: first `200 OK`, second `200 OK` with `reason: duplicate_event`.
   - Verify only one punch row created (`SELECT count(*) FROM punches WHERE line_event_id = '<id>'`).

3. **Active punch uniqueness**:
   - Simulate concurrent clock-in requests for same employee (e.g., `ab -n 10 -c 5`).
   - Verify only one row with `clocked_out_at IS NULL` exists per employee/tenant.

4. **Frontend behavior**:
   - Click button rapidly; confirm button disables and only one network request is sent.
   - Confirm toast/confirmation appears after success and button re-enables.
