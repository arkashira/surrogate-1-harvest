## Deployment
cf workers

## v0.1 scope
v0.1 ships a single CF Worker proxy that exposes a normalized /v1/chat/completions endpoint, routes to one configured provider (OpenAI) with a failover to a second (Anthropic), enforces global rate-limit and timeout via Redis, records RequestLog to Supabase Postgres, and emits basic Prometheus metrics. Includes health check, simple weighted routing config in Postgres, and zero-downtime deploy to CF Workers free tier.

## Open questions
- Which provider auth patterns must be supported at v0.1 (API key header, Authorization Bearer, custom signed requests)?
- What latency SLO (p95/p99) and throughput (RPS) targets for the CF Workers runtime?
- Will request/response bodies be logged in Postgres or only metadata/audit summaries (PII implications)?
- How to synchronize per-provider quota windows across distributed CF Workers (Redis strategy or edge KV)?
- Which failover triggers are required (status codes, timeouts, 429s, cost overruns) and retry budgets?
- Do we need per-customer API keys and tenant isolation at v0.1 or only a single global gateway key?
