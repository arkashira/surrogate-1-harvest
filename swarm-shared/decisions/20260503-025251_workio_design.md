# workio / design

## 1. Diagnosis

- **No server-side idempotency for LINE webhook events**: LINE retries on timeouts/slow 2xx can cause duplicate punches/leave/OT events; no deduplication key or uniqueness constraint exists.
- **Missing tenant-scoped uniqueness for punches within a deduplication window**: concurrent or retried webhooks can create multiple clock-in/out records for same tenant+user+date+event.
- **No structured request tracing / idempotency key propagation**: frontend retries and LINE retries cannot be correlated; no `X-Idempotency-Key` or `X-Request-ID` handling.
- **Race between optimistic UI and server replays**: frontend may optimistically apply state then receive duplicate server writes, causing inconsistent UI state.
- **Operational observability gap**: no logging of deduplication decisions, making debugging of duplicates difficult.

## 2. Proposed change

- **File**: `workio/server/src/routes/webhook/line.ts` (or equivalent webhook handler)
- **Scope**: add server-side idempotency layer for LINE webhook events (clock-in/out, leave/OT) with tenant-scoped uniqueness and short TTL deduplication store (Redis or DB). Add request tracing headers and structured logs.

```ts
// workio/server/src/routes/webhook/line.ts
import { Router, Request, Response } from 'express';
import { v4 as uuidv4 } from 'uuid';
import Redis from 'ioredis';
import { logger } from '../../lib/logger';

const router = Router();
const redis = new Redis(process.env.REDIS_URL || 'redis://localhost:6379');

// Deduplication TTL (minutes) - covers LINE retry window
const IDEMPOTENCY_TTL_MINUTES = 10;

function buildIdempotencyKey(tenantId: string, body: any): string {
  // Deterministic key from tenant + event type + user + timestamp (coarse) + LINE event id if present
  const event = body.events?.[0];
  const type = event?.type || 'unknown';
  const userId = event?.source?.userId || 'unknown';
  const ts = event?.timestamp ? new Date(event.timestamp).toISOString().slice(0, 16) : 'nots';
  const lineEventId = event?.webhookEventId || uuidv4();
  return `idem:workio:${tenantId}:${type}:${userId}:${ts}:${lineEventId}`;
}

async function withIdempotency(
  req: Request,
  res: Response,
  handler: (req: Request) => Promise<void>
) {
  const tenantId = (req as any).tenantId; // set by tenant middleware
  if (!tenantId) {
    logger.error({ msg: 'missing_tenant_in_webhook' });
    return res.status(400).json({ error: 'missing_tenant' });
  }

  const idemKey = buildIdempotencyKey(tenantId, req.body);
  const requestId = req.headers['x-request-id'] || uuidv4();

  // Try to acquire lock
  const existing = await redis.get(idemKey);
  if (existing) {
    logger.info({
      msg: 'duplicate_webhook_ignored',
      requestId,
      idemKey,
      tenantId,
      originalResponse: existing
    });
    return res.status(200).json(JSON.parse(existing));
  }

  // Store placeholder to prevent concurrent retries from racing
  await redis.set(idemKey, JSON.stringify({ status: 'processing' }), 'EX', IDEMPOTENCY_TTL_MINUTES * 60);

  try {
    // Capture response to cache
    const chunks: any[] = [];
    const originalSend = res.json;
    res.json = function (body) {
      chunks.push(body);
      // Cache successful response
      redis.set(idemKey, JSON.stringify({ status: 'ok', body }), 'EX', IDEMPOTENCY_TTL_MINUTES * 60);
      return originalSend.call(this, body);
    } as any;

    await handler(req);
  } catch (err) {
    // On error, remove placeholder so retry can attempt again
    await redis.del(idemKey);
    logger.error({ msg: 'webhook_handler_error', requestId, idemKey, error: err });
    throw err;
  }
}

router.post('/line', async (req, Res) => {
  await withIdempotency(req, Res, async (req) => {
    // existing LINE event handling logic here
    const events = req.body.events || [];
    for (const event of events) {
      // process clock-in/out, leave, OT...
      logger.info({ msg: 'processing_line_event', eventType: event.type, userId: event.source?.userId });
      // ... your business logic ...
    }
  });
});

export default router;
```

## 4. Verification

1. **Local test with duplicate POSTs**:
   - Start server and send identical LINE webhook payload twice within 10 minutes.
   - Expect: second request returns 200 with cached response and logs `duplicate_webhook_ignored`.

2. **Redis inspection**:
   - After first request, `redis.get <idemKey>` should show cached response.
   - After TTL expiry, key should be absent and retry should reprocess.

3. **Database uniqueness**:
   - Attempt to insert two punches with same tenant+user+date+type within short window via direct DB insert; unique index should reject duplicates.

4. **Structured logs**:
   - Check logs contain `requestId`, `idemKey`, `tenantId`, and `duplicate_webhook_ignored` or `processing_line_event`.

5. **End-to-end via LINE simulator**:
   - Configure LINE webhook to your dev endpoint.
   - Send same event multiple times (simulate retries) and verify only one punch record created and UI remains consistent.
