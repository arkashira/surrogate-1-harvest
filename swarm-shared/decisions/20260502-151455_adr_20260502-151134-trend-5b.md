## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: single deterministic workflow (research->signal->risk->execution) with Postgres-backed event sourcing, Redis Streams for queueing, one broker connector (sandbox mode), immutable audit trail, replay endpoint, and basic compliance pass/fail rules — deployable via CF Workers + Supabase + Upstash Redis free tiers.

## Open questions
- Deterministic replay: should Redis Streams + Postgres event-sourcing be the single source of truth, or do we snapshot workflow state to Postgres for cheaper long-tail replay?
- Idempotency and exactly-once semantics across connectors: which brokers/OMS support idempotent keys and how do we enforce dedupe at the orchestration layer?
- Compliance rule engine: declarative JSON rules vs. small DSL vs. pluggable Python predicates — what is the governance/latency trade-off?
- Latency SLOs: what is the p99 target for research->execution and which path (sync vs. queued) is acceptable for fee-generating workflows?
- Secret and credential management for connectors: use Supabase Vault-like approach, CF Workers KV/secrets, or per-tenant encrypted config in Postgres?
- Rate limits and circuit breaking per connector: centralized token bucket in Redis or per-worker local with gossip?
