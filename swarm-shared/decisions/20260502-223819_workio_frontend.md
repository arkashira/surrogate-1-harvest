**Diagnosis:**

1. The current implementation lacks an optimistic UI for clock in/out, resulting in perceived latency and poor user experience on flaky mobile networks.
2. There is no local-first draft state for leave/OT forms, causing unsaved form data to be lost on navigation or offline.
3. Clock state can become stale after background/foreground or network flaps due to the lack of SWR/revalidation or visibility-triggered refetch.
4. The absence of skeleton/loading affordance on action buttons leads to duplicate punches.
5. Form state is not persisted during submission, requiring users to retype everything if the request fails.

**Proposed Change:**

To address these issues, I propose the following changes:

1. **Optimistic Clock In/Out:**
	* Implement an optimistic UI for clock in/out using a `useOptimisticPunch()` hook.
	* This hook will manage the optimistic state, rollback on error, and local queue for offline support.
2. **Local-First Draft State:**
	* Introduce a `useDraftForm()` hook for leave/OT forms to persist form data in `localStorage` with debounced writes and TTL.
	* This hook will ensure that unsaved form data is recovered on navigation or offline.
3. **Clock State Revalidation:**
	* Implement `useVisibilityRevalidate()` to refetch punch data when the component becomes visible, ensuring that the clock state is up-to-date.
4. **Skeleton/Loading Affordance:**
	* Add a skeleton/loading affordance on action buttons to prevent duplicate punches.
5. **Form State Persistence:**
	* Use the `useDraftForm()` hook to persist form state during submission, allowing users to recover their input if the request fails.

**Implementation:**

The implementation will involve creating the `useOptimisticPunch()`, `useDraftForm()`, and `useVisibilityRevalidate()` hooks. These hooks will be used in the `ClockButton`, `LeaveForm`, and `OvertimeForm` components to address the identified issues.
