# workio / design

## Final synthesized solution

**Core diagnosis (merged, de-duplicated)**
- Non-idempotent LINE webhook ingestion allows retries and replays to create duplicate punches.
- Non-atomic upsert (read-then-insert/update) permits race duplicates under concurrent deliveries.
- No DB-level uniqueness or optimistic concurrency allows silent overwrites and duplicate active punches.
- Missing stable idempotency key prevents safe client/webhook retries.

**Chosen approach (correctness + actionability)**
- Enforce idempotency and business rules at the DB level with constraints and atomic upserts.
- Use a single, stable idempotency key (`line_message_id`) for LINE webhook replays.
- Use one atomic upsert path that handles both idempotent replay and active-punch uniqueness.
- Add optimistic concurrency (`version`) to protect updates from races and overwrites.
- Keep changes minimal and focused: one table, one upsert function, one webhook handler.

## 1. Schema change (`workio/server/src/db/schema.sql`)

```sql
-- Idempotency + optimistic concurrency columns
ALTER TABLE punches
  ADD COLUMN IF NOT EXISTS line_message_id VARCHAR(255),
  ADD COLUMN IF NOT EXISTS version INTEGER NOT NULL DEFAULT 1;

-- Idempotency: one punch per LINE message (global, not per-day)
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_line_msg
  ON punches (tenant_id, line_message_id)
  WHERE line_message_id IS NOT NULL;

-- Business rule: at most one active punch per tenant/employee/day
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_active_day
  ON punches (tenant_id, employee_id, DATE(clock_in_ts))
  WHERE clock_out_ts IS NULL;
```

## 2. Service: atomic upsert (`workio/server/src/services/punchService.ts`)

```ts
import { PoolClient } from 'pg';
import { getClient } from '../db';

export interface PunchUpsertInput {
  tenantId: string;
  employeeId: string;
  clockInTs: Date;
  clockOutTs?: Date | null;
  latitude?: number;
  longitude?: number;
  lineMessageId?: string;
  expectedVersion?: number; // optional optimistic concurrency guard
}

/**
 * Atomic upsert for punches:
 * - Idempotent by line_message_id when provided.
 * - Ensures at most one active punch per tenant/employee/day.
 * - Increments version on updates for optimistic concurrency.
 */
export async function upsertPunch(input: PunchUpsertInput, client?: PoolClient): Promise<{ id: string; version: number }> {
  const pg = client ?? await getClient();
  try {
    const result = await pg.query(
      `INSERT INTO punches (tenant_id, employee_id, clock_in_ts, clock_out_ts, latitude, longitude, line_message_id, version)
       VALUES ($1, $2, $3, $4, $5, $6, $7, 1)
       ON CONFLICT (tenant_id, line_message_id) DO UPDATE
         SET clock_out_ts = EXCLUDED.clock_out_ts,
             latitude = EXCLUDED.latitude,
             longitude = EXCLUDED.longitude,
             version = punches.version + 1
       ON CONFLICT (tenant_id, employee_id, DATE(clock_in_ts))
         WHERE clock_out_ts IS NULL
         DO UPDATE
           SET clock_out_ts = EXCLUDED.clock_out_ts,
               latitude = EXCLUDED.latitude,
               longitude = EXCLUDED.longitude,
               version = punches.version + 1
       RETURNING id, version`,
      [
        input.tenantId,
        input.employeeId,
        input.clockInTs,
        input.clockOutTs ?? null,
        input.latitude ?? null,
        input.longitude ?? null,
        input.lineMessageId ?? null,
      ]
    );

    return { id: result.rows[0].id, version: result.rows[0].version };
  } finally {
    if (!client) pg.release?.();
  }
}

/**
 * Optional helper for explicit optimistic-concurrency updates (e.g., edits).
 * Returns null if version mismatch (concurrent update).
 */
export async function updatePunchWithVersion(
  input: { id: string; tenantId: string; expectedVersion: number; clockOutTs?: Date | null; latitude?: number; longitude?: number },
  client?: PoolClient
): Promise<{ id: string; version: number } | null> {
  const pg = client ?? await getClient();
  try {
    const result = await pg.query(
      `UPDATE punches
       SET clock_out_ts = $1, latitude = $2, longitude = $3, version = version + 1
       WHERE id = $4 AND tenant_id = $5 AND version = $6
       RETURNING id, version`,
      [input.clockOutTs ?? null, input.latitude ?? null, input.longitude ?? null, input.id, input.tenantId, input.expectedVersion]
    );

    if (result.rowCount === 0) return null;
    return { id: result.rows[0].id, version: result.rows[0].version };
  } finally {
    if (!client) pg.release?.();
  }
}
```

## 3. Webhook handler (`workio/server/src/routes/webhook/line.ts`)

```ts
import { Router } from 'express';
import { upsertPunch } from '../../services/punchService';
import { verifyLineSignature } from '../../utils/line';
import { findEmployeeByLineUserId } from '../../services/employeeService';
import { getActivePunch } from '../../services/punchService';

const router = Router();

router.post('/line', verifyLineSignature, async (req, res) => {
  const events = req.body.events || [];

  for (const ev of events) {
    if (ev.type !== 'message' || ev.message?.type !== 'text') continue;

    const lineUserId = ev.source.userId;
    const employee = await findEmployeeByLineUserId(lineUserId);
    if (!employee) continue;

    const now = new Date();
    const lineMessageId = ev.message.id; // stable idempotency key

    const active = await getActivePunch(employee.tenant_id, employee.id);
    await upsertPunch({
      tenantId: employee.tenant_id,
      employeeId: employee.id,
      clockInTs: active ? active.clock_in_ts : now,
      clockOutTs: active ? now : null,
      latitude: ev.source?.metadata?.latitude ?? undefined,
      longitude: ev.source?.metadata?.longitude ?? undefined,
      lineMessageId,
    });
  }

  res.json({ ok: true });
});

export default router;
```

## 4. Verification (merged + concrete)

1. **Idempotency**: replay the same LINE webhook (same `message.id`) twice; confirm one row and `version` increments only once.
2. **Concurrency**: fire parallel deliveries for same tenant/employee/day; confirm one active punch and no duplicate rows (enforced by `idx_punches_active_day`).
3. **Constraint enforcement**: attempt direct insert with duplicate `line_message_id`; must be rejected by `idx_punches_line_msg`.
4. **Active-punch rule**: attempt to clock-in when an active punch exists for same tenant/employee/day; must update existing row (clock-out) rather than create a second active punch.
5. **Optimistic concurrency**: attempt edit with stale `expectedVersion`; update must be rejected (returns null from `updatePunchWithVersion`).
6. **E2E smoke**: send LINE text from registered user; verify punch created with correct `line_message_id` and that clock-out updates the same row.
