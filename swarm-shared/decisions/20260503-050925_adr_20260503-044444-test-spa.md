## Deployment

## v0.1 scope
v0.1 in 2 weeks: CLI-first MVP (FastAPI optional) that connects to one AWS account + one GCP project, lists all Lambda/Cloud Function schedules, detects duplicates (identical cron+target), flags orphaned entries (missing trigger/rule), exports a CSV/JSON report and a simple summary endpoint; stores results in Supabase Postgres with no queue and no frontend.

## Open questions
- Which identity model will we use for cross-account/platform access (assume IAM roles per customer or long-lived keys)?
- How will we detect 'drifted' schedules (timezone drift, expression drift, or invocation-time drift) and what tolerance is acceptable?
- What retention window do we need for schedule history vs. current state to balance cost and audit depth?
- Will customers require real-time webhook alerts or is periodic report export sufficient for v0.1?
- How do we define 'orphaned' for Cloud Functions (no trigger vs. trigger removed) and Lambda (no EventBridge rule vs. disabled rule)?
