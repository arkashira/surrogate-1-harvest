## Deployment
cf workers

## v0.1 scope
v0.1 ships in 2 weeks: a single-file Rust CLI daemon that creates a local SOCKS5 listener, hides its port from local port scans via userspace socket cloaking, tunnels traffic through an obfuscated TLS-like stream (shadowsocks 2022 AEAD) to a public bridge, spawns a TUN interface managed by boringtun with randomized parameters and packet padding to evade DPI, and enforces a process-level kill-switch that blocks non-VPN traffic if the stealth path fails; local SQLite stores session logs and minimal config, no server components required, and basic client-side WAU-ZB telemetry (opt-in) is emitted via CF Workers endpoint.

## Open questions
- Which DPI signatures are highest risk in target regions (e.g., TTL anomalies, MSS patterns, TLS fingerprinting) and which obfuscation primitives neutralize them?
- Can we rely on CF Workers for lightweight config/geo-fallback endpoints without introducing central points of failure or logging?
- What is the acceptable trade-off between stealth and throughput (e.g., fragmentation vs latency)?
- How will we implement a reliable kill-switch across desktop OSes (Windows/Linux/macOS) without admin/root in all cases?
- What telemetry schema and consent flow ensures WAU-ZB measurement without deanonymizing users?
