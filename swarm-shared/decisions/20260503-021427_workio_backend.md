# workio / backend

## Final Synthesized Solution

Below is the single, consolidated implementation that merges the strongest insights from both proposals, resolves contradictions in favor of correctness and concrete actionability, and provides a production-ready path to eliminate duplicate LINE attendance punches.

---

### 1) Diagnosis (merged)

- LINE webhook retries cause duplicate punches because ingestion is **not idempotent** and there is **no dedupe key or idempotency token** enforced.
- The clock-in/out path uses a **non-atomic find-then-write** pattern (`findOne` → conditional `insert`/`update`), permitting race conditions under concurrency or retries.
- There is **no tenant-scoped uniqueness constraint** to prevent cross-tenant or duplicate same-day punches at the database level.
- Missing **transactional boundary** when creating/updating attendance and associated shift/OT state, risking partial writes.
- No lightweight **idempotency table** to absorb retries cheaply before hitting business logic.
- Punch state transitions (clock-in → break → clock-out) are not enforced atomically, risking invalid sequences or double clock-ins.
- No server-side de-duplication window for LINE event IDs; repeated events within a short window can be accepted as new punches.

---

### 2) Proposed change (merged)

- Add **idempotent, atomic punch ingestion** with transactional upsert and tenant-scoped uniqueness.
- Add a **DB migration**:
  - `idempotency_keys` table with tenant-scoped uniqueness and TTL.
  - Add `uq_attendance_tenant_employee_day` unique constraint on `attendance`.
- Introduce **state-machine–aware upsert logic**:
  - Enforce valid transitions (e.g., prevent double clock-in without an intervening clock-out).
  - Allow break/resume updates within the same day without violating uniqueness.
- Add **short in-memory de-duplication cache** keyed by `(tenantId, lineEventId)` for fast, same-second retry suppression.
- Update route to call the idempotent service method and return **200 for duplicates** and **409 for conflicts**.
- Provide **migration + cleanup job** and operational verification steps.

```sql
-- Idempotency keys for LINE webhook retries (expires after 48h)
CREATE TABLE IF NOT EXISTS idempotency_keys (
  id          SERIAL PRIMARY KEY,
  tenant_id   INTEGER NOT NULL,
  key         TEXT    NOT NULL,
  payload_md5 TEXT    NOT NULL,
  created_at  TIMESTAMPTZ DEFAULT NOW(),
  expires_at  TIMESTAMPTZ NOT NULL,
  UNIQUE (tenant_id, key)
);

-- Index for fast expiry cleanup
CREATE INDEX IF NOT EXISTS idx_idempotency_expires ON idempotency_keys (expires_at);

-- Ensure one punch row per employee per tenant per day; clock_type can be updated atomically.
-- This supports clock-in → break → clock-out transitions via state-aware upsert.
ALTER TABLE attendance
  ADD CONSTRAINT uq_attendance_tenant_employee_day
  UNIQUE (tenant_id, employee_id, punch_date);
```

// Key: `${tenantId}:${lineEventId}`; Value: timestamp.
const recentDedupeCache = new Map<string, number>();
const DEDUPE_TTL_MS = 60_000; // 60s server-side dedupe window

function isRecentlySeen(tenantId: number, lineEventId: string): boolean {
  const key = `${tenantId}:${lineEventId}`;
  const now = Date.now();
  const seenAt = recentDedupeCache.get(key);
  if (seenAt && now - seenAt < DEDUPE_TTL_MS) return true;
  recentDedupeCache.set(key, now);
  // Simple expiry sweep (best-effort)
  if (recentDedupeCache.size > 10_000) {
    for (const [k, v] of recentDedupeCache.entries()) {
      if (now - v > DEDUPE_TTL_MS) recentDedupeCache.delete(k);
    }
  }
  return false;
}
