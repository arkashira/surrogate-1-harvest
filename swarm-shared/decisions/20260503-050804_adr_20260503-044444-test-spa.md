## Deployment
cf workers (free-tier Cloudflare Workers + D1 for lightweight metadata; primary Postgres remains Supabase free tier)

## v0.1 scope
v0.1 (2 weeks): CLI-first MVP that discovers Lambda and Cloud Functions schedules via customer-supplied credentials, persists normalized schedule metadata to Supabase Postgres, computes cron drift/duplicates via croniter, exposes a FastAPI backend (hosted on free-tier Cloudflare Workers + D1 or local dev), and outputs a JSON report of stale schedules to retire; no UI, no background jobs, no alerting.

## Open questions
- Which identity provider will customers use for read-only AWS/GCP access (IAM role ARN, service account key, or OIDC federation)?
- How frequently should discovery jobs run (hourly vs daily) to balance freshness and rate limits?
- What retention window for historical audit events is acceptable under the free-tier Postgres limits?
- Will we require customer-provided tagging conventions to reliably attribute ownership across platforms?
- How should we handle private/internal endpoints or VPC-restricted functions (assume customer-provided credentials with network reach)?
