## Deployment
cf workers (free-tier Cloudflare Workers + D1 (Postgres-ish) for API; Supabase for durable Postgres; optional HF space for lightweight UI prototype)

## v0.1 scope
v0.1 (2 weeks): CLI-first MVP that discovers Lambda and Cloud Functions cron/rate schedules via read-only credentials, persists them to Supabase Postgres, computes duplicates (by hash) and orphaned schedules (resource missing), exposes a minimal JSON API via Cloudflare Workers, and outputs a simple report of stale schedules to retire (no UI, no auth, no remediations).

## Open questions
- Which IAM scopes are minimally required to list/describe Lambda schedules and Cloud Functions across customer accounts (assume cross-account read-only role)?
- How will customers provide credentials (assume OIDC federation or service-account keys) and rotate them safely?
- What is the acceptable discovery cadence (hourly/daily) given rate limits and free-tier constraints?
- How do we define 'drifted' (tags/labels/code hash mismatch vs. cron change) and what heuristics will we use?
- Will orphaned detection rely on existence of the target function/resource at discovery time or require historical invocations?
