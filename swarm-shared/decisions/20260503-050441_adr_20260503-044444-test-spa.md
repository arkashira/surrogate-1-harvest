## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: CLI-first MVP (FastAPI service deployable to CF Workers + local dev) that connects to one AWS account and one GCP project via service-account/access-key, lists all schedules into Supabase Postgres, detects duplicates (same cron+target), marks orphans (target not found), and exports a CSV report plus simple counts endpoint; no UI, no scheduler, no auto-remediation.

## Open questions
- How will customers grant read-only cross-cloud credentials (least-privilege IAM roles vs. short-lived OIDC) without exposing secrets?
- What retention and compute model will we use for historical invocation telemetry (cold storage vs. sampling) to keep free-tier viable?
- Should orphan detection require target existence checks (extra API calls) or rely on platform metadata only?
- How do we define and compute 'drift' when customers customize runtime/timeouts outside our metadata model?
- Which SLAs/expectations for scan frequency (hourly vs. daily) fit the free-tier budget and rate limits?
