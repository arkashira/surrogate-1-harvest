# workio / quality

## Final synthesized implementation (correct + actionable)

**Core principle**: enforce idempotency at the API boundary and enforce uniqueness at the database layer; keep the frontend fast, safe, and user-friendly.

---

### 1) Database — enforce exactly-one punch per user per date per type

```sql
-- workio/server/src/db/schema.sql
-- Prevent duplicate punches for same user/date/type
ALTER TABLE punches
  ADD CONSTRAINT unique_user_date_type UNIQUE (user_id, date, type);

-- Idempotency table (lightweight, TTL-managed by app/janitor)
CREATE TABLE IF NOT EXISTS punch_idempotency (
  idempotency_key  TEXT      NOT NULL,
  user_id          INTEGER   NOT NULL,
  response_status  INTEGER   NOT NULL,
  response_body    JSONB     NOT NULL,
  created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (idempotency_key, user_id)
);

CREATE INDEX idx_punch_idempotency_lookup
  ON punch_idempotency (idempotency_key, user_id);
```

- `unique_user_date_type` is the source of truth to block duplicates from any origin (UI double-tap, retries, LINE webhook replays).
- `punch_idempotency` stores prior successful responses so retries with the same `Idempotency-Key` return the same result without re-executing logic.

### 2) Backend — idempotent upsert with race-safe handling

```ts
// workio/server/src/routes/punch.ts
import { Router } from 'express';
import { pool } from '../db';

const router = Router();

router.post('/', async (req, res) => {
  const idempotencyKey = req.header('Idempotency-Key');
  const userId = req.user?.id;
  if (!userId) return res.status(401).json({ error: 'Unauthorized' });

  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // Idempotency check: return prior response if key exists
    if (idempotencyKey) {
      const { rows } = await client.query(
        `SELECT response_status, response_body
         FROM punch_idempotency
         WHERE idempotency_key = $1 AND user_id = $2`,
        [idempotencyKey, userId]
      );
      if (rows[0]) {
        await client.query('ROLLBACK');
        return res.status(rows[0].response_status).json(rows[0].response_body);
      }
    }

    // Determine next punch type for today
    const today = new Date().toISOString().split('T')[0];
    const { rows: existing } = await client.query(
      `SELECT type FROM punches
       WHERE user_id = $1 AND date = $2
       ORDER BY created_at DESC LIMIT 1`,
      [userId, today]
    );
    const last = existing[0];
    const nextType = !last || last.type === 'clock_out' ? 'clock_in' : 'clock_out';

    // Insert punch (unique constraint prevents duplicates)
    const { rows: inserted } = await client.query(
      `INSERT INTO punches (user_id, date, type, created_at)
       VALUES ($1, $2, $3, NOW())
       RETURNING id, user_id, date, type, created_at`,
      [userId, today, nextType]
    );

    // Persist idempotency record if key provided
    if (idempotencyKey) {
      await client.query(
        `INSERT INTO punch_idempotency (idempotency_key, user_id, response_status, response_body, created_at)
         VALUES ($1, $2, $3, $4, NOW())`,
        [idempotencyKey, userId, 200, inserted[0]]
      );
    }

    await client.query('COMMIT');
    return res.json(inserted[0]);
  } catch (err: any) {
    await client.query('ROLLBACK');

    // Unique violation: race or duplicate -> return existing punch (idempotent)
    if (err.code === '23505') {
      const today = new Date().toISOString().split('T')[0];
      const { rows } = await client.query(
        `SELECT id, user_id, date, type, created_at
         FROM punches
         WHERE user_id = $1 AND date = $2
         ORDER BY created_at DESC LIMIT 1`,
        [userId, today]
      );
      return res.json(rows[0] || { error: 'duplicate' });
    }

    console.error(err);
    return res.status(500).json({ error: 'Internal server error' });
  } finally {
    client.release();
  }
});

export default router;
```

Key choices:
- Uses a transaction to couple idempotency lookup + punch insert.
- On `23505` (unique violation), returns the existing punch instead of an error — safe for concurrent/retried requests.
- Idempotency key is scoped by `(idempotency_key, user_id)` to avoid cross-user collisions.

### 3) Frontend — optimistic UI, in-flight lock, and idempotency key

```tsx
// workio/src/components/PunchButton.tsx
import { useState, useCallback } from 'react';

export function PunchButton() {
  const [submitting, setSubmitting] = useState(false);

  const punch = useCallback(async () => {
    if (submitting) return;
    setSubmitting(true);

    const idempotencyKey = `punch-${Date.now()}-${Math.random().toString(36).slice(2)}`;

    try {
      const res = await fetch('/api/punch', {
        method: 'POST',
        headers: {
          'Content-Type': 'application/json',
          'Idempotency-Key': idempotencyKey,
        },
        body: JSON.stringify({}),
      });

      if (!res.ok) {
        throw new Error('Punch failed');
      }

      const result = await res.json();
      // optimistic success handling (e.g., show last punch time)
      console.log('Punch recorded:', result);
    } catch (err) {
      // show toast/retry UI; allow user to retry
      console.error(err);
    } finally {
      setSubmitting(false);
    }
  }, [submitting]);

  return (
    <button
      onClick={punch}
      disabled={submitting}
      className={`btn ${submitting ? 'btn-disabled' : ''}`}
    >
      {submitting ? 'กำลังบันทึก...' : 'ลงเวลา'}
    </button>
  );
}
```

Why this is sufficient:
- `disabled` + `submitting` lock prevents multiple in-flight requests from the same component instance.
- Idempotency key ensures retries (manual or network) are safe.
- No localStorage nonce required for correctness (the backend unique constraint + idempotency key already guarantee safety), but you can add session-level de-dup with a short-lived client nonce if you want extra UX polish.

### 4) Verification checklist

- **DB uniqueness**: Try inserting two punches for same user/date/type — second insert must fail (or be caught by app and return existing row).
- **Idempotency key**: Send same `Idempotency-Key` twice — second request must return same punch row and not create a new one.
- **Concurrent requests**: Fire two simultaneous POSTs (e.g., from two tabs) — only one punch row should be created; both responses should return the same punch.
- **LINE webhook replay**: Replay a webhook payload — no duplicate row should appear.
- **UI behavior**: Button shows disabled/loading while submitting; rapid clicks do not queue multiple requests.
- **Logs**: Confirm `23505` handling returns existing punch without surfacing an error to the client.

This synthesis removes contradictions, keeps the strongest protections (DB unique constraint + idempotency table + frontend lock), and is immediately implementable.
