## Deployment
cf workers

## v0.1 scope
v0.1 ships in 2 weeks: CLI tool + FastAPI backend on Supabase that discovers Lambda and Cloud Functions cron/rate schedules, persists them to Postgres, detects exact-duplicate cron hashes and orphaned schedules (platform_id missing), and exports a summary report (JSON/CSV) with stale schedule counts and recommended retirements.

## Open questions
- Which AWS/GCP accounts/projects will be in scope for v0.1 and how will credentials be provided (OIDC vs long-lived keys)?
- What permission model is required to read schedules (read-only IAM roles vs cross-account roles)?
- How frequently should the auditor run (hourly vs daily) to balance freshness and cost?
- How will duplicates/drift be defined (exact cron match vs time-window overlap) and surfaced to users?
- Will customers prefer a CLI-first workflow or a hosted UI (CF Workers + Workers KV vs Supabase edge functions)?
