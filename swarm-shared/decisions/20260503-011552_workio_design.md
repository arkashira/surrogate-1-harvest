# workio / design

## Final synthesized implementation (correct + actionable)

**Core invariant**: *at most one open punch per user per calendar day* (`clock_out_at IS NULL`).  
**Primary defense**: enforce this at the database level so retries, races, and double-taps cannot violate correctness.  
**Secondary defense**: idempotent webhook handling so duplicate LINE deliveries are safe no-ops.  
**Tertiary polish**: optimistic UI to reduce perceived latency and double-taps (valuable, but must not be required for correctness).

---

### 1. Database schema (source of truth)

Apply to `workio/server/src/db/schema.sql`:

```sql
-- Idempotency for webhook replay protection (indexed)
ALTER TABLE punches ADD COLUMN IF NOT EXISTS idempotency_key TEXT;
CREATE INDEX IF NOT EXISTS idx_punches_idempotency_key
  ON punches (idempotency_key)
  WHERE idempotency_key IS NOT NULL;

-- Enforce: at most one open punch per user per calendar day
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_one_open_per_user_per_day
  ON punches (user_id, DATE(clock_in_at))
  WHERE clock_out_at IS NULL;

-- Fast lookup of open punch by user (covers common "am I clocked in?" query)
CREATE INDEX IF NOT EXISTS idx_punches_open_by_user
  ON punches (user_id)
  WHERE clock_out_at IS NULL;
```

**Why this is correct**  
- The partial unique index is the authoritative guard: concurrent inserts/updates that would create two open punches for the same user/day will be rejected by Postgres.  
- `idempotency_key` + index enables safe `ON CONFLICT DO NOTHING` for duplicate webhook deliveries without relying on app-layer locking.  
- `idx_punches_open_by_user` keeps the common “find my open punch” query fast at scale.

### 2. Idempotent webhook handler (LINE)

File: `workio/server/src/routes/line.ts` (or equivalent webhook entrypoint).

Key rules:
- Derive a stable idempotency key from LINE event + user + action (e.g., `eventId:userId:action`).  
- Use `INSERT ... ON CONFLICT (idempotency_key) DO NOTHING` for new events.  
- For clock-out, close the existing open punch idempotently; never create a second open punch.

```ts
import { db } from '../db';

async function handlePunchEvent(
  userId: string,
  action: 'clock_in' | 'clock_out',
  lineEventId: string
) {
  const idempotencyKey = `${lineEventId}:${userId}:${action}`;

  if (action === 'clock_in') {
    // Try to create an open punch; duplicate webhook => no-op
    await db.query(
      `INSERT INTO punches (user_id, clock_in_at, idempotency_key)
       VALUES ($1, NOW(), $2)
       ON CONFLICT (idempotency_key) DO NOTHING`,
      [userId, idempotencyKey]
    );
    return;
  }

  // action === 'clock_out'
  // Idempotently close the current open punch for this user/day.
  // Use a CTE so we only create an idempotency record if we actually close something.
  await db.query(
    `WITH closed AS (
       UPDATE punches
       SET clock_out_at = NOW()
       WHERE user_id = $1
         AND clock_out_at IS NULL
       RETURNING id
     )
     INSERT INTO punches (user_id, clock_in_at, clock_out_at, idempotency_key)
     SELECT user_id, clock_in_at, NOW(), $2
     FROM punches
     WHERE user_id = $1
       AND clock_out_at IS NOT NULL
       AND id IN (SELECT id FROM closed)
     ON CONFLICT (idempotency_key) DO NOTHING`,
    [userId, idempotencyKey]
  );

  // Fallback safety: ensure no open punch remains (handles races where insert above skipped)
  await db.query(
    `UPDATE punches
     SET clock_out_at = NOW()
     WHERE user_id = $1
       AND clock_out_at IS NULL
       AND idempotency_key IS DISTINCT FROM $2`,
    [userId, idempotencyKey]
  );
}
```

Notes:
- If your codebase already has a service layer (`punch.service.ts`), move the upsert/close logic there and keep the route thin.  
- If you prefer a dedicated idempotency table, replace `idempotency_key` on `punches` with that pattern; the principle is the same.

### 3. Clock-in path: optimistic UI (frontend)

File: `workio/frontend/src/hooks/usePunch.ts`

Goal: eliminate double-tap amplification by reflecting state immediately while keeping server as source of truth.

```ts
import { useState, useCallback } from 'react';
import { clockIn, clockOut } from '../api/punch';

export function usePunch(userId: string) {
  const [isClockedIn, setIsClockedIn] = useState(false);
  const [pending, setPending] = useState(false);

  const doClockIn = useCallback(async () => {
    if (pending) return;
    setPending(true);
    setIsClockedIn(true); // optimistic
    try {
      await clockIn(userId);
    } catch {
      setIsClockedIn(false); // revert on failure
    } finally {
      setPending(false);
    }
  }, [userId, pending]);

  const doClockOut = useCallback(async () => {
    if (pending) return;
    setPending(true);
    setIsClockedIn(false); // optimistic
    try {
      await clockOut(userId);
    } catch {
      setIsClockedIn(true); // revert on failure
    } finally {
      setPending(false);
    }
  }, [userId, pending]);

  return { isClockedIn, pending, doClockIn, doClockOut };
}
```

- Disable the button while `pending` to reduce double-taps.  
- On app start or focus, reconcile state from the server (e.g., `GET /punches/open`).

### 5. Verification checklist

1. **Schema**  
   ```bash
   psql workio -c "\d punches"
   psql workio -c "SELECT indexname, indexdef FROM pg_indexes WHERE tablename = 'punches';"
   ```
   Confirm `idx_punches_one_open_per_user_per_day` exists and is `UNIQUE`.

2. **Duplicate open punch is rejected**  
   ```sql
   BEGIN;
   INSERT INTO punches (user_id, clock_in_at, clock_out_at) VALUES (1, NOW(), NULL);
   INSERT INTO punches (user_id, clock_in_at, clock_out_at) VALUES (1, NOW(), NULL);
   -- Second insert must fail with unique_violation
   ROLLBACK;
   ```

3. **Idempotency**  
   - Replay the same LINE webhook payload twice.  
   - Verify only one punch row is created/updated and no duplicate open punches appear.  
   - Check logs for duplicate suppression messages.

4. **Application behavior**  
   - Rapid clock in/out via LINE multiple times.  
   - Confirm only one open punch per user per day in DB.  
   - Confirm UI reflects state quickly and recovers gracefully on errors.

5. **Query performance**  
   ```sql
   EXPLAIN ANALYZE SELECT * FROM punches WHERE user_id = 123 AND clock_out_at IS NULL;
   ```
   Should show index scan using `idx_punches_open_by_user`.

---

**Summary of choices**  
- Database partial unique index is non-negotiable for correctness.  
- Idempotency key + `ON CONFLICT` makes webhook retries safe.  
- Optimistic UI reduces double-taps but is not required for data integrity.  
- All changes are minimal, backward-compatible, and verifiable in production.
