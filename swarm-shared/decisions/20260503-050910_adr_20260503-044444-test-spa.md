## Deployment
cf workers (free-tier friendly; deploy API + lightweight scheduler; Postgres via Supabase free tier)

## v0.1 scope
v0.1 in 2 weeks: read-only inventory and audit across one AWS account and one GCP project per connection; detect duplicates (identical cron+target hash), detect orphaned schedules (target function not found), list schedules with state; REST API + JSON export; CLI to run an audit and print summary; no UI, no scheduler automation, no write-back.

## Open questions
- Which identity model will customers use to connect AWS/GCP (IAM role assumption vs long-lived keys)?
- How will we detect 'drifted' schedules (definition: cron changed from baseline or target changed)?
- What retention window for schedule history vs live inventory to balance cost and auditability?
- Will orphaned detection require cross-account/project graph traversal or only per-connection scope?
- Do we expose a write-back capability to disable/delete schedules or remain read-only auditor?
