## Deployment
cf workers

## v0.1 scope
v0.1 ships in 2 weeks: Cloudflare Workers gateway with a single POST /v1/chat/completions endpoint that normalizes OpenAI-compatible requests, routes to one configured provider (OpenAI) with auth via CF Secrets, enforces global RPM via Redis, records tokens and p95 latency to Supabase Postgres, and exposes /metrics for basic observability. No failover, no queueing, no multi-provider selection yet.

## Open questions
- Which provider auth secrets will be stored (Supabase Vault, CF Secrets, or external KV) and rotation cadence?
- Redis persistence strategy for token counters: TTL vs durable snapshots to survive restarts without double-counting?
- Global vs per-consumer rate limits: enforce at gateway edge (KV) or defer to Postgres+queue for accuracy?
- SLA definition for p95 latency: measured gateway-to-upstream or end-to-end including queueing?
- Failover backoff policy: exponential vs jittered and how to mark providers degraded without flapping?
- Schema for provider-specific parameter normalization (e.g., temperature ranges, stop sequences) to avoid silent drops?
