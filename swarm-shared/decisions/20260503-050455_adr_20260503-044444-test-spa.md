## Deployment
cf workers (free tier) + Supabase (free tier) for API + Postgres; CLI runs locally or via GitHub Actions on schedule

## v0.1 scope
v0.1 ships a CLI + CF Worker API that authenticates via supplied credentials, lists Lambda functions + Cloud Functions and their EventBridge/Scheduler triggers, persists snapshots to Supabase Postgres, detects duplicates (same cron targeting multiple functions), drifted schedules (cron changed since last snapshot), and orphaned triggers (no backing function), then outputs a retirement plan report (no automated deletes) — all runnable locally and via GitHub Actions on a daily cron, staying within Supabase and CF free tiers.

## Open questions
- Which accounts/projects and regions should be in-scope for v0.1 to stay within free-tier rate limits?
- How will we authenticate cross-platform (AWS IAM roles vs GCP service accounts) from CF Workers vs CLI?
- What staleness threshold (e.g., no invocation in 90 days) and drift rules (cron expression change vs schedule removal) do we adopt?
- Will customers prefer read-only reports or automated retirement actions, and what approval workflow is required?
- How to handle rate limits and pagination for large Lambda/Cloud Functions estates within free-tier constraints?
