# workio / frontend

## Final consolidated solution

**File changes**
- `workio/src/features/clock/ClockButton.tsx` (create/update) — optimistic UI, per-minute idempotency key, in-flight guard, 60s local debounce, clear error/retry UX.
- `workio/src/lib/useClockOptimistic.ts` — reusable hook that encapsulates idempotency, in-flight map, local debounce, and mutation orchestration (frontend-only).
- Minor update to your clock API client to accept and forward `idempotencyKey` (and optional `ts`).

**Backend requirement (non-negotiable for correctness)**
- Accept and enforce `idempotencyKey` (or reuse LINE `webhookEventId`) with a unique constraint / idempotency table so duplicate requests with the same key are rejected. Without this, frontend guards reduce but do not eliminate duplicates under race/retry scenarios.

### 1) Hook (reusable)

`src/lib/useClockOptimistic.ts`

```ts
import { useCallback, useRef, useState } from 'react';

const PENDING_TTL = 60_000; // 60s local debounce for same-minute actions

type Status = 'idle' | 'pending' | 'ok' | 'error';

export function useClockOptimistic() {
  const [status, setStatus] = useState<Status>('idle');
  const inFlightKeyRef = useRef<string | null>(null);
  const lastActionRef = useRef<number>(0);

  const genIdempotencyKey = useCallback((userId: string | undefined) => {
    const now = new Date();
    const dateKey = now.toISOString().slice(0, 10); // YYYY-MM-DD
    const minuteKey = now.getHours() * 60 + now.getMinutes();
    return `${dateKey}#${minuteKey}#${userId ?? 'unknown'}`;
  }, []);

  const perform = useCallback(
    async (
      action: 'in' | 'out',
      exec: (payload: { idempotencyKey: string }) => Promise<unknown>,
      userId: string | undefined
    ) => {
      const key = genIdempotencyKey(userId);

      // Local same-minute + in-flight guard
      const now = Date.now();
      if (inFlightKeyRef.current) {
        return;
      }
      if (now - lastActionRef.current < PENDING_TTL && inFlightKeyRef.current === key) {
        return;
      }

      inFlightKeyRef.current = key;
      setStatus('pending');
      lastActionRef.current = now;

      try {
        await exec({ idempotencyKey: key });
        setStatus('ok');
        // Keep key to block same-minute retries; allow new actions next minute
      } catch (err) {
        setStatus('error');
        inFlightKeyRef.current = null;
        setTimeout(() => setStatus('idle'), 3_000);
      }
    },
    [genIdempotencyKey]
  );

  const reset = useCallback(() => {
    setStatus('idle');
  }, []);

  return {
    optimisticStatus: status,
    perform,
    reset,
    canSubmit: status !== 'pending',
  };
}
```

### 2) Clock button component

`src/features/clock/ClockButton.tsx`

```tsx
import { useCallback } from 'react';
import { useClock } from './ClockProvider'; // exposes clockIn/clockOut and status
import { useClockOptimistic } from '../lib/useClockOptimistic';
import { Check, Clock, AlertCircle } from 'lucide-react';

export function ClockButton() {
  const { status: userStatus, clockIn, clockOut } = useClock();
  const { optimisticStatus, perform, canSubmit } = useClockOptimistic();

  const isWorking = userStatus?.state === 'checked_in';

  const handleAction = useCallback(async () => {
    const action = isWorking ? 'out' : 'in';
    const exec = isWorking ? clockOut : clockIn;
    await perform(action, exec, userStatus?.userId);
  }, [isWorking, clockIn, clockOut, perform, userStatus?.userId]);

  return (
    <div className="flex flex-col items-center gap-3">
      <button
        onClick={handleAction}
        disabled={!canSubmit}
        aria-busy={optimisticStatus === 'pending'}
        className={`
          relative flex h-20 w-20 items-center justify-center rounded-full font-semibold transition-all
          ${
            optimisticStatus === 'pending'
              ? 'cursor-wait bg-gray-200 text-gray-500'
              : optimisticStatus === 'ok'
              ? 'bg-green-500 text-white shadow-green-200'
              : optimisticStatus === 'error'
              ? 'bg-red-500 text-white'
              : isWorking
              ? 'bg-red-500 text-white hover:bg-red-600 shadow-red-200'
              : 'bg-blue-500 text-white hover:bg-blue-600 shadow-blue-200'
          }
          ${!canSubmit ? 'ring-2 ring-blue-300' : ''}
        `}
      >
        {optimisticStatus === 'pending' && (
          <span className="absolute inset-0 animate-spin rounded-full border-2 border-white/30" />
        )}

        <span className="relative flex items-center gap-1.5 text-sm">
          {optimisticStatus === 'ok' && <Check className="h-5 w-5" />}
          {optimisticStatus === 'error' && <AlertCircle className="h-5 w-5" />}
          {optimisticStatus === 'pending' ? (
            <Clock className="h-5 w-5 animate-pulse" />
          ) : isWorking ? (
            'ออก'
          ) : (
            'เข้า'
          )}
        </span>
      </button>

      <p className="text-xs text-gray-500">
        {optimisticStatus === 'pending' && 'กำลังส่ง...'}
        {optimisticStatus === 'ok' && 'บันทึกแล้ว'}
        {optimisticStatus === 'error' && 'เกิดข้อผิดพลาด ลองใหม่'}
        {optimisticStatus === 'idle' && (isWorking ? 'กดเพื่อออกงาน' : 'กดเพื่อเข้างาน')}
      </p>
    </div>
  );
}
```

### 3) Clock API client (minimal change)

Update your clock client to forward the idempotency key (and optional timestamp) so backend can enforce deduplication.

// Example: src/features/clock/clockApi.ts
export async function clockIn(payload: { idempotencyKey: string }) {
  return fetch('/api/clock/in', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ ...payload, ts: Date.now() }),
  });
}

export async function clockOut(payload: { idempotencyKey: string }) {
  return fetch('/api/clock/out', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },

  });
}

### 4) Backend enforcement (required for correctness)

- Accept `idempotencyKey` (and optionally `webhookEventId` from LINE) on clock endpoints.
- Store a unique constraint/index on `idempotencyKey` (or `webhookEventId`) in the clock events table or in a dedicated idempotency table.
- On insert, reject duplicates (HTTP 409 or 200 with no side effects) so retries and duplicate client taps cannot create extra records.
- Ensure daily totals are computed from the canonical event records so duplicates cannot inflate them.

---

### 5) Verification checklist

- Rapid taps (5–10x): Network tab shows only one request per minute per user. Button shows “กำลังส่ง...” → “บันทึกแล้ว” without duplicate state flips.
- Slow 3G: Optimistic state shows immediately; button disabled while pending.
- Error path: Force offline or server error → button shows error after timeout and allows retry
