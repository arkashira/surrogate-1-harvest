## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: CLI-first scanner (Python) that reads AWS Lambda and GCP Cloud Functions schedules for a single account/region, persists normalized schedules to Supabase Postgres, detects exact cron duplicates and orphaned schedules (present in inventory but not in live cloud), and exports a simple JSON report + summary counts; no frontend, no scheduler, no auth beyond service-account keys supplied by user.

## Open questions
- Which identity model will customers use to grant read-only cross-account access (AWS OIDC/GCP SA keys) without storing long-lived secrets?
- Do we normalize rate expressions (e.g., rate(5 minutes)) into cron for dedupe and drift logic?
- What retention window for schedule_run records balances auditability vs. free-tier Postgres limits?
- Will orphan detection require cloud event logs (CloudTrail/Cloud Audit Logs) or rely only on current inventory diffs?
- Do we expose a webhook or CF Worker binding for periodic scans, or keep it CLI-driven initially?
