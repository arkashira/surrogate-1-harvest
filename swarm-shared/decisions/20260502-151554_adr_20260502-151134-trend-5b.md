## v0.1 scope
v0.1 in 2 weeks: deterministic signal-to-execution pipeline (research→signal→risk guard→execution) with Redis Streams event journal, Postgres audit trail, mock broker adapter, one sample strategy, replay by run-id, and perf harness targeting median latency <500ms on free-tier infra (CF Workers + Supabase + Redis).

## Open questions
- Will brokers be accessed via FIX, REST, or WebSocket — and which brokers first?
- What is the deterministic clock source (NTD, exchange timestamps, or hybrid) for latency measurement?
- How will replay be triggered (time-range, run-id, or signal-id) and where will immutable audit logs be stored?
- What is the SLA for persistence vs. hot-path (Postgres write latency vs. Redis-only fast path)?
- Which risk rules must block execution vs. soft warnings, and can they be overridden deterministically?
- Will Cloudflare Workers run the full pipeline or act as an API gateway while workers run on HF Spaces or local runners?
