## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: native CLI (Linux/macOS) + WASM browser build that (1) exposes a local SOCKS5 listener with traffic obfuscation (shadowsocks-like stream cipher + packet length padding), (2) optionally creates a TUN interface with packet whitening and header randomization, (3) runs a local kill-switch that blocks real-IP egress if stealth path fails, (4) emits anonymized, batched telemetry to Supabase via CF Worker for WAU-ZB, and (5) pulls blocklist/obfs config from CF Workers KV — all free-tier (CF Workers, Supabase free, no external VPS).

## Open questions
- Which DPI fingerprints are highest-risk in target regions (SNI, packet lengths, timing, TUN MTU) and can we mitigate with pure client-side transforms?
- Can we distribute and auto-update stealth configs via CF Workers KV without centralized servers that become blocking targets?
- What is the acceptable tradeoff between stealth and latency when using meek/domain-front or shadowsocks-like obfs over HTTPS?
- How will we measure 'blocked' reliably on-device to compute WAU-ZB without leaking telemetry during detection events?
- What kill-switch enforcement is feasible in WASM vs native binaries to guarantee fail-closed privacy?
