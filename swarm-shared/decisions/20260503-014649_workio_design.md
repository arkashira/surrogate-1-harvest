# workio / design

**Final synthesized implementation (strongest parts, concrete & correct)**

**Scope**: Make LINE webhook punch processing **idempotent + atomic** at the database level, safe under retries and concurrency.

---

### 1) Database schema (migration)

Create a unique constraint on the business key and an idempotency table to deduplicate across retries:

```sql
-- One punch per tenant/employee/date/shiftType
ALTER TABLE punches
  ADD CONSTRAINT uniq_punch_per_shift
  UNIQUE (tenant_id, employee_id, clock_date, shift_type);

-- Idempotency ledger for LINE deliveryId (and other sources)
CREATE TABLE IF NOT EXISTS idempotency_keys (
  id         SERIAL PRIMARY KEY,
  key        TEXT NOT NULL,
  tenant_id  INTEGER NOT NULL,
  entity     TEXT NOT NULL,   -- e.g. 'punch'
  entity_id  INTEGER,         -- FK to punches.id (nullable)
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE (key, tenant_id, entity)
);
CREATE INDEX idx_idempotency_key ON idempotency_keys(key, tenant_id, entity);
```

### 2) Service layer (TypeScript) — atomic upsert + idempotency

File: `/opt/axentx/workio/server/src/services/punchService.ts`

```ts
import { PoolClient } from 'pg';
import { pool } from '../db';

export type PunchInput = {
  tenantId: number;
  employeeId: number;
  clockDate: string; // YYYY-MM-DD
  shiftType: 'in' | 'out';
  clockTime: Date;
  latitude?: number;
  longitude?: number;
  lineDeliveryId: string; // idempotency key from LINE
};

export async function createPunch(input: PunchInput) {
  const client: PoolClient = await pool.connect();
  try {
    await client.query('BEGIN');

    // 1) Idempotency fast-path (prevents duplicate work on retries)
    const idem = await client.query(
      `SELECT entity_id FROM idempotency_keys WHERE key = $1 AND tenant_id = $2 AND entity = 'punch'`,
      [input.lineDeliveryId, input.tenantId]
    );
    if (idem.rows.length > 0) {
      await client.query('ROLLBACK');
      return { created: false, existingId: idem.rows[0].entity_id };
    }

    // 2) Atomic insert protected by unique constraint
    const res = await client.query(
      `INSERT INTO punches (tenant_id, employee_id, clock_date, shift_type, clock_time, latitude, longitude)
       VALUES ($1, $2, $3, $4, $5, $6, $7)
       ON CONFLICT (tenant_id, employee_id, clock_date, shift_type) DO NOTHING
       RETURNING id`,
      [
        input.tenantId,
        input.employeeId,
        input.clockDate,
        input.shiftType,
        input.clockTime,
        input.latitude,
        input.longitude,
      ]
    );

    let punchId: number;
    if (res.rowCount === 0) {
      // Constraint blocked insert (concurrent/retried request won)
      const existing = await client.query(
        `SELECT id FROM punches
         WHERE tenant_id = $1 AND employee_id = $2 AND clock_date = $3 AND shift_type = $4`,
        [input.tenantId, input.employeeId, input.clockDate, input.shiftType]
      );
      if (existing.rows.length === 0) {
        // Should not happen, but fail safe
        await client.query('ROLLBACK');
        throw new Error('Constraint blocked insert but no row exists');
      }
      punchId = existing.rows[0].id;
    } else {
      punchId = res.rows[0].id;
    }

    // 3) Record idempotency key (makes retries safe)
    await client.query(
      `INSERT INTO idempotency_keys (key, tenant_id, entity, entity_id)
       VALUES ($1, $2, 'punch', $3)`,
      [input.lineDeliveryId, input.tenantId, punchId]
    );

    await client.query('COMMIT');
    return { created: res.rowCount > 0, punchId };
  } catch (err) {
    await client.query('ROLLBACK');
    throw err;
  } finally {
    client.release();
  }
}
```

### 3) Webhook handler (LINE)

File: `/opt/axentx/workio/server/src/controllers/webhookController.ts` (or `.js`)

```ts
import { createPunch } from '../services/punchService';

export async function lineWebhook(req, res) {
  try {
    // Validate signature if needed (LINE provides)
    const events = req.body.events || [];

    for (const ev of events) {
      if (ev.type !== 'message' && ev.type !== 'postback') continue;

      // Parse clock in/out from payload (adjust to your schema)
      const { tenantId, employeeId, action, clockTime, latitude, longitude } =
        parseLineEvent(ev); // implement this

      const clockDate = clockTime.toISOString().slice(0, 10); // YYYY-MM-DD
      const shiftType = action === 'clock_in' ? 'in' : 'out';

      await createPunch({
        tenantId,
        employeeId,
        clockDate,
        shiftType,
        clockTime,
        latitude,
        longitude,
        lineDeliveryId: ev.deliveryId, // LINE-provided idempotency key
      });
    }

    // LINE expects 200 OK even for duplicates/failures we handled
    res.status(200).send();
  } catch (err) {
    // Log and return 5xx so LINE may retry (idempotency prevents duplicates)
    console.error('LINE webhook failed', err);
    res.status(500).send();
  }
}
```

### 4) Retry & failure handling (operational)

- **Idempotency**: `lineDeliveryId` + `idempotency_keys` ensures retries never create duplicate punches.
- **Atomicity**: Single transaction + `ON CONFLICT DO NOTHING` + unique constraint prevents race duplicates.
- **Partial failures**: If downstream logic (notifications, etc.) fails after DB commit, retries are safe because idempotency key is already recorded.
- **LINE retries**: Return `2xx` for handled duplicates; return `5xx` only for unhandled errors to allow safe retry.
- **TTL for idempotency keys** (optional): Add a cron or DB job to purge keys older than your retention window (e.g., 30–90 days) if table grows large.

---

### 5) Verification checklist

1. **Duplicate deliveryId**: Send same `deliveryId` twice → second request returns existing punch, no new row.
2. **Concurrent requests**: Fire two near-simultaneous webhooks for same tenant/employee/date/shift → exactly one punch row.
3. **Constraint enforcement**: Try to insert conflicting punch via SQL/API → rejected by unique constraint.
4. **Retry after failure**: Simulate failure after DB commit but before LINE reply; retry creates no duplicate.
5. **Logs & metrics**: Monitor for `created: false` (idempotent hits) and constraint violations (should be zero).

This design combines the strongest ideas:  
- Candidate 1’s transaction + idempotency pattern,  
- Candidate 2’s `deliveryId`-based idempotency table and `ON CONFLICT` atomic upsert,  
- plus explicit schema enforcement and operational retry semantics.
