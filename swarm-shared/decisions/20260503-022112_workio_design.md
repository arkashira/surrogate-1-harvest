# workio / design

## Final Synthesized Solution

### Diagnosis (merged)
- LINE webhook retries create duplicate punches due to **missing message-level idempotency**.
- Punch upsert is **non-atomic** (`findOne` → conditional `insert`/`update`), risking races under concurrency.
- **No unique constraint** enforces one open punch per `(tenant, employee)`, allowing multiple concurrent clock-ins.
- Clock-out can be overwritten by late retries, corrupting duration calculations.
- Tenant isolation is incomplete in the write path (app-level filtering only).

### Chosen approach (correctness + actionability)
- Use a **single DB transaction** with **idempotency key table** keyed by `(tenant_id, line_message_id)`.
- Enforce **one open punch** via a **partial unique index** on `punches (tenant_id, employee_id) WHERE clock_out_at IS NULL`.
- Use **atomic upsert** for clock-in and targeted update for clock-out within the same transaction.
- Keep tenant scoping strict at the SQL level (all queries include `tenant_id`).
- Reject late/out-of-order clock-out attempts rather than silently overwriting; this avoids corrupting durations.

### 1) DB migration
File: `workio/server/src/db/migrations/20260504_punch_idempotency.sql`

```sql
BEGIN;

-- Idempotency table for LINE message-level deduplication
CREATE TABLE IF NOT EXISTS punch_idempotency_keys (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id       UUID NOT NULL,
  employee_id     UUID NOT NULL,
  line_message_id TEXT NOT NULL,
  punch_id        UUID NOT NULL,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  CONSTRAINT fk_punch_idempotency_tenant FOREIGN KEY (tenant_id) REFERENCES tenants(id) ON DELETE CASCADE,
  CONSTRAINT fk_punch_idempotency_employee FOREIGN KEY (employee_id) REFERENCES employees(id) ON DELETE CASCADE,
  CONSTRAINT fk_punch_idempotency_punch FOREIGN KEY (punch_id) REFERENCES punches(id) ON DELETE CASCADE,
  CONSTRAINT uq_tenant_line_message UNIQUE (tenant_id, line_message_id)
);

-- Ensure one open punch per employee/tenant
CREATE UNIQUE INDEX IF NOT EXISTS uq_tenant_employee_open_punch
ON punches (tenant_id, employee_id)
WHERE clock_out_at IS NULL;

COMMIT;
```

### 2) Service: atomic, idempotent punch upsert
File: `workio/server/src/services/punchService.ts`

```ts
import { pool } from '../db';

export type PunchEvent = 'clock_in' | 'clock_out';

export async function recordPunch(params: {
  tenantId: string;
  employeeId: string;
  lineMessageId: string;
  event: PunchEvent;
  latitude?: number;
  longitude?: number;
  recordedAt?: Date;
}) {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // 1) Idempotency: if already processed, return existing punch
    const dup = await client.query(
      `SELECT punch_id FROM punch_idempotency_keys
       WHERE tenant_id = $1 AND line_message_id = $2`,
      [params.tenantId, params.lineMessageId]
    );
    if (dup.rows.length > 0) {
      const existing = await client.query(
        `SELECT * FROM punches WHERE id = $1 AND tenant_id = $2`,
        [dup.rows[0].punch_id, params.tenantId]
      );
      await client.query('ROLLBACK');
      return existing.rows[0];
    }

    const now = params.recordedAt || new Date();
    const dateLocal = new Date(now.getTime() - now.getTimezoneOffset() * 60000)
      .toISOString()
      .split('T')[0];

    let result;
    if (params.event === 'clock_in') {
      // Atomic upsert: insert new open punch; if one exists, return it (no double clock-in)
      const insertRes = await client.query(
        `INSERT INTO punches (tenant_id, employee_id, date, clock_in_at, clock_in_lat, clock_in_lon)
         VALUES ($1, $2, $3, $4, $5, $6)
         ON CONFLICT (tenant_id, employee_id) WHERE clock_out_at IS NULL
         DO UPDATE SET clock_in_at = EXCLUDED.clock_in_at, clock_in_lat = EXCLUDED.clock_in_lat, clock_in_lon = EXCLUDED.clock_in_lon
         RETURNING *`,
        [params.tenantId, params.employeeId, dateLocal, now, params.latitude, params.longitude]
      );
      result = insertRes.rows[0];
    } else {
      // Clock-out: update the open punch for this tenant/employee
      const updateRes = await client.query(
        `UPDATE punches
         SET clock_out_at = $4, clock_out_lat = $5, clock_out_lon = $6, updated_at = now()
         WHERE tenant_id = $1 AND employee_id = $2 AND clock_out_at IS NULL
         RETURNING *`,
        [params.tenantId, params.employeeId, dateLocal, now, params.latitude, params.longitude]
      );
      if (updateRes.rowCount === 0) {
        await client.query('ROLLBACK');
        throw new Error('No open punch to clock out');
      }
      result = updateRes.rows[0];
    }

    // 2) Record idempotency key linked to the punch row
    await client.query(
      `INSERT INTO punch_idempotency_keys (tenant_id, employee_id, line_message_id, punch_id)
       VALUES ($1, $2, $3, $4)`,
      [params.tenantId, params.employeeId, params.lineMessageId, result.id]
    );

    await client.query('COMMIT');
    return result;
  } catch (err) {
    await client.query('ROLLBACK');
    throw err;
  } finally {
    client.release();
  }
}
```

### 4) Verification checklist
1. Apply migration:
   ```bash
   psql workio < server/src/db/migrations/20260504_punch_idempotency.sql
   ```

2. Idempotency test:
   - Replay the same LINE message id twice.
   - Assert: one punch row; second call returns the same punch; no duplicate idempotency key.

3. Concurrency test:
   - Fire concurrent `clock_in` requests for same `(tenant, employee)` with different LINE message ids.
   - Assert: only one open punch exists; partial index prevents two open punches.

4. Clock-out correctness:
   - Clock in, then clock out (same or different message id).
   - Assert: punch row has both `clock_in_at` and `clock_out_at`; no open punch remains.
   - Assert: late retry with same message id returns existing punch and does not overwrite `clock_out_at`.

5. Tenant isolation:
   - Create two tenants with overlapping employee identifiers.
   - Clock in on tenant A, then attempt clock in for same employee on tenant B.
   - Assert: tenant B gets its own open punch (tenant isolation preserved by composite keys and tenant-scoped queries).
