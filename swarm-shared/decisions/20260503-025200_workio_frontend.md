# workio / frontend

## 1. Diagnosis
- Frontend lacks **client-side idempotency guard** for clock-in/out: rapid clicks or retries can race and submit duplicate punches while server idempotency is resolving.
- No **optimistic UI + rollback** for punch actions: users see stale state until network round-trip completes, leading to double-taps and confusion.
- Missing **online/offline handling**: clock-in/out fails silently when offline; no queue/retry or clear feedback.
- Punch form does not **debounce + abort** in-flight requests: overlapping requests can corrupt local state and produce duplicates.
- No **persistent pending-punch lock** (localStorage) to survive page reloads/navigation during in-progress punch attempts.

## 2. Proposed change
File: `workio/src/features/punch/usePunch.ts` (new)  
Also touch: `workio/src/features/punch/PunchButton.tsx` and `workio/src/lib/api.ts` (add abort + idempotency key).  
Scope: implement a client-side punch manager with idempotency key, optimistic update, offline queue, and abort-on-unmount.

## 3. Implementation

### `workio/src/lib/api.ts` (add)
```ts
// Add to existing API utilities
export const punch = async (
  type: 'IN' | 'OUT',
  idempotencyKey: string,
  signal?: AbortSignal,
  location?: GeolocationCoordinates
) => {
  const res = await fetch('/api/punch', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ type, idempotencyKey, location }),
    signal,
  });
  if (!res.ok) throw new Error('Punch failed');
  return res.json();
};
```

### `workio/src/features/punch/usePunch.ts` (new)
```ts
import { useCallback, useRef } from 'react';
import { punch } from '@/lib/api';

const PENDING_PUNCH_KEY = 'workio:pendingPunch';

type PendingPunch = {
  idempotencyKey: string;
  type: 'IN' | 'OUT';
  location?: GeolocationCoordinates;
  timestamp: number;
};

export function usePunch() {
  const controllerRef = useRef<AbortController | null>(null);

  // Persist pending punch across reloads so we can reconcile
  const setPending = (p: PendingPunch | null) => {
    if (p) localStorage.setItem(PENDING_PUNCH_KEY, JSON.stringify(p));
    else localStorage.removeItem(PENDING_PUNCH_KEY);
  };

  const getPending = (): PendingPunch | null => {
    try {
      const raw = localStorage.getItem(PENDING_PUNCH_KEY);
      return raw ? JSON.parse(raw) : null;
    } catch {
      return null;
    }
  };

  const doPunch = useCallback(async (type: 'IN' | 'OUT', location?: GeolocationCoordinates) => {
    // Cancel any in-flight request
    if (controllerRef.current) controller.abort();
    const controller = new AbortController();
    controllerRef.current = controller;

    const idempotencyKey = `${type}:${Date.now()}:${Math.random().toString(36).slice(2, 9)}`;
    const pending: PendingPunch = { type, idempotencyKey, location, timestamp: Date.now() };
    setPending(pending);

    // Optimistic callback could be returned for UI to update immediately
    try {
      await punch(type, idempotencyKey, controller.signal, location);
      // Success: clear pending
      setPending(null);
      return { ok: true, idempotencyKey };
    } catch (err: any) {
      if (err.name === 'AbortError') {
        // Keep pending so user can retry explicitly if desired
        return { ok: false, aborted: true };
      }
      // Network/offline: keep pending for retry
      return { ok: false, error: err.message, pending };
    } finally {
      if (controllerRef.current === controller) controllerRef.current = null;
    }
  }, []);

  const retryPending = useCallback(async () => {
    const pending = getPending();
    if (!pending) return null;
    // reuse same idempotencyKey for server idempotency
    try {
      await punch(pending.type, pending.idempotencyKey, undefined, pending.location);
      setPending(null);
      return { ok: true };
    } catch (err: any) {
      return { ok: false, error: err.message };
    }
  }, []);

  return { doPunch, retryPending, getPending };
}
```

### `workio/src/features/punch/PunchButton.tsx` (example usage)
```tsx
import { usePunch } from './usePunch';
import { useEffect, useState } from 'react';

export function PunchButton({ type }: { type: 'IN' | 'OUT' }) {
  const { doPunch, retryPending, getPending } = usePunch();
  const [status, setStatus] = useState<'idle' | 'pending' | 'success' | 'error'>('idle');

  // On mount: if there's a pending punch, offer retry or auto-retry when online
  useEffect(() => {
    const pending = getPending();
    if (pending && pending.type === type && navigator.onLine) {
      // Auto-retry when online
      retryPending().then((r) => {
        if (r?.ok) setStatus('success');
      });
    }
  }, [type, retryPending, getPending]);

  const handleClick = async () => {
    setStatus('pending');
    // Optional: get location
    let location: GeolocationCoordinates | undefined;
    try {
      const pos = await getCurrentPositionPromise();
      location = pos.coords;
    } catch {
      // proceed without location; server will validate
    }
    const result = await doPunch(type, location);
    if (result?.ok) setStatus('success');
    else if (!result?.aborted) setStatus('error');
  };

  return (
    <button
      onClick={handleClick}
      disabled={status === 'pending'}
      className="btn btn-primary"
    >
      {status === 'pending' ? 'กำลังบันทึก...' : type === 'IN' ? 'Clock In' : 'Clock Out'}
    </button>
  );
}

function getCurrentPositionPromise(): Promise<GeolocationPosition> {
  return new Promise((resolve, reject) => {
    navigator.geolocation.getCurrentPosition(resolve, reject, {
      enableHighAccuracy: true,
      timeout: 10000,
    });
  });
}
```

## 4. Verification
- **Manual**: Click punch button rapidly — only one punch appears in list; localStorage shows pending during flight then clears on success.  
- **Offline**: Disable network, punch → `status=error` and localStorage retains pending; go online and reload/retry → punch succeeds and clears.  
- **Duplicate guard**: Inspect server logs — duplicate `idempotencyKey` values from same client should not create extra punches (server idempotency handles).  
- **Abort**: Navigate away mid-request — no unhandled rejections; pending remains for explicit retry.
