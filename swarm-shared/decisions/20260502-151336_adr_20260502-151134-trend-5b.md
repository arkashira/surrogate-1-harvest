## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: single deterministic workflow (research→signal→risk→execution) with Postgres audit trail, Redis-backed queue and cache, one broker connector (paper-trade mode), compliance guardrails (hard risk limits), replay endpoint to re-run by inputs_hash, and CI tests for contract and idempotency — deployable on CF Workers free tier + Supabase free tier.

## Open questions
- Deterministic replay: should inputs_hash include normalized clock/sequence sources or exclude volatile timestamps to guarantee idempotency?
- Latency SLA for sub-second orchestration: target p99 under 800ms end-to-end on free-tier infra — is CF Workers + Supabase + Redis sufficient or do we need edge WASM for risk checks?
- Compliance rule engine: use embedded deterministic Python rules vs portable DSL (e.g., JSON Logic) to minimize vendor lock-in?
- Connector auth model: per-customer managed tokens in Supabase vault vs CF Workers KV encrypted at rest — which satisfies auditability and rotation cadence?
- Replay storage: keep full event payloads in Postgres (row size) vs offload to HF Spaces object storage with pointer in audit_event — cost vs latency trade-off?
