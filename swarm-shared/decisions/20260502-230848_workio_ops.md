## Final consolidated solution (correct + actionable)

**Core problem**: duplicate clock-in/out punches caused by retries/races, with no storage-level enforcement or safe retry mechanism.

**Chosen approach** (resolve contradictions in favor of correctness + simplicity):
- Use **one atomic upsert** on the natural key `(employee_id, punch_type, date)` with a partial unique index that excludes soft-deleted rows.
- Add an optional `idempotency_key` column + unique index for explicit replay-safe deduplication when clients/webhooks can supply a stable key.
- Avoid client-side SELECT-then-INSERT and manual transaction choreography for the common path; rely on a single `INSERT ... ON CONFLICT DO NOTHING` and return the existing row when there’s a conflict. This removes races and is simpler to maintain.
- Keep the handler small and framework-agnostic; wrap only where necessary (e.g., when combining multiple writes).

---

### 1) Schema change (`server/src/db/schema.sql`)

```sql
-- Optional explicit idempotency key (nullable for backward compatibility)
ALTER TABLE punches ADD COLUMN IF NOT EXISTS idempotency_key TEXT;

-- Partial unique constraint on natural key (ignore soft-deleted rows)
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_unique_daily
  ON punches (employee_id, punch_type, date)
  WHERE deleted_at IS NULL;

-- Fast lookup + uniqueness for idempotency key when supplied
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_idempotency_key
  ON punches (idempotency_key)
  WHERE idempotency_key IS NOT NULL;
```

Notes:
- If `deleted_at` is not used, drop the partial clause and use a plain `UNIQUE` constraint/index.
- If you need to allow multiple punch types per day (e.g., clock_in/clock_out pairs), the natural-key index above is correct; if you need multiple punches of the same type per day, remove that index and rely on `idempotency_key` only.

### 2) Handler (framework-agnostic; adapt to Express/Line route)

```ts
import { pool } from '../db';

/**
 * Create or fetch a punch idempotently.
 * Prefer idempotency_key when available; fall back to natural-key uniqueness.
 */
export async function createPunch(payload: {
  employee_id: string | number;
  punch_type: 'clock_in' | 'clock_out';
  date: string; // ISO date (YYYY-MM-DD)
  timestamp?: Date | string;
  location?: string;
  idempotency_key?: string;
}) {
  const {
    employee_id,
    punch_type,
    date,
    timestamp,
    location,
    idempotency_key,
  } = payload;

  // 1) Try idempotency-key fast path when supplied
  if (idempotency_key) {
    const byKey = await pool.query(
      `INSERT INTO punches (employee_id, punch_type, date, timestamp, location, idempotency_key, created_at)
       VALUES ($1, $2, $3, $4, $5, $6, NOW())
       ON CONFLICT (idempotency_key) DO NOTHING
       RETURNING *`,
      [employee_id, punch_type, date, timestamp || new Date(), location, idempotency_key]
    );

    if (byKey.rows.length > 0) {
      return { punch: byKey.rows[0], _idempotent: true };
    }

    // Key existed but row not returned (DO NOTHING) — fetch existing
    const existingByKey = await pool.query(
      `SELECT * FROM punches WHERE idempotency_key = $1`,
      [idempotency_key]
    );
    if (existingByKey.rows.length > 0) {
      return { punch: existingByKey.rows[0], _idempotent: true };
    }

    // Key was supplied but collided with natural key (rare). Continue to natural upsert below.
  }

  // 2) Natural-key upsert (primary defense against duplicates)
  const upsert = await pool.query(
    `INSERT INTO punches (employee_id, punch_type, date, timestamp, location, idempotency_key, created_at)
     VALUES ($1, $2, $3, $4, $5, $6, NOW())
     ON CONFLICT (employee_id, punch_type, date)
     DO UPDATE SET
       location = COALESCE(EXCLUDED.location, punches.location),
       updated_at = NOW()
     WHERE punches.deleted_at IS NULL
     RETURNING *`,
    [employee_id, punch_type, date, timestamp || new Date(), location, idempotency_key]
  );

  // If DO UPDATE triggered (conflict), upsert returns the existing row; if insert, returns new row.
  return { punch: upsert.rows[0], _conflict: upsert.rowCount === 1 && upsert.rows[0] };
}
```

Why this is correct + actionable:
- **Atomic**: no race between check and insert; the database enforces uniqueness.
- **Idempotent**: retries with same `idempotency_key` or same natural key produce the same row.
- **Simple**: one statement for the common case; explicit key path only when supplied.
- **Safe with soft deletes**: partial index excludes `deleted_at` rows from uniqueness enforcement.

### 3) Minimal Express/Line route example

```ts
import express from 'express';
import { createPunch } from './punchService';

const router = express.Router();

router.post('/punches', async (req, res) => {
  const { employee_id, punch_type, date, timestamp, location, idempotency_key } = req.body;

  if (!employee_id || !punch_type || !date) {
    return res.status(400).json({ error: 'Missing required fields' });
  }

  try {
    const { punch, _idempotent, _conflict } = await createPunch({
      employee_id,
      punch_type,
      date,
      timestamp,
      location,
      idempotency_key,
    });

    const code = _idempotent || _conflict ? 200 : 201;
    return res.status(code).json(punch);
  } catch (err) {
    console.error(err);
    return res.status(500).json({ error: 'Failed to create punch' });
  }
});

export default router;
```

### 4) Verification checklist

- **Schema applied**: run migration and confirm indexes exist (`\d punches` in psql).
- **Duplicate insert rejected**: manually insert same `(employee_id, punch_type, date)` twice; second insert should be ignored or raise unique violation.
- **Idempotency key works**: insert with a specific `idempotency_key`, repeat with same key; no new row created.
- **Handler test**: unit test that calls `createPunch` twice with same payload/key and asserts exactly one row and same returned punch.
- **Soft-delete interaction**: if `deleted_at` is used, verify you can create a new punch after soft-deleting the previous one for the same natural key (partial index allows it).

---

**Summary**: enforce uniqueness at the DB layer with a partial unique index on `(employee_id, punch_type, date)` and use atomic `ON CONFLICT` upserts. Add `idempotency_key` for explicit replay safety when available. Keep handler logic minimal and rely on the database for correctness.
