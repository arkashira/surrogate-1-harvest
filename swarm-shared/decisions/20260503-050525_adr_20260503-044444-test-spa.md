## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: CLI-first API service (FastAPI on CF Workers) that connects to one AWS account and one GCP project, lists EventBridge rules + Lambda triggers and Cloud Scheduler jobs + Cloud Functions, normalizes cron expressions, detects exact duplicates and obvious orphans (no backing function), persists results to Supabase Postgres, exposes GET /schedules and GET /findings, and returns a summary JSON with counts of duplicate/orphaned schedules.

## Open questions
- Which AWS accounts/GCP projects and regions should be in scope for v0.1 and how will we authenticate (OIDC/workload identity vs long-lived keys)?
- How will we reliably observe last invocation timestamps (CloudWatch Logs Insights vs EventBridge/Stackdriver sinks vs scheduled metric queries) given free-tier limits?
- What deduplication tolerance (cron normalization, timezones, resource aliases) and drift threshold should we apply to avoid false positives?
- Who is the primary audience (single developer, small team, platform org) and how will they act on findings (CLI, UI, email/Slack)?
