# workio / backend

## 1. Diagnosis

- Clock-in/out endpoints accept duplicate requests within the same minute (retries, double-taps, LINE duplicate webhooks) → multiple punches for same tenant/user/day with no idempotency key.
- No server-side state machine enforcement: employee can clock-in twice or clock-out without clock-in; transitions are validated only in UI.
- Punch writes are not atomic with latest state check → race between two near-simultaneous requests can corrupt daily sequence (in→out→in→out).
- LINE webhook handler does synchronous DB writes and can exceed 3s SLA on contention → LINE retries and amplifies duplicates.
- No lightweight idempotency layer for write-heavy endpoints (punch, leave, OT) → retry storms under spotty mobile connectivity.

## 2. Proposed change

File: `/opt/axentx/workio/server/src/routes/punch.ts` (or equivalent route file)  
Scope: add idempotency token + state-machine guard to `POST /punch` (clock-in/out) with atomic upsert and unique constraint.

- Add table `punch_idempotency` (or extend punches with `idempotency_key` + unique constraint per tenant/user/day/action_window).
- Enforce state transition: for a given day, last punch must be `in` to allow `out`, and last must be `out` (or none) to allow `in`.
- Make handler async-acknowledge LINE quickly: accept, validate idempotency+transition, write atomically, return 200; background any heavy work.

2) Route handler diff (TypeScript/Express):

```ts
// server/src/routes/punch.ts
import { Request, Response } from 'express';
import { db } from '../db';
import { verifyLineSignature } from '../middleware/line';

type PunchBody = {
  tenantId: string;
  userId: string;
  action: 'in' | 'out';
  idempotencyKey?: string; // client or LINE dedupe id
  timestamp?: string; // ISO
  location?: { lat: number; lng: number };
};

export async function createPunch(req: Request, res: Response) {
  try {
    const { tenantId, userId, action, idempotencyKey = crypto.randomUUID() } = req.body as PunchBody;
    const now = new Date(req.body?.timestamp || Date.now());
    const date = now.toISOString().split('T')[0]; // YYYY-MM-DD

    // 1) Idempotency check (lightweight)
    const existing = await db.oneOrNone(
      `SELECT id, action_type FROM punches
       WHERE tenant_id = $1 AND user_id = $2 AND date = $3
       AND date_trunc('minute', created_at) = date_trunc('minute', $4)
       ORDER BY created_at DESC LIMIT 1`,
      [tenantId, userId, date, now]
    );

    if (existing) {
      // Same minute, same user/tenant: if same action treat as duplicate, if opposite and state allows, allow (race guard below)
      if (existing.action_type === action) {
        return res.status(200).json({ ok: true, duplicate: true, punchId: existing.id });
      }
    }

    // 2) State transition guard + atomic upsert using CTE
    const result = await db.tx(async (t) => {
      // Fetch latest punch for the day
      const latest = await t.oneOrNone(
        `SELECT id, action_type FROM punches
         WHERE tenant_id = $1 AND user_id = $2 AND date <= $3
         ORDER BY date DESC, created_at DESC LIMIT 1`,
        [tenantId, userId, date + ' 23:59:59']
      );

      if (action === 'out' && (!latest || latest.action_type !== 'in')) {
        throw new Error('Cannot clock out without a prior clock-in for the day.');
      }
      if (action === 'in' && latest && latest.action_type === 'in') {
        // Already clocked in — return existing to avoid duplicate
        return { punchId: latest.id, duplicate: true };
      }

      // Insert new punch
      const inserted = await t.one(
        `INSERT INTO punches (tenant_id, user_id, action_type, date, created_at, location, idempotency_key)
         VALUES ($1, $2, $3, $4, $5, $6, $7)
         RETURNING id`,
        [tenantId, userId, action, date, now, req.body?.location || null, idempotencyKey]
      );

      return { punchId: inserted.id, duplicate: false };
    });

    // 3) Fast LINE ack (fire-and-forget any notifications)
    // Optionally queue LINE notification via background job to keep handler fast
    // e.g., notifyPunchToLINE({ tenantId, userId, punchId: result.punchId, action }).catch(console.warn);

    return res.status(200).json({ ok: true, punchId: result.punchId, duplicate: result.duplicate });
  } catch (err: any) {
    console.error('Punch error:', err);
    return res.status(400).json({ ok: false, error: err.message });
  }
}
```

3) LINE webhook fast path:

- Keep webhook handler minimal: validate signature, parse userId/tenant, call `createPunch`, return 200 immediately.
- Offload LINE reply/notification to queue (Bull/SQS) or background promise to avoid blocking.

4) Client-side: generate stable idempotencyKey per user action (e.g., hash of tenant+user+date+action+minute) so retries are safe.

## 4. Verification

- Unit test: simulate concurrent requests (Promise.all 10x) for same tenant/user/minute/action → exactly one punch row created and others return duplicate flag.
- State machine test: attempt out→in→in→out sequence and assert only valid transitions persist.
- Manual: use Postman/curl to send duplicate idempotencyKey → second request returns `{ duplicate: true }` and same punchId.
- Load test: fire 50 req/s for same user/minute and confirm no duplicate rows and response times <500ms.
- LINE simulation: send duplicate LINE webhook events (same timestamp/user) and confirm only one punch and no timeout/retry amplification.
