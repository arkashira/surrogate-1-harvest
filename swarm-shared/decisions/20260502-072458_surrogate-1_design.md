# surrogate-1 / design

### Diagnosis
* The project lacks a robust implementation for handling Hugging Face API rate limits on the discovery side, which can block dataset training.
* The existing implementation may not be reusing existing Lightning Studio instances efficiently, leading to wasted resources.
* The project does not have a clear strategy for handling HF commit caps, which can limit the throughput of dataset ingestion.
* The current approach to downloading dataset files may not be optimized, potentially leading to slower training times.
* The project does not have a mechanism to automatically restart training processes if they are stopped due to idle timeouts.

### Proposed change
The proposed change is to implement a robust Hugging Face API rate limit handling mechanism, reuse existing Lightning Studio instances, and optimize dataset file downloads. This can be achieved by modifying the `train.py` script and the `ingestion.py` script.

### Implementation
To implement the proposed change, the following steps can be taken:
1. Modify the `train.py` script to use the HF CDN bypass mechanism to download dataset files, which can help avoid API rate limits.
2. Implement a retry mechanism with a 360-second wait time to handle API rate limit errors (429 status code).
3. Use the `list_repo_tree` method with `recursive=False` to pre-list file paths and embed them in the training script.
4. Reuse existing Lightning Studio instances by checking the status of studios before creating a new one.
5. Implement a mechanism to automatically restart training processes if they are stopped due to idle timeouts.
