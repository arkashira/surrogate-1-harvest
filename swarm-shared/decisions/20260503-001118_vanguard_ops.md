# vanguard / ops

### 1. Diagnosis
* No persisted manifest per `(repo, dateFolder)` exists, leading to repeated enumeration via authenticated HF API for every training launch, which burns quota and risks hitting the 429 rate limit.
* The current approach does not allow for pre-flight checks or caching of available files, resulting in users potentially selecting invalid or missing paths that cause training to fail late in the process.
* The absence of a CDN-only manifest for `(repo, dateFolder)` means that the frontend cannot efficiently manage or validate user input regarding dataset paths and availability.
* Training scripts likely utilize `load_dataset(streaming=True)` on repositories with heterogeneous file schemas, which can trigger PyArrow `CastError`.
* There's a lack of optimization in handling HF API rate limits and commit caps, which could significantly impact the efficiency and scalability of the training process.

### 2. Proposed change
The proposed change involves creating a mechanism to persist and utilize a CDN-only manifest for each `(repo, dateFolder)` pair. This will enable the frontend to pre-validate user input and reduce the reliance on authenticated HF API calls for dataset enumeration. Additionally, optimizing the training script to handle heterogeneous file schemas without triggering PyArrow errors and implementing strategies to mitigate HF API rate limits and commit caps will be crucial.

### 3. Implementation
To implement this change, the following steps can be taken:
1. **Modify the Training Script**: Update the training script to download dataset files individually using `hf_hub_download` and then project them to `{prompt, response}` only at parse time. This avoids the PyArrow `CastError` issue.
2. **Create a Manifest Generation Script**: Develop a script that generates a CDN-only manifest for each `(repo, dateFolder)` pair. This script should use `list_repo_tree(path, recursive=False)` to list files in a date folder without recursing, which helps avoid hitting the HF API rate limit.
3. **Store Manifests**: Store the generated manifests in a location accessible by the frontend, such as a database or a file storage system. This allows the frontend to query and validate user input against the manifest.
4. **Optimize HF API Usage**: Implement pagination and delay retries after hitting the 429 rate limit to avoid being blocked. For commit caps, spread writes across multiple sibling repositories to increase the aggregate commit limit.
5. **Update Frontend Logic**: Modify the frontend to query the stored manifests for available files and paths before initiating a training job. This ensures that users can only select valid paths, reducing late failures.

### 4. Verification
To verify that these changes work as expected:
1. **Test Manifest Generation**: Run the manifest generation script for a few `(repo, dateFolder)` pairs and verify that the manifests are correctly generated and stored.
2. **Validate Frontend Behavior**: Test the frontend by attempting to select various paths, including valid and invalid ones. Verify that the frontend correctly validates user input against the stored manifests.
3. **Monitor Training Jobs**: Run several training jobs with the updated script and verify that they complete successfully without hitting PyArrow errors or excessive HF API rate limits.
4. **Check API Rate Limit and Commit Cap Handling**: Monitor the system's behavior when approaching or hitting the HF API rate limits and commit caps. Verify that the implemented strategies (pagination, delayed retries, and spreading commits across repositories) effectively mitigate these issues.
