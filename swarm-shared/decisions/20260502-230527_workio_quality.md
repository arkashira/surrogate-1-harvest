# workio / quality

## Final consolidated solution (best parts merged, contradictions resolved)

### Diagnosis (merged, de-duplicated)
- No idempotency enforcement on clock-in/out endpoints: duplicate LINE webhook deliveries and client retries create multiple punches for the same logical day-punch, causing payroll drift.
- Non-transactional punch writes: concurrent retries can insert overlapping records without atomic upsert/locking, producing double punches or invalid state transitions (e.g., two “in” records without an “out”).
- Missing DB-level uniqueness: schema allows duplicate `(employee_id, punch_type, date)` rows; no guardrail at the database.
- No request deduplication: endpoint accepts raw payloads without checking a deduplication key (e.g., `line_delivery_id` or client-generated `idempotency_key`).
- Race condition between check-then-insert: app-level “find latest punch then insert if none” is not atomic under concurrent/retried requests.

### Design decisions (resolved)
- Use the natural key `(employee_id, punch_type, date)` as the primary uniqueness constraint for payroll correctness. This prevents duplicate punches per day per type regardless of retry source.
- Support an explicit `Idempotency-Key` header (or `idempotency_key` body field) for replay-safe retries from clients/LINE. Store it in a small idempotency table keyed by the idempotency key with a TTL for cleanup.
- Perform all writes inside a single transaction with `ON CONFLICT` upsert so concurrent requests cannot create duplicates.
- Keep punch_type values consistent with existing code: use `'clock_in'` / `'clock_out'` (not `'in'`/`'out'`) unless the rest of the codebase already uses the shorter forms. The implementation below uses the longer forms; adjust if necessary.

### Implementation

#### 1) DB schema changes (`workio/server/src/db/schema.sql`)
```sql
-- 1) Ensure punches uniqueness at the DB level
-- If creating fresh, include in CREATE TABLE; otherwise alter:
ALTER TABLE punches
  ADD CONSTRAINT uq_punch_employee_type_date
  UNIQUE (employee_id, punch_type, date);

-- 2) Idempotency table for replay protection (with TTL)
CREATE TABLE IF NOT EXISTS punch_idempotency (
  idempotency_key TEXT PRIMARY KEY,
  employee_id INTEGER NOT NULL,
  punch_type VARCHAR(20) NOT NULL,
  punch_date DATE NOT NULL,
  punch_record_id INTEGER NOT NULL REFERENCES punches(id),
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

#### 2) Idempotent punch endpoint (`workio/server/src/routes/punch.ts`)
```ts
import { Router, Request, Response } from 'express';
import pool from '../db';

const router = Router();

/**
 * Create or update a punch idempotently.
 * Idempotency key sources (in order):
 *  1) Idempotency-Key header
 *  2) req.body.idempotency_key
 *  3) Natural key (employee_id, punch_type, date) if none provided
 *
 * Natural-key uniqueness constraint prevents duplicates from concurrent/retried requests.
 * Explicit idempotency keys provide stable replay protection for clients/LINE.
 */
router.post('/punch', async (req: Request, res: Response) => {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    const { employee_id, punch_type, latitude, longitude, metadata, idempotency_key: bodyKey } = req.body;
    const idempotencyKey = (req.get('Idempotency-Key') || bodyKey || null) as string | null;
    const date = new Date().toISOString().split('T')[0]; // YYYY-MM-DD in UTC

    // Natural-key upsert: prevents duplicates from retries/races
    const upsert = `
      INSERT INTO punches (employee_id, punch_type, date, ts, latitude, longitude, metadata)
      VALUES ($1, $2, $3, NOW(), $4, $5, $6)
      ON CONFLICT (employee_id, punch_type, date)
      DO UPDATE SET
        ts = EXCLUDED.ts,
        latitude = EXCLUDED.latitude,
        longitude = EXCLUDED.longitude,
        metadata = EXCLUDED.metadata
      RETURNING *
    `;

    const result = await client.query(upsert, [
      employee_id,
      punch_type,
      date,
      latitude || null,
      longitude || null,
      metadata || null,
    ]);

    const punch = result.rows[0];

    // Record explicit idempotency key for stable replay protection
    if (idempotencyKey) {
      await client.query(
        `INSERT INTO punch_idempotency (idempotency_key, employee_id, punch_type, punch_date, punch_record_id)
         VALUES ($1, $2, $3, $4, $5)
         ON CONFLICT (idempotency_key) DO UPDATE
           SET punch_record_id = EXCLUDED.punch_record_id`,
        [idempotencyKey, employee_id, punch_type, date, punch.id]
      );
    }

    await client.query('COMMIT');
    res.json({ ok: true, punch });
  } catch (err: any) {
    await client.query('ROLLBACK');
    // Distinguish constraint violations for clearer client errors
    if (err.code === '23505') {
      res.status(409).json({ ok: false, error: 'Duplicate punch detected' });
      return;
    }
    console.error('Punch upsert failed', err);
    res.status(500).json({ ok: false, error: String(err) });
  } finally {
    client.release();
  }
});

export default router;
```

#### 3) Optional: idempotency read-through shortcut
If you want to short-circuit retries immediately using an idempotency key without attempting the upsert first:

```ts
if (idempotencyKey) {
  const existing = await client.query(
    `SELECT punch_record_id FROM punch_idempotency WHERE idempotency_key = $1`,
    [idempotencyKey]
  );
  if (existing.rows.length) {
    // Return the previously created punch without re-upserting
    const punch = (await client.query('SELECT * FROM punches WHERE id = $1', [existing.rows[0].punch_record_id])).rows[0];
    await client.query('ROLLBACK');
    client.release();
    return res.json({ ok: true, punch, reused: true });
  }
}
```
(Place this block right after `BEGIN` if you prefer this pattern.)

### Verification checklist
1. Apply schema changes to dev DB (`ALTER TABLE` + `CREATE TABLE punch_idempotency`).
2. Start backend: `cd workio/server && npm run dev`.
3. Send duplicate requests with the same idempotency key:
   ```bash
   curl -X POST http://localhost:3000/punch \
     -H "Content-Type: application/json" \
     -H "Idempotency-Key: test-123" \
     -d '{"employee_id":1,"punch_type":"clock_in","latitude":13.123456,"longitude":100.123456}'
   ```
   Repeat 3–5 times. Confirm:
   - HTTP 200 each time.
   - Exactly one row for `(employee_id=1, punch_type='clock_in', date=today)`.
   - Idempotency table maps `test-123` to the same `punch_id` across calls.
4. Concurrent race test:
   ```bash
   for i in {1..10}; do
     curl -s -X POST http://localhost:3000/punch \
       -H "Content-Type: application/json" \
       -d '{"employee_id":2,"punch_type":"clock_out"}' &
   done
   wait
   ```
   Then verify:
   ```sql
   SELECT COUNT(*) FROM punches WHERE employee_id=2 AND punch_type='clock_out' AND date=CURRENT_DATE;
   ```
   Must return `1`.
5. Simulate LINE webhook replay with same payload and confirm no duplicate punches.
