# workio / backend

## Final Synthesized Implementation

**Chosen approach**: Combine Candidate 1’s robust transactional flow and schema with Candidate 2’s stricter tenant/user validation and clearer idempotency key design. Favor correctness and concrete actionability.

---

### 1. Diagnosis (consolidated)

- **Non-idempotent LINE webhook**: retries on 5xx/timeouts create duplicate punches and invalid open/close state.
- **Non-atomic punch transitions**: `SELECT` then `INSERT/UPDATE` allows interleaving; concurrent/retried requests can double-open or double-close.
- **No deduplication layer**: no idempotency key or de-dup window, so LINE retries and network blips create phantom punches.
- **Race on clock state**: latest-punch checks without row locking allow two webhooks to both believe the user is “out” (or “in”) and both insert/close.
- **Missing transactional boundary**: punch updates and side-effects (audit/notification) are not atomic; partial writes can persist.
- **Weak tenant/user validation**: webhook can accept events without validating tenant context, risking cross-tenant pollution.

---

### 2. Proposed change

- **File**: `workio/server/src/routes/webhook/line.ts` (or `/src/routes/line/webhook.ts` depending on layout)
- **Scope**:
  - Add idempotency table with unique constraint on `(source, delivery_key)` (plus tenant-scoped variant if needed).
  - Perform punch open/close inside a single DB transaction.
  - Use `SELECT ... FOR UPDATE` on the latest open punch row (per user/tenant) to serialize transitions.
  - Validate tenant/user mapping strictly before any state change.
  - Record idempotency entry inside the same transaction.
  - Commit only after punch transition and idempotency record are persisted; enqueue notifications outside the transaction.

// workio/server/src/routes/webhook/line.ts
import { pool } from '../../db';
import { v4 as uuidv4 } from 'uuid';

export async function handleLineWebhook(req, res) {
  const { body } = req;
  const events = body?.events;
  if (!events || !Array.isArray(events)) return res.status(400).send('invalid');

  const client = await pool.connect();
  try {
    for (const ev of events) {
      if (ev.type !== 'message' || ev.message?.type !== 'text') continue;

      const userId = ev.source?.userId;
      const deliveryId = ev.delivery?.id || ev.message?.id || ev.message?.text;
      if (!userId || !deliveryId) continue;

      // Resolve tenant strictly; reject if unknown
      const tenantId = await resolveTenantIdByLineUserId(userId);
      if (!tenantId) {
        // Optionally log and continue or return 400 for this event
        continue;
      }

      await client.query('BEGIN');

      // 1) Idempotency check
      const idem = await client.query(
        `SELECT 1 FROM webhook_deliveries WHERE source = $1 AND delivery_key = $2`,
        ['line', deliveryId]
      );
      if (idem.rowCount > 0) {
        await client.query('ROLLBACK');
        continue;
      }

      // 2) Lock latest punch for this user+tenant to serialize transitions
      const latest = await client.query(
        `SELECT id, clock_in, clock_out, location_in, location_out
         FROM punches
         WHERE tenant_id = $1 AND user_id = $2
         ORDER BY clock_in DESC
         LIMIT 1
         FOR UPDATE`,
        [tenantId, userId]
      );

      const last = latest.rows[0];
      const now = new Date();
      const location = extractLocationFromLine(ev); // implement: parse text or location payload

      if (last && !last.clock_out) {
        // close open punch
        await client.query(
          `UPDATE punches
           SET clock_out = $1, updated_at = NOW(), location_out = $2
           WHERE id = $3`,
          [now, location, last.id]
        );
      } else {
        // open new punch
        await client.query(
          `INSERT INTO punches (tenant_id, user_id, clock_in, location_in, created_at)
           VALUES ($1, $2, $3, $4, NOW())`,
          [tenantId, userId, now, location]
        );
      }

      // 3) Record delivery as processed (inside tx)
      await client.query(
        `INSERT INTO webhook_deliveries (source, delivery_key, tenant_id, user_id, processed_at, payload_hash)
         VALUES ($1, $2, $3, $4, NOW(), $5)`,
        ['line', deliveryId, tenantId, userId, hashPayload(ev)]
      );

      await client.query('COMMIT');

      // 4) Enqueue notification outside transaction (best-effort)
      enqueueLineNotify(userId, last && !last.clock_out ? 'clock_out' : 'clock_in', now).catch((err) => {
        console.warn('Failed to enqueue LINE notify', err);
      });
    }
    res.status(200).json({ ok: true });
  } catch (err) {
    await client.query('ROLLBACK').catch(() => {});
    console.error('LINE webhook tx failed', err);
    res.status(500).send('error');
  } finally {
    client.release();
  }
}

### 4. Verification (actionable tests)

1. **Idempotency**
   - Send same LINE event payload twice (same delivery id) → second request must not modify punches.
   - Confirm exactly one `webhook_deliveries` row for that key and unchanged punch count.

2. **Concurrent/retry race**
   - Simulate two near-simultaneous POSTs for the same user (e.g., `Promise.all` with two clients).
   - Assert final state: exactly one open and one close (no double-open/close).

3. **State transitions**
   - No open punch → POST → creates open punch (`clock_out IS NULL`).
   - POST again → closes punch (`clock_out` set and `clock_out >= clock_in`).

4. **Rollback on error**
   - Force error after punch update but before commit (e.g., throw).
   - Verify no partial rows persist and no delivery row recorded.

5. **Tenant validation**
   - Send event for unmapped LINE user → handler must skip/return error and not create punches for other tenants.

6. **LINE retry behavior**
   - Configure LINE webhook retry and observe logs; duplicates must be ignored and final punch state consistent.
