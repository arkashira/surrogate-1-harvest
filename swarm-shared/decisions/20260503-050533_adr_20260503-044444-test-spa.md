## v0.1 scope
v0.1 (2 weeks): CLI-first MVP deployable to Cloudflare Workers. Connect one AWS account and one GCP project via encrypted service-account/role credentials. Discover and list all EventBridge (cron/rate) rules mapped to Lambda and all Cloud Scheduler jobs mapped to Cloud Functions in selected regions. Run a one-shot audit that flags duplicates (same target+expression within workspace), orphans (schedule exists with no target function), and drifted schedules (target function missing or expression changed). Output JSON report + seeded Supabase Postgres tables (workspace, credential, discovered_schedule, audit_run, audit_issue). Basic GET endpoints for listing runs/issues and marking resolved. No UI; auth via Supabase JWT. Free-tier only (CF Workers, Supabase free, OSS deps).

## Open questions
- How will customers grant least-privilege read-only access to Lambda/Cloud Functions metadata (IAM roles vs service accounts)?
- What is the minimal viable schedule discovery scope (region-level vs org-level) for v0.1?
- Will audit runs be triggered manually, on cron, or webhook-driven from cloud events?
- How should drift be defined (expression change, target function change, IAM/resource config change)?
- Retention policy for discovered_schedule history and audit_run artifacts
