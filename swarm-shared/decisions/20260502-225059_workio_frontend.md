# workio / frontend

### 1. Diagnosis
* The current implementation of the attendance queue in the Workio frontend lacks a robust and resilient storage solution, relying on fragile `localStorage` which is prone to size limitations and data loss.
* The system does not implement idempotency for clock actions, which can lead to inconsistent state if a user attempts to clock in or out multiple times.
* There is no durable retry mechanism in place, which means that failed clock-in/out attempts may not be retried, resulting in lost data.
* The current implementation is vulnerable to cross-tab conflicts, where multiple tabs or windows may interfere with each other's state.
* The system lacks a periodic flush mechanism to ensure that data is regularly synced with the server, even when the user is not actively interacting with the application.

### 2. Proposed change
To address these issues, we propose implementing an IndexedDB-backed queue with idempotency keys, durable retry state, exponential backoff + jitter, online-triggered + periodic flush, and cross-tab safe operations. The changes will be made in the `workio` frontend repository, specifically in the files responsible for handling clock-in/out logic and data storage.

### 3. Implementation
We will create a new IndexedDB database to store the attendance queue, with the following features:
```typescript
// db.ts
import { openDB } from 'idb';

const db = openDB('workio-attendance', 1, {
  upgrade(db) {
    db.createObjectStore('queue', { keyPath: 'id' });
  },
});

export const addClockEvent = async (event: ClockEvent) => {
  const tx = db.transaction('queue', 'readwrite');
  const store = tx.objectStore('queue');
  await store.add(event);
  await tx.done;
};

export const getQueue = async () => {
  const tx = db.transaction('queue', 'readonly');
  const store = tx.objectStore('queue');
  const queue = await store.getAll();
  await tx.done;
  return queue;
};
```

We will also implement idempotency for clock actions by generating a unique idempotency key for each clock event:
```typescript
// clock.ts
import { v4 as uuidv4 } from 'uuid';

const generateIdempotencyKey = () => {
  return uuidv4();
};

export const clockIn = async () => {
  const idempotencyKey = generateIdempotencyKey();
  const event = { type: 'clock-in', idempotencyKey };
  await addClockEvent(event);
  // ...
};
```

To ensure durable retry and exponential backoff + jitter, we will use a library like `retry-async`:
```typescript
// retry.ts
import retryAsync from 'retry-async';

const retryOptions = {
  retries: 5,
  factor: 2,
  minTimeout: 1000,
  maxTimeout: 30000,
  randomize: true,
};

export const retryClockEvent = async (event: ClockEvent) => {
  await retryAsync(async () => {
    // Send clock event to server
  }, retryOptions);
};
```

Finally, we will implement online-triggered + periodic flush and cross-tab safe operations using the `workio` frontend's existing infrastructure.

### 4. Verification
To verify that the changes work as expected, we can write tests to cover the following scenarios:
* Clock-in/out events are stored correctly in the IndexedDB queue.
* Idempotency keys are generated correctly and prevent duplicate clock events.
* Durable retry and exponential backoff + jitter work correctly for failed clock events.
* Periodic flush and cross-tab safe operations ensure that data is synced regularly and correctly.
