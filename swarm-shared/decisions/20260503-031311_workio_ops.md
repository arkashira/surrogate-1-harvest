# workio / ops

## Final consolidated implementation (strongest parts, resolved contradictions)

### 1. Diagnosis (merged)
- **Missing LINE signature verification** (`X-Line-Signature`) — allows spoofed/replayed events to corrupt attendance.
- **No idempotency/replay protection** — LINE retries on 5xx/timeouts create duplicate clock-in/out, leave, OT rows.
- **No event freshness enforcement** — stale/replayed events can be processed after retries or clock skew.
- **No structured audit trail** — makes incident investigation, rollback, and observability hard.
- **No safe retry semantics** — returning 5xx for non-transient errors can cause retry storms; returning 200 for invalid requests can mask attacks.

### 2. DB schema (single source of truth)
```sql
-- Idempotency + audit table
CREATE TABLE IF NOT EXISTS line_event_idempotency (
  line_event_id      TEXT PRIMARY KEY,
  received_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  status             TEXT NOT NULL CHECK (status IN ('processing','processed','failed')),
  event_type         TEXT,            -- LINE event type
  entity_type        TEXT,            -- 'clock','leave','ot'
  entity_id          BIGINT,
  user_id            TEXT,
  payload_hash       TEXT,
  error_message      TEXT
);

-- Optional: index for TTL cleanup
CREATE INDEX IF NOT EXISTS idx_line_event_idempotency_received_at
  ON line_event_idempetency(received_at);
```

### 3. Core invariants (resolved contradictions)
- **Signature verification is mandatory and first** — reject 401 on mismatch (do not proceed to idempotency).
- **Idempotency key is deterministic and stable** — use `source.userId:eventType:timestamp` (or explicit `eventId` if provided). This prevents collisions across users/types and ensures replays map to the same key.
- **Freshness window is enforced per event** — reject events older than `MAX_EVENT_AGE_MS` (e.g. 5 minutes) with 400 to prevent processing stale events.
- **Idempotency record is created atomically with a transaction that includes the business operation** — prevents race conditions between duplicate deliveries.
- **Return 200 only for accepted/processed or duplicate events** — return 4xx for invalid/stale/auth failures; return 5xx only for transient server errors where a LINE retry is safe and desired.
- **Structured audit/logging is emitted for every attempt** — include event id, status, outcome, and error details.

    .createHmac('sha256', CHANNEL_SECRET)
    .update(body, 'utf8')
    .digest('base64');
  return crypto.timingSafeEqual(Buffer.from(signature), Buffer.from(expected));
}

  const { source, type, timestamp } = ev || {};
  if (!source?.userId || !type || !timestamp) return null;
  // Prefer any explicit eventId from LINE payload when available
  const explicit = ev.eventId || ev.message?.id || ev.source?.requestId || null;
  return explicit ? `${explicit}` : `${source.userId}:${type}:${timestamp}`;
