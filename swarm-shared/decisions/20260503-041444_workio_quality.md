# workio / quality

## Final Synthesis — One Correct, Actionable Solution

**Core principle:**  
Keep the backend as the source of truth (idempotency + uniqueness + row-level locking), and make the frontend **fast, safe, and predictable** by adding a client-side lock, optimistic state, abort control, and a short temporal guard. All candidate insights are valid; where they differ, prefer correctness and concrete implementation.

---

### 1. Diagnosis (resolved)

- **Duplicate submissions** come from rapid taps and overlapping POSTs before idempotency keys settle.  
- **Missing client-side lock** allows multiple in-flight requests for the same user/action.  
- **No optimistic UI** causes confusing latency and invites retries.  
- **Out-of-order resolution** can overwrite newer state when slow responses resolve after newer ones.  
- **Backend races** on punch updates require row-level locking/atomic upserts keyed by idempotencyKey (tenant+userId+date+eventType).  
- **Grace window / temporal guard** prevents accidental back-to-back duplicates within a short window (2–5s) while still allowing legitimate corrections.

---

### 2. Backend requirements (non-negotiable)

1. **Idempotency enforcement**  
   - Unique constraint/index on `(tenant_id, user_id, date, event_type, idempotency_key)` (or a dedicated idempotency table with request hash).  
   - Accept idempotency key in request body/header; return existing record on duplicate key (200) rather than error.

2. **Row-level locking / atomic upsert**  
   - Use `INSERT ... ON CONFLICT DO NOTHING` (PostgreSQL) or equivalent with `SELECT ... FOR UPDATE` when updating last punch.  
   - Ensure updates to “current” punch state are atomic and serialized per user.

3. **Grace window enforcement (optional but recommended)**  
   - Reject or ignore punches of the same type within a configurable grace window (e.g., 2–5 minutes) unless explicitly allowed (correction flow).  
   - Return clear error codes (e.g., 409 with `code: TOO_SOON`).

4. **API contract**  
   - `POST /clock/in` and `POST /clock/out` accept:  
     ```json
     { "idempotencyKey": "string" }
     ```
   - Response: full `ClockRecord`.  
   - Support `AbortSignal` (standard HTTP).

  return `${type}-${Date.now()}-${crypto.randomUUID()}`;
