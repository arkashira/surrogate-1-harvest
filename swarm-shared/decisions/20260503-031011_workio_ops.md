# workio / ops

## Final Synthesis — Security + Idempotency + Actionable Delivery

**Chosen scope (minimal, high-impact):**  
- Add **LINE signature verification** (non-negotiable for integrity/non-repudiation).  
- Add **idempotency by `line_event_id`** enforced at the database to survive retries and replays within LINE’s retry window.  
- Keep changes small, testable, and deployable in <2 hours.  
- Defer frontend optimistic UI and interactive flows (Candidate 2) to a separate UX ticket; they are valuable but not required to fix the replay/spoofing bug.

---

### 1. Concrete implementation plan

#### 1.1 Add signature verification (security)
- Create `server/src/middleware/line-signature.ts` (as Candidate 1).  
- Use `crypto.timingSafeEqual` and require `LINE_CHANNEL_SECRET`.  
- In dev, allow bypass with a clear warning (avoid blocking local testing).

#### 1.2 Preserve raw body (required for valid signature)
- Add `raw-body` capture **before** JSON parsing.  
- Recommended: use `express.json({ verify: (req, _, buf) => { req.rawBody = buf.toString(); } })` rather than a manual stream collector (simpler, fewer edge cases).  
- Apply signature middleware **after** raw-body capture but **before** business logic.

#### 1.3 Enforce idempotency at the database (correctness)
- Add `line_event_id TEXT` column to `clock_events`, `leave_requests`, `ot_requests`.  
- Create unique partial indexes:  
  `CREATE UNIQUE INDEX CONCURRENTLY idx_clock_events_line_event_id ON clock_events (line_event_id) WHERE line_event_id IS NOT NULL;`  
  (repeat for leave_requests and ot_requests).  
- Use partial indexes to allow NULLs while enforcing uniqueness for non-NULL values.

#### 1.4 Update webhook handler (actionable)
- Validate `X-Line-Signature` via middleware (401 on mismatch).  
- Validate payload shape; return 400 for malformed requests.  
- For each event:  
  1. Skip if missing `ev.id`.  
  2. **Idempotency check**: attempt insert with `line_event_id`.  
  3. Catch unique violation (`23505`) and treat as success (idempotent replay).  
  4. Use a transaction per batch (or per event) to avoid partial application.  
- Return 200 only after successful processing or confirmed duplicate.

#### 1.5 Wire-up
- In main app:  
  ```ts
  app.use(express.json({
    limit: '1mb',
    verify: (req: any, _, buf) => { req.rawBody = buf.toString(); }
  }));
  app.use('/webhook/line', lineSignatureMiddleware, lineRouter);
  ```

### 2. Migration (SQL)
```sql
-- clock_events
ALTER TABLE clock_events ADD COLUMN IF NOT EXISTS line_event_id TEXT;
CREATE UNIQUE INDEX CONCURRENTLY IF NOT EXISTS idx_clock_events_line_event_id
  ON clock_events (line_event_id) WHERE line_event_id IS NOT NULL;

-- leave_requests
ALTER TABLE leave_requests ADD COLUMN IF NOT EXISTS line_event_id TEXT;
CREATE UNIQUE INDEX CONCURRENTLY IF NOT EXISTS idx_leave_requests_line_event_id
  ON leave_requests (line_event_id) WHERE line_event_id IS NOT NULL;

-- ot_requests
ALTER TABLE ot_requests ADD COLUMN IF NOT EXISTS line_event_id TEXT;
CREATE UNIQUE INDEX CONCURRENTLY IF NOT EXISTS idx_ot_requests_line_event_id
  ON ot_requests (line_event_id) WHERE line_event_id IS NOT NULL;
```
- Run during low traffic or with `CONCURRENTLY` to avoid locks.  
- Backfill existing rows with NULL `line_event_id` (no conflict).

---

### 3. Verification checklist
- **Signature verification**:  
  - Send request with correct signature → 200.  
  - Send request with incorrect/missing signature → 401.  
- **Idempotency**:  
  - Insert event with `line_event_id = 'test-123'` → row created.  
  - Replay same `line_event_id` → no new row, 200 returned.  
  - Verify with `SELECT count(*) FROM clock_events WHERE line_event_id = 'test-123'` (should be 1).  
- **Integration**:  
  - Configure LINE webhook to target endpoint; confirm 200 responses and no duplicate records under retries.

---

### 4. Why this wins
- **Correctness**: Signature + DB-level uniqueness is the only way to guarantee integrity and exactly-once semantics across retries.  
- **Actionability**: Each change is small, isolated, and testable.  
- **Safety**: Partial indexes and NULL handling avoid schema pitfalls; unique violation handling makes race conditions harmless.  
- **Scope control**: Frontend optimistic UI (Candidate 2) is deferred; it improves UX but does not fix the core replay/spoofing vulnerability.
