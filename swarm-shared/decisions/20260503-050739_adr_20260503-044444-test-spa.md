## Deployment
cf workers (free-tier Workers + D1 (Postgres-compatible) for API; Supabase for durable Postgres; optionally HF space for lightweight UI prototype)

## v0.1 scope
v0.1 in 2 weeks: CLI + FastAPI service that discovers Lambda and Cloud Functions schedules (read-only), normalizes cron expressions, persists inventory in Supabase Postgres, detects duplicates/drifted/orphaned entries, exposes REST endpoints and JSON report output, and provides a simple script to mark schedules as retired (soft delete). No UI, no auto-remediation, free-tier deploy on CF Workers + D1 + Supabase.

## Open questions
- Which identity/assumed-role strategy will we use to read Lambda schedules and Cloud Functions across customer accounts (cross-account IAM vs. customer-supplied keys)?
- How often should discovery scans run (hourly vs. daily) to balance freshness vs. rate limits and cost?
- Will we require write permissions to retire schedules automatically, or will we limit v0.1 to read-only detection and recommendations?
- What SLA/latency expectations exist for scan completion (sync API vs. async job + webhook)?
