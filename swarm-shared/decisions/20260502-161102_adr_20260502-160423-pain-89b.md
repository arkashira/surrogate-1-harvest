## Deployment
cf workers

## v0.1 scope
v0.1 ships a single static client binary (Linux x86_64) with: 1) SOCKS5 listener hidden behind shadowsocks-like AEAD obfuscation; 2) TUN-less mode (userspace tun via boringtun) that encapsulates traffic to appear as HTTPS to a CF Worker egress; 3) local kill-switch that blocks non-stealth interfaces on failure; 4) zero-config onboarding (QR/URL import for obfuscation keys + CF egress); 5) encrypted, opt-in telemetry to Supabase Postgres for WAU-ZB aggregation; 6) basic DPI probe resistance (utls mimic + traffic morphing padding); 7) no server infra required beyond CF Workers and Supabase free tier.

## Open questions
- Which obfuscation front (domain-fronted CF Workers, meek-like TLS, or uTLS-mimic) provides the best reach/stealth trade-off under active DPI in target regions?
- How will we validate WAU-ZB without collecting PII or risking deanonymization (client-side hashing strategy and aggregation cadence)?
- What is the kill-switch enforcement model (TUN/bpf drop vs. firewall rules) to guarantee no leaks across platforms (Linux/Windows/macOS)?
- Can we rely solely on outbound HTTPS to CF Workers for control-plane to avoid separate infra, and how do we rotate bridge endpoints safely?
- How do we handle captive portals and networks that block or intercept TLS without breaking stealth or user safety?
