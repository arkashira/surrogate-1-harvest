# workio / ops

## Final consolidated solution (correct + actionable)

**Core principle**: enforce uniqueness at the database level and make the write path idempotent so retries (webhooks, client retries, network flakes) cannot create duplicates.

---

### 1) Database changes (migration)

File: `workio/server/src/db/schema.sql` (or run as a migration)

```sql
-- 1) Clean up any exact duplicate rows (safe one-time)
DELETE FROM punches
WHERE ctid NOT IN (
  SELECT MIN(ctid)
  FROM punches
  GROUP BY employee_id, punch_type, date, ts
);

-- 2) Add idempotency support column (nullable for backward compatibility)
ALTER TABLE punches
ADD COLUMN IF NOT EXISTS idempotency_key TEXT;

-- 3) Enforce one punch per employee per type per date.
--    Use a partial unique index so existing rows without idempotency_key remain valid,
--    while new/updated rows that provide idempotency_key are strongly deduplicated.
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_unique_per_day
  ON punches (employee_id, punch_type, date)
  WHERE idempotency_key IS NOT NULL;

-- 4) Covering index for fast latest-punch lookups and date-range queries
CREATE INDEX IF NOT EXISTS idx_punches_employee_date_type_ts
  ON punches (employee_id, date, punch_type, ts DESC);
```

Apply:

```bash
psql workio < workio/server/src/db/schema.sql
```

Verify no duplicates remain:

```sql
SELECT employee_id, punch_type, date, COUNT(*)
FROM punches
GROUP BY employee_id, punch_type, date
HAVING COUNT(*) > 1;
```

### 2) Application changes (idempotent write path)

File: `workio/server/src/routes/punch.ts` (or equivalent)

```ts
import { Request, Response } from 'express';
import { pool } from '../db';

export async function createPunch(req: Request, res: Response) {
  const { employee_id, punch_type, date, ts, location, idempotency_key } = req.body;

  if (!employee_id || !punch_type || !date) {
    return res.status(400).json({ error: 'Missing required fields' });
  }

  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // Idempotency-key fast path: if provided and already exists, return existing
    if (idempotency_key) {
      const existing = await client.query(
        `SELECT id, employee_id, punch_type, date, ts, location, idempotency_key, created_at
         FROM punches
         WHERE idempotency_key = $1`,
        [idempotency_key]
      );

      if (existing.rows.length > 0) {
        await client.query('COMMIT');
        return res.status(200).json({ ...existing.rows[0], duplicate: true });
      }
    }

    // Try insert; conflict on (employee_id, punch_type, date) when idempotency_key present
    const result = await client.query(
      `INSERT INTO punches (employee_id, punch_type, date, ts, location, idempotency_key)
       VALUES ($1, $2, $3, $4, $5, $6)
       ON CONFLICT (employee_id, punch_type, date)
       DO NOTHING
       RETURNING id, employee_id, punch_type, date, ts, location, idempotency_key, created_at`,
      [employee_id, punch_type, date, ts || new Date(), location, idempotency_key || null]
    );

    await client.query('COMMIT');

    if (result.rows.length === 0) {
      // Duplicate detected by unique constraint
      return res.status(409).json({ error: 'Duplicate punch for this date' });
    }

    return res.status(201).json(result.rows[0]);
  } catch (err) {
    await client.query('ROLLBACK');
    console.error('Punch creation failed', err);
    return res.status(500).json({ error: 'Internal server error' });
  } finally {
    client.release();
  }
}
```

Key points:
- Use a transaction (`BEGIN`/`COMMIT`/`ROLLBACK`) to make check-then-insert atomic.
- If `idempotency_key` is supplied and exists, return `200` with the existing record (safe for LINE/webhook retries).
- If no `idempotency_key` but a duplicate `(employee_id, punch_type, date)` exists, `ON CONFLICT DO NOTHING` prevents insert and we return `409` so callers know it’s a duplicate.
- Never surface duplicate rows to clients or webhook retries.

### 4) Operational notes

- For existing code that cannot provide `idempotency_key`, the partial unique index still prevents duplicates at the database level, and the app returns `409` so callers can handle retries safely.
- If you use an ORM (Knex/Sequelize), map the same pattern: transaction → idempotency-key lookup → `ON CONFLICT DO NOTHING` → `409` on conflict.
- Treat `409` as “already recorded” in clients and webhook handlers; do not retry on `409` for the same `(employee_id, punch_type, date)`.
- Keep `idempotency_key` generation on clients/webhooks (e.g., UUID or deterministic hash of request) to maximize idempotency coverage.
