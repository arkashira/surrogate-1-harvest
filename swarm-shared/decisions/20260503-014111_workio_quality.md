# workio / quality

## 1. Diagnosis

- **No idempotency key on LINE webhook ingestion** — LINE retries on 5xx/timeouts; duplicate punches/leave/OT entries created on retry.
- **Race condition in punch state transitions** — `find-then-insert/update` allows multiple open punches per user/tenant when concurrent deliveries occur.
- **No transactional boundary across write + state update** — partial failures leave inconsistent state (e.g., punch recorded but status not updated).
- **Missing unique constraint to enforce one-open-punch-per-user** — DB-level safety absent, relies on app logic.
- **No deduplication store for LINE event IDs** — repeated events with same `deliveryId`/`timestamp` are reprocessed.

## 2. Proposed change

File: `/opt/axentx/workio/server/src/routes/webhook/line.ts` (or equivalent webhook handler)  
Scope:
- Add idempotency table `line_webhook_events` with unique constraint on `(channel_id, event_id)` (or stable hash of `deliveryId` + `timestamp` + `source.userId`).
- Wrap punch/leave/OT handling in a DB transaction with `SELECT ... FOR UPDATE` on user’s latest punch row to serialize concurrent transitions.
- Add unique partial index `one_open_punch_per_user` on `punches(user_id, tenant_id)` where `clock_out_at IS NULL`.
- Return 200 quickly after idempotency check and enqueue processing (or process inline inside transaction) so LINE stops retrying.

## 3. Implementation

```sql
-- server/src/db/migrations/20260504_idempotency_and_punch_constraint.sql
CREATE TABLE IF NOT EXISTS line_webhook_events (
  id              BIGSERIAL PRIMARY KEY,
  channel_id      TEXT NOT NULL,
  event_id        TEXT NOT NULL,
  event_type      TEXT NOT NULL,
  payload         JSONB NOT NULL,
  processed_at    TIMESTAMPTZ,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  CONSTRAINT uq_line_event UNIQUE (channel_id, event_id)
);

-- Prevent multiple open punches per user/tenant
CREATE UNIQUE INDEX IF NOT EXISTS idx_one_open_punch_per_user
ON punches (user_id, tenant_id)
WHERE clock_out_at IS NULL;
```

```typescript
// server/src/routes/webhook/line.ts
import { Router } from 'express';
import { pool } from '../../db';
import crypto from 'crypto';

const router = Router();

function stableEventId(body: any): string {
  // Prefer explicit id; fallback to hash of deliveryId/timestamp/source
  const id = body?.events?.[0]?.webhookEventId ||
             crypto.createHash('sha256')
               .update(JSON.stringify(body.events || []))
               .digest('hex');
  return id;
}

router.post('/line', async (req, res) => {
  const body = req.body;
  const channelId = process.env.LINE_CHANNEL_ID || 'default';
  const eventId = stableEventId(body);

  // Fast idempotency check + mark processed in one round
  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // Try to insert; if duplicate, this is a retry -> acknowledge and stop
    const insertEvent = await client.query(
      `INSERT INTO line_webhook_events (channel_id, event_id, event_type, payload)
       VALUES ($1, $2, $3, $4)
       ON CONFLICT (channel_id, event_id) DO NOTHING
       RETURNING id`,
      [channelId, eventId, 'line.webhook', body]
    );

    if (insertEvent.rowCount === 0) {
      await client.query('COMMIT');
      return res.sendStatus(200);
    }

    // Process each event serially inside transaction
    const events = body.events || [];
    for (const ev of events) {
      if (ev.type === 'message' && ev.message.type === 'text') {
        const text = ev.message.text.trim().toLowerCase();
        const userId = ev.source.userId;

        // Lock latest punch row for this user to prevent races
        const latest = await client.query(
          `SELECT id, clock_out_at FROM punches
           WHERE user_id = (SELECT id FROM users WHERE line_user_id = $1 AND tenant_id = $2)
           ORDER BY clock_in_at DESC LIMIT 1 FOR UPDATE`,
          [userId, 'default-tenant'] // tenant resolution TBD; placeholder
        );

        const hasOpen = latest.rows[0] && !latest.rows[0].clock_out_at;

        if (text === 'clock in' || text === 'เข้างาน') {
          if (hasOpen) {
            // Already clocked in — ignore or notify
            continue;
          }
          await client.query(
            `INSERT INTO punches (user_id, tenant_id, clock_in_at, line_event_id)
             VALUES (
               (SELECT id FROM users WHERE line_user_id = $1 AND tenant_id = $2),
               $2, NOW(), $3
             )`,
            [userId, 'default-tenant', eventId]
          );
        } else if (text === 'clock out' || text === 'เลิกงาน') {
          if (!hasOpen) {
            // No open punch — ignore or notify
            continue;
          }
          await client.query(
            `UPDATE punches
             SET clock_out_at = NOW(), line_event_id = $1
             WHERE id = $2`,
            [eventId, latest.rows[0].id]
          );
        }
      }
    }

    await client.query('COMMIT');
    res.sendStatus(200);
  } catch (err) {
    await client.query('ROLLBACK');
    console.error('LINE webhook processing failed', err);
    // Return 5xx so LINE retries (unless it's a client error we don't want retried)
    res.sendStatus(500);
  } finally {
    client.release();
  }
});

export default router;
```

## 4. Verification

1. **Idempotency test**  
   - Send same LINE webhook payload twice (simulate retry).  
   - Verify second request returns 200 and does not create duplicate punch rows.  
   - Confirm `line_webhook_events` has exactly one entry for the `event_id`.

2. **Race condition test**  
   - Fire two concurrent `clock in` requests for same user (e.g., `Promise.all` with 10 parallel calls).  
   - Verify only one open punch is created; check DB for unique constraint violation logs (should be none) and confirm only one row with `clock_out_at IS NULL`.

3. **Constraint enforcement**  
   - Manually attempt to insert two open punches for same user via SQL.  
   - Verify `idx_one_open_punch_per_user` prevents the second insert.

4. **Transaction rollback test**  
   - Force an error mid-handler (e.g., throw after first punch update).  
   - Confirm no partial writes remain (punch row count unchanged).

5. **LINE retry behavior**  
   - Return 500 on first request, then 200 on retry.  
   - Verify LINE stops retrying after 200 and state is consistent (one punch row, correct timestamps).
