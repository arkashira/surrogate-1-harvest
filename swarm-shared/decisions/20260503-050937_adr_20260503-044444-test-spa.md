## Deployment
cf workers (free tier; deploy API + lightweight scheduler; reuse existing infra)

## v0.1 scope
v0.1 in 2 weeks: CLI + FastAPI endpoint that scans one AWS account and one GCP project, lists schedules with state (active/disabled/orphaned), detects duplicates (identical cron targeting overlapping time windows), flags drifted schedules (expression changed since last scan), surfaces summary counts and per-resource details via JSON; stores results in Supabase Postgres; deploy to Cloudflare Workers free tier; no UI, auth via bearer token.

## Open questions
- Which accounts/projects and regions should be in-scope for week 1 to keep scope minimal?
- How will we authenticate to AWS and GCP in CI/on Cloudflare Workers (least-privilege keys vs OIDC)?
- What staleness threshold defines 'orphaned' (e.g., no last_run_at in 90 days) and 'drifted' (checksum mismatch or >5 min schedule slip)?
- Should duplicate detection include same expression across different accounts/regions or limit to same account/region?
- What retention policy for raw scan results vs aggregates to stay within Supabase free tier?
