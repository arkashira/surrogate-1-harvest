# workio / quality

## 1) Diagnosis

- **No idempotency enforcement at DB layer**: clock-in/out endpoints accept duplicate requests for the same `(employee_id, punch_type, date)` when clients retry on flaky mobile networks → double punches and payroll drift.
- **Non-transactional punch writes**: concurrent retries or race windows can insert two rows for the same punch window because there is no unique constraint or upsert.
- **No idempotency key in request contract**: frontend can retry safely but backend treats retries as new punches; missing stable key prevents replay-safe deduplication.
- **Missing defensive constraint in schema**: `punches` table lacks a natural unique key (e.g., `(employee_id, punch_date, punch_type)` or idempotency-key column) to reject duplicates at the database level.
- **Clock skew / timezone ambiguity**: punch timestamps stored without explicit timezone or normalized date boundary can cause same-day duplicates across day boundaries for late/early punches.

## 2) Proposed change

- **File scope**: `workio/server/src/db/schema.sql` (add constraint + idempotency column)  
- **File scope**: `workio/server/src/routes/punch.ts` (accept + enforce idempotency key, use transactional upsert)  
- **Change type**: Add `idempotency_key` column (unique), create unique constraint on `(employee_id, punch_date, punch_type)` as defensive fallback, and implement `INSERT ... ON CONFLICT DO NOTHING` (or upsert) in a single transaction.

## 3) Implementation

### Step 1 — Update schema (`workio/server/src/db/schema.sql`)

```sql
-- Add idempotency_key to punches if not present
ALTER TABLE punches
  ADD COLUMN IF NOT EXISTS idempotency_key TEXT;

-- Ensure uniqueness at the DB level for same-day same-type punches per employee
-- (defensive fallback; idempotency_key is the primary dedupe mechanism)
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_employee_date_type
  ON punches (employee_id, punch_date, punch_type);

-- Optional: enforce idempotency_key uniqueness when provided
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_idempotency
  ON punches (idempotency_key)
  WHERE idempotency_key IS NOT NULL;
```

### Step 2 — Update punch route (`workio/server/src/routes/punch.ts`)

```ts
import express from 'express';
import { pool } from '../db';
import { v4 as uuidv4 } from 'uuid';

const router = express.Router();

/**
 * POST /punch
 * Body: { employee_id: number, punch_type: 'in'|'out', latitude?: number, longitude?: number, idempotency_key?: string }
 *
 * Behavior:
 * - Accept idempotency_key from client (generated per user action). If absent, generate one.
 * - Use transactional upsert to prevent duplicates under (employee_id, punch_date, punch_type).
 * - Return existing punch row on conflict (idempotent response).
 */
router.post('/', async (req, res) => {
  const { employee_id, punch_type, latitude, longitude } = req.body;
  let { idempotency_key } = req.body;

  if (!employee_id || !punch_type || !['in', 'out'].includes(punch_type)) {
    return res.status(400).json({ error: 'Invalid payload' });
  }

  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // Normalize date in UTC to avoid timezone ambiguity for punch_date
    const punch_date = new Date().toISOString().split('T')[0]; // YYYY-MM-DD UTC
    if (!idempotency_key) idempotency_key = uuidv4();

    // Try insert; on conflict (employee_id, punch_date, punch_type) return existing row
    const upsertQuery = `
      INSERT INTO punches (employee_id, punch_type, punch_date, latitude, longitude, idempotency_key, created_at)
      VALUES ($1, $2, $3, $4, $5, $6, NOW())
      ON CONFLICT (employee_id, punch_date, punch_type)
      DO UPDATE SET idempotency_key = EXCLUDED.idempotency_key
      RETURNING *;
    `;

    const result = await client.query(upsertQuery, [
      employee_id,
      punch_type,
      punch_date,
      latitude ?? null,
      longitude ?? null,
      idempotency_key,
    ]);

    await client.query('COMMIT');
    return res.json({ punch: result.rows[0], idempotent: true });
  } catch (err: any) {
    await client.query('ROLLBACK');
    // If conflict on idempotency_key, try to fetch existing
    if (err.code === '23505' && err.constraint === 'idx_punches_idempotency') {
      const getQuery = 'SELECT * FROM punches WHERE idempotency_key = $1';
      const getResult = await pool.query(getQuery, [idempotency_key]);
      if (getResult.rows.length > 0) {
        return res.json({ punch: getResult.rows[0], idempotent: true });
      }
    }
    console.error('Punch upsert failed', err);
    return res.status(500).json({ error: 'Internal server error' });
  } finally {
    client.release();
  }
});

export default router;
```

### Step 3 — Frontend guidance (quick note)

- Generate a stable `idempotency_key` per user punch action (e.g., `uuidv4()` persisted in UI state until success) and include it in retries.  
- If key is omitted, backend generates one — but client-provided key enables cross-device/client replay safety.

## 4) Verification

1. **Schema check**  
   ```bash
   psql workio -c "\d punches"
   # Confirm idempotency_key column exists
   psql workio -c "\di"
   # Confirm idx_punches_employee_date_type and idx_punches_idempotency exist
   ```

2. **Duplicate prevention test**  
   - Send same `{ employee_id: 1, punch_type: "in" }` twice in quick succession (same or different idempotency_key).  
   - Expect: single row in DB; second request returns same row with `idempotent: true`.

3. **Concurrent race test**  
   - Use `ab` or a small script to fire 10 concurrent identical punch requests.  
   - Expect: exactly one row inserted; no duplicate `punch_date`+`punch_type` for same employee.

4. **Idempotency-key replay test**  
   - Create punch with `idempotency_key: "test-123"`.  
   - Replay same request later (same key).  
   - Expect: same punch row returned, no new row.

5. **Timezone/day-boundary sanity**  
   - Punch late UTC (e.g., 23:59) and early next day; verify `punch_date` boundaries are consistent and duplicates cannot be created across days for same punch_type.
