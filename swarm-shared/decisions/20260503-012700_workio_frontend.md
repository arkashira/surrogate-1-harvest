# workio / frontend

## Final synthesized implementation (best parts, corrected + actionable)

**Chosen approach**
- Use React Query for server-state, optimistic updates, rollback, and invalidation (Candidate 1).
- Add a client-side in-flight lock + idempotency key to prevent double submits and race retries (both candidates).
- Keep UI responsive with disabled + spinner, and handle 409 conflicts silently (Candidate 1).
- Rollback only on non-conflict errors; on conflict, accept server truth and reconcile via invalidation.
- Keep file locations consistent and minimal: `usePunch.ts` (hook) and `PunchButton.tsx` (component).

### workio/src/features/punch/usePunch.ts
```ts
import { useMutation, useQueryClient } from '@tanstack/react-query';
import { v4 as uuidv4 } from 'uuid';

export type PunchType = 'IN' | 'OUT';

export interface PunchResponse {
  id: string;
  type: PunchType;
  timestamp: string; // ISO
  status?: 'OK' | 'CONFLICT';
}

interface PunchRequest {
  type: PunchType;
  lat?: number;
  lng?: number;
  idempotencyKey: string;
}

export function usePunch() {
  const queryClient = useQueryClient();

  return useMutation<PunchResponse, Error, PunchRequest>({
    mutationFn: async (body) => {
      const res = await fetch('/api/punches', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify(body),
      });

      if (!res.ok) {
        const errPayload = await res.json().catch(() => ({}));
        const error = new Error(errPayload.message || `Failed to punch ${body.type}`);
        (error as any).status = res.status;
        throw error;
      }

      return res.json();
    },
    onMutate: async (newPunch) => {
      await queryClient.cancelQueries({ queryKey: ['punches'] });

      const previous = queryClient.getQueryData<PunchResponse[]>(['punches']);
      const optimistic: PunchResponse = {
        id: `optimistic-${newPunch.idempotencyKey}`,
        type: newPunch.type,
        timestamp: new Date().toISOString(),
        status: 'OK',
      };

      // Optimistic insert at top (most recent)
      queryClient.setQueryData<PunchResponse[]>(['punches'], (old) => [
        optimistic,
        ...(old || []),
      ]);

      return { previous };
    },
    onError: (err: any, _variables, context) => {
      // Rollback only on non-conflict errors.
      // 409 means server already has the correct state — keep server truth.
      const status = err?.status;
      if (status !== 409 && context?.previous) {
        queryClient.setQueryData(['punches'], context.previous);
      }
    },
    onSettled: () => {
      // Always reconcile with server after mutation settles.
      queryClient.invalidateQueries({ queryKey: ['punches'] });
    },
  });
}
```

### workio/src/features/punch/PunchButton.tsx
```tsx
import { useState, useCallback } from 'react';
import { usePunch } from './usePunch';
import { v4 as uuidv4 } from 'uuid';

export function PunchButton() {
  // isPunchedIn derived from server + optimistic state via React Query cache
  const { mutate, isPending } = usePunch();
  const [isPunchIn, setIsPunchIn] = useState(true);

  const handlePunch = useCallback(() => {
    // Client-side in-flight lock (redundant with disabled but safe)
    if (isPending) return;

    const idempotencyKey = uuidv4();
    mutate(
      {
        type: isPunchIn ? 'IN' : 'OUT',
        idempotencyKey,
      },
      {
        onSuccess: () => {
          // Flip UI state only after server confirms
          setIsPunchIn((prev) => !prev);
        },
        onError: (err: any) => {
          // 409 means already punched — accept server state and flip UI to match
          if (err?.status === 409) {
            setIsPunchIn((prev) => !prev);
          }
          // Other errors: rollback handled by hook; toast can be surfaced here if desired
        },
      }
    );
  }, [isPending, mutate, isPunchIn]);

  return (
    <button
      onClick={handlePunch}
      disabled={isPending}
      className={`
        inline-flex items-center gap-2 rounded-lg px-6 py-3 font-semibold text-white shadow transition
        ${isPending ? 'cursor-wait bg-gray-400' :
          isPunchIn ? 'bg-green-600 hover:bg-green-700' : 'bg-red-600 hover:bg-red-700'}
      `}
      aria-busy={isPending}
    >
      {isPending ? (
        <>
          <span className="h-4 w-4 animate-spin rounded-full border-2 border-white border-t-transparent" />
          Processing...
        </>
      ) : (
        <>{isPunchIn ? 'Clock In' : 'Clock Out'}</>
      )}
    </button>
  );
}
```

### Backend requirement (non-negotiable for correctness)
- `/api/punches` must accept `idempotencyKey` and enforce deduplication per employee (and optionally per time window) so retries or duplicate submissions cannot create multiple punch records.
- Return 409 when a punch would conflict with existing open/close rules (e.g., double IN without OUT), and include the canonical punch state in the response body when possible.

### Verification checklist (concrete, actionable)
1. **Optimistic UI**
   - Click → button disabled + spinner; new punch appears instantly at top of list (optimistic id).
   - Throttle to “Slow 3G” → UI remains responsive.

2. **Double-tap / race protection**
   - Rapid multi-click → only one request sent; no duplicate optimistic entries.

3. **Conflict (409) handling**
   - Simulate 409 (or backend duplicate rule) → optimistic entry replaced by server state after invalidation; button toggles to correct state without error toast.

4. **Error rollback**
   - Force 5xx → optimistic entry removed; button re-enabled; user can retry.

5. **Idempotency across retries**
   - Start request, cancel, retry (same or new idempotencyKey) → backend must prevent duplicate punches for same employee+window; verify single DB record.

6. **Visual feedback**
   - Disabled + spinner present during pending; aria-busy set for accessibility.
