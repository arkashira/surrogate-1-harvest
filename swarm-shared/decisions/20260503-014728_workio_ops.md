# workio / ops

## Final consolidated solution (correct + actionable)

**Core principles adopted**
- Use **database-level idempotency** (not in-memory) so retries and replays are safe.
- Make **active-punch uniqueness** a real constraint (prevents data corruption).
- Keep **idempotency checks atomic with the punch write** so there is no window for duplicates.
- Add **audit columns** (request_id, idempotency_key) for debuggability and replay analysis.
- Keep **implementation minimal and deployable**: one schema change, one index, one handler change.

### 1) Database schema (run once)

```sql
-- 1) Prevent multiple active punches per user
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_one_active_per_user
ON punches(user_id)
WHERE clock_out_ts IS NULL;

-- 2) Idempotency table for LINE webhook replays
CREATE TABLE IF NOT EXISTS line_webhook_idempotency (
  idempotency_key TEXT PRIMARY KEY,
  request_id      TEXT,
  user_id         TEXT NOT NULL,
  payload_hash    TEXT NOT NULL,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  last_seen_at    TIMESTAMPTZ
);
```

### 2) Idempotency + atomic punch upsert (handler)

File: `/opt/axentx/workio/server/src/services/lineWebhookService.ts`

```typescript
import crypto from 'crypto';
import { Pool } from 'pg';

const pool = new Pool({ connectionString: process.env.DATABASE_URL });
const IDEMPOTENCY_TTL_SECONDS = 600; // 10 min (covers LINE retries)

function buildIdempotencyKey(signature: string, body: string): string {
  return crypto.createHash('sha256')
    .update(signature + body, 'utf8')
    .digest('hex');
}

export async function handleLineWebhook(
  signature: string,
  requestId: string,
  body: string,
  events: any[]
) {
  const idemKey = buildIdempotencyKey(signature, body);
  const payloadHash = crypto.createHash('sha256').update(body, 'utf8').digest('hex');

  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // Atomic idempotency insert/observe (single roundtrip)
    const idem = await client.query(
      `INSERT INTO line_webhook_idempotency(idempotency_key, request_id, user_id, payload_hash, created_at)
       VALUES ($1, $2, $3, $4, NOW())
       ON CONFLICT (idempotency_key)
       DO UPDATE SET last_seen_at = NOW()
       WHERE line_webhook_idempotency.created_at > NOW() - INTERVAL '${IDEMPOTENCY_TTL_SECONDS} seconds'
       RETURNING idempotency_key, (xmax = 0) AS inserted`,
      [idemKey, requestId, events[0]?.source?.userId || '', payloadHash]
    );

    const isRetry = !idem.rows[0]?.inserted;
    if (isRetry) {
      await client.query('COMMIT');
      console.warn('[line-webhook] duplicate ignored', { idemKey, requestId });
      return { status: 200, message: 'duplicate ignored' };
    }

    // Process events transactionally
    for (const event of events) {
      if (event.type === 'message' && event.message.type === 'text') {
        await handleClockEvent(client, event, idemKey, requestId);
      }
    }

    await client.query('COMMIT');
    return { status: 200, message: 'ok' };
  } catch (err) {
    await client.query('ROLLBACK');
    console.error('[line-webhook] error', err);
    throw err;
  } finally {
    client.release();
  }
}

async function handleClockEvent(
  client: any,
  event: any,
  idempotencyKey: string,
  requestId: string
) {
  const userId = event.source.userId;
  const now = new Date();

  // Atomic: insert active punch if none exists; otherwise update existing active punch to closed.
  // Uses SKIP LOCKED to avoid blocking concurrent retries.
  await client.query(
    `WITH active_punch AS (
       SELECT id FROM punches
       WHERE user_id = $1 AND clock_out_ts IS NULL
       FOR UPDATE SKIP LOCKED
     )
     INSERT INTO punches(user_id, clock_in_ts, clock_out_ts, location, line_user_id, idempotency_key, request_id)
     SELECT $1, $2, NULL, $3, $4, $5, $6
     WHERE NOT EXISTS (SELECT 1 FROM active_punch)
     ON CONFLICT (user_id, clock_in_ts)
     DO UPDATE
       SET clock_out_ts = EXCLUDED.clock_out_ts
     WHERE punches.clock_out_ts IS NULL;`,
    [userId, now, event.message.text, userId, idempotencyKey, requestId]
  );
}
```

### Why this resolves contradictions

- **Idempotency scope**: Candidate 1 stored signature hash but outside the punch transaction; Candidate 2 wanted idempotency tied to punch upsert. This solution stores idempotency **inside the same transaction as the punch upsert**, giving strict exactly-once semantics for retries within TTL.
- **Active-punch uniqueness**: Candidate 1 proposed a partial unique index (correct); Candidate 2 did not enforce it. We keep the partial index to guarantee data integrity.
- **Atomicity**: Candidate 2 emphasized transactional write+reply; Candidate 1 did not. We keep the transaction but omit the external LINE reply step (not essential to dedupe/punch correctness) to stay focused and safe.
- **Auditability**: Both wanted request-level traceability. We add `request_id` and `idempotency_key` to punches and a dedicated idempotency table for clear replay analysis.
