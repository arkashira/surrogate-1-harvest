## Final Synthesis (Correctness + Actionability)

**Chosen diagnosis**: Candidate 1 is correct and concrete. The root cause is server-side: missing idempotency and missing database-level enforcement of “one active punch per user.” Client-side fixes (optimistic UI, idempotency keys generated on the client) help UX but do **not** prevent duplicates from LINE webhook retries, retries from the network, or double-taps that reach the server. Therefore, server-side idempotency + a unique constraint is mandatory; client-side idempotency and optimistic UI are complementary but secondary.

**Chosen implementation scope**: Apply Candidate 1’s server-side changes (idempotency middleware + DB unique constraint + upsert), and add minimal, safe client-side guidance (idempotency key generation + optimistic UI) to reduce perceived latency and double-taps without relying on the client for correctness.

## 2. Idempotency middleware (server)

```ts
// middleware/idempotency.ts
import { Request, Response, NextFunction } from 'express';
import { Pool } from 'pg';
import crypto from 'crypto';

const pool = new Pool({ connectionString: process.env.DATABASE_URL });

export const idempotency = (ttlMinutes = 60) => {
  return async (req: Request, res: Response, next: NextFunction) => {
    const key = req.header('X-Idempotency-Key');
    if (!key) {
      return res.status(400).json({ error: 'Missing X-Idempotency-Key' });
    }

    const keyHash = crypto.createHash('sha256').update(key).digest('hex');
    const expiresAt = new Date(Date.now() + ttlMinutes * 60_000).toISOString();

    const { rows } = await pool.query(
      'SELECT response_body FROM idempotency_keys WHERE key_hash = $1 AND expires_at > NOW()',
      [keyHash]
    );

    if (rows.length > 0) {
      return res.json(rows[0].response_body);
    }

    (req as any)._idempotencyKeyHash = keyHash;
    (req as any)._idempotencyExpiresAt = expiresAt;

    const originalJson = res.json.bind(res);
    res.json = (body: any) => {
      // Best-effort cache; failure must not break the request
      pool.query(
        `INSERT INTO idempotency_keys (key_hash, response_body, expires_at)
         VALUES ($1, $2, $3)
         ON CONFLICT (key_hash) DO UPDATE
         SET response_body = EXCLUDED.response_body, expires_at = EXCLUDED.expires_at`,
        [keyHash, body, expiresAt]
      ).catch(() => {});
      return originalJson(body);
    };

    next();
  };
};
```

## 3. Punch route with race-safe upsert and constraint enforcement

```ts
// routes/punch.ts
import { Router, Request, Response } from 'express';
import { Pool } from 'pg';
import { idempotency } from '../middleware/idempotency';

const router = Router();
const pool = new Pool({ connectionString: process.env.DATABASE_URL });

/**
 * POST /punch
 * Body: { userId: number, type?: 'in' | 'out', location?: { lat, lng } }
 * If type omitted, toggle based on current state.
 */
router.post('/punch', idempotency(60), async (req: Request, res: Response) => {
  const { userId, type, location } = req.body;

  if (!userId) {
    return res.status(400).json({ error: 'userId required' });
  }

  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // Get current active punch (clock_out IS NULL)
    const { rows: active } = await client.query(
      'SELECT id, clock_in FROM punches WHERE user_id = $1 AND clock_out IS NULL ORDER BY clock_in DESC LIMIT 1',
      [userId]
    );

    const now = new Date().toISOString();

    // Clock-out path
    if (type === 'out' || (!type && active.length > 0)) {
      if (active.length === 0) {
        await client.query('ROLLBACK');
        return res.status(400).json({ error: 'No active punch to clock out' });
      }

      const result = await client.query(
        `UPDATE punches
         SET clock_out = $1, location_out = $2
         WHERE id = $3 AND clock_out IS NULL
         RETURNING *`,
        [now, location, active[0].id]
      );

      await client.query('COMMIT');
      return res.json({ punch: result.rows[0], action: 'out' });
    }

    // Clock-in path — rely on unique index to prevent duplicates
    const result = await client.query(
      `INSERT INTO punches (user_id, clock_in, location_in)
       VALUES ($1, $2, $3)
       ON CONFLICT DO NOTHING
       RETURNING *`,
      [userId, now, location]
    );

    if (result.rowCount === 0) {
      await client.query('ROLLBACK');
      return res.status(409).json({ error: 'Already clocked in' });
    }

    await client.query('COMMIT');
    return res.json({ punch: result.rows[0], action: 'in' });
  } catch (err: any) {
    await client.query('ROLLBACK');
    // Unique violation from idx_one_active_punch_per_user
    if (err.code === '23505') {
      return res.status(409).json({ error: 'Already clocked in' });
    }
    console.error(err);
    return res.status(500).json({ error: 'Internal server error' });
  } finally {
    client.release();
  }
});

export default router;
```


## 4. Client-side guidance (recommended, not required for correctness)

To reduce perceived latency and accidental double-taps:

- Generate a UUID or crypto-random `X-Idempotency-Key` per user-initiated punch action (not per retry) and reuse it for immediate retries. Example (web/JS):

```ts
function makeIdempotencyKey(): string {
  return crypto.randomUUID();
}

// On punch button tap:
const key = makeIdempotencyKey();
setOptimisticPunchState(); // Immediately show "clocked in" locally
fetch('/punch', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'X-Idempotency-Key': key
  },
  body: JSON.stringify({ userId })
}).then(applyServerState).catch(revertOptimisticState);
```

- Use optimistic UI: update local state immediately, then reconcile with server response. If the server returns 409 (“Already clocked in”), refresh the current state from the server.

- For cross-tab/device freshness, poll or use a lightweight channel (e.g., periodic GET /punch/status) to keep the active punch state current.
