## Deployment
cf workers

## v0.1 scope
v0.1 ships in 2 weeks: a CF Worker gateway exposing a single /v1/chat/completions endpoint that normalizes requests to OpenAI format, routes to OpenAI and Anthropic (chat models only), enforces per-provider token quotas stored in Supabase Postgres, tracks token counts in KV with best-effort p95 latency measurement, returns standardized gateway trace IDs, and includes contract tests plus a README for local dev (no frontend).

## Open questions
- Which latency SLA target (p95 ms) is acceptable for v0.1 and how will we measure it end-to-end?
- Do we normalize to OpenAI-compatible routes or expose provider-native schemas with a translation layer?
- How do we synchronize token counters across CF Workers instances without centralized Redis (free-tier constraints)?
- Which subset of providers and models must be supported at launch to validate the gateway?
- What retry/backoff and idempotency semantics are required for quota exhaustion or provider outages?
