## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: single-binary client (Linux x86_64) with shadowsocks-like pluggable transport, local SOCKS5 listener on random port, TUN cloaking via traffic morphing, client-side kill-switch (fail-closed), encrypted config stored locally and optionally synced to Supabase Postgres via user-owned token, basic session telemetry (blocked/waivered) reported via CF Workers endpoint; no server infrastructure required beyond Supabase free tier and CF Workers free tier.

## Open questions
- Which pluggable transport (shadowsocks-like vs meek vs domain-front) offers best reach under Russian DPI with minimal latency?
- How will client-side kill-switch be implemented per-OS (Windows/WFP, Linux/nftables, macOS/pf) without root prompts breaking zero-config?
- What minimal client-side telemetry schema preserves privacy while enabling WAU-ZB measurement?
- Can TUN interface naming/routing be randomized to evade artifact detection without breaking DNS/MTU?
- How to store/rotate bridge endpoints without centralized discovery that can be blocked?
