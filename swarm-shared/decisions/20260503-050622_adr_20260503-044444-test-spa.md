## Deployment
cf workers

## v0.1 scope
v0.1 (2 weeks): read-only discovery and audit for a single linked AWS account and GCP project; list and detail views of schedules; detect duplicates (exact cron+target), drifted schedules (cron changed since last snapshot), and orphaned schedules (no target found); export CSV and JSON; CLI + hosted CF Worker (free tier) + Supabase free tier; no auto-remediation.

## Open questions
- Which IAM/service-account scopes are minimally required for safe read-only discovery across accounts/projects?
- How will customers map provider credentials (AWS keys, GCP SA keys) into the worker at runtime (secrets vs OAuth flow)?
- What retention and GDPR deletion policy should apply to discovered schedule snapshots and issue history?
- How to define 'duplicate' thresholds (same cron + same target vs same cron across different targets) without false positives?
- Should orphaned schedules be auto-retired or only flagged for manual review in v0.1?
