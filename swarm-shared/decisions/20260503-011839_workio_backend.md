# workio / backend

## Final synthesized solution

**Core problem**: duplicate open punches can occur because (1) LINE retries/deliveries lack idempotency handling, and (2) punch creation uses `find-then-insert/update` which is not race-safe.  
**Goal**: enforce “at most one open punch per employee per day” and make webhook processing idempotent with concrete, actionable changes.

---

### 1. Schema changes (`workio/server/src/db/schema.sql`)

```sql
-- 1) Idempotency column on punches to deduplicate LINE webhook deliveries
ALTER TABLE punches
  ADD COLUMN IF NOT EXISTS idempotency_key TEXT NULL;

-- Unique index on idempotency_key (only for non-null values)
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_idempotency_key
  ON punches (idempotency_key)
  WHERE idempotency_key IS NOT NULL;

-- 2) Partial unique index: at most one open punch per employee per calendar day
--    Uses DATE(clock_in_at) as the logical day.
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_one_open_per_day
  ON punches (tenant_id, employee_id, DATE(clock_in_at))
  WHERE clock_out_at IS NULL;

-- 3) Optional helper index to speed up “find open punch” and day-range queries
CREATE INDEX IF NOT EXISTS idx_punches_employee_open
  ON punches (tenant_id, employee_id, clock_out_at)
  WHERE clock_out_at IS NULL;
```

**Why this combination is strongest**  
- `idempotency_key` on the punches table gives simple, queryable deduplication tied to the record itself (no separate table to maintain and join).  
- The partial unique index enforces the business rule at the DB level, making race conditions impossible to violate correctness.  
- Keeping the idempotency index partial avoids bloat and allows multiple normal punches to coexist while still preventing duplicates from retries.

---

### 2. Backend implementation (atomic upsert + idempotency)

Use a single `BEGIN … COMMIT` transaction with:
1. Idempotency check/insert via `idempotency_key`.
2. Atomic upsert for clock-in/clock-out using `INSERT … ON CONFLICT` and targeted `UPDATE` for closing.

```ts
// workio/server/src/routes/punch.ts
import { pool } from '../db';

export async function createPunch(req, res) {
  const { tenantId, employeeId, lineEventId, action } = req.body;
  // action: 'clock_in' | 'clock_out'
  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // Idempotency guard using LINE eventId (or generate a stable key from payload+intent)
    if (!lineEventId) {
      throw new Error('lineEventId required for idempotency');
    }

    const idem = await client.query(
      `SELECT 1 FROM punches WHERE idempotency_key = $1`,
      [lineEventId]
    );
    if (idem.rowCount > 0) {
      await client.query('ROLLBACK');
      return res.json({ ok: true, message: 'duplicate webhook ignored' });
    }

    const now = new Date();

    if (action === 'clock_in') {
      // Try insert; if unique violation on open punch, treat as already clocked in
      const result = await client.query(
        `INSERT INTO punches (tenant_id, employee_id, clock_in_at, idempotency_key, created_at, updated_at)
         VALUES ($1, $2, $3, $4, $5, $6)
         ON CONFLICT (tenant_id, employee_id, DATE(clock_in_at))
         WHERE clock_out_at IS NULL
         DO NOTHING
         RETURNING *`,
        [tenantId, employeeId, now, lineEventId, now, now]
      );

      if (result.rowCount === 0) {
        // Fetch existing open punch to return it (already clocked in)
        const existing = await client.query(
          `SELECT * FROM punches
           WHERE tenant_id = $1 AND employee_id = $2 AND clock_out_at IS NULL
           ORDER BY clock_in_at DESC LIMIT 1`,
          [tenantId, employeeId]
        );
        await client.query('ROLLBACK');
        return res.json({ ok: true, punch: existing.rows[0], message: 'already clocked in' });
      }

      await client.query('COMMIT');
      return res.json({ ok: true, punch: result.rows[0] });
    }

    if (action === 'clock_out') {
      // Atomically close the open punch for this employee/tenant
      const result = await client.query(
        `UPDATE punches
         SET clock_out_at = $1, updated_at = $1, idempotency_key = COALESCE(idempotency_key, $2)
         WHERE tenant_id = $3 AND employee_id = $4 AND clock_out_at IS NULL
         RETURNING *`,
        [now, lineEventId, tenantId, employeeId]
      );

      if (result.rowCount === 0) {
        // No open punch to close; record idempotency anyway to prevent retries from creating noise
        await client.query(
          `INSERT INTO punches (tenant_id, employee_id, clock_in_at, clock_out_at, idempotency_key, created_at, updated_at)
           VALUES ($1, $2, $3, $3, $4, $3, $3)
           ON CONFLICT (idempotency_key) DO NOTHING`,
          [tenantId, employeeId, now, lineEventId]
        );
        await client.query('COMMIT');
        return res.json({ ok: true, message: 'no open punch to close (idempotent ack)' });
      }

      await client.query('COMMIT');
      return res.json({ ok: true, punch: result.rows[0] });
    }

    throw new Error('invalid action');
  } catch (err: any) {
    await client.query('ROLLBACK');
    // Unique violation on idempotency_key or open punch index -> treat as duplicate
    if (err.code === '23505') {
      return res.json({ ok: true, message: 'duplicate prevented by DB constraint' });
    }
    console.error(err);
    res.status(500).json({ error: 'internal error' });
  } finally {
    client.release();
  }
}

module.exports = { createPunch };
```

**Key choices and why they win**  
- **Idempotency on the punches table** (rather than a separate table) simplifies queries and cleanup while still preventing duplicate processing.  
- **Partial unique index** enforces “one open punch per employee per day” at the DB level — the only way to guarantee correctness under concurrency.  
- **Atomic upsert/close pattern**:  
  - For clock-in: try insert; if conflict on open punch, return existing (safe and deterministic).  
  - For clock-out: `UPDATE … WHERE clock_out_at IS NULL` targets exactly the open punch; if none exists, still record idempotency to absorb retries safely.  
- **Explicit transaction + idempotency check first** prevents race conditions between duplicate webhook deliveries.
