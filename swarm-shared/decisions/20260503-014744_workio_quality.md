# workio / quality

## 1. Diagnosis

- **No idempotency persistence**: LINE retries with same event payload produce duplicate punch rows; `X-Line-Signature` is verified but never stored.
- **Non-atomic upsert**: current `findOne → if-exists-update-else-insert` is racy under concurrency and retries.
- **Missing unique constraint**: DB allows duplicate `(user_id, date, punch_type)` or `(user_id, timestamp)` entries; no guard against duplicates.
- **Clock-in/out state derived on the fly**: race-prone when two events arrive nearly simultaneously (e.g., retry + original).
- **No transactional boundary**: punch write and related state (daily summary/OT accrual) are not atomic; partial updates possible on failure.

## 2. Proposed change

Make LINE webhook punch processing **idempotent + atomic** at the database level:

- File: `server/src/db/schema.sql` — add unique constraint and idempotency table.
- File: `server/src/services/punchService.ts` (or equivalent) — replace find-then-upsert with single `INSERT ... ON CONFLICT DO NOTHING` + returning.
- File: `server/src/controllers/lineWebhookController.ts` — store `X-Line-Signature` (or event id) in idempotency table inside same tx.

Scope: single transaction that writes idempotency key + punch row atomically; safe under retries and concurrency.

## 3. Implementation

```sql
CREATE TABLE IF NOT EXISTS line_event_idempotency (
  event_id        TEXT PRIMARY KEY,
  received_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()

-- Ensure one punch record per user per timestamp (or per date+punch_type if you prefer)
-- Adjust column names to match your punches table (assumed: punches)
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_user_timestamp
ON punches (user_id, timestamp);
```

If your punches table uses `date` + `punch_type` (clock_in/clock_out) and allows multiple per day, use this instead:

```sql
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_user_date_type
ON punches (user_id, date, punch_type);
```

### 3.2 Punch service (atomic upsert)

```ts
// server/src/services/punchService.ts
import { pool } from '../db';

export interface CreatePunchInput {
  userId: string;
  timestamp: Date;
  punchType: 'clock_in' | 'clock_out';
  latitude?: number;
  longitude?: number;
  lineEventId: string; // idempotency key
}

export async function createPunch(input: CreatePunchInput) {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // 1) Idempotency guard (use LINE event id or stable signature-derived id)
    const idem = await client.query(
      `INSERT INTO line_event_idempotency (event_id) VALUES ($1) ON CONFLICT DO NOTHING RETURNING event_id`,
      [input.lineEventId]
    );

    if (idem.rowCount === 0) {
      // already processed — return existing punch (optional)
      const existing = await client.query(
        `SELECT * FROM punches WHERE user_id = $1 AND timestamp = $2`,
        [input.userId, input.timestamp]
      );
      await client.query('COMMIT');
      return existing.rows[0];
    }

    // 2) Atomic punch insert (conflict -> do nothing; adjust unique index to match policy)
    const result = await client.query(
      `INSERT INTO punches (user_id, timestamp, punch_type, latitude, longitude)
       VALUES ($1, $2, $3, $4, $5)
       ON CONFLICT (user_id, timestamp) DO NOTHING
       RETURNING *`,
      [input.userId, input.timestamp, input.punchType, input.latitude, input.longitude]
    );

    // 3) If you maintain daily summaries/OT, update them here in same tx
    // await updateDailySummary(client, input.userId, input.timestamp, input.punchType);

    await client.query('COMMIT');
    return result.rows[0];
  } catch (err) {
    await client.query('ROLLBACK');
    throw err;
  } finally {
    client.release();
  }
}
```

### 3.3 Webhook controller (extract stable idempotency key)

```ts
// server/src/controllers/lineWebhookController.ts
import { createPunch } from '../services/punchService';

export async function handleLineWebhook(req, res) {
  try {
    // Use LINE event id if present; otherwise derive stable id from payload + signature
    const lineEventId = req.body.events?.[0]?.webhookEventId || req.body.events?.[0]?.deliveryId;
    const fallbackId = req.headers['x-line-signature'];
    const idempotencyKey = lineEventId || fallbackId;

    if (!idempotencyKey) {
      return res.status(400).json({ error: 'Missing idempotency key' });
    }

    const event = req.body.events[0];
    if (event.type !== 'message' || event.message.type !== 'text') {
      return res.status(200).end();
    }

    // Parse user/time from your LINE payload + GPS (adjust to your format)
    const userId = event.source.userId;
    const timestamp = new Date(event.timestamp);
    const punchType = await inferPunchType(userId, timestamp); // your logic

    await createPunch({
      userId,
      timestamp,
      punchType,
      latitude: event.message?.content?.metadata?.latitude,
      longitude: event.message?.content?.metadata?.longitude,
      lineEventId: idempotencyKey,
    });

    res.status(200).end();
  } catch (err) {
    console.error(err);
    res.status(500).json({ error: 'Processing failed' });
  }
}
```

## 4. Verification

1. **Schema applied**: confirm unique index and idempotency table exist.
   ```bash
   psql workio -c "\d punches"
   psql workio -c "\d line_event_idempotency"
   ```

2. **Idempotency test**: simulate duplicate LINE event (same `lineEventId`).
   - Send same payload twice (curl/postman) → second request must not create a new punch row and must return 200.
   - Check DB: `SELECT count(*) FROM punches WHERE user_id = '<id>' AND timestamp = '<ts>';` → must be 1.

3. **Concurrency test**: fire two requests with same `lineEventId` in parallel (e.g., `ab` or `hey`).
   - No duplicate rows; no constraint violations.

4. **Rollback test**: force error after idempotency insert (e.g., throw before commit) → verify tx rollback removes idempotency row (or keep it if you prefer to retain processed attempts — adjust policy as needed).

5. **LINE retry simulation**: use a saved real LINE event payload and re-POST with same `X-Line-Signature` header → confirm no duplicate punches and consistent state.
