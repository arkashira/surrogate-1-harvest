## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: CLI-first MVP that connects to one AWS account and one GCP project, lists schedules into Supabase, detects duplicates/drifted targets/orphaned entries, outputs a retirement report (JSON + CSV) and exposes a minimal FastAPI endpoint for polling results; no UI, no auto-remediation, scheduled via GitHub Actions or local cron.

## Open questions
- Which IAM scopes are minimally required to list/describe schedules and tags on Lambda and Cloud Functions without granting invoke permissions?
- How will customers authenticate (Supabase JWT, CF Workers OAuth, or API keys) and map tenants to platform projects/accounts?
- What retention and backfill policy should apply for schedule history to balance audit accuracy vs storage cost?
- How should drift be defined (cron change, target function change, timezone/region change) and how to avoid false positives from expected changes?
- Will orphaned schedules be detected by presence of target function only, or also by Cloud Scheduler/EventBridge side resources?
