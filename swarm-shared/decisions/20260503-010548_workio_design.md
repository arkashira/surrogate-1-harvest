# workio / design

### Diagnosis
* Missing database-level uniqueness guard for open punches, allowing duplicates under LINE webhook retries or client double-taps.
* Race-prone app upsert logic in clock-in handler, potentially causing multiple open punches per user per day.
* Lack of atomic, DB-enforced uniqueness guarantee and idempotent write path.
* Inadequate handling of concurrent taps or webhook retries, which can lead to inconsistent data.
* Insufficient validation and error handling for clock-in and clock-out operations.

### Proposed change
The proposed change is to implement a database-level uniqueness guard for open punches and modify the clock-in handler to use an idempotent write path. This will involve creating a partial unique index on the `clock_in_at` and `user_id` columns, as well as updating the clock-in handler to use a transactional approach.

### Implementation
To implement the proposed change, the following steps can be taken:

1. Create a partial unique index on the `clock_in_at` and `user_id` columns:
```sql
CREATE UNIQUE INDEX idx_user_id_clock_in_at ON punches (user_id, date(clock_in_at)) WHERE clock_out_at IS NULL;
```
This index will prevent duplicate open punches for the same user on the same day.

2. Update the clock-in handler to use a transactional approach:
```typescript
// server/src/controllers/ClockInController.ts
import { getRepository } from 'typeorm';
import { Punch } from '../entities/Punch';

const clockIn = async (userId: number) => {
  const punchRepository = getRepository(Punch);
  const queryRunner = punchRepository.manager.connection.createQueryRunner();

  await queryRunner.startTransaction();

  try {
    const existingPunch = await punchRepository.findOne({
      where: { userId, clockOutAt: null },
    });

    if (existingPunch) {
      // Handle existing open punch
      return;
    }

    const newPunch = new Punch();
    newPunch.userId = userId;
    newPunch.clockInAt = new Date();

    await queryRunner.manager.save(newPunch);

    await queryRunner.commitTransaction();
  } catch (error) {
    await queryRunner.rollbackTransaction();
    throw error;
  } finally {
    await queryRunner.release();
  }
};
```
This updated clock-in handler uses a transactional approach to ensure that either all or none of the changes are committed to the database, preventing inconsistent data.

```typescript
  const existingPunch = new Punch();

  existingPunch.clockOutAt = null;

    expect(error).toBeInstanceOf(Error);
  }

```
2. Test that creating a new punch with no existing open punch for the same user on the same day succeeds:
```typescript

  const userId = 1;

  expect(newPunch).toBeInstanceOf(Punch);
  expect(newPunch.userId).toBe(userId);
  expect(newPunch.clockInAt).toBeInstanceOf(Date);
  expect(newPunch.clockOutAt).toBeNull();
});
```
These tests will verify that the proposed change works as expected and prevents duplicate open punches for the same user on the same day.
