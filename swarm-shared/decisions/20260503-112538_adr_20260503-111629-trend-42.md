## v0.1 scope
v0.1 in 2 weeks: single-region Cloudflare Workers gateway with TypeScript/Fastify-like handlers on Workers runtime; Postgres (Supabase) for provider/model/policy config; Redis-backed token counters and per-provider rate-limit enforcement; /v1/chat/completions normalized endpoint that proxies to OpenAI and Anthropic (chat models) with auth passthrough; strategy 'balanced' (latency+cost) routing; basic retry and circuit-breaker; token ledger writes to Postgres; p95 latency and token-routed metrics emitted to Workers Analytics (or console + log drain); contract tests for provider payloads; no frontend, no multi-tenant isolation, no custom models — only provider passthrough with normalization and global policy enforcement.

## Open questions
- Which provider auth patterns must be supported at launch (api_key headers, OAuth2 client-credentials, signed requests)?
- What is the authoritative source-of-truth for per-provider rate limits and quota reset cadences?
- How should the gateway handle provider-specific non-200 success shapes (e.g., streaming partial failures, 429 with retry-after)?
- What p95 latency SLA target (ms) and error budget policy will NSM enforce for v0.1?
- Will token counting require normalization across providers (e.g., OpenAI vs Anthropic tokenizers) or best-effort counters?
- Do we need per-tenant isolation (multi-tenant keys) in v0.1 or single-org global routing?
