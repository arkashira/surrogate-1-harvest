# workio / quality

### 1. Diagnosis
* No client-side idempotency key on punch submissions, leading to duplicate punches on double-taps or LINE webhook replays.
* Clock-in/out button lacks optimistic state or debounce, allowing users to spam and queue multiple requests.
* No local validation of valid state transitions for clock-in/out actions.
* Punch writes are not transactional, potentially resulting in data inconsistencies.
* No server-side state machine to enforce valid sequences of clock-in/out actions.

### 2. Proposed change
To address the issues mentioned above, we will introduce a client-side idempotency key for punch submissions and implement a server-side state machine to enforce valid clock-in/out sequences. The changes will be made in the following files:
* `workio/server/src/controllers/PunchController.js`
* `workio/server/src/models/Punch.js`
* `workio/server/src/utils/idempotency.js`
* `workio/src/components/PunchButton.js`

### 3. Implementation
