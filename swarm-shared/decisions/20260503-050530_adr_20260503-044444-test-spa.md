## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: CLI tool (Python) that authenticates via Supabase Auth, scans a single AWS account + single GCP project for Lambda and Cloud Function schedules, persists findings into Supabase Postgres, and outputs a summary of duplicates, drifted cron expressions, and orphaned schedules (no UI, no queue, manual run).

## Open questions
- Which AWS/GCP credentials model minimizes friction while remaining least-privilege (read-only IAM role vs. org-level service account)?
- Will customers prefer CLI-first or scheduled CF Worker scans (cron on CF) for periodic audits?
- How will we reliably detect 'orphaned' schedules (schedule exists but target function missing) given eventual consistency and cross-account boundaries?
- What retention policy should we apply to raw platform metadata in Postgres to stay within Supabase free tier?
