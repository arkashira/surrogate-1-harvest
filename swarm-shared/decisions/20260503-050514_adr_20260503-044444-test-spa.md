## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: CLI + FastAPI service deployable on CF Workers (or local dev) that discovers Lambda + Cloud Functions schedules for a single customer sandbox, persists to Supabase Postgres, detects duplicates (same target+cron in same region) and orphans (target not found), exposes a GET /audit endpoint returning counts and a GET /schedules endpoint; no remediation actions, no UI, no queue, manual re-run via CLI.

## Open questions
- Which customer accounts/tenants and regions will be in scope for week 1 (sandbox AWS/GCP creds)?
- How will we authenticate to customer AWS/GCP (OIDC from CF Workers, short-lived keys, or customer-supplied tokens)?
- What permission model is required (read-only vs. optional remediation actions) and how will we scope least privilege?
- How do we define 'orphaned' (no target exists, target lacks permissions, or schedule disabled > N days)?
- What duplicate/drift thresholds and grouping keys should we use to avoid noise (exact cron match vs. semantic equivalence)?
