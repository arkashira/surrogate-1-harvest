## Deployment
cf workers

## v0.1 scope
v0.1 ships a single-region CF Worker gateway that exposes a /v1/chat/completions endpoint, normalizes OpenAI and Anthropic responses, supports API-key auth via Supabase-stored encrypted keys, enforces per-provider rate limits and monthly token quotas via Redis, records RequestLog to Supabase Postgres, provides basic retry/failover to a secondary provider, and emits p95 latency and token-usage metrics to logs — all deployable on Cloudflare free tier with Supabase free plan and Redis free tier.

## Open questions
- Which latency SLO (p95 ms) and token volume targets define the first SLA?
- Will we support per-customer keys and quota isolation (multi-tenant) in v0.1?
- Streaming vs non-streaming: do we normalize SSE/NDJSON across providers in v0.1?
- How will cost attribution work (per-customer vs aggregate) for token usage reporting?
- Which providers and operations are in scope for the first gateway surface (chat/completions only?)
- Do we enforce global rate limits at CF Workers edge or in a regional queue/service?
- What encryption/key management approach is acceptable for storing provider keys on the free tier?
