## Deployment
cf workers

## v0.1 scope
v0.1 (2 weeks): CLI + CF Worker job that lists Lambda functions and Cloud Functions, extracts schedule triggers, persists to Supabase Postgres, computes duplicates by hash and orphaned by last_run_at staleness (>30d), exposes a read-only REST endpoint for schedule list and audit summary, and outputs a CSV report of stale schedules to retire.

## Open questions
- Which IAM scopes are minimally required to list/describe Lambda triggers and Cloud Functions without granting invoke permissions?
- How will we detect 'drifted' schedules (e.g., code/config changed but cron unchanged) without invoking functions?
- What retention window for schedule history balances auditability vs storage on free tiers?
- Should orphaned detection rely on last_run_at staleness or explicit provider state (e.g., disabled/removed triggers)?
- How will customers map schedules to teams/tenants (tags, labels, or external directory)?
