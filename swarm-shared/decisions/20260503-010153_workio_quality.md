# workio / quality

## 1. Diagnosis

- Missing **database-level uniqueness guard** for open punches: no partial unique index prevents multiple `clock_out_at IS NULL` rows per `(user_id, date(clock_in_at))`, allowing duplicates under LINE webhook retries or client double-taps.
- App-layer “find-then-insert” in the clock-in handler is racy and relies on optimistic checks instead of a real constraint.
- No idempotency key on the LINE webhook handler, so retries from LINE (network blips, 5xx) create duplicate punches.
- Clock-in endpoint returns 200 before transaction commit, so downstream retries can interleave.
- No audit trail for punch state transitions (e.g., open→closed) to detect and reconcile duplicates.

## 2. Proposed change

- **File**: `workio/server/src/db/schema.sql`
  - Add partial unique index for open punches.
  - Add unique constraint on `(user_id, punch_date)` for closed punches to prevent duplicate closed entries per day.
- **File**: `workio/server/src/routes/punch.ts` (or wherever the clock-in handler lives)
  - Wrap clock-in in `INSERT ... ON CONFLICT DO NOTHING` and return existing open punch when conflicted (idempotent).
  - Add idempotency using `X-Idempotency-Key` (or derive from LINE `webhookEventId`/`deliveryId`) stored in a new `punch_idempotency` table to dedupe retries.
- **File**: `workio/server/src/middleware/transaction.ts` (or handler)
  - Return 202/200 only after commit; ensure handler is safe to retry.

## 3. Implementation

### 3.1 Schema changes (`workio/server/src/db/schema.sql`)

```sql
-- Ensure punches table exists (adjust if name differs)
-- Add partial unique index: at most one open punch per user per calendar day
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_open_per_user_day
ON punches (user_id, DATE(clock_in_at))
WHERE clock_out_at IS NULL;

-- Prevent duplicate closed punches per user per calendar day
-- (optional but improves data quality; keep if business allows only one punch pair/day)
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_closed_per_user_day
ON punches (user_id, DATE(clock_in_at))
WHERE clock_out_at IS NOT NULL;

-- Idempotency table for LINE webhook retries
CREATE TABLE IF NOT EXISTS punch_idempotency (
  idempotency_key TEXT NOT NULL PRIMARY KEY,
  punch_id INTEGER NOT NULL REFERENCES punches(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX IF NOT EXISTS idx_punch_idempotency_key ON punch_idempotency(idempotency_key);
```

### 3.2 Clock-in handler (example) (`workio/server/src/routes/punch.ts`)

```ts
import { Request, Response } from 'express';
import { db } from '../db';
import { sql } from 'drizzle-orm'; // or your query builder

export async function clockIn(req: Request, res: Response) {
  const { userId } = req.user; // authenticated user
  const idempotencyKey = req.headers['x-idempotency-key'] as string;

  if (!idempotencyKey) {
    return res.status(400).json({ error: 'Missing X-Idempotency-Key' });
  }

  return db.transaction(async (trx) => {
    // Check idempotency first
    const existing = await trx
      .select({ punchId: sql<number>`punch_id`.mapWith(Number) })
      .from(sql.table('punch_idempotency'))
      .where(sql`idempotency_key = ${idempotencyKey}`)
      .then((rows) => rows[0]);

    if (existing) {
      const punch = await trx
        .select()
        .from(sql.table('punches'))
        .where(sql`id = ${existing.punchId}`)
        .then((rows) => rows[0]);
      return res.status(200).json({ punch, idempotent: true });
    }

    // Try to insert new open punch; partial index will block duplicates
    const [punch] = await trx
      .insert(sql.table('punches'))
      .values({
        user_id: userId,
        clock_in_at: new Date(),
        clock_out_at: null,
        // location fields if present
      })
      .onConflictDoNothing()
      .returning();

    if (!punch) {
      // Conflict on partial index -> fetch existing open punch
      const existingOpen = await trx
        .select()
        .from(sql.table('punches'))
        .where(
          sql`user_id = ${userId} AND DATE(clock_in_at) = DATE(${new Date()}) AND clock_out_at IS NULL`
        )
        .then((rows) => rows[0]);

      if (!existingOpen) {
        // Should not happen, but be safe
        return res.status(500).json({ error: 'Failed to create or retrieve punch' });
      }

      // Record idempotency for this retry so future retries are fast
      await trx
        .insert(sql.table('punch_idempotency'))
        .values({ idempotency_key: idempotencyKey, punch_id: existingOpen.id });

      return res.status(200).json({ punch: existingOpen, idempotent: true });
    }

    // Record idempotency for this request
    await trx
      .insert(sql.table('punch_idempotency'))
      .values({ idempotency_key: idempotencyKey, punch_id: punch.id });

    return res.status(201).json({ punch, idempotent: false });
  });
}
```

### 3.3 LINE webhook adapter (if not using X-Idempotency-Key)

Derive idempotency key from LINE’s event identifiers to dedupe retries:

```ts
function lineEventIdempotencyKey(event: any): string {
  // Use deliveryId or webhookEventId if available; fallback to timestamp+userId
  return `line:${event.deliveryId || event.webhookEventId || `${Date.now()}:${event.source.userId}`}`;
}
```

Inject into handler:

```ts
const idempotencyKey = lineEventIdempotencyKey(lineEvent);
// then call clockIn-like logic with that key
```

## 4. Verification

1. **Schema check**  
   ```bash
   psql workio -c "\d punches"
   psql workio -c "\d punch_idempotency"
   ```
   Confirm partial indexes exist.

2. **Duplicate open punch prevention**  
   - Simulate two concurrent clock-ins for same user (same day):
     ```bash
     curl -X POST http://localhost:3000/punch/clock-in -H 'X-Idempotency-Key: test-123' -H 'Content-Type: application/json' -d '{}'
     curl -X POST http://localhost:3000/punch/clock-in -H 'X-Idempotency-Key: test-123' -H 'Content-Type: application/json' -d '{}'
     ```
   - Expect: first returns 201, second returns 200 with `idempotent: true` and same punch row.  
   - Verify only one row with `clock_out_at IS NULL` exists:
     ```sql
     SELECT count(*) FROM punches WHERE user_id = <uid> AND clock_out_at IS NULL;
     ```

3. **Retry safety with LINE-like retries**  
   - Send same request with same idempotency key after 1s/5s/30s — should return same punch and not create new rows.

4. **Closed punch uniqueness**  
   - Attempt to create two closed punches for same user same day (if your index covers closed punches) — second insert should fail or be prevented by app logic.

5. **Audit/idempotency table**  
   - Check `punch_idempotency` contains entries for each processed idempotency key and maps to correct punch.
