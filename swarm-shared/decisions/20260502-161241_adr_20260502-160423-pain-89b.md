## Deployment
cf workers

## v0.1 scope
v0.1 ships a local stealth agent (Rust CLI + Tauri desktop) that: 1) establishes a meek/QUIC masquerade tunnel to a public free-tier rendezvous (CF Worker + Supabase-backed config), 2) obfuscates local SOCKS5/TUN artifacts (randomized ports, browser-like TLS fingerprints, plausible packet timing), 3) enforces fail-closed kill-switch (block-all-on-failure) using local firewall rules where possible, 4) stores minimal encrypted session state in local SQLite, and 5) emits privacy-preserving telemetry (WAU-ZB signals) via CF Workers Queues to Supabase — all deployable client-side with zero server control.

## Open questions
- Which DPI signatures are highest risk in target regions (e.g., TCP fingerprint, TUN MTU, DNS patterns) and what local mitigations are feasible client-side only?
- Can we rely on free-tier CF Workers limits for telemetry volume under bursty WAU-ZB scenarios, or do we need sampling/aggregation strategy?
- How will we distribute signed config updates without centralized endpoints that can be blocked (e.g., GitHub releases + IPFS fallback vs. CF Workers)?
- What kill-switch enforcement is possible on desktop OSes without admin/root (e.g., Windows Filtering Platform vs. userspace firewall rules)?
- How do we measure 'blocked session' client-side without leaking metadata (local heuristics vs. remote canaries)?
