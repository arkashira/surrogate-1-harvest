# workio / backend

## 1. Diagnosis

- No idempotency layer for LINE webhook retries → duplicate clock-in/out rows when LINE retries delivery (network hiccups, 5xx).
- Clock-in/out endpoint uses naive upsert (`INSERT ... ON CONFLICT DO UPDATE`) without a deterministic uniqueness key → race between concurrent punches can create two rows.
- No request deduplication window (e.g., 5–10 min) for same employee/day → retries after “late” punch still create duplicates.
- Missing DB unique constraint to enforce exactly-once per (employee, punch_date, punch_type) → application logic races under load.
- No idempotency key persistence for non-LINE requests (manual/API punches) → client retries create duplicates.

## 2. Proposed change

- **File**: `server/src/routes/punch.ts` (or equivalent route handling clock-in/out)
- **File**: `server/src/db/schema.sql` (add unique constraint + idempotency table)
- **Scope**: add `punches.idempotency_key` (varchar, unique), create unique index on `(employee_id, punch_date, punch_type)`, and implement atomic upsert using the idempotency key derived from LINE message id or client-supplied key.

## 3. Implementation

### 3.1 DB schema change (`server/src/db/schema.sql`)

```sql
-- Add idempotency key to punches
ALTER TABLE punches ADD COLUMN IF NOT EXISTS idempotency_key VARCHAR(128);
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_idempotency
  ON punches (idempotency_key)
  WHERE idempotency_key IS NOT NULL;

-- Enforce one punch per employee per date per type (clock_in/clock_out)
-- Keep existing punch_time index; add composite unique constraint
CREATE UNIQUE INDEX IF NOT EXISTS idx_punches_employee_date_type
  ON punches (employee_id, punch_date, punch_type);
```

### 3.2 Route handler (`server/src/routes/punch.ts`)

```ts
import { Request, Response } from 'express';
import { pool } from '../db';
import { v4 as uuidv4 } from 'uuid';

// POST /api/punch
export async function createPunch(req: Request, res: Response) {
  const { employeeId, punchType, latitude, longitude, lineMessageId } = req.body;
  const clientIdempotencyKey = req.headers['idempotency-key'] as string | undefined;

  // Deterministic idempotency key: prefer LINE message id, then client header, then generate
  const idempotencyKey = lineMessageId || clientIdempotencyKey || uuidv4();
  const punchDate = new Date().toISOString().split('T')[0]; // YYYY-MM-DD in UTC

  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // Try to insert with idempotency key; if conflict on idempotency_key, return existing row
    const upsert = `
      INSERT INTO punches (employee_id, punch_type, punch_date, punch_time, latitude, longitude, idempotency_key)
      VALUES ($1, $2, $3, NOW(), $4, $5, $6)
      ON CONFLICT (idempotency_key) DO UPDATE
        SET punch_time = EXCLUDED.punch_time,
            latitude = EXCLUDED.latitude,
            longitude = EXCLUDED.longitude
      RETURNING *
    `;
    const result = await client.query(upsert, [
      employeeId,
      punchType,
      punchDate,
      latitude || null,
      longitude || null,
      idempotencyKey,
    ]);

    // Also enforce business rule: one punch per employee/date/type (clock_in vs clock_out)
    // If duplicate by business rule, reject unless it's an idempotent retry (same idempotency_key already handled above)
    const check = await client.query(
      'SELECT id FROM punches WHERE employee_id = $1 AND punch_date = $2 AND punch_type = $3 AND idempotency_key IS DISTINCT FROM $4',
      [employeeId, punchDate, punchType, idempotencyKey]
    );
    if (check.rows.length > 0) {
      await client.query('ROLLBACK');
      return res.status(409).json({
        error: 'Duplicate punch',
        message: `A ${punchType} for ${punchDate} already exists for this employee.`,
      });
    }

    await client.query('COMMIT');
    return res.status(201).json({ punch: result.rows[0], idempotencyKey });
  } catch (err: any) {
    await client.query('ROLLBACK');
    // Unique violation on employee/date/type (idx_punches_employee_date_type)
    if (err.code === '23505') {
      return res.status(409).json({ error: 'Duplicate punch', message: err.detail });
    }
    console.error('Punch creation failed', err);
    return res.status(500).json({ error: 'Internal server error' });
  } finally {
    client.release();
  }
}
```

### 3.3 Middleware: idempotency short-circuit (optional, fast path)

Add a tiny cache (e.g., node-cache or Redis) for recent idempotency keys to return 200 fast for retries within 5–10 min without hitting DB. Example (Redis):

```ts
import Redis from 'ioredis';
const redis = new Redis(process.env.REDIS_URL);

async function idempotentPunch(req: Request, res: Response, next: Function) {
  const key = req.headers['idempotency-key'] || req.body.lineMessageId;
  if (!key) return next();
  const cached = await redis.get(`idem:${key}`);
  if (cached) {
    return res.status(200).json(JSON.parse(cached));
  }
  // attach key to request for handler to store after success
  (req as any).idempotencyKey = key;
  next();
}
```

Store after successful response:

```ts
await redis.setex(`idem:${idempotencyKey}`, 600, JSON.stringify({ punch: result.rows[0], idempotencyKey }));
```

4. **Client idempotency header**
   - Send request with `Idempotency-Key: abc123`, then repeat.
   - Expect: second request returns cached/stored result (if Redis enabled) or DB idempotent upsert (same row).
