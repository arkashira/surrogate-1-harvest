# workio / quality

## 1) Diagnosis

- **No idempotency key enforcement** on clock-in/out endpoints: duplicate requests from flaky mobile networks create double punches for the same `(employee_id, punch_type, date)` causing payroll drift.
- **Non-transactional punch writes**: concurrent retries can insert two punches within the same second because there is no unique constraint or upsert at the DB layer.
- **No server-side deduplication window**: retries within minutes (common on mobile) are treated as new punches rather than idempotent repeats.
- **Missing unique constraint** on `(employee_id, punch_type, date)` or `(employee_id, punch_type, date, time_window)` allows duplicates to persist.
- **Clock-out/leave/OT endpoints share the same weakness**: same race and duplication risks exist across all state-changing endpoints.

## 2) Proposed change

- **File**: `workio/server/src/db/schema.sql` — add unique constraint and idempotency table.
- **File**: `workio/server/src/routes/punch.ts` (or equivalent) — add idempotency key header support and transactional upsert.
- **Scope**: clock-in/out endpoints only (highest value, lowest surface) — ~1–2h.

## 3) Implementation

### Step 1 — DB schema (idempotency + constraint)

```sql
-- workio/server/src/db/schema.sql

-- Idempotency keys (lightweight)
CREATE TABLE IF NOT EXISTS idempotency_keys (
  key         TEXT PRIMARY KEY,
  payload_hash TEXT NOT NULL,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- Optional: index for cleanup
CREATE INDEX IF NOT EXISTS idx_idempotency_keys_created_at ON idempotency_keys (created_at);

-- Prevent duplicate punches per employee per punch_type per date
-- (allows one in + one out per date by using punch_type discriminator)
ALTER TABLE punches
  ADD CONSTRAINT uq_employee_punch_type_date
  UNIQUE (employee_id, punch_type, date);
```

### Step 2 — Transactional upsert endpoint

```ts
// workio/server/src/routes/punch.ts
import { Request, Response } from 'express';
import { pool } from '../db';
import crypto from 'crypto';

export async function clockInOut(req: Request, res: Response) {
  const { employee_id, punch_type, latitude, longitude } = req.body;
  const idempotencyKey = req.headers['idempotency-key'] as string;

  if (!idempotencyKey) {
    return res.status(400).json({ error: 'idempotency-key header required' });
  }

  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // 1) Check idempotency
    const idRes = await client.query(
      'SELECT payload_hash FROM idempotency_keys WHERE key = $1',
      [idempotencyKey]
    );

    const payload = JSON.stringify({ employee_id, punch_type, date: new Date().toISOString().split('T')[0] });
    const payloadHash = crypto.createHash('sha256').update(payload).digest('hex');

    if (idRes.rowCount > 0) {
      // Idempotent replay — return existing punch
      if (idRes.rows[0].payload_hash !== payloadHash) {
        await client.query('ROLLBACK');
        return res.status(409).json({ error: 'idempotency-key payload mismatch' });
      }

      const punch = await client.query(
        `SELECT * FROM punches
         WHERE employee_id = $1 AND punch_type = $2 AND date = $3`,
        [employee_id, punch_type, new Date().toISOString().split('T')[0]]
      );

      await client.query('COMMIT');
      return res.json({ punch: punch.rows[0], replay: true });
    }

    // 2) Insert punch with upsert on conflict
    const date = new Date().toISOString().split('T')[0];
    const time = new Date().toISOString();

    const punchResult = await client.query(
      `INSERT INTO punches (employee_id, punch_type, date, time, latitude, longitude)
       VALUES ($1, $2, $3, $4, $5, $6)
       ON CONFLICT (employee_id, punch_type, date) DO UPDATE
         SET time = EXCLUDED.time,
             latitude = EXCLUDED.latitude,
             longitude = EXCLUDED.longitude
       RETURNING *`,
      [employee_id, punch_type, date, time, latitude, longitude]
    );

    // 3) Record idempotency key
    await client.query(
      'INSERT INTO idempotency_keys (key, payload_hash) VALUES ($1, $2)',
      [idempotencyKey, payloadHash]
    );

    await client.query('COMMIT');
    return res.json({ punch: punchResult.rows[0], replay: false });
  } catch (err) {
    await client.query('ROLLBACK');
    console.error('Punch transaction failed', err);
    return res.status(500).json({ error: 'internal server error' });
  } finally {
    client.release();
  }
}
```

### Step 3 — Lightweight cleanup job (optional, can be cron)

```sql
-- Retain idempotency keys for 48h (safe for mobile retries)
DELETE FROM idempotency_keys WHERE created_at < NOW() - INTERVAL '48 hours';
```

## 4) Verification

1. **Schema check**
   ```bash
   psql workio -c "\d punches"
   # should show UNIQUE CONSTRAINT on (employee_id, punch_type, date)
   psql workio -c "\d idempotency_keys"
   # should show PRIMARY KEY on key
   ```

2. **Duplicate request test**
   - Send same `POST /punch` twice with identical `idempotency-key` header and body.
   - Expect: first `200` with `replay: false`, second `200` with `replay: true` and identical punch row.
   - Verify only one row exists in `punches` for that `(employee_id, punch_type, date)`.

3. **Concurrent request test**
   - Fire two near-simultaneous requests with different `idempotency-key` values but same logical punch.
   - Expect: one wins insert, the other hits unique constraint and updates (or fails gracefully) — no duplicate rows.

4. **Payload mismatch test**
   - Reuse same `idempotency-key` with different body (e.g., different `punch_type`).
   - Expect: `409` with `payload mismatch`.
