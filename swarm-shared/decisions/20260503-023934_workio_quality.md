# workio / quality

## Final synthesized solution (correct + actionable)

**Core diagnosis (merged, resolved)**  
- Missing **database-level uniqueness** allows duplicates from LINE retries and client double-taps.  
- Application-level checks are racy (read-then-insert) and not safe under concurrency.  
- No **idempotency key** plumbing means retries (client or webhook) can create duplicates.  
- Returning 200 before commit and missing atomic upsert increase race windows.  
- Schema must match actual table columns; choose **one** canonical uniqueness definition and stick to it.

**Chosen canonical uniqueness (pick one and be consistent)**  
- If the table tracks per-punch **timestamps**, use `(employee_id, punch_ts, direction)` as the unique constraint (Candidate 1).  
- If the table tracks **per-day punch types** (in/out) and stores `punch_date` + `punch_type`, use `(employee_id, tenant_id, punch_date, punch_type)` (Candidate 2).  
- Do **not** mix both constraints unless you need both semantics; implement the one that matches your business rule.

**Required changes (minimal, high-leverage)**  
1. **Schema migration** — add unique constraint + supporting index + optional idempotency column.  
2. **Route handler** — accept idempotency key, perform atomic upsert, commit before response, return correct status codes.  
3. **Client** — generate and send `X-Idempotency-Key` for punch calls; optimistic UI update.  
4. **Optional** — short-lived idempotency cache for fast duplicate rejection (use Redis for multi-instance).

### 2) Backend route (atomic upsert + idempotency)

Key rules:
- Require `X-Idempotency-Key` (or generate one server-side).  
- Use a single atomic `INSERT ... ON CONFLICT DO NOTHING` (or minimal update) inside a transaction.  
- Commit before sending the response.  
- Return `201` on insert, `200` on idempotent hit with existing punch.

```ts
// workio/server/src/routes/punch.ts
import express from 'express';
import { pool } from '../db';
import { v4 as uuidv4 } from 'uuid';

const router = express.Router();

/**
 * POST /punch
 * Body: { employeeId: number, direction: 'in'|'out', punchTs?: string }
 * Headers: X-Idempotency-Key (recommended)
 *
 * Atomic upsert using the unique constraint.
 */
router.post('/', async (req, res) => {
  const { employeeId, direction, punchTs } = req.body;
  const idempotencyKey = req.header('X-Idempotency-Key') || uuidv4();
  const ts = punchTs ? new Date(punchTs) : new Date();

  if (!employeeId || !direction || !['in', 'out'].includes(direction)) {
    return res.status(400).json({ error: 'Invalid payload' });
  }

  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // Try to insert; if duplicate, return existing row
    const upsert = `
      INSERT INTO punches (employee_id, punch_ts, direction, created_by_idempotency_key)
      VALUES ($1, $2, $3, $4)
      ON CONFLICT (employee_id, punch_ts, direction) DO NOTHING
      RETURNING id, punch_ts, direction;
    `;
    const upsertResult = await client.query(upsert, [employeeId, ts, direction, idempotencyKey]);

    let row = upsertResult.rows[0];
    let inserted = true;

    if (!row) {
      // Duplicate detected — fetch existing
      const get = `
        SELECT id, punch_ts, direction
        FROM punches
        WHERE employee_id = $1 AND punch_ts = $2 AND direction = $3
        LIMIT 1;
      `;
      const existing = await client.query(get, [employeeId, ts, direction]);
      if (!existing.rows.length) {
        // Should not happen, but handle gracefully
        throw new Error('Duplicate conflict but no existing row found');
      }
      row = existing.rows[0];
      inserted = false;
    }

    await client.query('COMMIT');

    return res.status(inserted ? 201 : 200).json({
      id: row.id,
      employeeId,
      direction,
      punchTs: row.punch_ts.toISOString(),
      inserted,
    });
  } catch (err: any) {
    await client.query('ROLLBACK').catch(() => {});
    console.error('Punch upsert failed', err);
    return res.status(500).json({ error: 'Internal server error' });
  } finally {
    client.release();
  }
});

export default router;
```

**Notes for multi-instance deployments**  
- For stronger idempotency across instances, persist idempotency keys in Redis with a TTL (e.g., 5–10 minutes) and perform a fast cache check before DB upsert.  
- The DB unique constraint remains the source of truth; the cache is only an optimization.

### 4) Verification checklist

- **Schema**: confirm unique constraint and index exist (`\d punches` in psql).  
- **Duplicate prevention**: send identical punch twice; second request must return `200` with same `id` and `inserted: false`.  
- **Race test**: fire 10–20 concurrent identical punch requests; only one row should be created.  
- **Idempotency key**: reuse the same `X-Idempotency-Key` across retries; same punch row must be returned.  
- **Commit-before-response**: ensure 200/201 is sent only after DB commit.  
- **Frontend**: optimistic UI should not break on retries; network throttling/offline→online replay must not create duplicates.

---

**Summary of decisions**  
- Enforce uniqueness at the **database level** (single source of truth).  
- Use **atomic upsert** (`ON CONFLICT DO NOTHING`) + fetch-existing fallback to avoid races.  
- Require **idempotency keys** and commit before responding.  
- Keep changes minimal: one migration + one route + small client update.  
- Choose **one** uniqueness definition that matches your schema and business rule; do not add redundant constraints unless strictly required.
