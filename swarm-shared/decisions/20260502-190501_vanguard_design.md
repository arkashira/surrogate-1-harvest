# vanguard / design

## 1. Diagnosis

- No canonical discovery entrypoint exists to surface top-hub insights (e.g., MOC) before planning — violates established `#knowledge-rag #graph #hub` pattern.
- Missing CDN-bypass file-list strategy for HF datasets; any future training ingestion will immediately hit HF API rate limits (429) instead of using `resolve/main/` CDN paths.
- No Lightning Studio reuse guard — each run risks recreating studios and burning 80+ hours/month quota on idle/duplicate instances.
- No executable discovery entrypoint to prepare context (top-hub docs + HF file manifest + studio reuse) before design/planning work begins.
- No local data projection step to enforce `{prompt,response}` schema and filename pattern (`batches/mirror-merged/{date}/{slug}.parquet`) before upload — risks mixed-schema pollution.

## 2. Proposed change

Create `/opt/axentx/vanguard/discovery/run_discovery.sh` as the single-pass discovery entrypoint that:
- Queries knowledge-rag for the top-hub (e.g., MOC) and emits a concise insight file.
- Produces an HF CDN-bypass file list JSON for a given dataset repo/date folder (single API call) to embed in training scripts.
- Detects and reuses a running Lightning Studio (L40S preferred, fallback to available) and writes its connection info.
- Projects local parquet data to `{prompt,response}` only and renames to `batches/mirror-merged/{date}/{slug}.parquet`.
