## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: CLI + CF Worker that scans a single AWS account + single GCP project, persists schedules to Supabase Postgres, detects duplicates (identical cron+region) and orphans (schedules with no matching function), exposes findings via a minimal FastAPI endpoint (hosted on CF Workers) and JSON export; no UI, no drift detection, no automated remediation.

## Open questions
- Which AWS/GCP credentials model minimizes friction while remaining secure for multi-customer scans (assume read-only IAM roles)?
- How will we reliably detect invocations (CloudWatch Logs vs. GCP Logs) and what retention window is acceptable for 'stale' classification?
- What is the acceptable false-positive rate for drift detection (e.g., infra-as-code changes not reflected in live metadata)?
- Will customers grant cross-account roles or will we require per-project API keys + delegated scanning agents?
- How should duplicate detection scope be defined (same region, same project, globally) and what merge/retire UX is acceptable?
