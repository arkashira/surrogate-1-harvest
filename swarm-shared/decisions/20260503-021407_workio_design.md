# workio / design

Below is the **single, merged, production-ready solution**.  
It keeps the strongest, most correct parts from both candidates and removes contradictions by preferring **database-level safety + concrete, copy-pasteable code**.

---

## 1. Diagnosis (merged, prioritized)

- **LINE retries cause duplicates** because ingestion has no idempotency key or uniqueness guard.
- **Race window** between “check active punch” and “insert/update” allows duplicate rows under concurrency/retries.
- **Missing tenant-scoped uniqueness** allows multiple active or same-day punches per employee.
- **Application-only checks are bypassable**; database constraints are required for correctness.
- **Leave/OT endpoints have identical risks** and must use the same pattern.

---

## 2. Chosen approach (why)

- Use **one atomic upsert** (not `findOne` → `insert`/`update`) to remove race windows.
- Add **partial unique index** to enforce “one active punch per tenant+employee” at the database level.
- Add **idempotency table keyed by (tenant_id, line_event_id)** so repeated LINE deliveries are no-ops.
- Apply the **same pattern to leave/OT** (shared idempotency table, atomic upsert, constraints).
- Return **200 for duplicates** (idempotent success), 500 only for real failures.

---

## 3. Implementation

### 3.1 Database schema (single migration)

```sql
-- server/src/db/schema.sql  (or run as migration)

-- Idempotency for LINE webhook events (tenant + line_event_id)
CREATE TABLE IF NOT EXISTS line_webhook_idempotency (
  tenant_id      BIGINT  NOT NULL,
  line_event_id  TEXT    NOT NULL,
  handler        TEXT    NOT NULL,            -- e.g. 'clock', 'leave', 'ot'
  created_at     TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (tenant_id, line_event_id)
);

-- Ensure only one active punch per tenant+employee
-- Assumes punches has: id, tenant_id, employee_id, clock_in_at, clock_out_at
CREATE UNIQUE INDEX IF NOT EXISTS idx_one_active_punch
ON punches (tenant_id, employee_id)
WHERE clock_out_at IS NULL;
```

### 3.2 Idempotent clock-in/out handler (atomic upsert)

```ts
// server/src/routes/attendance.ts
import { db } from '../db';
import type { Request, Response } from 'express';

export async function handleLineClock(req: Request, res: Response) {
  const { tenantId } = res.locals; // set by auth middleware
  const { userId: employeeId, timestamp, lineEventId } = req.body;

  if (!tenantId || !employeeId || !lineEventId) {
    return res.status(400).json({ error: 'missing required fields' });
  }

  try {
    await db.query('BEGIN');

    // 1) Idempotency check (atomic)
    const dup = await db.queryOne(
      `SELECT 1 FROM line_webhook_idempotency
        WHERE tenant_id = $1 AND line_event_id = $2`,
      [tenantId, lineEventId]
    );

    if (dup) {
      await db.query('COMMIT');
      return res.status(200).json({ ok: true, duplicate: true });
    }

    // 2) Atomic upsert punch (single statement)
    // If active punch exists -> clock-out; else -> clock-in.
    const now = new Date(timestamp || Date.now()).toISOString();
    const result = await db.queryOne(
      `WITH active AS (
         SELECT id FROM punches
         WHERE tenant_id = $1 AND employee_id = $2 AND clock_out_at IS NULL
         FOR UPDATE
       ),
       closed AS (
         UPDATE punches
         SET clock_out_at = $3
         WHERE tenant_id = $1 AND employee_id = $2 AND clock_out_at IS NULL
         RETURNING id, clock_in_at, clock_out_at
       ),
       inserted AS (
         INSERT INTO punches (tenant_id, employee_id, clock_in_at, clock_out_at)
         SELECT $1, $2, $3, NULL
         WHERE NOT EXISTS (SELECT 1 FROM closed)
         RETURNING id, clock_in_at, clock_out_at
       )
       SELECT id, clock_in_at, clock_out_at FROM closed
       UNION ALL
       SELECT id, clock_in_at, clock_out_at FROM inserted`,
      [tenantId, employeeId, now]
    );

    // 3) Record idempotency key
    await db.query(
      `INSERT INTO line_webhook_idempotency (tenant_id, line_event_id, handler)
        VALUES ($1, $2, 'clock')`,
      [tenantId, lineEventId]
    );

    await db.query('COMMIT');
    return res.status(200).json({ ok: true, punch: result });
  } catch (err: any) {
    await db.query('ROLLBACK');
    console.error('Clock handler error', err);

    // Unique violation on idx_one_active_punch or idempotency table -> treat as success
    if (err.code === '23505') {
      return res.status(200).json({ ok: true, duplicate: true });
    }
    return res.status(500).json({ error: 'internal' });
  }
}
```

### 3.3 Leave / OT handlers (same pattern)

- Add `line_event_id` column to `leave_requests` and `ot_requests` **or** reuse the shared idempotency table above.
- Use the same flow:
  1. `BEGIN`
  2. Idempotency check
  3. Atomic upsert/insert (with constraints/indexes)
  4. Record idempotency
  5. `COMMIT`
- Add tenant-scoped uniqueness/indexes on leave/OT tables as needed (e.g., one approved leave per day per employee).

## 4. Verification checklist

1. **Schema applied**
   - Confirm `line_webhook_idempotency` and `idx_one_active_punch` exist.

2. **Idempotency smoke test**
   - Send identical payload twice (same `lineEventId`, tenant, employee):
     - First → 200, `duplicate: false`, punch created.
     - Second → 200, `duplicate: true`, no new punch.

3. **Race test**
   - Fire 10 concurrent requests for same tenant/employee/lineEventId:
     - Exactly one punch row created.
     - No 500s from unique violations (or handled as 200 duplicate).

4. **Active-punch invariant**
   - Clock-in → row with `clock_out_at IS NULL`.
   - Clock-out → same row updated.
   - Second clock-in without clock-out → new row inserted.
   - Two concurrent clock-outs on same active punch → one succeeds, one no-op.

5. **LINE retry simulation**
   - Replay same webhook after delay → treated as duplicate, no double pay/leave/OT side effects.

---

## 5. Notes / decisions

- **Why partial unique index for active punches**: database-enforced, cannot be bypassed.
- **Why idempotency table**: cheap, explicit, works across handlers, easy to audit/expire.
- **Why atomic upsert with CTE + FOR UPDATE**: removes race window without extra round-trips.
- **Why 200 for duplicates**: idempotent APIs should not return errors for repeated deliveries.
