## Deployment

## v0.1 scope
CLI-first v0.1 in 2 weeks: authenticate to Supabase project, run local scan against one AWS region + one GCP project, upsert schedules into Postgres, detect duplicates (same cron+payload+overlapping time windows), detect orphaned schedules (target function not found), emit schedule_audit rows, and print a concise report with counts and suggested retirements.

## Open questions
- Which AWS/GCP credentials model minimizes friction while remaining secure for multi-customer scans (assume read-only roles)?
- Do we normalize all schedules to standard cron (5-field) or preserve vendor-specific rate syntax and map on read?
- What retention window for historical schedule metadata balances auditability with storage costs?
- How will customers define 'stale' thresholds (e.g., no last_run_at in 30/60/90 days) — config per workspace or global default?
