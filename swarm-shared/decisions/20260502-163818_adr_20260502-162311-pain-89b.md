## Deployment
cf workers

## v0.1 scope
v0.1 in 2 weeks: Android-only client with hidden SOCKS5 listener on randomized ephemeral port, randomized TUN interface name, Noise-protocol handshake to a CF Worker edge endpoint, signed config fetched via Worker KV, and Supabase-backed user/session metadata (free-tier). Includes basic DPI mock tests and a minimal revocation path via Worker Queue. No kill-switch, no split-tunneling, and no MDM/Knox integration.

## Open questions
- Which pluggable transport primitives (meek, obfs4-like framing, or domain-fronting) are least likely to trigger DPI alerts in target jurisdictions given current DPI signatures?
- Can we reliably cloak TUN interface names and associated sockets on stock Android without root while maintaining throughput (and which syscalls/proc paths must be filtered)?
- What is the acceptable trade-off between battery usage and stealth (e.g., background traffic shaping, keepalive frequency, TUN idle behavior)?
- How will revocation and config rotation be delivered under active inspection (Worker queues vs. in-band signaling) without leaking metadata?
- What telemetry can we safely collect for WAUS/U without creating user-identifiable logs under legal pressure?
