# workio / quality

## Final Consolidated Solution (Correct + Actionable)

I merged the strongest, non-contradictory parts from both proposals and resolved conflicts in favor of correctness and concrete actionability.

### Key decisions (why)
- **Idempotency key + unique index** (Candidate 1) is strictly safer than idempotency-by-`line_event_id` alone (Candidate 2) because LINE can retry with the same event payload; a deterministic hash of `(userId, eventTimestamp, punchType)` is stable across retries and independent of LINE’s internal event ID stability.
- **Atomic upsert via `ON CONFLICT`** (Candidate 1) is simpler and safer than the multi-step “BEGIN/idempotency check/upsert” pattern (Candidate 2) while providing equivalent correctness when paired with the right constraints.
- **Partial unique index** (Candidate 2) is useful if you soft-delete punches (`deleted_at`). If you do not soft-delete, use a plain unique constraint. I include both options and mark which to choose.
- **UI disable-on-submit** (both) is necessary but not sufficient; server-side enforcement is the source of truth.

### 1. Database schema (run as migration)

Choose **A** if you do **not** soft-delete punches; choose **B** if you do.

**A) Hard-delete model (simpler)**
```sql
-- workio/server/src/db/schema.sql  (or migration file)
ALTER TABLE punches
  ADD COLUMN IF NOT EXISTS idempotency_key TEXT;

CREATE UNIQUE INDEX IF NOT EXISTS uniq_punch_employee_date_type
  ON punches (employee_id, punch_date, punch_type)
  WHERE (deleted_at IS NULL); -- keep this line even if no soft-delete; it’s harmless

CREATE UNIQUE INDEX IF NOT EXISTS uniq_punch_idempotency
  ON punches (idempotency_key)
  WHERE (idempotency_key IS NOT NULL);
```

**B) Explicit partial unique (if you rely on soft-delete)**
```sql
-- Alternative: strict partial unique for active rows only
CREATE UNIQUE INDEX IF NOT EXISTS uq_employee_punch_active
  ON punches (employee_id, punch_date, punch_type)
  WHERE deleted_at IS NULL;
```

Add `line_event_id` only if you need to reference LINE events externally; otherwise prefer `idempotency_key`.

### 2. Idempotency key derivation (deterministic)

```ts
// workio/server/src/lib/idempotency.ts
import crypto from 'crypto';

export function makePunchIdempotencyKey(
  userId: string,
  eventTimestamp: number,
  punchType: 'in' | 'out'
): string {
  // Stable across retries: userId + exact event timestamp + punchType
  return crypto
    .createHash('sha256')
    .update(`${userId}:${eventTimestamp}:${punchType}`)
    .digest('hex');
}
```

### 3. Atomic upsert in LINE webhook handler

```ts
// workio/server/src/routes/line.ts
import { Request, Response } from 'express';
import { pool } from '../db';
import { makePunchIdempotencyKey } from '../lib/idempotency';

export async function handleLineWebhook(req: Request, res: Response) {
  try {
    const { events } = req.body;
    if (!Array.isArray(events)) return res.status(400).json({ error: 'invalid_payload' });

    for (const ev of events) {
      if (ev.type !== 'message' || ev.message?.type !== 'text') continue;
      const text = (ev.message.text || '').trim().toLowerCase();
      if (!['clock in', 'clock out'].includes(text)) continue;

      const userId = ev.source?.userId;
      const punchType = text === 'clock in' ? 'in' : 'out';
      const eventTimestamp = Number(ev.timestamp || Date.now());
      const now = new Date();
      const punchDate = now.toISOString().split('T')[0]; // YYYY-MM-DD in UTC

      if (!userId) continue;

      const idempotencyKey = makePunchIdempotencyKey(userId, eventTimestamp, punchType);

      // Atomic insert-if-not-exists by idempotency key.
      // The partial unique index prevents duplicate active (employee,date,type).
      const result = await pool.query(
        `INSERT INTO punches (
           employee_id, punch_date, punch_type, punched_at, idempotency_key
         )
         VALUES ($1, $2, $3, $4, $5)
         ON CONFLICT (idempotency_key) DO NOTHING
         RETURNING id, employee_id, punch_date, punch_type, punched_at`,
        [userId, punchDate, punchType, now, idempotencyKey]
      );

      // Optional: if you want to allow location updates within a grace window,
      // use ON CONFLICT (employee_id, punch_date, punch_type) DO UPDATE
      // with a WHERE ABS(EXTRACT(EPOCH FROM (now - punched_at))) < 60.
      // For strict correctness, DO NOTHING is safest.
    }

    return res.status(200).json({ ok: true });
  } catch (err) {
    console.error('LINE webhook error', err);
    return res.status(500).json({ error: 'internal' });
  }
}
```

### 4. Frontend UI guard (prevent double-tap)

```tsx
// workio/src/components/ClockButton.tsx
import React, { useState } from 'react';
import { clockIn, clockOut } from '../api';

export function ClockButton({ type }: { type: 'in' | 'out' }) {
  const [loading, setLoading] = useState(false);

  const handleClick = async () => {
    if (loading) return;
    setLoading(true);
    try {
      if (type === 'in') await clockIn();
      else await clockOut();
    } finally {
      setLoading(false);
    }
  };

  return (
    <button
      onClick={handleClick}
      disabled={loading}
      className={`
        px-6 py-3 rounded font-medium transition-opacity
        ${loading ? 'opacity-60 cursor-not-allowed' : ''}
        ${type === 'in' ? 'bg-green-600 text-white' : 'bg-red-600 text-white'}
      `}
    >
      {loading ? 'กำลังดำเนินการ...' : type === 'in' ? 'Clock In' : 'Clock Out'}
    </button>
  );
}
```

### 5. Verification checklist (run these)

1. **Apply migration**  
   Run the chosen schema change against your database.

2. **Idempotency test**  
   - Send two identical LINE webhook payloads (same `userId`, same `timestamp`, same `type`) within seconds.  
   - Assert: `SELECT COUNT(*) FROM punches WHERE employee_id = ? AND punch_date = ? AND punch_type = ?` returns `1`.

3. **Concurrency test**  
   - Fire two near-simultaneous requests for the same employee/date/type (slightly different timestamps).  
   - Assert: only one active punch exists (partial unique index prevents two active rows).

4. **UI test**  
   - Rapid double-click Clock In/Out; confirm button disables and only one network request is sent; only one punch appears.

5. **Rollback/delete behavior (if using soft-delete)**  
   - If you soft-delete (`deleted_at` set), verify you can create a new punch for the same employee/date/type after deletion (partial unique allows it).  
   - If you hard-delete, verify uniqueness still holds.

### Notes on extensibility
- If you later want to allow one “in” and one “out” per day (but not two “in”s), the partial unique index on `(employee_id, punch_date, punch_type)` already enforces this exactly.
- If you need grace-window updates (e.g., correcting location within 60s), replace `ON CONFLICT (idempotency_key) DO NOTHING` with `ON CONFLICT (employee_id, punch_date, punch_type) DO UPDATE SET ... WHERE ABS(EXTRACT(EPOCH FROM (now - punched_at))) < 60`. Keep idempotency key for replay safety.
